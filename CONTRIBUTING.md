# 贡献指南

91ai 是面向开发者的中文 AI 工程知识库。欢迎补充可验证的资源、实践经验和评测，也欢迎修正文档错误。

## 可以贡献什么

- 新的 AI 编程、Agent、RAG、评测与安全资源。
- 有适用条件、限制和风险说明的真实使用经验。
- 工具对比、生产案例、学习路径和避坑记录。
- 失效链接、过时信息、事实错误和导航问题的修复。

推荐资源可使用 [资源推荐表单](https://github.com/mindcarver/91ai/issues/new?template=resource-suggestion.yml)，内容问题可使用 [内容问题表单](https://github.com/mindcarver/91ai/issues/new?template=content-report.yml)。

## 内容标准

提交前请确认：

1. 说明内容适合谁、解决什么问题，以及为什么值得收录。
2. 事实与个人判断分开表述。事实附来源，判断说明适用条件和局限。
3. 能力、价格、许可、迁移和安全行为等信息优先引用官方来源。
4. 对快速变化的信息标注核验日期，统一使用 `YYYY-MM-DD`。
5. 社区文章、第三方教程和非官方实现应明确标注其性质。
6. 不提交密钥、令牌、私有 URL、内部文档或未授权内容。
7. 涉及脚本、Skill、MCP Server 或命令执行时，说明权限、供应链和凭据风险。

时效等级、历史快照和待复核的处理方式见 [内容保鲜规则](docs/maintenance/content-freshness.md)。

## 文档约定

- `README.md` 保持简短，主要承担项目定位和导航。
- 内容地图放在 `docs/README.md`，AI 编程内容放在 `docs/ai-coding/`。
- 新文件使用小写 kebab-case，例如 `tool-name.md`。
- 使用清晰标题、短段落和必要的比较表格，避免营销话术和链接堆砌。
- 图片放在 `assets/`，并使用相对路径引用。

## 源文档与生成物

- `docs/` 是公开正文的单一真实源。
- `assets/` 存放公开文档直接引用的媒体资产。
- `post-to-wechat/` 是由正文生成的发布副本，不应反过来作为正文修改源。
- 发布结果 JSON、本地绝对路径、临时目录和调试日志不应提交。

## 提交流程

1. 在提交前搜索仓库，避免重复资源或重复专题。
2. 只修改与本次贡献有关的文件，不顺带重排或重写其他内容。
3. 在本地运行：

   ```sh
   ./scripts/check.sh all
   ```

4. 如果新增或修改了外部链接，可在网络条件允许时运行 `./scripts/check.sh all-full`。
5. 在 Pull Request 中说明变更、关键来源、核验日期以及仍无法确认的内容。

检查通过只代表格式、内部链接和统计一致，不替代人工事实核验。

## 安全问题

不要在公开 Issue 中披露凭据、可利用细节或其他敏感信息。请按 [安全策略](SECURITY.md) 私下报告。
