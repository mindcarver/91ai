# 上下文与记忆系统设计

**TL;DR:** Agent 能不能长期稳定工作，取决于上下文和记忆的管理质量。2025 年 Letta（MemGPT）的四层记忆架构（Message Buffer → Core Memory → Recall → Archival）成为主流范式，LangGraph 用 Checkpointer + Store 实现了类似分层。关键转变是从 Prompt Engineering 到 **Context Engineering**——设计整个信息环境，而不是优化措辞。记忆系统的核心不是"保存更多"，而是选择、组织、读取和遗忘。

## 为什么上下文与记忆是关键能力

没有上下文管理能力的 Agent，会出现三类问题：

**短期失忆。** 用户刚说过限制条件，下一步就忘了；上一个工具已返回结果，下一步又重复调用；前面已排除的方案，后面又重新提出。

**长期失忆。** 用户反复强调的偏好、项目背景、行业术语，系统始终沉淀不下来。每次新任务从零开始。

**记忆污染。** 系统什么都记——旧信息、错误信息、临时猜测、过期偏好不断影响新任务。不是更聪明，是更固执、更混乱。

记忆系统的核心不是"保存更多"，是**选择、组织、读取和遗忘**。

一个成熟 Agent 要能回答四个问题：

1. 当前任务需要看什么？
2. 哪些历史信息值得长期保存？
3. 什么时候把记忆拿出来？
4. 旧信息和新信息冲突时谁优先？

答不清这四个问题，记忆越多，系统越危险。

## 上下文和记忆不是一回事

工程上必须分开。

**上下文（Context）**——模型在本次调用时实际看到的信息。服务当前决策，生命周期短，强调相关性和精简。

**记忆（Memory）**——系统跨轮次、跨任务、跨时间保存的信息资产。不一定每次都进入上下文，按需检索、筛选、注入。

```
上下文 = 桌面上摊开的材料（服务当前工作）
记忆   = 档案柜里的资料（分类和索引）
日志   = 摄像头记录的全过程（可以很全，但不给模型看）
```

常见错误是把日志当记忆。系统保存了所有聊天记录、工具调用、原始文档，然后按 token 上限截断塞给模型。这不是记忆系统，是"带截断的历史堆积"。

真正的记忆系统要有治理能力：哪些进入短期状态，哪些进入长期记忆，哪些只留日志，哪些应该丢弃。

## Context Engineering：从 Prompt Engineering 到信息环境设计

2025 年出现的新范式：**Context Engineering**。

Prompt Engineering 优化输入文本的措辞。Context Engineering 设计整个信息环境——哪些信息进入上下文、以什么格式、什么顺序。

包含：工具定义选择、记忆检索策略、示例筛选、输出格式约束。

这是 Agent 设计的核心竞争力。

## 上下文窗口为什么总是不够

很多人问：模型上下文窗口已经很长了，为什么还要管理？

因为约束不只是容量，还有质量。

**窗口再长也有限。** 多轮对话、多个文档、多个工具结果叠加后，上下文总会膨胀。

**信息越多，噪声越多。** 模型看到大量无关信息时，真正关键的约束被稀释。

**上下文越长，成本越高。** 每次调用都重复输入大量历史，是反复为旧信息付费。

**上下文越长，调试越难。** 模型行为异常时，很难判断哪段历史影响了它。

成熟系统追求的：

> 当前模型调用只看到完成当前判断所需的**最小充分信息**。

"最小"为了降低噪声和成本，"充分"为了不丢关键条件。

## 四类记忆

### 1. 会话记忆（Session Memory）

服务当前对话或任务链路：

- 当前目标
- 用户刚给出的限制
- 已上传文件
- 最近工具结果
- 当前步骤和待办
- 临时假设

价值是保持短期连贯。通常不应长期保存，任务结束后归档成摘要或丢弃。

### 2. 用户记忆（User Memory）

围绕特定用户长期沉淀：

- 输出风格偏好
- 常用语言和表达方式
- 关注领域
- 反复出现的业务背景
- 明确说过的工作习惯

价值是让系统越来越懂这个用户。但必须谨慎写入——一次临时要求不一定是长期偏好，模型推测不应该直接写成用户记忆。

### 3. 任务记忆（Task Memory）

围绕某个项目、工单、研究线沉淀：

- 已确认的目标
- 决策记录
- 边界条件
- 阶段性结论
- 风险和待办
- 已拒绝的方案

比会话记忆更稳定，但仍然绑定具体任务，不应随便跨项目复用。

### 4. 知识记忆（Knowledge Memory）

稳定、可复用的信息：

- 企业制度
- 操作手册
- 领域术语
- 标准流程
- 历史案例

这一层常和 RAG 系统重叠。但 RAG 只是"检索知识"的机制，记忆系统还要处理用户偏好、任务状态、决策历史和人工确认。

## Letta/MemGPT 四层记忆架构

Letta（前身 MemGPT）建立了当前主流的记忆概念框架。核心思想：**把 LLM 上下文视为"内存（RAM）"，外部存储视为"磁盘"，通过分页/交换机制管理无限信息。**

| 层级 | 类型 | 机制 | 类比 |
|------|------|------|------|
| L1 | **Message Buffer** | 最近 N 条消息，FIFO 滑动窗口 | CPU 缓存 |
| L2 | **Core Memory** | 上下文内的可编辑 JSON 块，Agent 自主读写 | RAM |
| L3 | **Recall Memory** | 全量对话历史的可搜索索引 | 硬盘索引 |
| L4 | **Archival Memory** | 外部数据库存储，向量检索 | 归档存储 |

### Core Memory 的设计

Core Memory 是上下文内始终存在的可编辑 JSON 块。每个块有：

- `label`：名称
- `value`：内容
- `limit`：大小上限（字符数）
- `description`：用途

典型结构：

```json
{
  "human": "用户偏好：先结论后展开，关注半导体行业",
  "persona": "Agent 身份和行为准则",
  "project_context": "当前项目：某半导体企业 AI 诊断"
}
```

Agent 可以通过工具调用修改这些块。关键创新：**记忆块可以跨 Agent 共享。** Agent A 写入一个块，Agent B 读取它——实现跨 Agent 记忆协调，无需消息传递。

### Recall Memory

全量对话历史存在外部。Agent 用搜索工具查询——不是全部加载到上下文，而是按需检索。支持关键词和语义搜索。

### Archival Memory

显式存储的知识（事实、文档、学习到的规则）。Agent 决定什么值得归档，使用专门的工具调用操作。

与 Recall 的区别：Recall 是对话历史的索引，Archival 是主动整理的知识。

### Sleep-Time Compute

Letta 2025 年提出的概念：**Agent 在"空闲时"异步整理记忆。**

背景 Agent 在用户不在时：

1. 回顾最近的对话
2. 提取和整合信息
3. 更新 Core Memory 块
4. 归档重要事实
5. 为下次对话准备优化后的上下文

这把记忆管理延迟从用户交互时间中分离出来。类似人类的睡眠记忆巩固过程。

## LangGraph 的记忆实现

LangGraph v1 用两套互补系统实现记忆：

### Short-Term Memory（Checkpointer）

线程级状态持久化。每个执行步骤产生检查点。支持：

- **恢复**：中断后从检查点继续
- **时间旅行调试**：从任意检查点回放
- **分支**：在任意点分叉对话

后端：MemorySqlite（开发）、PostgresSaver（生产）、Redis（低延迟）。

### Long-Term Memory（Store）

跨线程持久化，用命名空间组织的键值对。每条记忆有：

- namespace（用户 ID 或 Agent ID）
- key（唯一标识）
- value（数据）
- metadata（可选元数据）

支持向量语义搜索。Agent 可以在不同线程间检索同一用户的历史信息。

## 上下文管理策略

### 1. 分层组织上下文

不要把所有信息平铺进一个 Prompt：

| 层级 | 内容 | 优先级 |
|------|------|--------|
| 系统层 | 角色、原则、安全边界 | 必须保留 |
| 任务层 | 当前目标、范围、验收标准 | 必须保留 |
| 状态层 | 已完成步骤、当前进度、待办 | 必须保留 |
| 证据层 | 相关文档、工具结果、引用来源 | 按需裁剪 |
| 记忆层 | 与当前任务相关的历史偏好 | 检索后筛选 |
| 交互层 | 最近几轮对话 | 保留最近 N 轮 |

分层的好处：模型出错时，可以判断是任务目标不清、状态过载、证据不足，还是记忆误用。

### 2. Token 预算分配

显式分配上下文空间：

| 类别 | 预算占比 |
|------|----------|
| 系统提示 | 20% |
| 核心记忆 | 30% |
| 检索上下文 | 30% |
| 对话缓冲 | 20% |

没有预算，系统会被最近、最长、最吵的信息占满，而不是被最重要的信息驱动。

### 3. 滚动摘要

会话变长时，不是机械保留全部历史，而是生成滚动摘要。

有效摘要包含：当前目标、已确认事实、已拒绝方案、用户偏好、关键约束、待办、未解决问题。

无效摘要只是"用户先说了什么，然后系统回答了什么"的流水账。省了字数，没提高可执行性。

### 4. 相关性检索

长期记忆不常驻上下文，按需检索。触发方式：

- 用户提到某个项目 → 检索该项目任务记忆
- 用户要求延续之前风格 → 检索用户偏好
- Agent 进入某个分析节点 → 检索相关案例和术语

检索后二次筛选：是否相关？是否过旧？是否被新结论覆盖？是否引入权限问题？

### 5. 摘要与原文并存

- 摘要常驻或优先注入
- 原文按需回取
- 关键证据保留引用和定位

类似搜索系统：列表页显示标题和摘要，需要时再打开原文。

## 记忆写入策略

什么时候写记忆，比"能不能存"更重要。

**应该写入的：**

- 用户明确表达的稳定偏好
- 已确认的重要事实
- 经人工确认的最终结果
- 高价值、可复用的中间结论
- 明确的决策记录
- 已被拒绝且未来容易重复探索的方案

**不应该直接写入的：**

- 模型尚未验证的猜测
- 临时闲聊
- 一次性格式要求
- 失败工具返回的噪声
- 已知错误但尚未清理的结果

核心原则：

> **日志可以宽，记忆必须严。**

日志记录发生过什么；记忆会影响未来判断。写入记忆应该比写日志更谨慎。

每条长期记忆至少包含：

| 字段 | 说明 |
|------|------|
| content | 内容 |
| type | 用户偏好 / 项目事实 / 决策 / 知识 / 风险 |
| source | 用户 / 工具 / 模型 / 人工审核 |
| scope | 用户级 / 项目级 / 组织级 |
| created_at | 写入时间 |
| confidence | 置信度或确认状态 |
| expires_at | 过期时间或更新规则 |

没有来源和时间的记忆，未来无法判断是否可信。

## 记忆读取策略

读取不是"存了就拿出来"。需要触发条件和筛选规则。

常见触发：

- **显式键值**：按用户 ID、任务 ID 查询
- **语义检索**：根据当前任务内容找相关历史
- **流程节点触发**：进入某类节点时加载对应规则
- **用户指令触发**："延续上次那个方案"
- **异常触发**：当前任务失败时检索类似失败案例

**冲突检查**——旧记忆说"用户喜欢详细解释"，新输入说"这次只要一句话"。当前输入优先于旧偏好。

记忆优先级（从高到低）：

1. 当前用户本轮明确指令
2. 当前任务已确认状态
3. 人工确认的项目记忆
4. 稳定用户偏好
5. 历史案例和知识记忆
6. 未确认的模型摘要

## 记忆系统失败模式

| 失败模式 | 表现 | 解决方法 |
|----------|------|----------|
| **记忆膨胀** | 检索越来越脏，上下文越来越重 | 写入门槛、过期机制、定期清理 |
| **记忆污染** | 未验证推测进入长期记忆 | 区分草稿、低置信度、人工确认 |
| **记忆冲突** | 旧偏好和新偏好并存 | 版本、时间、优先级规则 |
| **记忆泄漏** | 跨用户/租户/项目信息错误复用 | 权限隔离和范围检查 |
| **记忆不可追溯** | 不知道某条记忆从哪来 | 记录来源、时间、写入者、确认状态 |
| **过度个性化** | 历史偏好压过当前任务要求 | 当前明确指令优先 |
| **记忆僵化** | 一直引用旧规则 | 过期时间、更新机制、人工纠错入口 |

## 工程实现的六个阶段

不要一开始就做复杂记忆图谱。分阶段建设：

**第一阶段：最近消息 + 会话摘要。** 解决短期连贯。保留最近几轮对话，维护一个任务状态摘要。

**第二阶段：结构化任务状态。** 把目标、约束、待办、已确认结论、风险项拆成字段，不放在长文本里。

**第三阶段：用户偏好和项目记忆。** 只写入明确、稳定、可复用的信息。记录来源和适用范围。

**第四阶段：按需检索长期记忆。** 根据任务、项目、用户和语义检索，再做二次筛选。

**第五阶段：审计与纠错。** 能回答：这条记忆是什么？从哪来？为什么被读取？什么时候失效？错了怎么改？

**第六阶段：高风险记忆人工确认。** 涉及客户事实、合规规则、长期用户画像的记忆，经过人工确认再长期生效。

## 最小可用设计

快速做可用的 Agent 记忆系统，从这个结构开始：

```
Session State（会话状态）
├─ current_goal
├─ constraints
├─ completed_steps
├─ pending_steps
├─ recent_tool_results
└─ open_questions

Task Memory（任务记忆）
├─ confirmed_facts
├─ decisions
├─ rejected_options
├─ risks
└─ final_outputs

User Memory（用户记忆）
├─ stable_preferences
├─ domain_context
└─ communication_style

Knowledge Memory（知识记忆）
├─ documents
├─ cases
├─ rules
└─ glossary
```

每次模型调用前，构造上下文：

1. 放入当前目标和验收标准
2. 放入任务状态摘要
3. 检索与当前步骤相关的记忆
4. 过滤过期、冲突、越权内容
5. 只注入最相关信息
6. 保留来源和置信度

每次任务结束后，处理写入：

1. 把完整执行过程放入日志
2. 把可复用结论提炼成记忆候选
3. 过滤临时噪声和未验证猜测
4. 对高风险记忆要求人工确认
5. 写入时记录来源、时间和适用范围

这个设计不复杂，但已经比"保存全部聊天记录"可靠得多。

## 结构化 vs 非结构化记忆

| 维度 | 结构化（Letta blocks、LangGraph Store） | 非结构化（向量存储、RAG） |
|------|---------------------------------------|------------------------|
| 访问模式 | 可预测（按标签、命名空间） | 灵活（按相似度） |
| Schema | 需要提前设计 | 不需要 |
| 噪声 | 低（精确匹配） | 较高（语义近似可能误导） |
| 适合 | 已知、频繁访问的信息 | 探索性知识检索 |
| 代价 | 前期设计成本 | 检索质量依赖 embedding |

趋势是**混合**：结构化存已知、高频信息（用户画像、Agent 身份、任务状态），非结构化存知识文档。

## 总结

上下文和记忆决定了 Agent 能不能从一次性问答走向长期协作。

- **上下文管理**解决"当前这一步该看什么"
- **记忆系统**解决"过去哪些东西值得影响未来"
- **日志**解决"发生过什么、如何追溯"

三者不能混用。一个好的 Agent 不是记得越多越好，而是能在正确的时间拿出正确的历史信息，并知道它从哪来、是否还可信、是否应该继续影响当前任务。
