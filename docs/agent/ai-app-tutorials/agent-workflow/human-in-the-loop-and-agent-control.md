# Human-in-the-loop 与 Agent 可控性

**TL;DR:** HITL 不是"AI 不够好所以让人兜底"，而是一种控制机制——当系统进入高风险、高不确定性或高责任节点时，让人类把关。2025 年的生产实践验证了五种 HITL 模式（审批门、升级梯、置信度路由、协作起草、惰性审计），实现了 78% 的关键错误减少。可控性包括六个维度，人工审核只是其中一项。

## HITL 不是补丁

Human-in-the-loop（HITL）指在 AI/Agent 执行过程中引入人工判断、审核、修改或批准。

把 HITL 理解成"AI 不够好，所以让人兜底"太窄了。更准确地说，HITL 是一种**控制机制**：当系统进入高风险、高不确定性或高责任节点时，让人类把关。

Agent 越自主，越需要 HITL。Agent 不只生成内容，还会做决策、调用工具、改变外部状态。没有人工介入点，系统一旦跑偏，错误会连续传递到后续步骤。

HITL 的价值有四个：

**补充业务上下文。** 模型知道行业常识，但不知道客户的预算、组织文化、政治约束、真实痛点。

**控制高风险动作。** 发邮件、改数据、提交订单、触发生产操作——不能只靠模型判断。

**建立责任链。** 企业系统需要知道每个关键决策是谁批准的。没有责任链，系统很难进入真实业务流程。

**修正模型盲点。** 模型会幻觉、过度自信、忽略边界。人工审核在关键节点阻断错误放大。

HITL 不是自动化的反面，是让自动化可用、可信、可追责的条件。

## 五种生产验证的 HITL 模式

一个分析了 420 万次 Agent 任务、14 个月运行的生产案例，验证了五种 HITL 模式的效果：

### 1. 审批门（Approval Gate）

每个 Agent 动作经过决策门：自动批准 / 排队审核 / 拒绝。

关键设计：配置 `confidence_threshold`（置信度阈值）和 `escalation_rules`（升级规则）。低于阈值 → 排队；匹配升级规则 → 强制人工审核，无论置信度多高。

### 2. 升级梯（Escalation Ladder）

动作按严重性分级：LOW → MEDIUM → HIGH → CRITICAL。每级映射到不同审核人和 SLA。

关键设计：升级规则可以跳级。比如金融交易超过 $10K，直接从 MEDIUM 升到 CRITICAL，跳过中间层。

### 3. 置信度路由（Confidence-Based Routing）

**不能只靠模型自报置信度**——LLM 自我评估不可靠。使用多信号复合评分：

| 信号 | 权重 |
|------|------|
| LLM 置信度分数 | 15% |
| 规则验证器结果 | 30% |
| 同类任务历史准确率 | 25% |
| 数据质量分数 | 20% |
| 用户偏好设置 | 10% |

多信号方案直接解决了 LLM 自我评估不可靠的问题。

### 4. 协作起草（Collaborative Drafting）

不是二元的通过/拒绝，而是 Agent 产出可编辑的结构化草稿（`DraftBlock`：段落类型 + 内容 + 置信度）。人类编辑特定段落，Agent 观察修改并学习。

Agent 为每个段落类型维护一个"人类修改缓冲区"，用于逐步学习用户的偏好和标准。

### 5. 惰性审计（Audit Trail with Lazy Review）

动作立即执行，但每次都记录完整上下文（输入、推理、输出、置信度）。`SmartSampler` 事后抽样审核：

- 所有低置信度动作
- 所有新任务类型
- 高置信度动作随机 10%
- 异常检测标记的动作

### 实测效果

| 指标 | 改善 |
|------|------|
| 关键错误 | 减少 78% |
| 问题发现平均时间 | 减少 91% |
| 年净节省 | $745K |
| 人工审核量 | 从 100% 降到 12% |

关键发现：**分级策略把人工审核量降到总动作的 12%，同时显著提升了错误拦截率。** 人的注意力被集中到最值得看的地方。

## 哪些节点需要人工介入

不是所有节点都应该审核。审核太多失去效率；审核太少风险不可控。

判断一个节点是否需要人工介入，看四个维度：

**影响范围。** 出错会影响多少后续步骤？基础事实类节点通常影响范围大。

**可逆性。** 出错后能不能恢复？生成草稿容易恢复，删除数据、外发邮件难恢复。

**主观性。** 是否涉及偏好、权衡、策略选择？方案优先级、风险接受度、客户沟通口径。

**外部性。** 是否影响外部系统或他人？只读分析风险低，写入和外发风险高。

实用分级：

| 风险等级 | 处理方式 |
|----------|----------|
| 低风险 | 自动执行，事后抽样审计 |
| 中风险 | 自动生成，人类确认后进入下一步 |
| 高风险 | 必须人工审批，审批前不得执行 |
| 不可逆 | 默认禁止自动执行，除非有明确授权和回滚方案 |

## 审核不是"看一眼"

很多 HITL 设计失败，是因为审核界面只是把模型原文丢给人看。审核者面对几千字输出，只能凭感觉点通过。

好的审核像决策表单，不是长文本。至少包含五部分：

1. **任务背景**：当前任务、所处节点、为什么需要审核
2. **关键信息摘要**：压缩成审核者需要判断的要点
3. **证据与来源**：关键事实从哪来，哪些是强证据，哪些只是推断
4. **风险提示**：系统自动标出可能有问题的地方
5. **操作选项**：通过、拒绝、要求修改、改后通过、转交他人

原则：**让人判断，而不是让人重新读完整个任务。**

## 中断与继续

HITL 会让自动流程暂停，系统必须设计"中断与继续"机制。

中断时保存完整状态：

- 当前任务目标
- 当前节点
- 已完成节点输出
- 待审核内容
- 工具调用历史
- 暂停原因
- 状态版本

审核完成后的处理：

**通过** → 从暂停点继续。继续前检查状态是否过期（审核隔了两天，外部数据可能需要刷新）。

**拒绝** → 终止或回退，记录拒绝原因。

**要求修改** → 明确修改范围：Agent 重做当前节点？人直接编辑结果？修改后是否需要再次审核？

**超时未审** → 提醒、升级、自动取消，或走降级路径。

没有这些状态机制，HITL 会变成流程中的黑洞：任务暂停了，但没人知道怎么恢复。

## 人工修改如何进入状态

人工审核不只是"通过/拒绝"。很多时候人会直接改内容或给修改意见。

三个原则：

**人工确认优先于模型草稿。** 一旦人类修改并确认，后续节点使用人工确认版本，不是模型原始版本。

**保留差异。** 记录人改了什么。复盘模型常犯错误，改进 Prompt 或规则。

**标注来源。** 状态里区分：模型生成、工具返回、人工修改、人工确认。不同来源的可信度和责任不同。

比如行业概览节点中，模型写"市场规模约 5000 亿"，审核者改成"根据某报告，2024 年中国大陆半导体设备市场约 X"。后续 Agent 使用审核者确认后的版本，并保留原始模型输出供追溯。

## 可控性的六个维度

HITL 是可控性的关键组成，但不是全部。一个可控 Agent 系统从设计上就限制风险：

| 维度 | 含义 |
|------|------|
| 目标可控 | 目标明确、边界清楚、完成标准可检查 |
| 路径可控 | 系统知道当前处于哪个阶段，下一步为什么执行 |
| 工具可控 | 工具有权限、参数限制、审批和审计 |
| 成本可控 | 有步数、时间、Token、API 调用上限 |
| 状态可控 | 草稿、确认结果、长期记忆分层管理 |
| 人工可接管 | 人可以暂停、查看、修改、重试、终止 |

如果工具权限完全放开、状态不可见、成本无限制，即使有审核点，系统仍然危险。

## 渐进式自主（Progressive Autonomy）

2025 年 HITL 领域的一个重要模式：**Agent 不应该一开始就有完整自主权。**

渐进式自主：

- Agent 在高监督模式下启动
- 通过在特定任务类型上的可靠表现"赚取"自主权
- 自主权级别是**按任务类型**设定的，不是全局开关
- 出错后自动降级回高监督模式

新能力上线时可以用**影子模式**：Agent 与人工并行执行，输出被比较但不被采用，直到可靠性被证明。

## Agent 运行状态

一个成熟 Agent 不应该只有"运行中"和"完成"两个状态：

```
pending → running → waiting_review → revision_requested → running
                ↓                        ↓
              paused                   paused
                ↓                        ↓
            completed / failed / cancelled
```

状态设计清楚，用户和系统才知道下一步该做什么。`waiting_review` 需要提醒审核人；`revision_requested` 需要重新执行某个节点；`paused` 需要等待人工恢复。

状态要和权限绑定。不是所有人都能恢复、取消或批准高风险任务。

## 审计与责任链

每个关键节点都应该留下：

- 谁发起任务
- Agent 做了什么
- 用了哪些工具
- 输出了什么
- 谁审核
- 审核意见
- 审核后改了什么
- 什么时候继续执行

这些记录不只是为了合规，也用于系统改进：某个节点经常被要求修改 → Prompt、工具或数据源可能有问题；某类风险经常被人工补充 → 可以把它变成自动检查规则。

## HITL 的分层策略

| 层级 | 策略 | 适用场景 |
|------|------|----------|
| 必审 | 高风险、不可逆、正式外发、合规 | 关键决策节点 |
| 抽审 | 中风险、大批量、质量相对稳定 | 常规产出节点 |
| 事后审计 | 低风险自动化节点 | 只读分析、内部草稿 |
| 免审 | 无副作用、可恢复、低影响 | 格式转换、简单摘要 |

分层的目的不是少审核，是把人的注意力放在最值得看的地方。

## 设计检查清单

- 哪些节点必须人工审核？
- 每个审核点的通过标准是什么？
- 审核者需要看哪些摘要、证据和风险提示？
- 审核结果有哪些选项？
- 拒绝后流程如何处理？
- 要求修改后是 Agent 重做还是人工直接改？
- 审核超时怎么办？
- 人工修改如何进入状态？
- 谁有权限审批高风险动作？
- 审核日志是否可追溯、防篡改？
- 是否有渐进式自主机制？

HITL 的目标不是把人塞进流程里，而是让人出现在最关键、最需要判断、最能阻断风险的位置。
