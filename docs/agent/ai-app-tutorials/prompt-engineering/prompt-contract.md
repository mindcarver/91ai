# Prompt Contract 与 Prompt 管理

## 核心概念

### 一、Prompt Contract 思维

Prompt Contract 是最核心的概念。

什么是 Prompt Contract？它是一种思维方式，把 Prompt 看作你与大模型之间的一份契约。这份契约包含：

输入约定（你承诺提供什么）：你会给模型提供哪些信息？格式是什么？范围是什么？输入的信息质量如何保证？

输出约定（模型承诺返回什么）：模型应该返回什么格式的内容？包含哪些字段？每个字段的含义和约束是什么？输出的质量标准是什么？

边界约定（什么情况怎么处理）：当输入不完整时怎么做？当信息不足无法回答时怎么做？当任务超出能力范围时怎么做？当遇到安全风险时怎么做？

为什么需要 Prompt Contract 思维？

因为它把 Prompt 从一段"随意的文字"变成了一份"可管理的规范"。在企业应用中，一个 Prompt 不是写完就结束的——它需要被测试、被评估、被迭代、被版本管理。如果你没有一个清晰的 Contract，你根本不知道怎么评估 Prompt 的好坏，更不知道该往哪个方向优化。

举个例子。你写了一个行业分析 Prompt，模型输出了分析结果。你怎么判断这个结果好不好？如果你有 Contract，你可以对照 Contract 来检查：输出包含了约定的所有字段吗？格式符合约定吗？当信息不足时有按照约定标注吗？没有 Contract，你只能凭感觉说"看起来不错"或"好像不太行"。

Prompt Contract 思维的另一个价值是它让 Prompt 变成了团队协作的基础。如果多个开发者都在维护同一套 Prompt，没有 Contract 就会出现不一致。有了 Contract，每个人都知道这个 Prompt 的输入是什么、输出是什么、边界是什么，协作效率会高得多。

### 二、企业级 Prompt 的结构

一个企业级 Prompt 不是一段文字，而是一个有结构的文档。它通常包含以下几个部分：

元数据——Prompt 的 ID、名称、版本号、创建者、最后更新日期、用途描述。这些信息用于管理和追踪 Prompt。

输入定义——这个 Prompt 接受哪些输入参数？每个参数的类型、格式、取值范围是什么？哪些是必填的，哪些是选填的？

System Prompt 模板——包含角色定义、行为规范、输出格式要求等固定内容，其中可以嵌入变量（比如行业名称、公司名称等），在运行时动态替换。

Few-shot 示例——提供 2-3 个高质量的输入输出示例，帮助模型理解期望的模式。

质量标准——怎么判断这个 Prompt 的输出质量？有哪些评估维度？每个维度的合格标准是什么？

边界处理——遇到异常输入怎么处理？信息不足时怎么输出？任务超出范围时怎么回复？

测试用例——一组用于验证 Prompt 效果的输入输出对，每次修改 Prompt 后都要跑一遍测试用例，确保没有退化。

变更记录——每次修改了什么、为什么修改、修改前后的效果对比。

这种结构化的 Prompt 管理方式，把 Prompt 从"一段文字"升级成了"一个可管理的组件"。在企业应用中，你可能需要维护几十个甚至上百个 Prompt，没有这种结构化管理，Prompt 就会变成一团乱麻。

### 三、Prompt 版本管理意识

Prompt 是需要迭代的。你写了第一版，测试后发现效果不理想，修改出第二版，再测试，再修改，直到满意。这个过程和软件开发的迭代过程一模一样。

但很多团队不重视 Prompt 的版本管理，随意修改 Prompt 而不记录变更历史。这就导致几个问题：

无法回滚。新版本 Prompt 效果更差了，你想回到旧版本，但旧版本的内容已经找不到了。

无法评估。你不知道这次修改到底改了什么，无法评估修改的效果。

无法协作。其他团队成员不知道 Prompt 被修改了，可能会基于旧版本做开发。

无法复现。用户报告了一个问题，但你无法复现当时的 Prompt 版本，无法诊断问题。

版本管理的基本实践：

给每个 Prompt 一个唯一 ID 和版本号。每次修改都更新版本号。

记录每次修改的内容、原因、修改人。

保留旧版本的完整内容。

每次修改后跑一遍测试用例，记录结果。

建立一个 Prompt 库，集中管理所有 Prompt 的版本。

### 四、Prompt Contract 的评估闭环

Contract 的价值不只是"写清楚输入输出"，而是让 Prompt 可以被持续评估。没有评估闭环，Contract 很容易变成一份没人看的文档。

评估闭环包括五个步骤：

1. 建立基准测试集。每个 Prompt 至少保留 5-10 个固定测试输入，覆盖标准场景、边界场景、异常输入和高风险输入。
2. 定义可检查的质量标准。不要只写"输出要好"，要写"字段齐全率 100%"、"JSON 校验通过率 98% 以上"、"高风险场景必须给出理由"。
3. 每次改动后跑回归测试。新版本不能只在一个样例上表现更好，还要确保旧样例不退化。
4. 记录 Prompt diff。每次修改要记录改了角色、任务、上下文、约束、示例还是输出格式，方便定位效果变化来自哪里。
5. 做失败归因。失败要分类记录：输入缺失、Prompt 指令冲突、Schema 太复杂、模型能力不足、知识缺失、下游校验过严。

一个成熟的 Contract 不应该只保存"当前 Prompt 是什么"，还应该保存"为什么这样写、怎么证明它有效、什么情况下它会失败"。

### 五、Prompt Contract 的生命周期

企业里的 Prompt 不是个人草稿，而是一个可发布、可回滚、可审计的组件。它应该有完整生命周期。

草稿阶段。允许快速试验，但不能进入生产调用。此阶段重点验证任务是否成立、输入输出是否合理。

评审阶段。由业务负责人、开发者和质量负责人一起检查：业务目标是否清楚、输出是否能被下游消费、边界处理是否充分、安全风险是否可接受。

发布阶段。通过测试后标记为稳定版本，绑定模型版本、Schema 版本、上线时间和调用场景。

监控阶段。上线后持续记录调用量、失败率、重试率、人工评分、用户反馈和异常样例。

迭代阶段。根据失败样例和业务变化修改 Prompt。每次修改都要有变更原因、测试结果和回滚方案。

废弃阶段。当业务场景消失、模板被新版本替代、或质量长期不达标时，要标记为 deprecated，而不是直接删除。保留旧版本是为了复现历史结果和排查线上问题。

生命周期管理的关键不是流程复杂，而是责任清晰。每个 Contract 至少要有 owner、reviewer、使用方、当前状态和最近一次评估结果。

## 实战分析

### 设计 Prompt Contract 模板

模板应该是一个标准化的文档结构，包含以下部分：Contract 名称和版本、输入约定（参数名、类型、是否必填、格式要求、示例值）、输出约定（输出格式、字段定义、字段约束、示例输出）、边界约定（信息不足时的处理方式、超出范围时的回复方式、安全风险的处理策略）、质量标准（准确率要求、完整性要求、格式合规率要求）。

这个模板不是写一次就固定的，它应该随着你的实践不断迭代。每次使用后发现新的边界情况，就补充到 Contract 里。

### Prompt Contract 示例

下面是一个简化版 Contract，用于"行业 AI 机会评估"。

```yaml
id: prompt.industry_ai_opportunity.v1
name: 行业 AI 机会评估
status: stable
owner: AI 应用顾问
version: 1.0
model_family: general_llm

purpose: >
  根据行业名称和简要上下文，评估该行业是否值得继续做 AI 应用机会深挖。

inputs:
  industry_name:
    type: string
    required: true
    description: 行业名称
    example: 半导体制造
  context:
    type: string
    required: false
    description: 用户提供的补充背景、痛点或目标

outputs:
  format: json
  required_fields:
    - industry_name
    - opportunity_score
    - top_scenarios
    - recommendation
  constraints:
    opportunity_score: 1 到 10 的数字
    recommendation: 只能是 值得深入 / 需要调研 / 暂不建议

boundaries:
  insufficient_context: 标注"需要进一步调研"，不要编造具体数据
  out_of_scope: 如果不是行业或业务场景分析，返回超出范围说明
  safety: 不输出个人隐私、密钥、未授权企业内部信息

quality_gates:
  json_valid_rate: 100%
  required_fields_rate: 100%
  scenario_count: 1 到 5 个
  no_fabricated_numbers: 必须通过人工抽检

test_cases:
  - industry_name: 半导体制造
    expected: 应覆盖良率、设备、工艺、检测等生产场景
  - industry_name: 一个不存在的行业名称
    expected: 不编造，输出需要进一步调研

change_log:
  - version: 1.0
    date: 2026-05-11
    change: 初始版本
    reason: 建立行业机会评估模板
```

这个示例的重点不是 YAML 语法，而是把输入、输出、边界、质量门槛和测试样例放在同一份契约里。这样开发、业务和评估人员看到的是同一个标准。

### 企业级 Prompt 设计要点

企业级 Prompt 和个人使用的 Prompt 有三个核心差异：

可管理性。个人 Prompt 写完就用了，企业级 Prompt 需要被版本管理、被追踪、被团队共享。这就要求 Prompt 有清晰的元数据和结构。

可测试性。个人 Prompt 靠"看一眼"判断好坏，企业级 Prompt 需要可量化评估。这就要求 Prompt 有明确的输出约定和质量标准。

可维护性。个人 Prompt 改了就改了，企业级 Prompt 的每次修改都要记录原因和效果。这就要求 Prompt 有完整的变更记录和测试用例。

每个 Prompt 的设计过程都应该遵循 Role-Task-Context-Constraint 框架，写完后用至少 3 个不同的输入测试，根据测试结果迭代优化。
