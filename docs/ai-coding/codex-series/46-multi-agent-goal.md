# Codex CLI /goal 多 Agent 目标协调：让并行工作保持对齐

> TL;DR：`/goal` 在会话中设置一个持久化的目标文本，Codex 在后续所有操作中都会参考这个目标。在多 Agent 场景下，目标可以传递给子代理，确保所有并行工作都朝同一个方向走。`/goal <文本>` 设置目标，`/goal` 查看，`/goal pause` 暂停，`/goal resume` 恢复，`/goal clear` 清除。目标最多 4000 字符，超过的话可以用文件引用。本文覆盖 /goal 的用法、在多 Agent 中的传递机制、目标分解策略和三个实战场景。

---

## 1. 为什么多 Agent 需要目标协调

让三个子代理并行审查一个 PR：一个看安全，一个看性能，一个看测试。每个子代理独立工作，可能产生这样的结果：

- 安全审查员发现了一个 SQL 注入漏洞，建议重写整个数据库访问层
- 性能审查员建议把所有同步调用改成异步
- 测试审查员建议为每个函数补充边界条件测试

三个子代理各自都做了认真分析，但它们的建议互相矛盾——重写数据库访问层和改异步调用会大量修改代码，同时补充测试会让改动范围更大。如果你同时采纳所有建议，改动量大到可能引入新 bug。

问题出在哪里？三个子代理没有一个共同的方向。它们不知道你的优先级——是先修安全漏洞？还是先补测试？还是先优化性能？

`/goal` 解决的就是这个问题。它让你在派生子代理之前先声明："这次审查的首要目标是找出安全漏洞并给出最小修复方案，性能和测试是次要的。"所有子代理在执行时都会参考这个目标，它们的建议会自动对齐到你的优先级。

## 2. /goal 命令详解

### 设置目标

```
> /goal 完成认证模块从 JWT 到 Session 的迁移，保持所有现有测试通过
```

Codex 确认目标已设置，后续所有操作都会参考这个目标。

### 查看目标

```
> /goal
```

显示当前设置的目标文本。如果没有设置过目标，会提示"没有活跃的目标"。

### 暂停目标

```
> /goal pause
```

暂停后，Codex 暂时不再参考目标做决策。适用于你想让 Codex 做一些和当前目标无关的临时工作。

### 恢复目标

```
> /goal resume
```

恢复之前暂停的目标。

### 清除目标

```
> /goal clear
```

永久删除当前目标。

### 内联目标

你可以把目标直接写在命令后面：

```
> /goal 修复登录页面的所有 TypeScript 类型错误，不改变任何运行时行为
```

也可以只输入 `/goal` 不带参数来查看当前目标。

## 3. 目标的结构与限制

### 字符限制

目标文本必须非空，最多 **4000 字符**。这个限制是硬性的——超过 4000 字符会被截断或拒绝。

4000 字符看起来不少，但对于复杂项目可能不够。如果你的目标涉及多个阶段的详细描述，推荐的做法是把详细内容写到一个文件里，然后在目标中引用文件：

```
> /goal 按照 docs/migration-plan.md 执行数据库迁移，每个阶段完成后跑一遍测试套件
```

这样目标文本很短（引用了文件路径），但 Codex 可以读取文件获取完整的分阶段描述。

### 目标内容的写法

好的目标应该包含：

1. **最终成果**：你想要什么结果
2. **约束条件**：什么不能做
3. **验证方式**：怎么判断完成了

反面例子（太模糊）：

```
> /goal 改进代码质量
```

正面例子（具体、可验证）：

```
> /goal 把 src/auth/ 下所有函数的 TypeScript 类型覆盖率提高到 100%，不改变运行时行为，每个文件改完后跑 pnpm test 确认测试通过
```

## 4. 目标在多 Agent 间的传递

### 传递机制

当你设置了 `/goal` 后派生子代理时，目标文本会被注入到子代理的上下文中。这意味着：

1. **主线程有目标** → 派生的子代理自动获得目标
2. **子代理的行为会参考目标** → 优先做和目标相关的分析，忽略无关内容
3. **主线程更新目标** → 后续派生的新子代理获得更新后的目标

注意：已经运行的子代理**不会**实时接收目标更新。目标是在子代理创建时注入的，不是实时同步的。

### 传递的边界

```
主线程 (goal: 修复所有 TypeScript 类型错误)
  ├─ 子代理 A (获得目标) — 扫描 src/ 下的类型错误
  ├─ 子代理 B (获得目标) — 扫描 tests/ 下的类型错误
  └─ 子代理 C (获得目标) — 生成修复补丁
```

子代理的嵌套深度受 `agents.max_depth` 限制（默认 1）。这意味着子代理不能再派生下一级子代理（除非你把 `max_depth` 设为 2 或更高）。在默认配置下，目标只传递一层。

### /side 中的目标

`/side` 对话**不支持**目标。如果你在主线程中设置了目标，然后开了一个 `/side` 对话，side 对话中 `/goal` 不可用。这是设计上的限制——`/side` 是临时的旁路对话，不需要持久化的目标。

## 5. /goal 与 /plan 的配合

`/goal` 和 `/plan` 是互补的工具：

| 工具 | 作用 | 何时用 |
|------|------|--------|
| `/goal` | 声明要达到的最终状态 | 任务开始前 |
| `/plan` | 生成具体的执行步骤 | 目标设定后 |

推荐的工作流：

```
1. /goal 完成认证模块迁移，测试全部通过
2. /plan 设计迁移方案，列出步骤和影响文件
3. （审查方案）
4. 按方案执行
```

目标告诉你"要去哪里"，规划告诉你"怎么走"。

## 6. 目标分解策略

对于大型任务，直接设一个笼统的目标不如分解为多层目标。

### 单一大目标的问题

```
> /goal 重构整个认证系统
```

这个目标太大了。Codex 可能在子代理间分配不清楚——安全审查员、性能审查员、测试审查员都认为自己应该覆盖整个认证系统，导致重复工作。

### 分解策略

**按模块分解**：

```
> /goal 按以下优先级完成认证模块重构：
> 1. 先修 auth/token.ts 中的 JWT 过期处理 bug
> 2. 然后把 auth/session.ts 从回调改为 Promise
> 3. 最后更新 auth/ 下的所有测试
> 每一步完成后跑 pnpm test 确认不引入新问题
```

**按维度分解**：

```
> /goal 对本次 PR 的审查目标是：
> 1. 安全：找出所有可能被恶意利用的输入点
> 2. 测试：找出没有测试的新增函数
> 3. 其他维度的问题记录但不展开
> 汇总时按安全 > 测试 > 其他排序
```

**按阶段分解**（配合文件引用）：

```
> /goal 按照 .claude/phases/migration.md 中的三阶段计划执行，每个阶段有独立的验收标准
```

## 7. 实战场景

### 场景一：大型重构任务

```
> /goal 把项目从 REST API 迁移到 tRPC，分三步：
> 1. 先定义 tRPC router 和类型
> 2. 逐个迁移 API 路由，每次迁移一个
> 3. 删除旧的 REST 路由文件
> 每步完成后运行 pnpm build && pnpm test，确保不破坏现有功能
```

然后派生子代理：

```
> 派两个 agent：
> Agent 1：扫描所有 REST API 路由，列出每个路由的请求参数、返回类型和中间件
> Agent 2：阅读 tRPC 文档，总结我们项目需要的 tRPC 模式
> 等两个都完成后，汇总信息，开始规划迁移步骤
```

### 场景二：多步骤功能开发

```
> /goal 实现用户通知系统：
> 1. 数据库表和 model
> 2. 通知发送的后端服务
> 3. 前端通知面板 UI
> 4. WebSocket 实时推送
> 按 1→2→4→3 的顺序实现，每步完成后跑测试
```

### 场景三：跨文件 bug 修复

```
> /goal 修复生产环境中用户报告的"订单状态偶尔不一致"问题。
> 约束：
> - 只改最小必要的代码
> - 不重构现有的订单状态机
> - 修复后补充一个回归测试
> - 用 git diff 验证改动范围不超过 20 行
```

然后派生子代理排查：

```
> 派两个 agent：
> Agent 1：审查 orders/ 下的所有并发操作，找出可能的竞态条件
> Agent 2：检查数据库事务的使用，找出缺少事务保护的写操作
> 汇总后给出最可能的原因和最小修复方案
```

## 8. 常见问题

### 目标太长怎么办？

写到文件里，然后在目标中引用文件路径。Codex 会读取文件内容作为目标的详细说明。

### 子代理会自动更新目标吗？

不会。目标是你在主线程设置的，子代理只能读取，不能修改或清除。

### 可以同时设多个目标吗？

不能。每个线程只能有一个活跃目标。如果你想跟踪多件事，把它们写在同一个目标文本里，或者按优先级分阶段设置——完成一个后 clear 掉，再设下一个。

### /goal pause 后子代理还有目标吗？

暂停后，主线程不再参考目标，但已经创建的子代理不受影响——它们在创建时就已经获得了目标文本。
