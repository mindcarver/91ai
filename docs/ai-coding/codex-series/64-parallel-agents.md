# 多 Agent 并行实战：/fork /side /spawn 的真实用法

> TL;DR：Codex 的多 Agent 能力不只是概念——/fork 复制会话探索不同方案、/side 开独立对话做辅助调查、spawn_agent 在 Agent 内部派生子任务。实战中，这三个机制组合使用能显著缩短复杂任务的处理时间。本文用三个真实场景（并行探索重构方案、边开发边调查、多文件同时修改）展示组合用法，以及 Agent 间通信和结果合并的技巧。

---

## 1. 三种多 Agent 机制的本质区别

在实战之前，先搞清楚这三个机制各自的定位。它们经常被混淆，但本质上是不同的工具。

### 一句话区分

- **`/fork`**：克隆当前对话到新线程，在新线程里自由探索。适合"我想试试另一条路"
- **`/side`**：在主对话旁边开一个临时对话，问完就消失。适合"我顺便查个东西"
- **`spawn_agent`**：在当前 Agent 内部派生一个子 Agent 处理子任务，完成后把摘要返回给当前 Agent。适合"这个大任务里有个子任务可以独立处理"

### 对比表

| 维度 | /fork | /side | spawn_agent |
|------|-------|-------|-------------|
| 触发方式 | 你手动输入 `/fork` | 你手动输入 `/side` 或 `/btw` | 你在指令中要求，Codex 自动派生 |
| 线程关系 | 完全独立的线程 | 临时分支，附属在主线程上 | 子线程，向主线程汇报 |
| 持久性 | 持久化，可以 `/resume` 恢复 | 临时，结束后不保存 | 随主线程持久化 |
| 上下文 | 克隆完整历史 | 克隆当前状态 | 独立上下文，只获得必要的项目信息 |
| 返回值 | 不自动返回，需要手动切线程 | 自动返回主线程 | 子 Agent 完成后返回摘要给主 Agent |
| 控制粒度 | 你控制方向和节奏 | 你问一个问题，Codex 回答 | 你描述任务，Codex 自己决定怎么拆 |
| 对主线程的影响 | 无（上下文完全隔离） | 无（临时分支不影响主线程） | 只收到摘要，不被中间输出污染 |
| 典型场景 | 方案对比、探索性重构 | 快速查资料、验证假设 | 并行分析、多维度审查 |

### 什么时候用哪个

**用 /fork 当**：
- 你有两条路想同时探索，需要多轮对话才能得出结论
- 你想做一个有风险的大改动，失败了可以扔掉整个分支
- 你需要持久化探索结果，可能第二天还要继续

**用 /side 当**：
- 你想快速问一个问题（30 秒内能回答的）
- 你想验证一个假设，不需要多轮对话
- 你在等 Codex 跑任务时顺手查个东西

**用 spawn_agent 当**：
- 任务可以拆成多个独立子任务，每个子任务不需要你的中间输入
- 你不需要实时看每个子任务的中间输出，只要最终结果
- 子任务之间不需要严格协调

## 2. 场景一：并行探索重构方案

### 问题背景

你的认证模块用 JWT，但你怀疑 Session + Cookie 方案可能更适合当前的业务场景（Web 端为主、需要主动吊销 token、用户量不大）。你不想在当前对话里直接试——万一不行还得回滚。

### 第一步：在主线程里分析现状

```
/mention src/auth/jwt.ts
/mention src/middleware/auth.ts
/mention src/routes/login.ts

分析当前认证架构的优缺点，特别关注：
1. Token 吊销的实现难度
2. 横向扩展时的问题
3. 移动端适配的成本
```

等 Codex 完成分析。

### 第二步：fork 出两条探索路径

```
/fork
```

TUI 切换到新线程。这个新线程完整复制了刚才主线程里的分析结果。

**Fork A（在新线程里）**：

```
基于刚才的分析，设计 Session + Cookie 的认证方案：
1. Session 存储用什么（Redis / 内存 / 数据库）
2. Cookie 的安全配置（httpOnly、secure、sameSite）
3. CSRF 防护策略
4. 登录、验证、登出的完整流程
5. 和现有 JWT 方案的迁移路径

先不要改代码，只出设计文档。
```

切回主线程（按 `[` 键或 `/agent` 选择主线程）。

**主线程**：

```
基于当前 JWT 架构的分析，设计改进方案：
1. 引入 Token 黑名单实现吊销
2. 优化 refresh token 的轮转策略
3. 减少 token 体积（去掉不必要的 claims）
4. 加上 token 续期的平滑过渡

先不要改代码，只出设计文档。
```

### 第三步：等待两条路径完成

两条路径并行推进。你可以用 `/agent` 在两个线程间切换，看看各自的进度。

```
/agent
# 显示：
# ● 主线程 (active) — JWT 改进方案设计
#   ○ Fork 1 — Session 方案设计
```

### 第四步：对比结果

两个方案都有设计文档后，在主线程里做对比：

```
把两个方案放在一起对比。我的评估维度是：
1. Token 吊销的实现复杂度
2. 横向扩展能力
3. 安全性（XSS、CSRF）
4. 迁移成本
5. 长期维护成本

/Fork 1 的方案摘要在你的上下文里（因为我们是从那里 fork 出来的）。
JWT 改进方案就是你刚才设计的。

用表格对比，每个维度打分（1-5），给出推荐。
```

### 第五步：决策和执行

选定方案后，在对应的线程里继续执行。如果选 JWT 改进方案，在主线程里继续。如果选 Session 方案，切到 Fork 1 继续。

```
（在选定的线程里）
按设计方案开始实施。每改完一个文件跑一次测试。
```

### 关键点

- 两条路径共享 fork 点之前的分析上下文，不需要重复解释项目背景
- 失败的路径直接放弃 fork，主线程不受影响
- 如果两个方案都不满意，可以再 fork 出第三条路径

## 3. 场景二：边开发边调查

### 问题背景

你正在开发一个通知系统。写代码的过程中发现需要确认几个技术细节，但不想打断开发流程。

### 主线程：写代码

```
/goal 实现用户通知系统，包括：
1. 数据库表设计（notifications 表）
2. 后端 API（创建通知、标记已读、获取未读列表）
3. 前端通知面板组件

先从数据库表和后端 API 开始。

/mention src/models/
/mention src/routes/
```

Codex 开始实现。

### /side：查技术细节

在 Codex 实现的过程中，你有几个问题需要确认：

```
/side 我们用的 PostgreSQL 14 支持 LISTEN/NOTIFY 吗？
如果支持，用它做实时通知推送是否合适？
和 WebSocket 相比各有什么优劣？
```

Codex 在临时分支里回答。你得到答案后，继续在主线程里决策：

```
通知推送用 WebSocket，不用 LISTEN/NOTIFY。
理由：需要支持浏览器端，LISTEN/NOTIFY 只能在服务端用。
```

过一会儿，又有一个问题：

```
/side 检查 package.json，我们项目已经有哪些 WebSocket 相关的依赖？
/mention package.json
```

```
/side notifications 表需要存已读/未读状态。
如果用户有 10000 条通知，标记全部已读的 SQL 怎么写最有效率？
是全量 UPDATE 还是加一个 is_read 索引？
```

每个 `/side` 问题几秒内得到答案，主线程的开发节奏完全不受影响。

### /fork：深入探索分支

开发到一半，你对通知的存储方案产生了疑虑。关系数据库可能不是最好的选择——用户可能积累大量通知，未读计数需要实时更新。

```
/fork
```

在 fork 里探索替代方案：

```
我在考虑把通知系统从关系数据库迁移到 Redis。
分析这个方案的可行性：
1. 用 Redis Sorted Set 存通知（score 为时间戳）
2. 未读计数用 Redis 的 hyperloglog 或者 set
3. 历史通知定期归档到 PostgreSQL

对比当前方案（纯 PostgreSQL），两种方案在以下场景的表现：
- 通知量：每用户每天 10-100 条
- 未读数查询频率：每次页面加载
- 用户量：10 万级

给出推荐，不需要写代码。
```

如果探索结论是"PostgreSQL 够用"，放弃 fork，回到主线程继续。如果结论是"Redis 更合适"，在 fork 里细化方案，然后手动把结论带回主线程。

### 主线程和调查线程的配合模式

```
主线程：持续推进开发任务
  ├── /side 问题 1 → 快速答案 → 主线程继续
  ├── /side 问题 2 → 快速答案 → 主线程继续
  ├── /fork → 深度探索（可能持续几十分钟）
  │     └── 探索完成后，结论手动带回主线程
  └── /side 问题 3 → 快速答案 → 主线程继续
```

### 关键点

- `/side` 不打断主线程的上下文——问完就消失
- `/fork` 的探索结果不会自动合并回主线程——你需要手动带回关键结论
- 用 `/goal` 保持主线程的方向感，避免被调查任务带偏

## 4. 场景三：多文件同时修改

### 问题背景

你需要在一个 monorepo 里给 8 个模块的 API 层统一加上请求参数验证。每个模块的代码结构类似，但细节不同。串行做需要改 8 次，每次都要读文件、分析接口、加验证逻辑。

### 第一步：让 Codex 分析一个模块作为模板

```
/mention packages/user-api/src/routes/index.ts

分析这个模块的 API 路由：
1. 有多少个路由
2. 每个路由接收什么参数
3. 当前有没有参数验证
4. 参数验证应该加在哪里（中间件 vs 路由处理函数内）

只分析，不改代码。
```

等 Codex 分析完后，确认分析正确。

### 第二步：定义验证规范

```
基于上面的分析，定义统一的参数验证规范：
1. 用 zod 做验证
2. 每个路由的参数 schema 定义在 routes/ 目录下的 schemas.ts 文件中
3. 验证失败返回 400，错误信息格式统一
4. 验证中间件放在路由处理函数之前

先把 user-api 模块的验证做了，作为模板。
```

等 Codex 完成 user-api 模块的修改，确认效果符合预期。

### 第三步：派生子 Agent 并行处理其他模块

```
user-api 的验证模板已经做好了。
现在对其他 7 个模块做同样的事情：
packages/order-api, packages/product-api, packages/payment-api,
packages/auth-api, packages/notification-api, packages/analytics-api,
packages/admin-api

派 7 个子 Agent，每个处理一个模块：
1. 读取该模块的路由文件
2. 按照和 user-api 相同的模式添加参数验证
3. 每个模块的 schema 放在各自的 schemas.ts 中
4. 不要修改 user-api（已经做好了）

完成后每个 Agent 报告修改了哪些文件、加了哪些验证。
```

Codex 会派生 7 个子 Agent 并行工作。每个子 Agent：

1. 读取对应模块的路由文件
2. 分析路由和参数
3. 按模板添加验证逻辑
4. 完成后返回摘要

你可以用 `/agent` 查看各个子 Agent 的进度：

```
/agent
# 显示：
# ● 主线程 (active) — 协调多模块验证
#   ○ Agent 1 — order-api 验证 (completed)
#   ○ Agent 2 — product-api 验证 (completed)
#   ○ Agent 3 — payment-api 验证 (in progress)
#   ○ Agent 4 — auth-api 验证 (in progress)
#   ○ Agent 5 — notification-api 验证 (in progress)
#   ○ Agent 6 — analytics-api 验证 (pending)
#   ○ Agent 7 — admin-api 验证 (pending)
```

### 第四步：验证结果

所有子 Agent 完成后，在主线程里做整体验证：

```
所有模块的验证都已添加。现在做整体检查：
1. 确认每个模块都有 schemas.ts 文件
2. 确认错误返回格式一致
3. 跑一遍全局的类型检查 (pnpm typecheck)
4. 跑一遍测试 (pnpm test)

报告结果。
```

### 限制和注意事项

**并发数限制**：默认 `max_threads = 6`（包括主线程），所以最多同时跑 5 个子 Agent。如果你需要更多并发，修改 `config.toml`：

```toml
[agents]
max_threads = 10
```

**文件冲突**：如果多个子 Agent 需要修改同一个文件（比如共享的 types.ts），会产生冲突。解决方法：

1. 提前规划，确保每个子 Agent 只修改自己模块的文件
2. 如果必须修改共享文件，在子 Agent 之外单独处理
3. 用 `/agent` 逐个检查子 Agent 的 diff，手动解决冲突

**模板一致性**：子 Agent 不一定能完全复制模板的做法。完成后逐个检查：

```
逐个检查每个模块的 schemas.ts，确认：
1. 导出格式和 user-api/schemas.ts 一致
2. 错误处理逻辑相同
3. 没有遗漏的路由
```

## 5. Agent 间通信机制

理解 Agent 间的信息流动方式，是组合使用这三个机制的前提。

### 上下文共享

三种机制的上下文共享方式不同：

**`/fork` 的上下文共享**：

fork 时，当前对话的**完整历史**被复制到新线程。之后两个线程完全独立——新线程里的对话不会出现在主线程里，反之亦然。

```
时间线：
  t1: 主线程 — 讨论方案 A
  t2: /fork → 新线程获得 t1 的完整历史
  t3: 主线程 — 继续推进方案 A
  t4: 新线程 — 探索方案 B
  t5: 主线程不知道新线程在 t4 说了什么
     新线程也不知道主线程在 t3 说了什么
```

**`/side` 的上下文共享**：

side 对话从当前状态创建临时分支。side 可以访问主线程的文件系统（读项目文件），但 side 里的对话不会保存到主线程。

```
时间线：
  t1: 主线程 — 正在开发
  t2: /side 问题 → 临时分支获得当前上下文
  t3: 临时分支 — Codex 回答问题
  t4: 临时分支结束，你回到主线程
  t5: 主线程 — 继续开发（side 的对话记录不会出现在主线程上下文中）
```

**`spawn_agent` 的上下文共享**：

子 Agent 获得独立的上下文，包含必要的项目信息（AGENTS.md 等），但不包含主线程的完整对话历史。子 Agent 完成后只返回摘要给主线程。

```
时间线：
  t1: 主线程 — 分析任务，决定拆分
  t2: 主线程 — spawn_agent A 和 B
  t3: Agent A — 独立工作（看不到主线程 t1 之后的对话）
  t4: Agent B — 独立工作（看不到 Agent A 的过程）
  t5: Agent A 完成 → 返回摘要给主线程
  t6: Agent B 完成 → 返回摘要给主线程
  t7: 主线程 — 收到两个摘要，做汇总
```

### 结果传递

| 机制 | 结果如何传递 | 是否自动 |
|------|------------|---------|
| `/fork` | 手动切线程查看，或 `/copy` 复制文本 | 否 |
| `/side` | 自动回到主线程，但对话不保存 | 是 |
| `spawn_agent` | 子 Agent 返回摘要到主线程 | 是 |

### 消息格式

子 Agent 返回的摘要格式取决于你给它的指令。建议在派生子 Agent 时明确要求输出格式：

```
派 3 个 agent，每个分析一个微服务的接口安全性。
完成后按以下格式返回摘要：

## <服务名> 安全分析

### 高风险
- [文件:行号] 问题描述

### 中风险
- [文件:行号] 问题描述

### 低风险
- 问题描述（不需要行号）

### 总结
一句话总结这个服务的安全状况。
```

格式化要求让摘要更容易在主线程中汇总和比较。

## 6. 结果合并策略

多个 Agent 的结果最终需要合并。根据场景不同，有三种策略。

### 手动对比

适用于 `/fork` 场景——你有两个或多个独立线程的结果，需要人工判断。

操作流程：

1. 用 `/agent` 切到 Fork 1，看方案 A 的结论
2. 用 `/copy` 复制关键结论
3. 切回主线程，粘贴 Fork 1 的结论作为参考
4. 看主线程的方案 B
5. 做对比决策

适合 fork 数量少（2-3 个）的情况。

### 自动汇总

适用于 `spawn_agent` 场景——多个子 Agent 的结果在主线程中自动汇总。

关键是在指令中要求汇总格式：

```
等所有子 Agent 完成后：
1. 按模块分组列出所有修改的文件
2. 统计总共修改了多少文件、新增多少行代码
3. 标出有潜在问题的修改（比如和模板差异较大的）
4. 给出下一步建议（需要手动检查的文件列表）
```

主线程的 Codex 会自动读取所有子 Agent 的摘要并汇总。

### 冲突处理

当多个 Agent 修改了同一个文件时，需要手动处理冲突。

检查冲突的方法：

```
检查本次所有 Agent 的修改，有没有多个 Agent 修改了同一个文件：
git diff --name-only | sort | uniq -d
```

如果有冲突：

1. 用 `git diff` 查看冲突内容
2. 手动决定保留哪个 Agent 的修改
3. 或者让 Codex 基于两个 Agent 的意图合并：

```
Agent A 和 Agent B 都修改了 src/types/api.ts。
Agent A 加了 Order 相关的类型定义，Agent B 加了 Payment 相关的类型定义。
合并两者的修改：保留 Agent A 的 Order 类型，加上 Agent B 的 Payment 类型。
不要删掉任何一个 Agent 的修改。

/mention src/types/api.ts
```

## 7. 资源管理

多 Agent 并行不是免费的。每个 Agent 有独立的上下文和模型调用，token 消耗是叠加的。

### Token 消耗控制

**并发 Agent 的 token 消耗**：

假设一个任务在单线程下需要消耗 X tokens。如果你把它拆成 5 个子 Agent 并行处理：

- 每个子 Agent 需要独立的上下文初始化：约 0.2X tokens/Agent
- 每个子 Agent 的处理：约 0.3X tokens/Agent（因为任务被拆分了）
- 主线程汇总：约 0.1X tokens
- 总消耗：5 * 0.2X + 5 * 0.3X + 0.1X = 2.6X

多 Agent 的 token 消耗大约是单线程的 **2-3 倍**。换来的是时间上的节省（5 个并行 vs 5 个串行）。

**控制策略**：

1. 只在子任务真正独立时才拆分。如果子任务之间有依赖，串行更省
2. 用便宜的模型做简单的子任务。在 config.toml 中为子 Agent 配置合适的模型
3. 限制每个子 Agent 的输出长度。在指令中明确要求"摘要不超过 500 字"

```toml
# 在自定义 Agent 角色中控制模型
[agents.quick-scanner]
description = "快速代码扫描"
config_file = "agents/quick-scanner.config.toml"

# agents/quick-scanner.config.toml
model = "gpt-5.4-mini"
model_reasoning_effort = "low"
```

### 并发数限制

```toml
[agents]
max_threads = 6  # 默认值，包括主线程
```

实际可用子 Agent 数 = `max_threads - 1`。如果任务需要更多并发：

```toml
[agents]
max_threads = 10  # 最多 9 个子 Agent 并行
```

调大 `max_threads` 前，确认你的 API 配额够用。每个子 Agent 都会独立调用 API。

### 上下文窗口管理

每个子 Agent 有独立的上下文窗口。如果子任务涉及大文件或大量代码，可能撑满上下文窗口导致输出被截断。

**应对方法**：

1. 在指令中只指定必要的文件，不要让子 Agent 扫描整个项目
2. 对于大文件，先用 `grep` 或 `awk` 提取相关部分，把提取后的内容给子 Agent
3. 明确要求子 Agent "只分析指定的文件，不要读取其他文件"

```
派一个子 Agent 分析 src/auth/ 目录的安全性。
约束：
- 只读 src/auth/ 下的文件
- 不要读取 tests/、docs/、node_modules/
- 摘要不超过 800 字
```

## 8. 常见陷阱

### 8.1 Agent 间信息不一致

两个 fork 出来的线程可能在同一个文件上做了不同的修改。因为 fork 只是对话的分支，不是文件系统的隔离。

**例子**：
- 主线程：修改了 `auth.ts` 第 10 行
- Fork 1：也修改了 `auth.ts` 第 10 行（不同的修改）
- 结果：磁盘上的 `auth.ts` 是最后那个线程的版本

**对策**：
- 避免在多个 fork 中修改同一个文件
- 如果必须修改，用 git branch 在文件层面隔离
- 在 fork 中用 `/plan` 模式先规划，确认不冲突后再执行

### 8.2 重复工作

如果子 Agent 的任务定义不够清晰，多个子 Agent 可能做了相同的工作。

**例子**：
- Agent A：分析 src/ 下的所有 API 接口
- Agent B：分析 src/ 下的所有安全风险
- 两者都读了 src/routes/ 下的文件，Agent A 分析了接口参数，Agent B 分析了接口安全性

这不是大问题（只是浪费了一些 token），但如果想避免：

**对策**：
- 明确划分每个子 Agent 的职责边界
- 指定不同的文件或目录给不同的子 Agent
- 在指令中说明"不要做其他 Agent 负责的部分"

### 8.3 上下文爆炸

每个子 Agent 有独立的上下文，如果子 Agent 之间需要共享大量信息，可能无法高效实现。

**例子**：
- Agent A 分析了 50 个文件的架构
- Agent B 需要 Agent A 的分析结果才能开始工作

这种情况下，子 Agent 并行没有意义——B 依赖 A 的结果。

**对策**：
- 只对**真正独立**的子任务使用并行
- 如果子任务之间有依赖，用串行或者让主线程协调
- 把共享信息写到一个文件里，让所有子 Agent 读这个文件

```
先让主线程分析架构，把结果写到 /tmp/architecture.md。
然后派子 Agent，每个子 Agent 先读 /tmp/architecture.md，再各自处理。
```

### 8.4 过度拆分

把一个简单的任务拆成 10 个子 Agent，每个子 Agent 只做一点点事。结果：创建子 Agent 的开销（上下文初始化、API 调用）比任务本身还大。

**对策**：
- 每个子 Agent 的任务量至少值得一次独立的上下文初始化（大约 2000-5000 tokens 的开销）
- 如果一个任务 5 分钟能完成，不值得拆
- 经验法则：预计耗时超过 15 分钟的任务才考虑并行拆分

## 9. 效率对比

### 场景一：8 个模块统一修改

| 方式 | 耗时 | Token 消耗 | 质量 |
|------|------|-----------|------|
| 串行：一个一个改 | 约 40 分钟（每个 5 分钟） | 1x | 高（可以逐个审查） |
| 并行：7 个子 Agent | 约 10 分钟（最慢的那个决定总时间） | 约 2.5x | 中（需要事后统一检查一致性） |
| 混合：先做一个模板，再并行其余 | 约 15 分钟 | 约 2x | 高（模板保证一致性） |

混合方案通常是最优的——先串行做好模板，确认质量，再并行复制到其他模块。

### 场景二：边开发边调查

| 方式 | 开发流畅度 | 信息获取速度 |
|------|-----------|------------|
| 全部串行：开发到一半停下来查资料，查完继续 | 低（频繁中断） | 慢（每次切换都要重新聚焦） |
| /side 边开发边查 | 高（不打断主线程） | 快（几秒内得到答案） |
| /fork 开独立线程查资料 | 中（需要手动切换线程） | 中（可以深度调查但需要等） |

`/side` 在快速问答场景下最合适，`/fork` 在需要深度调查的场景下更合适。

### 场景三：方案探索

| 方式 | 探索深度 | 风险 |
|------|---------|------|
| 在主线程直接试 | 只能试一个方向，失败要回滚 | 高（失败会污染上下文） |
| /fork 两条路并行 | 两个方向同时推进 | 低（失败的路径直接放弃） |
| spawn_agent 并行探索 | 可以更多方向同时探索 | 中（子 Agent 的探索深度有限） |

`/fork` 适合需要多轮对话的深度探索，`spawn_agent` 适合"快速扫描多个方向"。

## 10. 三种机制的组合模式

实战中最有效的用法不是单独使用某个机制，而是组合使用。

### 模式一：/fork + /side

主线程和 fork 并行推进的同时，用 /side 快速查资料。

```
主线程：开发功能 A
  ├── /side 问题 1 → 快速答案
  ├── /side 问题 2 → 快速答案
  └── /fork → Fork 1：探索替代方案
        ├── Fork 1 深度推进
        └── /side 问题 3 → Fork 里也可以用 /side
```

### 模式二：spawn_agent + /side

派生子 Agent 做重活的同时，用 /side 做轻量的调查。

```
主线程：协调任务
  ├── spawn_agent A → 处理模块 1
  ├── spawn_agent B → 处理模块 2
  ├── spawn_agent C → 处理模块 3
  ├── /side 问题 → 等子 Agent 时顺便查资料
  └── 子 Agent 完成后汇总
```

### 模式三：/fork + spawn_agent

在 fork 出来的分支里用 spawn_agent 做更细粒度的并行。

```
主线程：方案 A 的实现
  └── /fork → Fork 1：方案 B 的探索
        └── spawn_agent 3 个子 Agent：
            - 子 Agent 1：分析方案 B 的数据库层面
            - 子 Agent 2：分析方案 B 的 API 层面
            - 子 Agent 3：分析方案 B 的前端层面
```

这种方式适合非常复杂的决策——两个大方向各自深入分析。

### 模式四：/goal 约束下的多 Agent

用 /goal 约束所有 Agent 的方向，防止并行工作时各自跑偏。

```
/goal 选择认证方案（JWT 改进 vs Session），
确保两种方案的对比基于相同的评估维度：
安全性、扩展性、实现成本、迁移难度。

（在主线程）
按 JWT 改进方案设计，等我对比后决定。

/fork
（在 Fork 1）
按 Session 方案设计，等我对比后决定。
```

两个线程都有 /goal 的约束，确保它们的输出是可对比的。

## 11. 总结

多 Agent 并行不是万能的。它解决的是**时间效率**问题——当多个独立子任务可以同时处理时，并行能显著缩短总耗时。

三种机制各有定位：

- `/fork`：需要持久化、多轮对话的探索性工作
- `/side`：快速问答、不打断主流程的信息查询
- `spawn_agent`：结构化的子任务拆分和并行处理

组合使用的核心原则：

1. **先串行做模板，再并行复制**——避免子 Agent 的输出不一致
2. **子任务必须独立**——有依赖关系的任务不要强行并行
3. **控制 token 消耗**——用合适的模型、限制输出长度、控制并发数
4. **用 /goal 保持方向**——并行工作时特别容易跑偏
