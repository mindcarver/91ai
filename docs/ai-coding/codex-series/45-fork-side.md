# Codex CLI /fork 与 /side 分叉对话：不丢失上下文的探索方式

> TL;DR：`/fork` 把当前对话克隆到一个新线程——原始对话完全不变，你可以在新线程里自由探索，失败了也不影响主进度。`/side`（别名 `/btw`）更轻量——在主对话旁边开一个临时分支，问完就结束，主线程的上下文不受任何影响。两者都能让你在不丢失已有上下文的前提下"跑偏"探索，区别在于 `/fork` 是持久化的独立线程，`/side` 是临时的旁路对话。本文详解两个命令的机制、区别和使用场景。

---

## 1. 为什么需要分叉对话

用 Codex 做过中等规模任务的人都遇到过这种纠结：对话进行到一半，你突然想到一个可能更好的实现方案。但你不确定这个方案行不行，需要花几轮对话去探索。

问题是：如果你在当前对话里直接探索，失败的尝试会污染上下文。Codex 会记住之前尝试过的方案，在后续对话中可能被这些失败尝试误导。而且对话变长后，`/compact` 压缩时可能丢掉你真正想保留的信息。

另一种情况：对话进行中你想问一个无关的小问题（"这个函数的时间复杂度是多少？"）。如果直接在主对话里问，这个问题的上下文会插进当前的工作流，打断 Codex 的思路。

分叉对话解决的就是这两个问题：

- **`/fork`**：克隆整个对话到新线程，在新线程里自由探索，原始对话不受影响
- **`/side`**：开一个临时的旁路对话，问完就消失，主对话继续

## 2. /fork 详解

### 基本用法

在 TUI 中输入：

```
> /fork
```

Codex 会把当前对话**完整克隆**到一个新线程。新线程有自己的 ID，包含原始对话的所有历史记录。原始对话保持不变。

```
# fork 之前
● 主线程 — 你和 Codex 正在讨论认证模块的重构方案

# fork 之后
● 主线程 — 仍然在讨论认证模块（未受影响）
  ○ Fork 1 — 克隆了主线程的全部历史，可以自由探索新方案
```

### /fork 发生了什么

从技术角度看 `/fork` 做了这些事：

1. 复制当前对话的所有消息（用户输入 + Codex 回复 + 工具调用结果）
2. 创建一个新的线程，把复制的消息放入
3. 新线程获得独立的上下文窗口——它有自己的 token 预算
4. TUI 显示新线程，你可以立即开始在新线程中输入

新线程和主线程之间**没有共享状态**。你在新线程里的修改不会影响主线程，反之亦然。

### 什么时候用 /fork

- **探索替代方案**：你在主线程里有一个正在进行的方案，想同时尝试另一个方案
- **失败隔离**：你想做一个有风险的大改动，失败了可以扔掉整个分支
- **多人协作**：你和同事想从同一个对话起点出发，分别探索不同方向

### 命令行 fork

除了 TUI 中的 `/fork` 命令，你还可以在命令行中直接 fork 一个已保存的会话：

```bash
codex fork
```

这会打开一个会话选择器，让你选择要 fork 的历史会话。选完后 Codex 会启动一个新的 TUI，包含所选会话的完整历史。

这在你想要从一个之前的对话中分叉时很有用——比如你昨天做了一个技术调研，今天想从那个调研的结论出发探索两个不同的实现方向。

## 3. /side 详解

### 基本用法

```
> /side
```

或者带内联文本：

```
> /side 这个方案的并发安全性有风险吗？
> /side Check whether this plan has an obvious risk
```

`/btw` 是 `/side` 的别名，完全一样：

```
> /btw 顺便问一下，这个函数的时间复杂度是多少？
```

### /side 和 /fork 的关键区别

`/side` 比 `/fork` 更轻量，但有更多限制：

| 维度 | /fork | /side |
|------|-------|-------|
| 独立性 | 完全独立的线程 | 临时分支，附属在主线程上 |
| 上下文 | 克隆完整历史 | 克隆当前对话状态 |
| 持久性 | 可以保存、恢复、继续 | 临时的，结束后不保存 |
| 嵌套 | fork 出来的线程可以继续对话 | 不能在 side 里再开 side |
| TUI 状态 | 完全切换到新线程 | 主线程状态继续显示 |
| 审查模式 | 可用 | 不可用 |
| 返回方式 | `/agent` 切换回主线程 | 自动返回主线程 |

### /side 的行为

1. Codex 从当前对话创建一个临时分支
2. 你的内联文本（如果有）作为第一条消息发送到分支
3. Codex 在分支中处理你的问题
4. 处理完成后，分支关闭，你回到主线程
5. 分支的对话不会保存到历史中

在 side 对话进行中，TUI 仍然显示主线程的状态（比如主线程是否还在运行某个任务）。这样你可以一边问 side 问题，一边不丢失主线程的进度感知。

### /side 的限制

- **不能嵌套**：在 side 对话中不能再开 `/side`
- **不能在审查模式中使用**：如果你正在用 `/review`，`/side` 不可用
- **不持久化**：side 对话结束后不会出现在历史列表中，无法 `/resume`

## 4. 三种分叉方式对比

Codex 提供了三种"偏离主对话"的方式：

| 方式 | 命令 | 生命周期 | 上下文 | 适用场景 |
|------|------|---------|--------|---------|
| 新对话 | `/new` | 独立、空白 | 全新的上下文 | 完全不同的任务 |
| 克隆 | `/fork` | 独立、持久 | 克隆全部历史 | 从当前点探索替代方案 |
| 旁路 | `/side` | 临时、一次性 | 克隆当前状态 | 快速问答、小探索 |

### 选择指南

**用 `/new` 当**：你想开始一个完全不同的任务，不需要之前的上下文

**用 `/fork` 当**：你想从当前对话的基础上探索不同的方向，可能需要多轮对话，想保留探索结果

**用 `/side` 当**：你想快速问一个问题或做一个小的探索，不需要保留结果，不想离开主对话

## 5. 实战工作流

### 场景一：方案对比

你在做一个认证模块的重构，正在讨论方案 A（JWT + Redis）。突然想到方案 B（Session + Cookie）可能更合适。

```
# 主线程：正在实施方案 A
> /fork
# Fork 1：探索方案 B
> 如果改用 Session + Cookie 做认证，和当前 JWT 方案相比，在以下方面有什么差异：
> 1. 横向扩展性
> 2. CSRF 防护
> 3. 移动端适配
> 4. 实现复杂度
```

主线程继续推进方案 A，Fork 1 评估方案 B。两个线程可以并行工作，互不干扰。最后你对比两个线程的结论，决定用哪个方案。

### 场景二：快速验证假设

在写一个复杂的状态管理逻辑时，你想确认一个假设：

```
# 主对话：正在实现状态管理
> /side 当两个组件同时触发同一个 action，Redux 的 reducer 是按顺序执行还是并发执行？
```

Codex 在临时分支中回答你的问题。你得到答案后，继续在主对话中实现状态管理，不需要切换线程。

### 场景三：团队分工

你和一个同事在同一个项目上用 Codex 工作。你做前端，他做后端。

```
# 你从一个共同的技术讨论开始
> /fork
# 你的线程：实现前端组件
# 同事的线程：实现后端 API
```

两个人从同一个技术讨论出发，各自在自己的 fork 中工作。代码修改不会冲突（因为 fork 只是对话的分支，不是代码的分支——代码修改在工作目录中是共享的）。

### 场景四：实验性重构

你想尝试一个大范围的重构，但不确定值不值得：

```
# 主对话：当前代码能正常工作
> /fork
# Fork：尝试重构——把所有 class 组件改成函数式组件
# 如果重构成功，merge 回主对话
# 如果重构失败，放弃这个 fork，主对话不受影响
```

这比直接在主对话中尝试更安全——如果重构方案行不通，你不会浪费主对话的上下文。

## 6. 分叉与 Git 的关系

分叉是对话层面的概念，不是代码层面的。理解两者的区别很重要：

| 维度 | 对话分叉 | Git 分支 |
|------|---------|---------|
| 作用域 | 对话历史和上下文 | 代码文件 |
| 独立性 | 完全独立的上下文 | 独立的文件版本 |
| 合并 | 不支持自动合并 | 支持合并 |
| 冲突 | 不会冲突（对话独立） | 可能产生文件冲突 |

如果你在 Fork 1 中修改了 `auth.ts`，然后切回主线程也修改了 `auth.ts`，后者的修改会覆盖前者的修改——因为代码文件在工作目录中是唯一的，分叉不隔离文件系统。

要避免这个问题：
- **用 git stash 或 git branch** 保护 Fork 中的修改，再切回主线程
- **避免在多个 fork 中同时修改同一个文件**
- **用 `/plan` 模式先规划**，确认分叉方向后再做实际修改

## 7. 常见问题

### fork 后能合并回主线程吗？

不能自动合并。对话分叉是单向的——你可以手动把 Fork 中的结论复制到主对话（用 `/copy`），但没有"合并对话"的功能。

### side 对话的输出能保存吗？

不能。`/side` 是临时的，结束后输出不保存。如果你需要保留 side 的结论，在 side 结束前用 `/copy` 复制输出。

### 能 fork 一个历史会话吗？

能。用命令行的 `codex fork` 命令，会打开历史会话选择器。

### side 中能执行文件修改吗？

能。`/side` 中的 Codex 有完整的工具权限（取决于你的审批模式）。但修改的文件会直接反映在工作目录中——没有隔离。
