# Skill 评测：欠触发、误触发和执行失败怎么修

**TL;DR：** Skill 需要评测。至少记录三类问题：该触发时没触发、不该触发时触发、触发后执行质量差。

## 问题

很多团队写完 Skill 就开始用，但没有评测样例。结果是 Skill 有时不加载，有时乱加载，有时加载后仍然输出不稳定。

## 三类失败

| 失败类型 | 表现 | 修复方向 |
| --- | --- | --- |
| 欠触发 | 用户明确需要它，但 Claude 没用 | description 增加真实触发词 |
| 误触发 | 无关任务也加载 | 增加 Do Not Use When |
| 执行失败 | 触发了但结果差 | 改步骤、加示例、加验证 |

## 最小评测集

准备 15 条请求：

- 5 条应该触发。
- 5 条不应该触发。
- 5 条边界模糊。

每次修改 Skill 后跑一遍，记录变化。

## 落地练习

给一个已有 Skill 建一个 `eval.md`：

```md
| 请求 | 期望 | 实际 | 结论 |
| --- | --- | --- | --- |
| Review this diff | 触发 | 触发 | pass |
| Implement this feature | 不触发 | 触发 | fail |
```

## 权衡

Skill 评测不需要一开始自动化。先用人工表格能更快发现描述问题。等 Skill 稳定后，再考虑脚本化回归。
