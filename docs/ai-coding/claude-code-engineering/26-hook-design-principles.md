# Hook 设计原则：小、确定、可解释、可回滚

**TL;DR：** Hook 是治理层，不是第二个 Agent。它应该小、确定、可解释、可回滚。

## 问题

很多 Hook 系统会越写越复杂：自动判断任务类型、自动选择测试、自动修改输出、自动提交。最后出了问题没人知道是模型错还是 Hook 错。

## 四条原则

### 小

一个 Hook 只做一件事。

### 确定

同样输入应得到同样输出。

### 可解释

阻断时说明哪条规则触发。

### 可回滚

每个 Hook 都能单独禁用，不影响主流程。

## 推荐分层

| 层级 | 示例 |
| --- | --- |
| 记录 | 保存工具调用 |
| 提醒 | 修改后提示测试 |
| 阻断 | 禁止改 `.env` |
| 审批 | 高风险命令要求人工确认 |

## 落地练习

审查团队已有 Hook，给每个 Hook 标注：

- 触发事件。
- 输入。
- 输出。
- 是否阻断。
- 如何禁用。

## 权衡

Hook 太少，治理不足；Hook 太多，系统脆弱。优先治理高风险、低误判的动作。
