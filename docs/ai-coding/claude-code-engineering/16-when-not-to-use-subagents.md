# 不该用 Subagent 的场景：共享细节和连续编辑

**TL;DR：** Subagent 不适合所有复杂任务。需要频繁共享细节、连续编辑同一批文件、上下文高度耦合的任务，留在主会话更稳。

## 问题

看到 Subagent 后，很容易把所有任务都拆出去。结果是多个代理重复读文件、给出冲突建议，主会话还要花时间整合。

## 不适合的场景

- 一个小 bug，需要连续读写同一文件。
- UI 微调，需要不断看截图和改样式。
- 需求还没澄清，代理不知道目标。
- 多个模块强耦合，拆开会丢上下文。
- 用户需要实时参与决策。

## 判断问题

使用 Subagent 前问三句：

1. 这个任务能独立完成吗？
2. 它只需要返回摘要，而不是持续协作吗？
3. 它的工具权限能被明确限制吗？

如果答案不是三个“是”，先不要拆。

## 落地练习

找一个小 bug，分别让主会话和 subagent 处理。比较总耗时、上下文重复、最终 diff。大多数小任务会证明：不拆更快。

## 权衡

不用 Subagent 不代表不工程化。工程化的目标是降低风险和认知负担，不是增加代理数量。
