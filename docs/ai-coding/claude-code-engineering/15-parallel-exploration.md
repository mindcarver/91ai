# 并行探索：让多个代理独立研究再汇总

**TL;DR：** 并行探索适合大任务的前期判断。让多个 subagent 分别研究不同方向，再由主会话整合方案。

## 问题

大型改造开始前，最耗时的是理解：哪些模块相关、现有模式是什么、风险在哪里、测试怎么跑。一个会话串行探索会慢，也容易遗漏。

## 使用场景

- 多服务调用链分析。
- 迁移方案调研。
- 大型 bug 根因排查。
- 多个实现方案对比。
- 安全和性能风险并行审查。

## 推荐流程

```txt
主会话定义问题
  -> Explorer A 查业务路径
  -> Explorer B 查数据模型
  -> Reviewer C 查风险
  -> Test D 查验证方式
主会话汇总方案
```

## 落地练习

对一个“新增权限字段”的需求，让 3 个 subagent 并行研究：

- 数据库影响。
- API 影响。
- 前端影响。

最后要求主会话输出实施计划和风险清单。

## 权衡

并行探索会增加 token 和协调成本。只有任务可以拆成相互独立的问题时才值得用。
