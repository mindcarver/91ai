# 组织级治理：版本、审计、评测、禁用和升级策略

**TL;DR：** 组织级 Claude Code 落地不是“全员装工具”，而是管理规则、权限、评测、插件版本和事故回滚。

## 问题

个人使用 Claude Code，只要自己承担风险。组织使用时，风险变成共享的：错误代码进入主干、敏感数据泄露、CI token 滥用、插件供应链出问题。

## 治理对象

- `CLAUDE.md` 和 rules。
- Skills 和 Plugins。
- MCP servers。
- Subagents。
- Hooks。
- GitHub Actions。
- Agent SDK 服务。

## 治理清单

| 维度 | 问题 |
| --- | --- |
| 版本 | 谁能升级插件和 Skill |
| 审计 | 工具调用和文件修改是否留痕 |
| 权限 | 哪些工具可读，哪些可写 |
| 评测 | 更新后是否跑回归任务 |
| 禁用 | 出问题能否快速关闭 |
| 培训 | 团队是否理解边界 |

## 落地练习

建立一份组织级 AI Coding register：

```md
| 能力 | owner | 权限 | 仓库 | 版本 | 风险 | 回滚方式 |
| --- | --- | --- | --- | --- | --- | --- |
```

每个 MCP、Plugin、Skill、Hook 都登记。

## 权衡

治理会降低短期速度，但能保护长期信任。团队一旦经历过 AI 自动化事故，再补治理成本更高。
