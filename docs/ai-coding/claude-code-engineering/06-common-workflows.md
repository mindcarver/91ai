# 常用工作流：解释代码、修 Bug、补测试、写文档

**TL;DR：** Claude Code 最适合从高频低风险任务开始：解释代码、定位 bug、补测试、更新文档。这些任务能快速验证项目上下文和测试闭环。

## 问题

团队导入 Claude Code 时，常犯的错误是直接让它做大需求。更稳的路线是先把日常开发的四个任务跑顺。

## 四类基础工作流

| 工作流 | 目标 | 验证方式 |
| --- | --- | --- |
| 解释代码 | 快速 onboarding | 人工确认结构是否准确 |
| 修 Bug | 小范围行为修复 | 最小相关测试 |
| 补测试 | 提高回归保护 | 测试能失败再通过 |
| 写文档 | 同步使用说明 | Markdown 链接和示例命令检查 |

## 推荐提示

```text
先解释相关模块和调用链，不要修改文件。
确认你理解后，提出最小修改方案。
修改后运行最小相关测试。
最后总结 diff、验证结果和剩余风险。
```

## 落地练习

找一个最近提交过的 bug 修复，让 Claude Code 重新实现一次。比较它的 diff 和人类 diff：

- 是否改到同一层。
- 是否多改无关文件。
- 是否补了测试。
- 是否解释了失败路径。

## 权衡

基础工作流看起来不炫，但它能暴露最关键的问题：项目规则是否有效、测试是否可运行、Claude Code 是否会越界修改。
