# MCP 风险：Token、越权、工具投毒和提示注入

**TL;DR：** MCP 把 Claude Code 接到真实系统，也把真实风险带进来了。最小权限、只读优先、工具审计和人工确认是底线。

## 问题

MCP server 可能访问代码、issue、数据库、文档、监控和内部 API。只要权限设计不当，AI 的一次误操作就可能影响真实资产。

## 主要风险

- token 权限过大。
- 外部 PR 触发高权限 workflow。
- MCP 工具描述被污染。
- 工具返回内容包含 prompt injection。
- 数据库或云服务写操作缺少人工确认。
- 大输出污染上下文。

## 防护清单

| 风险 | 防护 |
| --- | --- |
| token 泄露 | 使用 secret manager，不写入仓库 |
| 权限过大 | 只读优先，按工具拆 token |
| 工具投毒 | 审计 MCP server 来源和代码 |
| prompt injection | 不信任工具返回的指令性文本 |
| 误写操作 | PreToolUse Hook + 人工确认 |

## 落地练习

列出当前所有 MCP server：

- 来源。
- 拥有哪些权限。
- 是否可写。
- 是否访问敏感数据。
- 谁负责维护。

## 权衡

过度限制会降低效率，但默认开放会放大事故半径。正确做法是从只读开始，用真实需求证明写权限必要。
