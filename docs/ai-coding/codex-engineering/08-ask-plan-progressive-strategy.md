# Ask 与 Plan 模式：从探索到执行的渐进策略

**TL;DR：** Codex 提供三种核心执行模式——Ask（只调研不动手）、Plan（先规划再执行）和 Execute（直接动手改代码）。大多数效率问题不是因为模型能力不足，而是模式选择错误。在 Ask 阶段省下的时间如果不该省，会在 Execute 阶段以 3-5 倍的成本偿还。在 Plan 阶段多花的一轮对话，能在 Execute 阶段省下 5-10 轮返工。正确的策略是 Ask -> Plan -> Execute 的渐进式推进：先建立理解，再规划路径，最后受控执行。本文给出三种模式的机制、适用场景、成本分析和决策表，以及最常见的五个错误模式。

## 三种执行模式的本质区别

Codex CLI 提供三种执行模式，对应三种截然不同的工作方式。很多开发者把三种模式当成"同一个功能的不同参数"，但它们的本质差异在于**模型被赋予的行动权限**和**上下文的使用方式**。

### Ask 模式：只读调研

Ask 模式下的 Codex 是一个纯粹的代码阅读者和分析师。它可以读取文件、搜索代码、执行只读命令（如 `git log`、`grep`），但不能修改任何文件、不能执行写操作。

```bash
# Ask 模式的基本用法
codex --ask "分析 src/payment/ 目录的模块结构和依赖关系"

# Ask 模式下模型只能做以下操作
# - 读取文件内容
# - 搜索代码模式
# - 执行只读 shell 命令（git status, git log, grep, find 等）
# - 输出分析结果

# Ask 模式下模型不能做以下操作
# - 修改任何文件
# - 运行写操作命令（git commit, npm install 等）
# - 执行测试（测试可能有副作用，如写入数据库）
```

Ask 模式的核心价值不是"安全"（虽然安全是附带的好处），而是**强制你和模型先建立理解再动手**。没有 Ask 模式时，很多开发者习惯直接让模型修改代码。模型在不完全理解代码库的情况下动手，产出质量不可预测。Ask 模式把"理解"从可选步骤变成了必须步骤。

Ask 模式的另一个隐性价值是**低成本试错**。一次 Ask 调用的 token 消耗通常是 Execute 调用的 1/3 到 1/5，因为它不需要生成代码、运行测试、处理错误。你可以在 Ask 模式下多次追问、调整方向、补充上下文，直到确认理解正确，再进入下一阶段。这些"便宜的"迭代如果推迟到 Execute 阶段，每次方向调整都伴随着代码修改和测试运行，成本会成倍增长。

### Plan 模式：结构化规划

Plan 模式下的 Codex 会输出一份结构化的执行计划，但不会执行任何修改操作。你审查计划后决定是否让 Codex 按计划执行。

```bash
# Plan 模式的基本用法
codex --plan "重构 payment/ 模块的错误处理，统一使用 Result 类型"

# Plan 模式的输出格式示例
# ========================================
# 执行计划：重构 payment/ 模块错误处理
# ========================================
#
# 第一阶段：调研（预计 3k-5k token）
# 1. 并行读取以下文件了解现有错误处理模式
#    - payment/service.ts
#    - payment/handlers.ts
#    - payment/types.ts
#    - utils/errors.ts
#
# 第二阶段：设计（预计 5k-8k token）
# 2. 设计 Result 类型和工具函数
#    - 定义 Result<T, E> 类型
#    - 实现 ok() 和 err() 工厂函数
#    - 设计迁移策略（渐进式 vs 一次性）
#
# 第三阶段：迁移（预计 10k-15k token）
# 3. 逐模块迁移（按依赖顺序）
#    a. payment/types.ts（定义新类型）
#    b. payment/repository.ts（底层先改）
#    c. payment/service.ts（中间层）
#    d. payment/handlers.ts（最上层）
#    每完成一个文件运行 pnpm test -- payment/ 确认无回归
#
# 第四阶段：验证（预计 3k-5k token）
# 4. 运行完整测试套件
# 5. 搜索旧模式残留引用
# 6. 输出迁移总结
#
# 预计总 token：21k-33k
# 预计总时间：15-30 分钟
# 风险等级：中（涉及错误处理，影响面广）
# ========================================
```

Plan 模式的核心价值是**让错误暴露在规划阶段而不是执行阶段**。一次 Plan 的成本是一次 API 调用加上你的几分钟审查时间。但如果跳过 Plan 直接执行，模型可能在第 8 步发现第 3 步的方案有问题，然后回退修改，重新执行后续步骤。回退和重做的成本远高于一次 Plan。

Plan 模式还解决了一个更隐蔽的问题：**范围蔓延**（scope creep）。直接执行时，模型在执行过程中可能"发现"需要额外修改的文件，然后自行扩展任务范围。这种范围蔓延在单次对话中很难被察觉，直到你看到 git diff 才发现改了 15 个文件而你的原始需求只涉及 3 个。Plan 模式把所有计划的改动暴露在执行之前，你可以在审查时砍掉不需要的部分。

### Execute 模式：直接执行

Execute 模式是 Codex 的默认模式。模型拥有完整的文件修改和命令执行权限（受沙箱和审批策略约束）。

```bash
# Execute 模式（默认）
codex "修复 auth/login.ts 中 handleTokenExpiry 的缓存清除问题"

# Execute 模式下模型可以做所有操作
# - 读取和修改文件
# - 运行任何 shell 命令
# - 执行测试、构建、部署脚本
# - 创建新文件、删除文件
```

Execute 模式不是"更高级"的模式，而是"承担更多风险"的模式。它的适用条件是：任务目标明确、实现路径清晰、错误成本可控。当这三个条件同时满足时，跳过 Ask 和 Plan 直接执行是合理的。当任何一个条件不满足时，直接执行就是在赌博。

## 为什么需要渐进策略

理解三种模式后，下一个问题是：什么时候用哪种模式？能不能对所有任务都用同一种模式？

答案是渐进策略。渐进策略的核心思想是：**在信息不足时选择低成本模式，在信息充足时切换到高效率模式**。这不是保守——而是让每种模式在正确的时机发挥正确的价值。

### 理解成本和修改成本的不对称性

软件工程中存在一个根本性的不对称：理解代码的成本远低于修改代码的成本，但修改代码出错的成本远高于理解代码出错的成本。

```text
理解阶段：
  成本：1x（读取文件、分析结构、识别模式）
  出错成本：1x（分析结论错误，修正结论即可）

修改阶段：
  成本：3-5x（修改代码、运行测试、处理错误、验证结果）
  出错成本：5-10x（定位错误、回退修改、重新实现、重新验证）

不对称的后果：
  如果在理解阶段犯错 → 在修改阶段被发现 → 总成本 = 1x（理解）+ 3-5x（修改）+ 5-10x（返工）= 9-16x
  如果在理解阶段就做对 → 修改阶段一次成功 → 总成本 = 1x（理解）+ 3-5x（修改）= 4-6x

  差异：在理解阶段多花 1x 时间，可以节省 5-10x 的返工成本
```

这就是为什么 Ask 模式不是"浪费时间"，而是在用 1x 的成本消除 5-10x 的风险。Plan 模式同样如此——一份计划可能花 5 分钟审查，但它避免的返工可能需要 30-60 分钟。

### 信息充足度模型

渐进策略的决策基础是信息充足度。在任务开始时，你对以下问题的答案通常是不确定的：

| 维度 | 需要回答的问题 | 信息不足时的后果 |
|------|--------------|----------------|
| 目标 | 任务的具体完成标准是什么？ | 模型按照自己的理解定义"完成"，可能与你的期望不同 |
| 范围 | 涉及哪些文件和模块？ | 模型可能漏改文件或改了不该改的文件 |
| 依赖 | 修改会影响哪些其他模块？ | 改动引发连锁反应，破坏其他功能 |
| 模式 | 现有代码的编码风格和约定是什么？ | 新代码与现有代码风格不一致 |
| 约束 | 有哪些不能触碰的边界？ | 模型可能修改公共 API、引入新依赖、破坏兼容性 |

信息充足度决定你应该用哪种模式：

```text
信息充足度评估：

低（对 3 个以上维度不确定）
  → 必须用 Ask 模式先调研
  → 调研后可能还需要 Plan 模式规划

中（对 1-2 个维度不确定）
  → Plan 模式规划执行路径
  → 或者 Ask 模式快速补充信息后直接执行

高（对所有维度都确定）
  → 直接 Execute
  → 但仍需在 prompt 中写明验证步骤
```

### 渐进策略的推进节奏

渐进策略不是教条，而是一个可以根据实际情况灵活调整的框架。核心原则是**每一步都在增加信息充足度，且每一步的成本都在可控范围内**。

```text
完整的渐进流程：

Step 1: Ask（建立理解）
  输入：初步的任务描述
  输出：代码结构分析、依赖关系、潜在风险
  成本：3k-15k token，5-20 分钟
  决策点：信息是否足够进入 Plan？

Step 2: Plan（规划路径）
  输入：Ask 阶段的分析结果 + 明确的目标
  输出：结构化的执行计划（步骤、文件、顺序、验证点）
  成本：5k-20k token，5-15 分钟（含审查时间）
  决策点：计划是否合理？是否需要调整？

Step 3: Execute（受控执行）
  输入：经过审查的执行计划
  输出：代码修改 + 测试结果
  成本：10k-50k token，10-60 分钟
  决策点：执行结果是否满足完成条件？

任何一步发现问题：
  → 回退到上一步（而不是在当前步骤内强行推进）
```

回退是这个策略中最关键的设计。如果 Plan 阶段发现任务比预期复杂，应该回到 Ask 阶段补充调研，而不是硬着头皮规划一个不确定的方案。如果 Execute 阶段发现结果不对，应该回到 Plan 阶段重新规划，而不是在执行过程中反复修补。每次回退看似浪费了时间，实际上避免了更大的浪费。

## Ask 模式深度解析

### 适用场景

Ask 模式适合四种典型场景：

**场景一：理解陌生代码。** 接手新项目、跨团队协作、review 不熟悉的模块。目标是建立正确的心理模型，而不是修改代码。

```text
"分析 src/core/engine.ts 模块的结构。输出格式：

1. 模块职责：一句话描述这个模块做什么
2. 公共接口：列出所有 export 的函数/类，每个用一句话说明
3. 依赖关系：这个模块依赖哪些内部模块，被哪些模块依赖
4. 关键数据流：描述主要的数据处理链路
5. 潜在风险：你认为需要特别注意的代码段和原因

不要修改任何文件。"
```

**场景二：评估影响范围。** 在做改动之前，先搞清楚改动会影响哪些模块。这对重构和 API 变更尤其重要。

```text
"分析将 UserService.getUserById 的返回类型从 User | null 改为 Result<User, NotFoundError>
会影响哪些模块。

步骤：
1. 搜索所有调用 getUserById 的位置
2. 分析每个调用点当前如何处理 null 返回值
3. 列出需要修改的文件和每个文件的预期改动
4. 评估是否有循环依赖或间接引用被遗漏

不要修改任何文件。"
```

**场景三：调查问题根因。** 已知 bug 的表面现象，但不确定根因在哪。Ask 模式让模型专注调查，而不是急于修复。

```text
"调查以下 bug 的根因：

症状：用户反馈'偶尔'出现订单金额为 0 的情况
错误日志：[贴出相关日志]

调查方向：
1. 分析 order/createOrder 函数的完整执行路径
2. 检查金额计算涉及的并发访问（多个请求同时修改同一订单？）
3. 检查金额字段的类型定义（是否有精度丢失的可能？）
4. 检查数据库层面的约束（金额字段是否允许为 0？）

输出：列出所有可能的根因，按可能性排序，每个根因给出证据支持。
不要修改任何文件。"
```

**场景四：探索备选方案。** 存在多种技术路径，需要评估利弊后再决定方向。

```text
"评估在当前项目中引入缓存层的三个方案：

方案 A：Redis（独立缓存服务）
方案 B：Node.js 进程内缓存（如 node-cache）
方案 C：SQLite 本地文件缓存

评估维度：
1. 性能：读写延迟、吞吐量、内存占用
2. 一致性：缓存失效策略、多实例同步
3. 运维复杂度：额外依赖、监控、故障恢复
4. 代码改动量：需要修改哪些模块

先读取以下文件了解当前架构：
- src/services/data.ts（数据访问层）
- src/config.ts（配置）
- docker-compose.yml（基础设施）
- package.json（当前依赖）

输出格式：对比表格 + 推荐方案 + 理由。
不要修改任何文件。"
```

### Ask 模式的输出质量控制

Ask 模式的输出没有测试可以验证正确性，需要通过其他方式控制质量。

**指定输出格式。** 不指定格式时，模型倾向于输出流畅但松散的散文。指定格式后，模型被迫按结构组织信息，遗漏和不一致更容易被发现。

```text
差的 Ask prompt：
"帮我看看 payment 模块"

好的 Ask prompt：
"分析 payment/ 模块。输出格式：
1. 模块职责（一段话）
2. 公共接口列表（表格：函数名 | 参数 | 返回值 | 一句话说明）
3. 依赖关系图（文本格式，标注文件路径）
4. 风险清单（表格：风险 | 严重程度 | 位置 | 说明）"
```

**要求标注不确定性。** 模型在不确定时倾向于给出看起来确定的回答。强制标注不确定性可以避免把推测当成事实。

```text
"分析完成后，对以下内容标注置信度：
- 确定正确的标记 [HIGH]
- 有一定依据但未验证的标记 [MED]
- 基于推测的标记 [LOW]
不要对 [LOW] 的内容给出确定的结论。"
```

**分阶段追问。** 一次 Ask 不够深入时，不要重新发起任务，而是在同一会话中追问。追问利用了已有的上下文，比重新开始更高效。

```text
# 第一次 Ask：建立骨架
codex --ask "分析 src/payment/ 的模块结构，列出所有导出函数和依赖关系"

# 第二次 Ask：深入细节（在同一会话中）
"基于刚才的分析，详细解释 PaymentService.processRefund 的完整执行流程。
重点关注：金额校验逻辑、并发处理、错误恢复机制"

# 第三次 Ask：聚焦风险
"processRefund 中有哪些潜在的并发问题？具体到代码行级别说明。"
```

## Plan 模式深度解析

### 适用场景

Plan 模式适合三种典型场景：

**场景一：多步骤复杂任务。** 任务涉及 3 个以上的文件，或者需要按特定顺序执行多个步骤。Plan 模式强制模型先想清楚"怎么做"再动手。

```text
codex --plan "在用户系统中添加'两步验证'功能。

需要修改的模块：
1. auth/ - 验证流程变更
2. user/ - 用户设置存储
3. api/ - 新增 API 端点
4. notification/ - 验证码发送
5. tests/ - 测试覆盖

约束：
- 不修改现有的登录流程，两步验证是可选功能
- 验证码使用 TOTP 标准（RFC 6238）
- 需要兼容 Google Authenticator 等 app"
```

**场景二：高风险改动。** 涉及数据库 schema、公共 API、安全相关代码。Plan 模式让你在执行前审查方案的完整性和正确性。

```text
codex --plan "将用户表从单租户架构迁移到多租户架构。

风险等级：极高
影响范围：所有涉及用户数据的查询和写入

需要评估：
1. 数据库 schema 变更方案（新建 tenant_id 列 vs 新建 tenant 表）
2. 数据迁移方案（在线迁移 vs 停机迁移）
3. API 兼容性（现有接口是否需要加 tenant 参数）
4. 回滚方案（如果迁移出问题怎么恢复）"
```

**场景三：不确定实现路径。** 任务目标明确但存在多种实现路径，需要模型先分析选项再选择路径。

```text
codex --plan "优化 src/search/ 的搜索性能。

当前症状：10 万条文档时搜索响应 > 2 秒
优化选项：
A. 数据库索引优化（改动小，效果有限）
B. 引入 Elasticsearch（改动大，效果好）
C. 内存倒排索引（中等改动，效果显著）

在计划中包含：
1. 每种方案的改动范围和预期效果
2. 推荐的方案和理由
3. 分阶段实施计划（如果方案较大）"
```

### Plan 模式的审查要点

Plan 模式输出的计划需要你审查。审查的目的是确认计划合理，而不是逐字推敲。重点检查以下五个方面：

| 审查维度 | 检查什么 | 危险信号 |
|---------|---------|---------|
| 完整性 | 计划是否覆盖了所有需要修改的文件 | 计划中只列出了你提到的文件，没有模型自己发现的额外文件 |
| 顺序 | 步骤的执行顺序是否合理 | 先改上层模块再改底层模块（应该反过来） |
| 验证 | 每个步骤是否有对应的验证点 | 只有最后一步有测试，中间步骤没有验证 |
| 回退 | 出错时如何恢复 | 计划中没有回退策略 |
| 范围 | 改动范围是否与任务匹配 | 计划包含了你没有要求的额外优化或重构 |

一个高质量的 Plan 应该满足以下标准：

```text
好计划的特征：
- 每个步骤有明确的输入和输出
- 文件级别的修改列表（不只说"修改 payment 模块"，而是"修改 payment/service.ts 的 processRefund 函数"）
- 步骤间有验证点（不是全部改完才测）
- 依赖顺序正确（底层先于上层）
- 有预计的 token 消耗和时间

坏计划的特征：
- 只有笼统的阶段描述（"第一阶段：调研"），没有具体步骤
- 没有文件级别的修改列表
- 所有验证集中在最后
- 没有考虑依赖顺序
- 没有 token 和时间估算
```

### Plan 输出后的三种处理方式

审查计划后，你有三种处理方式：

**方式一：批准执行。** 计划合理，让 Codex 按计划执行。

```text
"按计划执行。每完成一个阶段后报告进度，不要跳过任何验证步骤。"
```

**方式二：调整后执行。** 计划整体可行但需要局部调整。

```text
"按计划执行，但做以下调整：
- 第二步和第三步合并为一个步骤（它们的改动是相关的）
- 在第三步后增加一个验证点：运行 pnpm test -- integration/
- 跳过第五步（那是额外的重构，不在本次任务范围内）"
```

**方式三：否决并重新规划。** 计划有根本性问题。

```text
"这个计划的执行顺序有问题——先改 handlers 再改 service 会导致中间状态不一致。
正确的顺序应该是：
1. 先在 types.ts 中定义新类型
2. 改 service.ts（底层）
3. 改 handlers.ts（上层）
4. 每步之后运行测试
请重新规划。"
```

## Ask -> Plan -> Execute 阶梯策略

### 阶梯一：Ask 阶段——建立理解

Ask 阶段的目标是回答"这是什么"和"改了会怎样"。它的产出是一份对代码库的结构化理解，包含模块关系、调用链、现有模式和潜在风险。

```text
Ask 阶段的标准流程：

1. 明确调研目标
   - "我需要理解什么才能安全地完成这个任务？"
   - 列出需要回答的关键问题

2. 并行读取关键文件
   - 在一条 prompt 中列出所有需要读取的文件
   - 模型并行读取，建立完整的上下文理解

3. 结构化输出
   - 指定输出格式（表格、列表、调用关系图）
   - 要求标注不确定性

4. 追问补充
   - 对不够深入的部分追问细节
   - 对不确定的部分要求验证

5. 输出决策依据
   - "基于以上分析，这个任务涉及 X 个文件，Y 个依赖，Z 个风险点"
   - 为 Plan 阶段提供输入
```

Ask 阶段的典型 token 消耗和耗时：

| 调研规模 | 文件数量 | Token 消耗 | 耗时 |
|---------|---------|-----------|------|
| 小（单模块理解） | 2-3 个 | 3k-8k | 5-10 分钟 |
| 中（跨模块分析） | 4-7 个 | 8k-20k | 10-20 分钟 |
| 大（全局影响评估） | 8-15 个 | 15k-40k | 20-40 分钟 |

### 阶梯二：Plan 阶段——规划路径

Plan 阶段的目标是回答"怎么做"和"按什么顺序做"。它的输入是 Ask 阶段的分析结果，产出是一份结构化的执行计划。

```text
Plan 阶段的标准流程：

1. 基于 Ask 结果明确执行目标
   - 将 Ask 阶段的理解转化为具体的执行步骤
   - 确定每个步骤的输入、操作和输出

2. 设计文件级修改列表
   - 每个需要修改的文件
   - 每个文件的预期改动内容
   - 文件间的修改依赖关系

3. 排列执行顺序
   - 底层模块优先于上层模块
   - 独立模块可以并行处理
   - 每个步骤后有验证点

4. 评估风险和回退策略
   - 哪些步骤可能出错
   - 出错时如何回退
   - 是否需要 feature flag 保护

5. 输出结构化计划
   - 提交给用户审查
   - 包含 token 估算和时间估算
```

Plan 阶段的成本通常是整个任务成本的 10-20%，但它能避免 30-50% 的返工。这是一个高杠杆的投资。

### 阶梯三：Execute 阶段——受控执行

Execute 阶段的目标是按照经过审查的计划，安全地完成代码修改。关键词是"受控"——不是盲目执行，而是每一步都有验证和检查点。

```text
Execute 阶段的受控策略：

1. 逐步执行，不要一口气跑完
   - 按计划的步骤顺序执行
   - 每完成一个步骤后确认结果
   - 不要跳过任何验证点

2. 持续验证
   - 每个文件修改后运行相关测试
   - 每个阶段完成后运行更广范围的测试
   - 全部完成后运行完整测试套件

3. 异常处理
   - 测试失败时立即停止
   - 分析失败原因（是计划的问题还是执行的问题）
   - 如果是计划的问题，回退到 Plan 阶段
   - 如果是执行的问题，在当前步骤内修复

4. 范围守卫
   - 定期检查 git diff --name-only
   - 确认改动文件列表与计划一致
   - 发现计划外的改动立即调查
```

### 快速通道：什么时候可以跳过阶段

渐进策略不是每次都必须走完三步。以下情况可以跳过部分阶段：

**跳过 Ask，直接 Plan 的条件：**
- 你对相关代码已经非常熟悉（比如你自己写的模块）
- 任务的目标和范围都很明确
- 改动的影响可以精确预估

```text
可以跳过 Ask 的例子：
"重构 src/utils/format.ts 中的日期格式化函数，
把 moment.js 替换为 date-fns。
你已经在之前几次改动中熟悉了这个文件。"
```

**跳过 Plan，直接 Execute 的条件：**
- 任务只涉及 1-2 个文件
- 改动逻辑简单直接（不超过 20 行）
- 有充分的测试覆盖
- 错误成本很低（改错了很容易回退）

```text
可以跳过 Plan 的例子：
"修复 src/utils/validator.ts 中 isEmail 函数对
超长字符串的正则回溯问题。修完后运行测试。"
```

**跳过 Ask 和 Plan，直接 Execute 的条件：**
- 所有条件同时满足：目标明确、范围可控、实现路径清晰、错误成本极低

```text
可以跳过一切的例子：
"在 package.json 中添加 engines 字段，
指定 node >= 18.0.0"
```

## 决策表和成本分析

### 模式选择决策表

根据任务特征和你的信息充足度，选择合适的执行模式组合：

| 任务特征 | 信息充足度 | 推荐路径 | 典型场景 |
|---------|-----------|---------|---------|
| 目标明确、范围可控、单文件 | 高 | 直接 Execute | 配置修改、简单 bug 修复 |
| 目标明确、范围可控、多文件 | 高 | Plan -> Execute | 添加新功能（遵循现有模式） |
| 目标明确、范围不确定 | 中 | Ask -> Plan -> Execute | 重构、迁移、API 变更 |
| 目标模糊、需要探索 | 低 | Ask -> (决策) | 技术选型、架构评估 |
| 复杂 bug、根因不明 | 低 | Ask -> Plan -> Execute | 生产环境问题排查 |
| 高风险改动（数据库、安全） | 任意 | Ask -> Plan -> Review -> Execute | 数据库迁移、认证变更 |

### 成本对比分析

不同路径的成本差异显著。以下是三类典型任务在不同路径下的成本对比：

**简单任务：修复一个已知的小 bug（单文件，改动 < 10 行）**

| 路径 | Token 消耗 | 时间 | 返工概率 |
|------|-----------|------|---------|
| 直接 Execute | 5k-10k | 5 分钟 | 10-20% |
| Plan -> Execute | 8k-15k | 10 分钟 | 5-10% |
| Ask -> Plan -> Execute | 12k-25k | 20 分钟 | < 5% |

结论：简单任务直接 Execute 是合理的。多出来的返工概率对应的返工成本（5-10 分钟）可以接受。

**中等任务：添加一个新功能（3-5 个文件，改动 50-200 行）**

| 路径 | Token 消耗 | 时间 | 返工概率 |
|------|-----------|------|---------|
| 直接 Execute | 15k-30k | 15-30 分钟 | 30-50% |
| Plan -> Execute | 20k-35k | 25-40 分钟 | 10-20% |
| Ask -> Plan -> Execute | 25k-45k | 35-55 分钟 | < 10% |

结论：中等任务至少需要 Plan。直接 Execute 的 30-50% 返工概率意味着你有 1/3 到 1/2 的概率需要返工 15-30 分钟，期望返工成本 5-15 分钟。Plan 阶段多花的 10 分钟是值得的。

**复杂任务：重构一个模块（5+ 个文件，改动 200+ 行）**

| 路径 | Token 消耗 | 时间 | 返工概率 |
|------|-----------|------|---------|
| 直接 Execute | 30k-80k | 30-90 分钟 | 50-70% |
| Plan -> Execute | 35k-60k | 40-70 分钟 | 20-30% |
| Ask -> Plan -> Execute | 40k-70k | 50-80 分钟 | < 15% |

结论：复杂任务必须走 Ask -> Plan -> Execute 全流程。直接 Execute 的返工概率超过 50%，且每次返工的成本是 30-60 分钟。跳过 Ask 和 Plan 节省的 10-20 分钟，很可能被返工的 30-60 分钟（甚至更多）完全抵消。

### Token 成本优化策略

如果 token 预算是硬约束，可以采用以下优化策略：

**策略一：Ask 阶段精简输出。** Ask 模式不要求模型给出完整分析报告，而是只回答关键问题。

```text
差的 Ask prompt（token 浪费）：
"全面分析 src/payment/ 模块的所有方面"

好的 Ask prompt（精准输出）：
"回答以下两个问题，不需要其他分析：
1. processRefund 函数被哪些其他函数调用？列出调用者
2. 修改 processRefund 的返回类型会影响多少个文件？"
```

**策略二：Plan 阶段只规划关键步骤。** 对于部分步骤明确的任务，只让模型规划不确定的部分。

```text
"我只需要你规划'如何安全地迁移 processRefund 的返回类型'。
其他步骤我已经知道怎么做了，不需要规划。"
```

**策略三：合并 Ask 和 Plan 为一步。** 当任务不是特别复杂时，可以在一条 prompt 中同时要求分析和规划。

```text
codex --plan "分析 src/payment/service.ts 中 processRefund 函数的错误处理模式，
然后规划如何将其从 throw 异常改为返回 Result 类型。
在计划中包含：1) 现有错误处理的分析 2) 迁移步骤 3) 验证方式"
```

## 审批模式与执行模式的组合

Codex 的执行模式（Ask / Plan / Execute）和审批模式（Suggest / Auto-edit / Full-auto）是两个正交的维度。它们的组合决定了 Codex 的行为边界。

### 组合矩阵

| 执行模式 \ 审批模式 | Suggest | Auto-edit | Full-auto |
|-------------------|---------|-----------|-----------|
| Ask | 只读分析 + 不执行任何命令 | 只读分析 + 不执行任何命令 | 只读分析 + 不执行任何命令 |
| Plan | 规划 + 不执行 | 规划 + 不执行 | 规划 + 不执行 |
| Execute | 展示修改建议 + 不执行 | 自动修改文件 + 审批命令 | 自动修改文件 + 自动执行命令 |

Ask 和 Plan 模式下，审批模式的选择影响不大，因为这两种模式本身就不执行修改。但在 Execute 模式下，审批模式的选择至关重要：

- **Suggest**：适合高风险改动的审查阶段。模型只展示建议的修改，你自己决定是否应用。
- **Auto-edit**：默认推荐模式。模型可以自动修改文件，但执行命令（如运行测试、安装依赖）需要你审批。
- **Full-auto**：适合低风险、高重复性的任务。模型自动完成所有操作。

### 推荐组合策略

```text
推荐组合策略：

1. 探索阶段（信息不足）：
   Ask + Auto-edit（实际上 Ask 模式下不会修改文件）

2. 规划阶段（方案设计）：
   Plan + Auto-edit（Plan 模式下也不会修改文件）

3. 执行阶段——低风险任务：
   Execute + Full-auto
   适用于：单文件修改、有测试覆盖、改动可回退

4. 执行阶段——中风险任务：
   Execute + Auto-edit
   适用于：多文件修改、需要运行命令验证

5. 执行阶段——高风险任务：
   Execute + Suggest
   适用于：数据库迁移、安全相关、生产配置
```

## 常见错误模式

### 错误一：用 Execute 模式做 Ask 的事

这是最常见的错误。开发者想要"理解代码"或"评估方案"，但用的是 Execute 模式而不是 Ask 模式。

```text
错误做法：
codex "看看 auth/ 模块有没有安全问题"

后果：
1. 模型读了代码，分析安全问题
2. 模型"顺便"修复了它认为有问题的地方
3. 你得到的是一堆未经审查的代码修改，而不是安全分析报告
4. 这些修改可能引入新问题，或者不符合团队的编码规范

正确做法：
codex --ask "分析 auth/ 模块的安全问题，列出每个问题的严重程度和修复建议。
不要修改任何文件。"
```

这个错误的根因是把"看"这个动词当成了 Ask 的信号，但实际上模型在 Execute 模式下的默认行为是"看完就动手"。解决方法很简单：如果任务不需要修改文件，就用 Ask 模式。

### 错误二：跳过 Plan 直接执行复杂任务

当任务涉及多个文件时，跳过 Plan 直接 Execute 的返工率显著增加。

```text
错误做法：
codex "把 src/api/ 目录下的所有 Express handler 迁移到 Fastify"

后果：
1. 模型从第一个文件开始改
2. 改到第三个文件时发现前两个文件的迁移方式不一致
3. 回去修改前两个文件
4. 改到第五个文件时发现需要改 types.ts 中的类型定义
5. 但 types.ts 被另一个模块依赖，改了之后其他模块也坏了
6. 雪球越滚越大，最终放弃了这次改动

正确做法：
codex --plan "把 src/api/ 目录下的所有 Express handler 迁移到 Fastify。
在计划中包含：文件依赖分析、迁移顺序、每个文件的预期改动、验证策略。"

审查计划后：
codex "按审查过的计划执行迁移。每完成一个文件后运行测试确认无回归。"
```

这个错误的根因是低估了多文件改动的协调复杂度。Plan 模式的价值就在于它让模型先分析依赖关系再定顺序，而不是边做边发现依赖。

### 错误三：在 Ask 阶段过早进入细节

Ask 阶段的正确节奏是先建立骨架认知再深入细节。过早追问细节会导致模型在上下文不完整的情况下给出不准确的分析。

```text
错误做法（第一次 Ask 就追问细节）：
codex --ask "分析 processRefund 函数第 42 行的 if 判断是否正确"

后果：
模型只看了 processRefund 函数的局部代码，没有理解它在整个调用链中的位置，
给出了一个看似合理但忽略上下文的分析。

正确做法（先骨架后细节）：
第一次 Ask：
codex --ask "分析 payment/ 模块的整体结构和核心流程。输出模块职责、公共接口、依赖关系"

第二次 Ask（在同一会话中追问）：
"基于刚才的分析，详细解释 processRefund 的执行流程，重点关注金额校验和并发处理"

第三次 Ask（进一步深入）：
"processRefund 第 42 行的 if 判断，结合整个调用链来看是否正确？
是否有调用者已经做了相同的检查导致冗余？"
```

### 错误四：Plan 输出后不做审查直接执行

Plan 模式的价值在于"执行前审查"。如果跳过审查环节，Plan 模式就退化成了一次更贵的 Execute。

```text
错误做法：
codex --plan "重构 payment/ 模块"
# 看到计划输出后直接说：
"执行吧"

后果：
计划中可能包含你没注意到的额外改动（"顺便优化了错误处理"），
或者执行顺序有问题（先改上层后改底层），
这些在审查时很容易发现，但执行后发现就需要返工。

正确做法：
1. 审查计划中的文件列表——是否有遗漏或多出的文件？
2. 审查执行顺序——底层是否先于上层？
3. 审查验证策略——每个步骤后是否有测试？
4. 审查范围——是否有计划外的"顺便"改动？
5. 确认后再说"按计划执行"
```

### 错误五：在同一个 prompt 中混合 Ask 和 Execute

这是效率最低的用法。一条指令同时要求分析和修改，模型的行为不可预测。

```text
错误做法：
codex "看看 auth/login.ts 有没有问题，有的话帮忙修一下"

后果（取决于模型的判断）：
可能性 A：模型只分析了问题，没有修改代码。
  → 你还需要再发一条指令让它修改，多了一轮对话。
可能性 B：模型分析后直接修改了代码。
  → 修改未经你审查，可能不符合你的预期。
可能性 C：模型花大量 token 在分析上，然后草率地给了一个修复。
  → 分析和修复的质量都不如分开做。

正确做法：
第一步（Ask）：
codex --ask "分析 auth/login.ts 的潜在问题，列出每个问题的严重程度和修复建议"

第二步（根据分析结果决定是否修复）：
codex "修复刚才发现的第一个问题。修完后运行 pnpm test -- auth/ 确认。"
```

混合指令的核心问题是：模型不知道你的优先级是什么。是先保证分析的全面性，还是先给出一个能用的修复？这两个目标对 token 的分配方式是矛盾的。分开执行让模型在每个阶段专注于一个目标。

## 实践检查清单

把渐进策略落地到日常工作中，可以在每次任务开始时用以下检查清单快速决策：

```text
任务开始前：

1. 任务是否需要修改文件？
   否 → Ask 模式
   是 → 继续

2. 我是否完全理解相关代码？
   否 → Ask 模式先调研 → 回到这里
   是 → 继续

3. 任务涉及几个文件？
   1 个，改动 < 20 行 → 直接 Execute
   2-3 个 → Plan -> Execute
   4+ 个 → Ask -> Plan -> Execute

4. 改动的风险等级？
   低（有测试、可回退）→ Auto-edit 或 Full-auto
   中（多文件、无完整测试）→ Plan + Auto-edit
   高（数据库/安全/生产）→ Plan + Suggest

5. 任务完成后：
   - 运行完整测试套件确认
   - 检查 git diff --name-only 确认范围
   - 如果结果不理想，回退到上一个阶段重新规划
```

这套检查清单的执行时间不超过 30 秒，但可以避免 80% 的模式选择错误。渐进策略的核心不是死板地走完三步，而是在正确的时机用正确的模式，用最低的成本获取最高的确定性。
