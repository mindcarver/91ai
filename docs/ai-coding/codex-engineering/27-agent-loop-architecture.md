# Agent Loop 架构：执行流、Prompt 构造和 Responses API

**TL;DR：** Codex 的核心运行机制是一个循环：构造 Prompt → 调用 Responses API → 执行工具调用 → 把结果追加回上下文 → 再次调用 API，直到模型输出最终文本回复。理解这个循环内部发生了什么——Prompt 如何分层构造、API 请求长什么样、工具调用怎么执行、上下文怎么压缩——是从"会用 Codex"到"能调优 Codex"的关键。本文拆解 Agent Loop 的完整执行流，覆盖 Prompt 六层构造、Responses API 端点与 SSE 流式机制、工具调用循环、上下文压缩策略、Prompt 缓存原理，以及 GPT-5.4 的上下文参数规格。

---

## Agent Loop 总览：一次 API 调用就是一轮循环

Codex 的运行模型可以用一句话概括：**ReAct 风格的单代理循环**。ReAct（Reasoning + Acting）指的是模型在每一轮循环中先"思考"（决定下一步做什么），再"行动"（调用工具执行），然后把行动结果反馈回来，进入下一轮"思考"。

每一轮循环的本质是一次 API 调用。模型的推理发生在 OpenAI 的服务器上，Codex CLI 负责在本地组织请求、执行工具、管理状态。这个过程可以用以下流程图表示：

```text
用户输入 "修复 login.ts 中的 token 过期 bug"
        |
        v
+--------------------------------------------------+
|              Agent Loop 循环体                      |
|                                                    |
|  1. Prompt 构造                                    |
|     System Prompt                                  |
|     + AGENTS.md 内容                               |
|     + Skill 指令                                   |
|     + 对话历史                                     |
|     + 用户输入                                     |
|     + 上轮工具结果                                  |
|                                                    |
|  2. 调用 Responses API                             |
|     POST /v1/responses                             |
|     → SSE 流式返回                                 |
|                                                    |
|  3. 解析响应                                       |
|     → 纯文本？输出给用户，循环结束                    |
|     → 工具调用？执行工具，结果追加到上下文              |
|                                                    |
|  4. 返回第 1 步                                     |
+--------------------------------------------------+
        |
        v
  最终文本输出：任务完成
```

这个循环有几个关键特征：

**每次迭代是一次完整的 API 调用。** 不是一次 HTTP 连接复用，不是 WebSocket 长连接。每一轮都是独立的 POST 请求，携带完整的对话上下文。这意味着每一轮的输入 Token 量等于系统提示词 + AGENTS.md + Skill + 所有历史对话 + 所有历史工具结果的 Token 总和。

**循环终止条件是模型输出纯文本。** 当模型认为任务完成、不需要再调用任何工具时，它输出一条纯文本消息。Codex CLI 收到这条消息后，结束循环，把文本展示给用户。如果模型输出的是工具调用（tool_call），CLI 执行工具后继续循环。

**上下文只增不减（除非触发压缩）。** 每一轮的工具调用结果、模型的中间推理文本，全部追加到对话历史中。这意味着一个 10 轮工具调用的任务，第 10 轮的输入 Token 量远大于第 1 轮。当累积上下文超过阈值时，压缩机制介入——后面会详细讲。

## Prompt 构造：六层叠加的洋葱模型

每一轮 API 调用的输入是一个消息数组（messages）。这个数组的构造不是简单的"用户说什么就发什么"，而是六层内容按固定顺序叠加的结果。理解这六层结构，才能理解为什么 AGENTS.md 的内容会影响每次调用的 Token 消耗，为什么 Skill 激活会增加成本，以及为什么长对话会越来越贵。

```text
Messages 数组结构（按顺序）：

[1] System Prompt          ← Codex 的基础行为指令（~2000 Token）
[2] AGENTS.md 内容          ← 项目特定上下文（数百到数千 Token）
[3] Skill 指令              ← 激活的 Skill 附加指令（~500-2000 Token/个）
[4] 对话历史 - 用户消息      ← 历史轮次的用户输入
[5] 对话历史 - 模型回复      ← 历史轮次的模型输出
[6] 对话历史 - 工具调用      ← 历史轮次的工具调用请求与结果
...
[N] 当前用户输入             ← 本轮用户的任务描述
```

### 第一层：System Prompt

System Prompt 是 Codex CLI 内置的基础指令，定义了模型的角色、能力和行为约束。这部分内容在每次 API 调用中都完全相同，因此是 Prompt 缓存的主要命中对象。

System Prompt 包含以下内容：

- **角色定义**：你是一个编码代理，任务是帮助用户完成软件工程任务
- **工具使用说明**：每个内置工具的调用格式、参数说明、使用时机
- **安全约束**：不能执行什么操作、什么时候需要用户确认
- **输出格式**：如何组织代码修改、如何报告执行结果
- **行为准则**：行动优先、验证后报告、遇到错误时的处理策略

System Prompt 的 Token 量大约在 2000-3000 Token 之间，取决于 Codex 版本和启用的工具数量。这部分是固定的，用户无法修改。

### 第二层：AGENTS.md 内容

AGENTS.md 是用户在项目根目录放置的项目上下文文件。Codex 读取这个文件的内容，将其作为系统级上下文注入到 Prompt 中。这个机制在前面第 3 篇和第 4 篇有详细讨论，这里从 Prompt 构造的角度再补充几个要点：

**注入位置在 System Prompt 之后。** 这意味着模型先读取 Codex 的基础行为指令，再读取项目特定上下文。AGENTS.md 中的指令如果与 System Prompt 冲突，后者的优先级更高。

**每次调用都完整注入。** 不管 AGENTS.md 有多长，它的完整内容在每次 API 调用中都会发送。一个 2000 Token 的 AGENTS.md，在 10 轮循环中累计消耗 20000 Token 的输入配额（实际因为 Prompt 缓存，消耗会低得多——后面讲）。

**多层继承的合并。** 如果项目存在多层 AGENTS.md（全局、组织级、项目级、子目录级），Codex 会按优先级合并所有层级的内容，作为一个整体注入。

### 第三层：Skill 指令

Skill 是 Codex 的扩展能力模块。当用户通过斜杠命令或自动触发激活一个 Skill 时，Skill 的指令内容会被注入到 Prompt 中。这部分内容的特征：

- 每个 Skill 的指令长度在 500-2000 Token 之间
- 多个 Skill 可以同时激活，指令叠加
- Skill 指令在激活后的每一轮循环中持续存在
- Skill 指令通常包含特定任务的执行模板、输出格式要求、判断标准

例如，当用户触发 `/gsd-review` 这个 Skill 时，Prompt 中会注入一套代码审查的详细模板和评判标准。这套模板在后续每一轮循环中都占据 Token 配额，直到会话结束或 Skill 被显式停用。

### 第四到六层：对话历史和当前输入

对话历史是 Token 消耗增长最快的部分。每一轮循环产生的模型输出和工具调用结果，全部追加到对话历史中。这部分的增长速度取决于：

- **工具调用的输出长度**：shell_command 的输出可以轻易达到数千 Token
- **模型的中间推理**：模型在思考过程中的输出也计入历史
- **轮次数量**：复杂任务可能需要 10-20 轮工具调用

当前用户输入是最后一层。在第一轮循环中，这就是用户输入的任务描述。在后续轮次中，这一层为空——因为用户的输入已经作为历史记录在第四层中了。

### Token 预算分配实例

以一个中等复杂度的任务为例，假设 AGENTS.md 有 800 Token，激活了一个 Skill（600 Token），任务执行了 8 轮循环：

```text
轮次 1（首轮）：
  System Prompt:   ~2500 Token
  AGENTS.md:        ~800 Token
  Skill:            ~600 Token
  用户输入:          ~200 Token
  ─────────────────────────
  合计输入:         ~4100 Token

轮次 5（中间轮）：
  System Prompt:   ~2500 Token
  AGENTS.md:        ~800 Token
  Skill:            ~600 Token
  历史（4轮累积）:  ~12000 Token
  ─────────────────────────
  合计输入:         ~15900 Token

轮次 8（末轮）：
  System Prompt:   ~2500 Token
  AGENTS.md:        ~800 Token
  Skill:            ~600 Token
  历史（7轮累积）:  ~28000 Token
  ─────────────────────────
  合计输入:         ~31900 Token
```

可以看到，对话历史的增长是非线性的。如果工具执行产生大量输出（比如一次 shell_command 返回了几千行的测试日志），Token 消耗的增长会更加陡峭。

## Responses API：不是 Chat Completions

Codex 使用的后端 API 不是 Chat Completions API（`/v1/chat/completions`），而是 Responses API（`/v1/responses`）。这是一个关键区别，影响端点选择、请求格式、流式传输方式和数据留存策略。

### 两个端点，两种认证

Codex CLI 支持两种认证方式，对应不同的 API 端点：

| 认证方式 | API 端点 | 认证凭证 | 适用场景 |
|----------|---------|---------|---------|
| ChatGPT 登录 | `chatgpt.com/backend-api/codex/responses` | ChatGPT 会话 Token（自动获取） | 个人用户、ChatGPT Plus/Pro 订阅者 |
| API Key | `api.openai.com/v1/responses` | OPENAI_API_KEY 环境变量 | 团队使用、API 计费、自动化集成 |
| 开源模型 | `localhost:11434/v1/responses` | 不需要认证（`--oss` 标志） | 本地部署、隐私敏感场景 |

ChatGPT 登录模式下，Codex CLI 通过浏览器 OAuth 流程获取会话 Token，后续请求自动携带。用户不需要手动配置 API Key。这种方式的计费走 ChatGPT 订阅额度，不是按 Token 计费。

API Key 模式走标准的 OpenAI API 计费。Token 消耗按输入/输出分别计费，价格取决于模型版本。这种方式适合团队使用和自动化集成，因为 API Key 可以通过环境变量或配置文件管理，不依赖浏览器交互。

`--oss` 标志允许连接本地运行的开源模型。Codex CLI 把请求格式化为兼容 OpenAI Responses API 的格式发送到本地端点。这意味着你可以在完全离线、数据不出本地的环境下使用 Codex 的工具执行能力，配合本地模型（如通过 Ollama 运行的模型）。

### SSE 流式传输

Responses API 使用 Server-Sent Events（SSE）进行流式传输。这意味着 API 响应不是等全部生成完毕后一次性返回，而是以事件流的形式逐步推送。

```text
SSE 事件流示例：

event: response.created
data: {"id": "resp_abc123", "status": "in_progress"}

event: response.output_item.added
data: {"type": "message", "role": "assistant", "content": []}

event: response.content_part.delta
data: {"type": "output_text", "text": "我来"}

event: response.content_part.delta
data: {"type": "output_text", "text": "分析一下"}

event: response.content_part.delta
data: {"type": "output_text", "text": "这个 bug..."}

event: response.output_item.done
data: {"type": "message", "content": [{"type": "text", "text": "我来分析一下这个 bug..."}]}

event: response.completed
data: {"id": "resp_abc123", "status": "completed", "usage": {...}}
```

SSE 流式传输带来三个好处：

**实时输出。** 用户在模型生成过程中就能看到中间文本，不需要等整个响应完成。这对于长时间推理的任务尤其重要——用户可以即时看到模型在做什么，而不是盯着空白屏幕等几十秒。

**流式工具调用。** 当模型决定调用工具时，工具调用参数也通过 SSE 流式推送。Codex CLI 可以在参数生成完毕后立即执行工具，不需要等整个响应完成。这在模型同时调用多个工具时尤其有价值。

**提前中断。** 如果用户发现模型走偏了，可以在流式输出过程中按 Escape 中断当前循环。已经生成和执行的部分不会回滚，但后续的循环会被取消。

### 没有 previous_response_id：ZDR 兼容

Responses API 设计上支持 `previous_response_id` 参数——一种让服务端复用上一轮响应上下文的机制，可以减少重复传输。但 Codex CLI 不使用这个参数。

这是一个有意的设计决策，目的是支持 Zero Data Retention（ZDR）模式。在 ZDR 模式下，OpenAI 服务端不会存储任何请求和响应数据。如果使用 `previous_response_id`，服务端需要保留上一轮的完整响应来支持复用，这与 ZDR 的零留存原则直接冲突。

因此 Codex CLI 的实现方式是：**每轮请求携带完整的对话上下文**。不依赖服务端的状态保持，所有状态管理都在客户端（CLI 本地）完成。这种方式的代价是每轮请求的输入 Token 量更大（因为要重新传输所有历史），但换来了对 ZDR 模式的完全兼容。

对于使用 API Key 计费的用户，这意味着没有服务端缓存带来的成本节省。但 Codex 通过 Prompt 缓存机制在客户端实现了类似效果——后面详细讲。

## 工具调用循环：从模型决策到本地执行

工具调用是 Agent Loop 的核心驱动力。模型通过工具调用与外部世界交互——读写文件、执行命令、搜索代码。理解工具调用的完整生命周期，是理解 Codex 执行效率和成本结构的基础。

### 单次工具调用流程

```text
1. 模型生成 tool_call
   {
     "type": "function_call",
     "name": "shell_command",
     "arguments": "{\"command\": \"npm test -- auth.test.ts\"}"
   }

2. Codex CLI 接收 tool_call

3. 审批检查
   ├─ auto-approve 模式：直接执行
   ├─ suggest 模式：提示用户确认
   └─ legacy 模式：逐条审批

4. 沙箱执行
   └─ 在隔离环境中执行 shell_command

5. 输出截断（如果需要）
   └─ 超过 ~10k Token 的输出只保留头尾

6. 结果格式化为 tool_result
   {
     "type": "function_call_output",
     "call_id": "call_xyz",
     "output": "Test Suites: 2 passed, 2 total\nTests: 8 passed, 8 total"
   }

7. tool_result 追加到对话历史

8. 进入下一轮循环（回到 Prompt 构造）
```

这个过程的关键节点是第 3 步的审批检查和第 4 步的沙箱执行。审批模式决定了工具调用是否需要用户确认，沙箱决定了工具在什么环境中执行。这两个机制在前面第 2 篇和第 16 篇有详细讨论。

### 并行工具调用

Codex 支持 `parallel_tool_calls`——模型可以在一轮循环中同时发起多个工具调用。这些工具调用在 CLI 端并行执行，结果一起追加到对话历史中。

典型的并行调用场景：

```text
模型在一轮中同时发起：

tool_call 1: read_file("src/auth/login.ts")
tool_call 2: read_file("src/auth/login.test.ts")
tool_call 3: search_files("clearCache", "src/auth/")

三个调用并行执行，结果同时返回
→ 模型一次性获得三个信息源，基于完整上下文做出下一步决策
```

并行调用对 Token 消耗没有直接影响——每个工具调用的 Token 量与串行执行相同。但它对执行效率有显著影响：三个并行调用与三个串行调用相比，总耗时从 3T 降低到接近 T。对于需要读取多个文件、执行多个搜索的任务，并行调用可以把执行时间缩短数倍。

并行调用的限制在于：工具之间不能有依赖关系。如果工具 B 的参数依赖工具 A 的执行结果，它们就不能并行——模型必须把 A 放在一轮、B 放在下一轮。模型被训练为自动判断哪些工具可以并行、哪些必须串行，但这个判断不是百分百准确。当模型错误地把有依赖的工具并行执行时，可能导致基于不完整信息的决策。

### 工具调用的 Token 成本

工具调用的 Token 成本分三部分：

| 成本项 | 计入位置 | 说明 |
|--------|---------|------|
| 工具调用请求 | 输入 Token | 模型生成的 tool_call JSON（含函数名和参数） |
| 工具执行结果 | 下一轮输入 Token | 工具输出被追加到对话历史 |
| 历史累积 | 每轮输入 Token | 所有历史工具调用和结果在每轮重新发送 |

第三项是工具调用最隐性的成本。一个在轮次 3 执行的 shell_command 产生了 5000 Token 的输出，这笔输出在轮次 4、5、6、7... 的每一轮中都会作为历史被重新发送，每次都计入输入 Token。如果一个任务执行了 15 轮，那轮次 3 的 5000 Token 输出实际消耗了 5000 x 13 = 65000 Token 的输入配额。

这就是为什么 shell_command 的输出截断机制如此重要。如果不截断，一个返回 20000 Token 日志的 shell_command，在 10 轮循环中会消耗 200000 Token 的输入配额。截断到 10000 Token 后，消耗降低到 100000 Token——虽然仍然可观，但至少是可控的。

## 上下文压缩：当历史太长时的应急措施

每一轮循环都在累积上下文。当累积的上下文接近模型的上下文窗口上限时，Codex 会触发压缩（compaction）机制。

### 压缩的触发条件

压缩的触发基于一个简单的判断：**当前输入 Token 量是否接近上下文窗口的安全阈值**。这个阈值通常设置在上下文窗口的 70%-80% 左右。对于 GPT-5.4 的默认 272K 上下文，压缩阈值大约在 190K-215K Token。

为什么不在上下文窗口 100% 满时才压缩？因为需要为模型的输出预留空间。如果输入占满了 272K，模型就没有空间生成回复了。预留 20%-30% 的窗口给输出和推理链，是确保模型能正常工作的安全裕度。

### 压缩的工作方式

压缩的核心操作是：**用一条摘要替换多轮历史对话**。具体流程：

```text
压缩前：

  System Prompt (2500 Token)
  AGENTS.md (800 Token)
  Skill (600 Token)
  ────────────────────────────
  用户消息 1: "修复 login.ts 的 token bug" (200 Token)
  模型回复 1: "让我先读取文件..." (150 Token)
  工具调用 1: read_file → 3000 Token
  模型回复 2: "找到问题了，在 handleTokenExpiry..." (200 Token)
  工具调用 2: apply_patch → 80 Token
  工具调用 3: shell_command → 5000 Token
  模型回复 3: "修复完成，运行测试..." (100 Token)
  工具调用 4: shell_command → 4000 Token
  用户消息 2: "测试还有个失败..." (100 Token)
  模型回复 4: "让我检查..." (100 Token)
  工具调用 5: read_file → 2000 Token
  ... (更多轮次)
  ────────────────────────────
  总历史: ~35000 Token

压缩后：

  System Prompt (2500 Token)
  AGENTS.md (800 Token)
  Skill (600 Token)
  ────────────────────────────
  压缩摘要: "用户要求修复 login.ts 的 token 过期缓存未清除 bug。
             模型读取了文件，定位到 handleTokenExpiry 函数缺少
             clearCache 调用。已通过 apply_patch 添加了调用。
             首次运行测试有 1 个失败（test case: 正常登录不应清缓存）。
             模型正在检查失败原因。" (~300 Token)
  ────────────────────────────
  总历史: ~300 Token
```

压缩摘要由模型自身生成。Codex 在检测到上下文接近阈值时，会构造一个特殊的压缩请求，把需要压缩的历史对话发送给模型，要求模型生成一条包含关键信息的摘要。这条摘要替换原来的多轮对话，释放出大量 Token 空间。

### 压缩的信息损失

压缩是有损的。摘要保留了任务的主要脉络（目标是什么、做了什么、当前进展），但丢弃了大量细节（具体的工具调用参数、完整的输出内容、中间推理过程）。这种信息损失可能导致：

- 模型忘记了早期执行的具体代码修改细节
- 模型无法精确引用之前读取的文件内容
- 模型对之前的失败尝试只有模糊记忆，可能重复尝试已被证实无效的方案

在实践中，压缩对任务完成质量的影响取决于任务的复杂度和后续需要的上下文深度。如果压缩发生在任务的收尾阶段（只剩最后的验证步骤），影响很小。如果压缩发生在任务的中间阶段（还需要参考早期读取的大量代码），影响可能显著。

### 减少压缩需求的策略

既然压缩有信息损失，最好的策略是避免触发压缩。几个实用做法：

**精简 AGENTS.md。** 只包含项目真正需要的上下文，避免冗长的模板和注释。每节省 100 Token 的固定 Prompt 开销，在整个会话生命周期中都会被乘以轮次数量。

**控制工具输出长度。** 在 AGENTS.md 中指导模型使用精准的搜索命令（如 `grep -n "pattern" file` 而非 `cat file`），减少不必要的输出。

**拆分长任务。** 一个需要 20 轮工具调用的任务，拆成两个 10 轮的任务分别执行。每个子任务的上下文独立，不会累积到需要压缩的程度。

**及时清理无用历史。** 某些工具调用的结果（如一次性的项目结构探索）在后续轮次中不再需要。虽然 Codex 目前没有提供手动删除特定历史轮次的机制，但通过拆分任务可以有效规避这个问题。

## Prompt 缓存：前缀匹配的隐性优化

前面提到，Codex 不使用 `previous_response_id` 来减少重复传输。那有没有其他机制来缓解每轮重新发送完整上下文的成本？答案是 Prompt 缓存（Prompt Caching）。

### 缓存原理

OpenAI 的 Prompt 缓存机制基于前缀匹配。当多个 API 请求的输入消息数组具有相同的前缀时，服务端可以复用已处理的前缀部分，只对差异部分重新计算。这个缓存是服务端自动完成的，不需要客户端做任何配置。

```text
轮次 1 的 Messages:

  [System Prompt] [AGENTS.md] [Skill] [用户输入]
   └─── 前缀 A ───┘

轮次 2 的 Messages:

  [System Prompt] [AGENTS.md] [Skill] [历史1] [工具结果1]
   └─── 前缀 A ───┘  └── 新增 B ──┘

轮次 3 的 Messages:

  [System Prompt] [AGENTS.md] [Skill] [历史1] [工具结果1] [历史2] [工具结果2]
   └─── 前缀 A ───┘  └────── 新增 C ──────┘
```

在轮次 2 中，前缀 A（System Prompt + AGENTS.md + Skill）与轮次 1 完全相同，服务端可以直接复用已缓存的计算结果。只有新增部分 B 需要重新计算。

在轮次 3 中，前缀 A + B 与轮次 2 的完整输入相同，服务端可以复用。只有新增部分 C 需要重新计算。

### 缓存命中条件

Prompt 缓存的命中需要满足以下条件：

| 条件 | 说明 |
|------|------|
| 前缀完全匹配 | 从第一条消息开始，逐 Token 匹配。任何差异都会导致缓存失效 |
| 缓存未过期 | 缓存通常在 5-10 分钟后过期。如果两轮间隔过长，缓存可能已被清除 |
| 缓存槽位可用 | 服务端的缓存容量有限。高负载时可能没有可用槽位 |

前缀完全匹配这个条件意味着：**System Prompt、AGENTS.md、Skill 指令的顺序和内容在每轮中必须完全一致**。如果 Codex 在某轮中修改了 System Prompt 的内容（比如因为某个配置变化），整个前缀缓存链就会断裂。

这也是为什么 AGENTS.md 的稳定性很重要。如果 AGENTS.md 的内容在不同轮次之间有微小变化（比如包含时间戳或随机数），前缀匹配就会失效，导致每轮的完整输入都需要重新计算，Token 成本显著增加。

### 缓存带来的成本节省

Prompt 缓存不改变输入 Token 的计量数量，但改变了计费费率。OpenAI 对缓存命中的 Token 使用更低的费率（通常是正常费率的 50%）。

以 GPT-5.4 的定价为例（假设输入 $2/M Token，缓存命中 $1/M Token）：

```text
无缓存场景（10 轮循环）：

  每轮固定前缀: 3900 Token (System + AGENTS.md + Skill)
  每轮新增内容: 平均增长 3000 Token
  总输入 Token: ~3900 x 10 + 3000 x 55 = 204000 Token
  成本: 204000 x $2/M = $0.408

有缓存场景（10 轮循环）：

  固定前缀: 3900 Token x 10 = 39000 Token（缓存命中）
  新增内容: ~3000 x 55 = 165000 Token（正常费率）
  成本: 39000 x $1/M + 165000 x $2/M = $0.039 + $0.330 = $0.369

节省: ~10%
```

固定前缀越长（AGENTS.md 越详尽），缓存的绝对节省越多。但也要平衡——过长的 AGENTS.md 增加了每轮的输入 Token 量，即使缓存命中率高，总成本也可能更高。核心原则是：AGENTS.md 包含必要信息即可，不要为了"以防万一"堆砌内容。

## GPT-5.4 的上下文参数规格

GPT-5.4 是 Codex 当前使用的默认模型之一。它的上下文参数直接决定了 Agent Loop 的运作边界。

### 上下文窗口

| 参数 | 值 | 说明 |
|------|-----|------|
| 默认上下文窗口 | 272K Token | 标准任务的上下文上限 |
| 长上下文窗口 | 1.05M Token | 通过配置启用，适用于需要大量代码上下文的任务 |
| 最大输出 Token | 128K Token | 单次 API 响应的最大输出长度 |

272K 的默认上下文窗口对于大多数日常工程任务足够。一个典型的 Codex 会话，如果 AGENTS.md 精简、工具输出控制得当，在 10-15 轮循环后通常消耗 50K-100K Token 的输入上下文，远未触及 272K 的上限。

1.05M 的长上下文窗口适用于需要同时处理大量代码文件的场景。例如，跨仓库的全局重构、需要理解整个 monorepo 结构的架构分析、或者需要参考大量历史对话的持续集成任务。长上下文的代价是更高的成本和更长的响应延迟。

128K 的最大输出 Token 是一个重要的设计参数。这意味着模型在单次响应中可以生成非常长的内容——足够输出一个完整的工具调用参数（比如一个大型 diff），或者在回复中提供详尽的分析。对于大多数任务，实际输出 Token 量远低于这个上限。

### 上下文窗口的组成

上下文窗口不是全部给输入使用的。它被分为三部分：

```text
+────────────────────────────────────────────────+
│                  上下文窗口 (272K)               │
+──────────────────────────┬─────────────────────+
│      输入 Token          │    输出 Token        │
│  (System + History +     │  (模型生成的文本     │
│   工具结果 + 用户输入)    │   和工具调用)        │
│                          │                     │
│  ~70-80%                 │  ~20-30%            │
+──────────────────────────┴─────────────────────+
```

输入 Token 的上限约为 190K-215K（272K 的 70%-80%）。剩余的 57K-82K 留给模型输出。这就是为什么压缩阈值设在 70%-80% 的位置——确保模型有足够的输出空间。

当使用长上下文（1.05M）时，输入上限相应提升到约 735K-840K Token。这为需要大量历史代码和对话上下文的复杂任务提供了充裕的空间，但成本也按比例增加。

## 从架构理解到实践优化

理解 Agent Loop 的内部机制后，几个实践优化方向变得直观：

**固定前缀的稳定性。** System Prompt、AGENTS.md、Skill 指令构成每轮请求的固定前缀。保持这个前缀的稳定性，最大化 Prompt 缓存的命中率。避免在 AGENTS.md 中放入动态内容（时间戳、随机值、运行时变量）。

**工具输出的精简。** 每次工具调用的输出不仅在本轮消耗 Token，还在后续每轮中被重复传输。在 AGENTS.md 中指导模型使用精准的命令，减少不必要的输出。例如，用 `grep -n "pattern" file` 替代 `cat file | grep "pattern"`，用 `head -50 file` 替代读取整个文件。

**上下文窗口的利用率监控。** 复杂任务中留意上下文增长速度。如果 AGENTS.md 很长（超过 2000 Token），或者工具频繁产生大量输出，上下文可能在 10-15 轮内接近压缩阈值。拆分任务或精简上下文，比触发压缩后承受信息损失更可取。

**并行工具调用的利用。** 在 AGENTS.md 的 Working Rules 中鼓励模型使用并行调用。例如，"读取多个文件时，一次性并行读取，不要串行等待。"这不会改变 Token 消耗，但能显著缩短执行时间。

**模型选择与上下文窗口的匹配。** 不是所有任务都需要 272K 上下文。简单的小修改在 10K-30K 上下文内就能完成。对于已知范围很小的任务，考虑使用更小的模型（如果可用）来降低成本。对于需要处理大量代码文件的任务，确保使用足够大的上下文窗口，避免频繁压缩。

Agent Loop 的架构决定了 Codex 的行为边界。理解这些边界，你就能在 Prompt 构造、工具使用、任务拆分三个层面做出更明智的决策，让 Codex 在 Token 预算内更高效地完成任务。
