# 常用工作流模板：理解代码、重构迁移、补测试和性能优化

**TL;DR：** 工程化使用 Codex 的核心不是学会更多参数，而是为每类常见任务建立标准化的 prompt 模板和验证方法。本文提供理解代码、Bug 修复、重构迁移、测试补全、文档更新和性能优化六类工作流的完整模板。每个模板包含 prompt 四要素写法、推荐的审批模式、预期 token 消耗、完成时间估算、常见失败原因和对应的解决方法。最后给出工作流选择决策表和从零开始的上手阶梯。

## 为什么需要工作流模板

上一篇文章讲了 prompt 四要素（目标、上下文、约束、完成条件），这一篇把四要素落到具体场景。没有模板的时候，每次任务都要重新组织语言，写法不一致，质量波动大。有了模板之后，同类任务用同一套结构，差异只在于填入的具体内容。

模板的价值不在于限制发挥，而在于减少低级错误。一份好的工作流模板做了三件事：

1. **保证四要素齐全。** 新手最容易遗漏的是完成条件和约束，模板把它们强制包含在内。
2. **预设合理的审批模式。** 不同任务的风险等级不同，模板为每类任务推荐了对应的审批策略。
3. **内置验证步骤。** 模板在 prompt 里写明了怎么验证结果，而不是改完代码就算完成。

下面六类工作流按照使用频率从高到低排列。理解代码和 Bug 修复是日常高频操作，重构迁移和测试补全是周期性任务，文档更新和性能优化频率较低但复杂度高。

## 工作流一：理解代码

### 场景

接手新项目、跨团队协作、review 陌生模块、定位问题需要先读懂上下文。这类任务的特点是只读不改，目标是建立正确的心理模型。

### Prompt 模板

```
阅读 [目录/文件] 下的代码，用中文总结：

1. 模块职责：这个模块解决什么问题，提供什么能力
2. 主要数据流：数据从哪里进来，经过哪些处理，到哪里出去
3. 关键函数和调用关系：列出核心函数及其调用者/被调用者
4. 潜在风险点：哪些地方可能存在并发问题、错误处理缺失或性能瓶颈

约束：
- 只分析，不修改任何文件
- 如果代码量超过 500 行，优先分析入口文件和导出接口
- 对不确定的部分标注"待确认"，不要猜测
```

### 执行策略

理解代码这类任务强烈建议先用 Ask 模式。Ask 模式下 Codex 只做调研不执行任何修改，天然安全，即使 prompt 写得不够精确也不会产生破坏。如果第一次输出的分析不够深入，可以在同一会话中追问细节，而不需要重新发起任务。

对于大型模块（超过 1000 行），分阶段执行比一次性读完更有效。先让 Codex 读入口文件和类型定义建立骨架认知，再逐个深入关键函数的实现。

```
# 第一步：建立骨架认知
codex --ask "阅读 src/auth/ 目录，列出所有导出的函数/类及其职责。只分析 index.ts 和 types.ts"

# 第二步：深入核心流程
codex --ask "基于上一轮的分析，详细解释 authenticate() 函数的完整执行流程，包括所有分支和错误处理"
```

### 验证方法

理解代码的验证不同于代码修改任务——没有测试可以跑。验证方式是让 Codex 基于分析结果画出调用关系图，然后人工核验关键路径。

```
codex --ask "基于对 src/auth/ 的分析，画出 authenticate 和 authorize 两个核心流程的调用关系图，用文本格式表示：

函数A -> 函数B -> 函数C
         |
         v
       函数D -> 函数E

标注每个函数所在的文件和行号"
```

人工核验时重点检查两点：调用关系是否遗漏了跨文件调用、数据流描述是否与实际入口出口一致。如果 Codex 标注了"待确认"的部分，应该手动阅读对应代码确认。

### 失败模式

| 失败现象 | 根因 | 解决方法 |
|---------|------|---------|
| 分析停留在表面，只列出函数签名 | prompt 没有要求解释实现逻辑 | 追加"详细解释每个核心函数的实现策略和分支逻辑" |
| 遗漏跨模块调用 | 分析范围限定在单一目录 | 在 prompt 中补充"如果发现外部依赖，追踪到对应模块并标注" |
| 生成大段不确定的推测 | 代码量过大导致上下文不够 | 缩小范围或分阶段分析 |
| 调用关系图与代码不符 | Codex 对间接调用（事件、回调）识别不准 | 手动检查事件监听和回调注册点 |

### 参考指标

| 指标 | 参考值 |
|------|-------|
| 推荐审批模式 | Ask（只读调研） |
| 预期 Token 消耗 | 3k-15k（取决于代码量和追问轮次） |
| 预期完成时间 | 5-20 分钟 |
| 适用代码规模 | 单文件到中等模块（50-1500 行） |

## 工作流二：Bug 修复

### 场景

已知 bug 需要定位根因并修复。这类任务的特点是目标明确（修好指定问题）、约束严格（不引入新问题）、验证标准清晰（测试通过）。

### Prompt 模板

```
修复 [文件路径] 中 [函数名] 的 [问题描述]。

Bug 描述：[具体的问题表现，包含错误信息、复现步骤或失败的测试]

相关代码：
- 主文件：[路径]
- 测试文件：[路径，如果有]
- 相关依赖：[其他相关文件路径]

约束：
- 不要修改 [列出不应改动的部分，如公共 API 签名、其他模块的代码]
- 修复应最小化，只解决描述的问题，不做额外的"顺便"优化
- 如果修复涉及数据库 schema 或 API 接口变更，先报告方案再动手

完成条件：
- 运行 [测试命令] 全部通过
- 运行 [lint/typecheck 命令] 无新增错误
- 解释修复的原理和为什么选择这个方案
```

四要素对应关系：Bug 描述是目标，相关代码是上下文，约束部分是约束，测试命令是完成条件。

### 执行策略

Bug 修复推荐分步执行：先用 Ask 模式定位根因，再用 Plan 模式规划修复方案，最后才执行修改。

```
# 第一步：定位根因
codex --ask "分析 [错误信息] 这个 bug 的根因。错误出现在 [函数名] 中，复现步骤是 [步骤]。只分析不动手改"

# 第二步：规划修复方案
codex --plan "基于上一轮的根因分析，规划修复方案。目标是 [具体目标]。约束：[约束]"

# 第三步：执行修复
codex "按规划的方案修复 [函数名] 中的 bug。完成后运行 [测试命令] 验证"
```

分步执行的好处是可以在每一步检查 Codex 的理解是否正确。如果 Ask 阶段的根因分析就偏离了方向，可以立即纠正，避免基于错误理解写出有问题的修复代码。

对于简单明确的 bug（比如拼写错误、条件判断反了），可以跳过 Ask 和 Plan 阶段直接执行。判断标准是：你能在脑中想出修复方案并且方案不超过 5 行代码改动。

### 验证方法

Bug 修复的验证分两层：

**自动验证层：** 在 prompt 中写明的测试命令和 lint/typecheck 命令。这些是最低标准，必须通过。

**人工验证层：** 审查 `git diff` 输出，重点检查三个维度：

1. **改动范围是否最小化。** 修复代码是否只改了与 bug 相关的逻辑，没有"顺便"重构其他代码。多余的改动增加 review 负担，也增加引入新 bug 的风险。
2. **是否引入新的边界问题。** 修复是否考虑了空值、并发、类型不匹配等边界情况。
3. **方案是否正确。** 修复是治标（掩盖症状）还是治本（消除根因）。比如一个 null pointer 错误，加 null check 是治标，找到为什么会产生 null 才是治本。

```
# 查看 diff
git diff

# 确认改动文件列表（检查是否改了不该改的文件）
git diff --name-only
```

### 失败模式

| 失败现象 | 根因 | 解决方法 |
|---------|------|---------|
| 修复了症状但没修复根因 | prompt 缺少根因分析要求 | 先用 Ask 模式定位根因再执行修复 |
| 改动范围过大，"顺便"重构了其他代码 | 约束不够明确 | 在约束中强调"最小化改动"并明确不应修改的范围 |
| 测试通过但引入了新的边界问题 | 测试覆盖不够 | 在完成条件中增加边界值测试 |
| 修复方案正确但实现有细节错误 | 复杂逻辑的细节容易出错 | 分步执行，先 Plan 再实现，实现后立即验证 |
| Codex 找不到 bug 根因 | 上下文不够或错误信息模糊 | 提供更完整的复现步骤和日志，必要时贴出完整的 stack trace |

### 参考指标

| 指标 | 参考值 |
|------|-------|
| 推荐审批模式 | Ask 定位 -> Plan 规划 -> Auto/Review 执行 |
| 预期 Token 消耗 | 5k-25k（取决于 bug 复杂度） |
| 预期完成时间 | 5-30 分钟 |
| 典型改动规模 | 1-50 行 |

## 工作流三：重构/迁移

### 场景

代码重构或技术栈迁移。典型场景包括：把 REST API 迁移到 GraphQL、把 class 组件改成 hooks、把 callback 风格改成 async/await、把一个 ORM 换成另一个。这类任务的特点是改动面大、容易引入回归、需要分步执行并逐步验证。

### Prompt 模板

```
将 [模块/目录] 从 [旧方案] 迁移到 [新方案]。

迁移范围：
- 涉及文件：[文件列表或目录]
- 变更内容：[具体描述，如"把所有 fetch 调用替换为 axios 实例"]
- 不变内容：[明确列出不应改变的部分]

约束：
- 保持所有现有测试通过
- 每完成一个文件的迁移后运行 [测试命令] 确认无回归
- 不改变公共 API 的行为语义
- 如果发现需要修改接口签名，先报告再动手

完成条件：
- 所有 [测试命令] 通过
- [lint/typecheck 命令] 无新增错误
- 列出所有被修改的文件和每个文件的改动摘要
```

### 执行策略

重构/迁移是六类工作流中风险最高的。强烈建议使用 Plan 模式先规划迁移步骤，而不是直接执行。Plan 模式下 Codex 会输出分步计划，你可以审查计划是否合理再决定是否执行。

```
# 第一步：规划迁移方案
codex --plan "将 src/api/ 目录下所有接口从 Express handler 迁移到 Fastify handler。
现有测试在 tests/api/ 下，使用 Jest。
保持所有现有测试通过。
先输出迁移计划，包括文件级别的迁移顺序和每个文件的预期改动。"

# 审查计划后执行
codex "按规划的迁移计划执行。每完成一个文件后运行 pnpm test 确认无回归"
```

迁移顺序的设计很关键。正确的顺序是：先迁移底层依赖最少的模块，再迁移依赖它的上层模块。如果反过来，底层的改动会导致上层全部需要重做。Plan 模式的价值就在于让 Codex 先分析依赖关系再定顺序。

对于大型迁移（超过 10 个文件），考虑使用 subagents 并行处理独立的模块。但要注意：subagents 之间不共享上下文，如果多个 subagents 修改了同一个文件的依赖，可能产生冲突。解决方法是在 Plan 阶段就把文件划分到不同的 subagent，确保没有文件被多个 subagent 同时修改。

### 验证方法

重构/迁移的验证重点是回归测试。

**分层验证：**

1. **文件级验证。** 每完成一个文件的迁移，立即运行相关测试确认无回归。这比全部改完再测要好得多——如果某个文件的迁移引入了问题，你能立刻定位到具体文件。
2. **模块级验证。** 完成一个完整模块的迁移后，运行模块级别的集成测试。
3. **全量验证。** 全部迁移完成后，运行完整的测试套件 + lint + typecheck。

**人工验证：** 重点审查以下文件：

- 公共 API 入口文件（确认接口行为语义不变）
- 错误处理路径（迁移过程中最容易遗漏的是错误处理）
- 测试文件的变更（如果测试也被修改了，需要确认是新方案需要改测试，还是为了通过测试而改测试——后者是危险的信号）

```
# 文件级验证（每完成一个文件）
pnpm test -- path/to/migrated/file.test.ts

# 模块级验证
pnpm test -- tests/api/

# 全量验证
pnpm test && pnpm lint && pnpm typecheck
```

### 失败模式

| 失败现象 | 根因 | 解决方法 |
|---------|------|---------|
| 迁移后测试大量失败 | 迁移改变了行为语义 | 在约束中明确"不改变公共 API 行为"，检查失败的测试是否在验证旧行为 |
| 改动雪球越滚越大 | 迁移范围不够明确，依赖链蔓延 | Plan 阶段严格限定迁移范围，使用 Adapter 模式隔离新旧方案 |
| 部分文件漏迁移 | 目录下存在隐藏的依赖（动态 import、反射调用） | Ask 阶段先用 grep 搜索所有引用点 |
| 新旧方案混用导致不一致 | 迁移不彻底，部分代码仍在用旧方案 | 在完成条件中增加"确认无旧方案的残留引用" |

### 参考指标

| 指标 | 参考值 |
|------|-------|
| 推荐审批模式 | Plan 规划 -> Review 逐步执行 |
| 预期 Token 消耗 | 15k-80k（大型迁移可能更多） |
| 预期完成时间 | 30 分钟 - 数小时 |
| 典型改动规模 | 50-1000+ 行 |

## 工作流四：测试补全

### 场景

给现有代码补充测试覆盖。典型场景包括：新功能缺少测试、历史代码测试覆盖率低、重构前需要先补测试作为安全网。

### Prompt 模板

```
给 [文件/模块路径] 补充测试。

被测代码分析：
- 文件：[路径]
- 主要函数/类：[列出]
- 依赖项：[列出外部依赖和 mock 策略]

覆盖要求：
1. 正常路径：每个公开函数的主流程至少一个测试
2. 边界值：空输入、最大值、最小值、null/undefined、空数组/对象
3. 错误处理：异常输入、依赖失败、网络错误等错误场景

测试框架：[如 Jest、Vitest、pytest 等]
测试文件位置：[如 tests/ 目录，或与源文件同目录]
Mock 策略：[如"对外部 API 调用使用 jest.mock"，"对数据库操作使用内存数据库"]

约束：
- 不修改被测代码
- 测试应该是确定性的，不依赖外部服务或随机值
- 每个测试用例独立，不依赖执行顺序

完成条件：
- 运行 [测试命令] 全部通过
- 新增测试覆盖 [列出关键函数]
- 输出测试覆盖率报告
```

### 执行策略

测试补全可以直接执行，因为不修改业务代码，风险较低。但如果被测模块的依赖关系复杂（比如依赖数据库、外部 API、消息队列），建议先用 Ask 模式分析依赖，确定 mock 策略后再写测试。

```
# 简单场景：直接执行
codex "给 src/utils/validator.ts 补充测试。使用 Vitest，测试文件放在 tests/utils/validator.test.ts。覆盖正常路径、边界值和错误处理。完成后运行 pnpm test 确认通过"

# 复杂场景：先分析依赖
codex --ask "分析 src/services/payment.ts 的依赖关系。列出所有外部依赖（数据库、API、第三方 SDK），推荐每个依赖的 mock 策略"
# 然后基于分析结果写测试
codex "基于上一轮的依赖分析，给 src/services/payment.ts 补充测试..."
```

### 验证方法

测试补全的验证分两步：

**第一步：测试通过。** 新写的测试全部通过是基本要求。但还要检查一个容易忽略的问题：测试是否真的在验证被测代码的行为。用以下方法排查：

- 故意破坏被测代码的逻辑（比如把返回值改成固定值），看测试是否失败。如果测试仍然通过，说明测试没有真正验证行为。
- 检查测试中是否有大量的 `expect.any()` 或 `expect.anything()`，这些可能是测试写得不够精确的信号。

**第二步：覆盖率报告。** 覆盖率报告告诉你哪些代码路径没有被测试到。

```
# 运行测试并生成覆盖率报告
pnpm test -- --coverage

# 检查覆盖率
# 重点关注：行覆盖率（statements）和分支覆盖率（branches）
# 分支覆盖率通常低于行覆盖率，因为边界条件和错误分支容易被遗漏
```

| 覆盖率类型 | 关注点 | 最低标准 |
|-----------|--------|---------|
| 行覆盖率（Statements） | 是否每行代码都被执行过 | 80% |
| 分支覆盖率（Branches） | 是否每个 if/else 分支都被覆盖 | 70% |
| 函数覆盖率（Functions） | 是否每个函数都被调用过 | 90% |
| 行覆盖（Lines） | 与语句覆盖率类似 | 80% |

注意覆盖率只是参考指标，不是目标。100% 覆盖率不等于没有 bug——它只能告诉你哪些代码被执行过，不能告诉你断言是否充分。

### 失败模式

| 失败现象 | 根因 | 解决方法 |
|---------|------|---------|
| 测试通过但没验证行为（假绿） | 断言不够严格，只检查函数不抛异常 | 在 prompt 中强调"每个测试必须有明确的断言，验证返回值或副作用" |
| 测试依赖外部服务，CI 环境下失败 | 缺少 mock 或 mock 不完整 | 在约束中明确"所有外部依赖必须 mock" |
| Mock 过度导致测试与实现强耦合 | 连被测代码的内部实现细节也 mock 了 | 区分"依赖"和"实现细节"，只 mock 外部依赖 |
| 覆盖率低，边界和错误场景缺失 | prompt 没有明确要求边界值和错误处理 | 使用上面的模板，明确列出三类覆盖要求 |

### 参考指标

## 工作流五：文档更新

### 场景

代码变更后需要同步更新相关文档。典型场景包括：API 接口变更需要更新接口文档、新增配置项需要更新配置文档、重构后需要更新架构文档。

### Prompt 模板

```
检查 [变更范围] 的代码变更，更新 [文档路径] 中的相关文档。

变更范围：[git commit hash、PR 编号、或文件路径列表]
目标文档：[文档文件路径]

更新要求：
- 保持文档与代码一致：API 签名、参数类型、返回值必须与代码实现同步
- 保持文档的现有风格和结构不变
- 只更新与变更相关的部分，不重写整篇文档
- 如果变更涉及破坏性改动，在文档中标注迁移指南

约束：
- 不修改代码文件，只修改文档文件
- 不添加文档中原本没有的章节
- 技术术语保持原文（英文），不在文档中翻译术语

- 列出所有被更新的文档位置
- 标注每处更新对应的代码变更
```

### 执行策略

文档更新的工作流相对简单，但有一个容易忽略的前置步骤：先让 Codex 理解代码变更的具体内容，再让它更新文档。直接让 Codex "更新文档"而不提供变更范围，它可能会编造不存在的内容。

```
# 方式三：基于文件路径（适合手动指定变更范围）
```


### 验证方法

文档验证没有自动化工具可以跑，需要人工对照。检查三个维度：

1. **准确性。** 文档中描述的接口签名、参数类型、返回值是否与代码实现一致。可以抽查关键接口，手动对照文档和代码。
2. **完整性。** 代码变更中涉及的改动是否都在文档中有对应更新。检查方法是让 Codex 列出"代码变更列表"和"文档更新列表"，然后逐项对照。
3. **风格一致性。** 更新部分的写作风格是否与文档其余部分一致（术语使用、格式、详略程度）。

### 失败模式

## 工作流六：性能优化

### 场景

识别和优化性能瓶颈。典型场景包括：接口响应慢、页面加载时间长、内存占用过高、CPU 密集型操作拖慢整体吞吐量。

### Prompt 模板

```
分析 [模块/文件路径] 的性能瓶颈。

当前症状：[描述性能问题的具体表现，如"API 响应时间 p99 > 2s"，"内存占用持续增长"]
数据来源：[性能监控数据、日志、用户反馈]

分析步骤：
1. 先用 [性能分析工具/方法] 识别热点
   - 工具：[如 Chrome DevTools Profiler、Node.js --prof、py-spy、ab 压测工具]
   - 命令：[具体的分析命令]
2. 基于热点分析结果，提出优化方案
   - 每个方案说明：预期提升、实施风险、改动范围
3. 对选定的优化方案实施改动

约束：
- 优化不能改变公共 API 的行为语义
- 优先选择低风险的优化方案
- 不做过早优化——只优化有数据证明的热点
- 优化前后都要跑基准测试对比

完成条件：
- 输出优化前后的基准测试数据对比
- 所有 [测试命令] 通过
- 说明优化原理和为什么选择这个方案
```

### 执行策略

性能优化是唯一一个必须在 Ask 阶段先收集数据的工作流。没有数据的优化是盲猜，浪费时间且可能让性能更差。

```
# 第一步：收集性能数据
codex --ask "分析 src/services/search.ts 的性能。先用 console.time/timeEnd 标记关键步骤的执行时间，设计一个性能测试脚本。只设计不动手改业务代码"

# 第二步：运行性能测试获取基准数据
# 这一步需要手动运行，因为需要真实的执行环境

# 第三步：基于数据分析瓶颈
codex --ask "基于以下性能数据，分析瓶颈所在：[贴出性能测试结果]"

# 第四步：制定优化方案
codex --plan "基于瓶颈分析，制定优化方案。每个方案说明预期提升和实施风险。[模板的其余部分]"

# 第五步：实施优化
codex "按方案实施优化。完成后运行基准测试对比。[模板的其余部分]"
```

### 验证方法

性能优化的验证必须量化。用数据说话，不用体感说话。

**基准测试对比模板：**

```
优化前：
- 操作 X 平均耗时：xxx ms（n=100）
- 内存占用峰值：xxx MB
- p50/p90/p99 延迟：xx/xx/xx ms

优化后：
- 操作 X 平均耗时：xxx ms（n=100）
- 内存占用峰值：xxx MB
- p50/p90/p99 延迟：xx/xx/xx ms

提升比例：xx%
```

关键检查点：

1. **基准测试是否稳定。** 多次运行基准测试，确认结果波动在 5% 以内。如果波动太大，说明测试本身有问题（比如受系统负载影响），需要排除干扰因素。
2. **优化是否可重现。** 在不同环境（开发机、CI 环境）下都能复现提升效果，而不是只在特定条件下有效。
3. **是否引入新问题。** 优化后运行完整测试套件，确认没有回归。特别关注：精度损失（浮点运算优化）、并发问题（缓存引入的竞态）、内存泄漏（对象池或缓存未释放）。

### 失败模式

| 失败现象 | 根因 | 解决方法 |
|---------|------|---------|
| 优化效果不明显 | 瓶颈定位不准确，优化了非热点代码 | 先收集性能数据，用数据驱动优化目标 |
| 优化后出现新的 bug | 优化改变了边界行为（如缓存导致数据不一致） | 优化后运行完整测试套件 + 重点测试相关场景 |
| 基准测试结果波动大 | 测试方法不够稳定，受系统因素干扰 | 增加测试次数取平均值，排除系统负载干扰 |
| 过度优化，代码可读性下降 | 对低频路径做了过度优化 | 只优化有数据证明的热点，对低频路径保持代码清晰 |

| 指标 | 参考值 |
|------|-------|
| 推荐审批模式 | Ask 数据收集 -> Plan 方案制定 -> Review 执行 |
| 预期 Token 消耗 | 10k-50k |
| 预期完成时间 | 30-60 分钟（不含人工运行基准测试的时间） |
| 典型改动规模 | 10-200 行 |

### 跨工作流组合场景

实际工作中经常需要组合多个工作流。以下是常见组合模式：
