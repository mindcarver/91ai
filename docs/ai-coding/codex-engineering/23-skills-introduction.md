# Skills 入门：什么时候该创建可复用技能

**TL;DR：** Skill 是 Codex 工程化体系中从"反复描述"到"一次定义反复调用"的关键跃迁。但它不是默认选项——大部分任务写好 prompt 就够了，重复出现的规则应该放 AGENTS.md。只有当一个工作流满足三个条件（已经手动执行过至少三次、包含三个以上步骤且有配套脚本或引用文件、流程本身趋于稳定）时，才值得投资封装成 Skill。过早创建 Skill 和从不创建 Skill 一样有害：前者制造维护负担，后者浪费重复描述的成本。本文建立一条清晰的决策路径：prompt 适合什么、AGENTS.md 适合什么、Skill 适合什么，以及 Skill 和 Automation 的边界在哪里。

---

## 三层递进：从 prompt 到 AGENTS.md 到 Skill

工程化不是一步到位的跳跃，而是逐层递进的沉淀过程。在 Codex 的体系中，知识有三种承载形态：prompt、AGENTS.md 和 Skill。每一层解决不同的问题，有不同的维护成本和复用价值。理解这三层递进是判断"什么时候该创建 Skill"的前提。

### 第一层：prompt -- 解决一次性问题

当你给 Codex 下达一条指令时，知识存在于 prompt 中。prompt 是最轻量的知识载体，不需要创建任何文件，不需要维护任何结构，写完就用，用完即弃。

适用场景：

- 修复一个特定的 bug
- 在某个文件中添加一个新函数
- 分析一段代码的性能瓶颈
- 回答一个关于项目的技术问题

这些任务的共同特征是"一次性的"。你不需要记住这次任务的执行方式，下次遇到类似问题时很可能上下文已经完全不同，需要重新描述。

关键判断标准：如果一条指令在未来一周内大概率不需要以相同的形式再次执行，它就应该留在 prompt 里。不要因为"可能还会用到"就急于固化。可能性不等于必然性。

### 第二层：AGENTS.md -- 沉淀重复出现的规则

当某条规则在多次任务中反复出现时，它应该从 prompt 迁移到 AGENTS.md。AGENTS.md 是常驻上下文，每次 Codex 启动时自动加载。它承载的是"每次任务都需要知道的常识"。

典型内容包括：

- 构建命令（`pnpm install`、`pnpm test`、`pnpm build`）
- 编码约定（使用 named exports、用 Zod 做输入验证）
- 安全边界（不要碰 `.env` 文件、不要 commit 到 main）
- 目录结构（哪个目录放什么类型的文件）

AGENTS.md 的核心约束是篇幅。控制在约 100 行以内，占用 3000-5000 tokens。超过这个范围，它会挤压模型处理实际任务的上下文空间，反而降低表现。所以 AGENTS.md 只放最精炼的规则，而不是完整的操作流程。

关键判断标准：如果一条规则在超过 50% 的任务中都需要被模型知道，它应该在 AGENTS.md 里。如果它只在特定类型任务中需要，不应该常驻，而应该按需加载。

### 第三层：Skill -- 封装可复用的完整工作流

当一个工作流满足以下所有条件时，它应该从 AGENTS.md 的简单规则升级为独立 Skill：

1. 工作流包含多个步骤（通常三个以上）
2. 工作流有配套的脚本、参考文件或模板资源
3. 工作流已经被手动验证过至少三次，流程趋于稳定
4. 多个人或多个项目需要执行同样的工作流

Skill 和 AGENTS.md 的根本区别在于加载方式。AGENTS.md 是常驻的，每次任务都占用上下文空间。Skill 是按需加载的，只在被触发时才进入上下文。这意味着你可以创建任意数量的 Skill，而不用担心每个都增加常驻的 token 成本。系统只在识别到相关条件时，才把对应的 Skill 加载进来。

三层递进的判断矩阵：

| 条件 | prompt | AGENTS.md | Skill |
|------|--------|-----------|-------|
| 只需要执行一次 | 是 | 否 | 否 |
| 每次任务都需要知道 | 可能 | 是 | 否 |
| 特定类型任务需要 | 是 | 可能 | 是 |
| 包含多步骤和脚本 | 不适合 | 不适合 | 是 |
| 需要按条件自动触发 | 否 | 否 | 是 |
| 需要版本化和跨团队共享 | 否 | 部分 | 是 |

## "第三次"规则

什么时候从 prompt 升级到 Skill？最实用的判断依据是"第三次"规则。

### 规则定义

当同一个工作流出现了第三次，就是时候考虑创建 Skill 了。

第一次出现时，你在 prompt 中完整描述这个工作流。比如"审查这个 PR，先看整体变更范围，再重点审查架构变更，然后审查逻辑变更，最后检查测试覆盖"。这是探索阶段——你在验证这个工作流本身是否合理。

第二次出现时，你发现自己几乎在复用第一次的 prompt。措辞可能略有不同，但步骤和判断标准基本一致。这是确认阶段——你确认了这个工作流有复用价值。

第三次出现时，你开始觉得重复描述有点烦了。你知道自己之前写过类似的指令，但不确定措辞细节，不得不重新组织语言。这是触发点——重复描述的成本已经超过了封装 Skill 的成本。

### 为什么是三次而不是两次或五次

两次不够。第二次可能只是巧合——相似的场景碰巧连续出现。如果只有两次就急于封装，可能发现第三次时工作流已经变了，Skill 需要大改甚至废弃。

五次太多。如果在五次重复之后才封装，你已经浪费了至少两次重复描述的时间。对于高频工作流（比如每天执行的 PR Review），这个浪费会快速累积。

三次是一个经过实践验证的平衡点。它足以排除巧合，又不会让你等太久。

### "第三次"的具体计数方式

"同一个工作流出现三次"不是模糊的感觉，而是可以具体追踪的。记录方式很简单：

```text
工作流追踪记录：

工作流：PR 审查
第 1 次（2026-04-12）：手动描述完整审查流程，输出结果基本符合预期
第 2 次（2026-04-15）：微调了审查顺序，补充了边界条件检查标准
第 3 次（2026-04-18）：流程基本稳定，和第 2 次的描述高度一致
→ 触发条件满足，准备创建 Skill
```

三次之间的差异如果很大（比如步骤顺序完全不同、判断标准互相矛盾），说明工作流本身还不稳定，不应该封装。三次之间的差异如果只是措辞微调，说明工作流已经收敛，可以封装。

## Skill 提供了什么 prompt 不能提供的

理解 Skill 的价值不在于"比 prompt 写得更长"，而在于它提供了一组 prompt 无法实现的结构化能力。

### 持久化定义和元数据

一个 Skill 由 SKILL.md 文件定义，其中包含 YAML 格式的元数据和 Markdown 格式的流程描述。这些元数据包括名称、描述、触发条件、适用范围等。

```yaml
---
name: pr-review
description: "对 Pull Request 进行结构化审查，覆盖架构变更、逻辑变更和测试覆盖三个维度"
trigger: "当用户请求审查 PR 时激活"
when:
  file_pattern:
    - "*.ts"
    - "*.tsx"
    - "*.py"
---

## 执行步骤

### 阶段一：架构和设计变更审查
...
```

prompt 没有这种结构。一条 prompt 就是一段自由文本，没有标准的元数据格式，没有触发条件声明，没有文件模式匹配。你可以把一条长长的 prompt 保存在某个文件里，但它只是一个文本片段，没有 Skill 的结构化语义。

元数据的核心价值是让系统理解这个 Skill 的适用场景。当 `when.file_pattern` 声明了 `*.ts, *.tsx, *.py` 时，系统可以在检测到相关文件操作时自动建议加载这个 Skill。这不是 prompt 能做到的。

### 关联资源：脚本、参考文件和模板

Skill 不仅仅是一个 SKILL.md 文件。一个完整的 Skill 是一个目录，包含流程定义和关联资源。

```text
skills/
  pr-review/
    SKILL.md              # 流程定义和元数据
    scripts/
      extract-diff.sh     # 提取 PR diff 的辅助脚本
      check-ci-status.sh  # 检查 CI 状态的辅助脚本
    references/
      review-checklist.md # 审查检查清单
      severity-guide.md   # 问题严重程度判定指南
    assets/
      report-template.md  # 审查报告模板
      example-output.md   # 期望输出的示例
```

prompt 可以引用外部文件（比如"参考 review-checklist.md 进行审查"），但它不能把引用关系结构化地声明出来。Skill 的目录结构让这些关联变成显式的、可版本控制的、可以随 Skill 一起分发的。

当团队成员 clone 一个包含 Skill 的仓库时，他们得到的不仅仅是一条流程描述，而是完整的工作流工具包：流程定义、辅助脚本、参考文档和输出模板。

### 条件激活机制

Skill 支持基于文件模式、项目类型和上下文条件自动激活。这是 Skill 和 prompt 之间最本质的区别之一。

```yaml
when:
  file_pattern:
    - "src/api/**/*.ts"       # 当操作涉及 API 目录下的 TypeScript 文件
    - "tests/api/**/*.test.ts" # 或 API 测试文件
  project_type: "nextjs"       # 且项目类型为 Next.js
```

当 Codex 检测到当前任务涉及 `src/api/` 下的文件变更时，系统会检查是否有匹配这个模式的 Skill。如果有，Skill 的流程描述会被加载到上下文中，模型按照 Skill 定义的步骤执行。

这种条件激活机制解决了两个问题：

第一，它解决了"每次都要记住加载"的问题。如果流程知识只存在于某个 prompt 模板中，你需要每次手动引用它。有了条件激活，系统自动判断什么时候需要什么知识。

第二，它解决了上下文浪费问题。AGENTS.md 的内容无论是否相关都常驻在上下文中。Skill 只在条件匹配时加载，不匹配时不占任何空间。这意味着你可以创建几十个 Skill，每个只在特定场景下被激活，而不会互相干扰。

### 版本化和跨团队共享

Skill 文件存储在 Git 仓库中，天然具备版本控制能力。每次修改 Skill 的流程或资源，都有完整的变更历史。这意味着：

- 团队可以像 review 代码一样 review Skill 的变更
- Skill 的演进可以追溯（"上个月审查流程增加了安全维度检查"）
- 不同分支可以有不同的 Skill 版本（开发分支可以测试新的 Skill 流程）
- Skill 可以跨仓库复用（通过 Git submodule 或直接复制）

prompt 存在聊天记录里，不可版本化，不可 review，不可跨会话追溯。AGENTS.md 虽然在 Git 中，但它是全局性的，不适合放特定工作流的完整流程描述。Skill 填补了这个空白——它是可以被独立版本化、独立演进、独立分发的工作流知识单元。

## 什么放 Skill，什么留在 AGENTS.md

Skill 和 AGENTS.md 的边界不清是实际操作中最常见的困惑。以下对比帮助明确各自的范围。

### AGENTS.md 的职责：常驻常识

AGENTS.md 承载的是"每次任务都需要的常识"。这些信息有一个共同特征：无论你在做什么类型的任务，它们都相关。

| 内容类型 | 是否属于 AGENTS.md | 理由 |
|----------|-------------------|------|
| 构建命令 | 是 | 每次修改代码后都需要跑构建和测试 |
| 目录结构 | 是 | 每次搜索和定位代码都需要知道目录用途 |
| 通用编码约定 | 是 | 每次写代码都应该遵循 |
| 安全边界 | 是 | 每次操作都需要遵守 |
| PR 审查流程 | 否 | 只有审查 PR 时才需要 |
| 部署验证步骤 | 否 | 只有部署后才需要 |
| 数据库迁移 SOP | 否 | 只有改 schema 时才需要 |
| 文档更新流程 | 否 | 只有代码变更影响文档时才需要 |

AGENTS.md 应该控制在约 100 行以内。如果发现 AGENTS.md 超过 100 行，大概率是混入了本应属于 Skill 的流程描述。

### Skill 的职责：特定场景的工作流

Skill 承载的是"在特定条件下需要的完整工作流"。这些信息有一个共同特征：只在特定类型的任务中出现，但出现时需要一套完整的、经过验证的操作流程。

一个完整的 Skill 通常包含：

1. 触发条件：什么情况下应该执行这个工作流
2. 前置检查：执行前需要确认哪些前提条件
3. 执行步骤：按顺序描述每一步要做什么
4. 判断标准：每一步的结果怎么评判
5. 错误处理：遇到异常怎么兜底
6. 输出格式：最终产出的格式和内容要求
7. 关联资源：脚本、参考文件和模板

这些内容如果全部塞进 AGENTS.md，会轻易让文件膨胀到 300 行以上。但作为独立的 Skill，它们只在被触发时才占用上下文空间。

### 边界判断的具体方法

当你在 AGENTS.md 中写某条规则时，问自己一个问题："这条规则只在特定类型的任务中才相关吗？"如果答案是"是"，它可能更适合作为 Skill 的一部分。

反过来，当你在创建一个 Skill 时，检查其中是否有内容应该属于 AGENTS.md。具体检查方式：扫描 Skill 中的每个步骤，如果某个步骤的约束（比如"用 pnpm 跑测试"）在所有 Skill 中都出现，那它应该在 AGENTS.md 的 Commands 部分，而不是重复写在每个 Skill 里。

```text
判断流程：

1. 这条信息在 > 50% 的任务中都需要吗？
   → 是：放 AGENTS.md
   → 否：继续判断

2. 这条信息有配套的脚本或参考文件吗？
   → 是：放 Skill
   → 否：继续判断

3. 这条信息只在特定条件下才相关吗？
   → 是：放 Skill
   → 否：它可能只是一条 prompt 级别的临时指令
```

## 什么时候不要创建 Skill

创建 Skill 有成本。不是文件创建本身的技术成本（那很低），而是维护成本、认知成本和上下文加载成本。以下四种情况不应该创建 Skill。

### 情况一：工作流只有一到两步

如果一个工作流只有"读一个文件、做一个修改"这么简单，它不需要 Skill。一条写好的 prompt 就够了。

```text
不需要 Skill 的示例：
- "读 src/config.ts，把 LOG_LEVEL 从 info 改成 debug"
- "在 auth/types.ts 中添加 mfa_enabled?: boolean 字段"
- "运行 pnpm test，报告失败用例"

需要 Skill 的示例：
- "审查这个 PR"（涉及架构审查、逻辑审查、测试审查三个阶段）
- "部署验证"（涉及健康检查、日志检查、指标对比、冒烟测试四步）
- "数据库迁移"（涉及 schema 变更分析、迁移脚本生成、回滚方案编写、测试执行）
```

判断标准：如果工作流的步骤可以用一句话完整描述，不需要 Skill。如果需要三句话以上才能说清楚步骤和判断标准，值得考虑 Skill。

### 情况二：工作流每次都不一样

Skill 的前提是流程稳定。如果一个工作流每次执行的步骤、判断标准或输出格式都不同，封装 Skill 会制造麻烦——你要么每次都修改 Skill，要么 Skill 的描述过于抽象而失去指导价值。

典型的不适合 Skill 化的工作流：

- "分析这段代码的设计意图" -- 分析方法随代码类型和复杂度变化
- "优化这个函数的性能" -- 优化策略取决于具体的瓶颈，无法预设步骤
- "评估是否应该重构这个模块" -- 评估标准高度依赖上下文，没有通用流程

这些任务需要的是好的 prompt 模板，而不是固定的 Skill。prompt 模板可以提供思考框架，但保留足够的灵活性来适应不同情况。

### 情况三：工作流还没有被手动验证过

在手动验证之前创建 Skill，等于在没有测试的情况下写代码。你封装的流程可能包含错误的步骤、遗漏的检查项或不切实际的判断标准。一旦 Skill 被创建并投入使用，团队成员会倾向于信任它的输出，即使这个输出是基于未经验证的流程。

正确做法：

1. 先在 prompt 中完整描述工作流
2. 手动执行至少三次，观察结果
3. 根据观察结果修正流程描述
4. 确认流程描述在过去三次执行中没有修改
5. 此时才提取为 Skill

### 情况四：工作流不需要按条件自动触发

如果你的工作流只在人工明确请求时才执行，而且执行频率很低（比如每月一次），创建 Skill 的价值有限。一条保存在文档中的 prompt 模板可能更合适。Skill 的核心价值之一是条件激活——当系统检测到特定文件模式或任务类型时自动加载。如果这个能力用不上，Skill 的结构化优势就被浪费了。

这不意味着低频工作流永远不该创建 Skill。如果低频工作流的步骤很复杂（比如年度安全审计，包含几十个检查项和判断标准），Skill 的价值在于把复杂的流程知识持久化，而不是自动触发。

## Skill 的命名和组织约定

Skill 的目录组织直接影响团队成员发现和使用 Skill 的效率。以下约定经过实践验证，适用于大多数团队。

### 目录结构

```text
.claude/skills/                   # 或 skills/ 目录（取决于项目配置）
  pr-review/
    SKILL.md
    scripts/
      extract-diff.sh
    references/
      review-checklist.md
    assets/
      report-template.md
  deploy-verify/
    SKILL.md
    scripts/
      health-check.sh
      smoke-test.sh
    references/
      metrics-baseline.json
    assets/
      report-template.md
  doc-update/
    SKILL.md
    references/
      doc-mapping.md
```

每个 Skill 一个独立目录，目录名就是 Skill 的标识符。目录名使用小写字母和连字符（kebab-case），与 URL 和 CLI 参数的常见格式一致。

### 命名原则

| 原则 | 正确示例 | 错误示例 | 理由 |
|------|---------|---------|------|
| 用动词开头 | `pr-review`、`deploy-verify` | `review`、`verification` | 动词明确表达"做什么" |
| 用领域限定 | `db-migrate`、`api-doc-gen` | `migrate`、`doc-gen` | 避免歧义 |
| 保持简短 | `pr-review` | `pull-request-code-review` | 过长的名称增加认知负担 |
| 避免缩写 | `deploy-verify` | `dep-ver` | 缩写降低可发现性 |
| 不用技术实现名 | `api-doc-gen` | `swagger-gen` | 技术选型可能变化 |

### 子目录用途

- `scripts/` -- 辅助脚本。Skill 的某些步骤可能需要执行自定义脚本，比如数据提取、格式转换或特定工具的调用封装。脚本应该有执行权限，并在 SKILL.md 中声明调用方式。
- `references/` -- 参考文件。审查检查清单、判断标准、配置参数等需要模型在执行时查阅的文件。参考文件不是每次都全量加载，而是由模型根据需要选择性地读取。
- `assets/` -- 静态资源。输出模板、示例文件、默认配置等不需要模型理解的辅助资源。模型只需要知道"把结果填入这个模板"。

## Skill 与 Automation 的关系

Skill 和 Automation 经常被混淆，但它们的职责边界很清晰。

### 职责分离

```
Skill 定义 HOW -- 怎么做
  → 具体步骤、判断标准、输出格式、错误处理

Automation 定义 WHEN -- 什么时候做
  → 触发时间（cron 表达式）或触发事件（PR 创建、部署完成）
  → 执行后的通知和告警策略
```

一个 Skill 可以被手动触发，也可以被 Automation 包装成定时任务。但 Automation 不应该定义流程本身——它只定义触发条件和执行后的处理策略。

### 演进路径

正确的演进路径是：先创建 Skill，手动验证稳定后，再考虑加 Automation。

```text
阶段一：手动执行 Skill
  → 创建 SKILL.md 和关联资源
  → 手动触发执行（如输入 /pr-review）
  → 检查输出质量，修正流程
  → 至少成功执行 5 次

阶段二：评估是否需要自动化
  → 检查四个条件（详见 Automations 章节）：
    1. Skill 本身稳定
    2. 不需要人工判断
    3. 结果可自动验证
    4. 失败时有回滚策略
  → 四项全部满足才继续

阶段三：创建 Automation
  → 配置触发方式（定时或事件驱动）
  → 配置输出方式（报告、PR、通知）
  → 配置失败告警和回滚策略
  → 先以影子模式运行（执行但不提交结果），观察一周
  → 确认无误后正式启用
```

最常见的错误是在 Skill 还不稳定时就急于加 Automation。结果自动化执行一次出问题的代价远大于手动执行出问题——手动执行时人可以即时干预，自动化出问题时可能在无人监督时持续产生错误结果。Automation 的门槛要远高于 Skill。Skill 允许偶尔失败，Automation 必须几乎不失败。

### 哪些 Skill 适合转为 Automation

不是所有 Skill 都适合自动化。适合自动化的 Skill 有一个共同特征：步骤完全确定性，不需要人工判断。

| Skill 类型 | 适合自动化 | 原因 |
|------------|-----------|------|
| 每日代码审查扫描 | 是 | 步骤固定，规则明确，输出是报告 |
| 依赖安全检查 | 是 | 工具输出驱动，无需判断 |
| PR Review（架构评估） | 否 | 架构评估需要人类判断 |
| 部署验证 | 部分 | 健康检查和指标对比可自动化，异常判断需要人 |
| 文档同步 | 部分 | 简单不一致可自动修复，复杂不一致需要人 |
| 代码重构 | 否 | 重构策略高度依赖上下文 |

## Skill 的按需加载机制

理解 Skill 的按需加载机制，有助于正确设计 Skill 的触发条件和内容。

### AGENTS.md 的常驻加载 vs Skill 的按需加载

AGENTS.md 在每次 Codex 启动时自动加载，无论当前任务是否相关。这意味着 AGENTS.md 中的每一行都永久占用上下文窗口的一部分。这是为什么 AGENTS.md 必须控制在约 100 行以内——它是一个固定成本。

Skill 的加载方式完全不同。系统维护一个 Skill 索引，记录每个 Skill 的触发条件。当用户下达一个任务时，系统检查任务上下文（涉及的文件、目录、任务类型），与所有已注册 Skill 的 `when.file_pattern` 进行匹配。只有匹配成功的 Skill 才会被加载到上下文中。

```text
启动阶段：
  加载 AGENTS.md（常驻，~3000-5000 tokens）
  建立 Skill 索引（仅读取 frontmatter，不加载完整内容）

任务执行阶段：
  用户输入："审查 PR #247"
  系统匹配：PR 审查 → 命中 pr-review Skill
  加载：skills/pr-review/SKILL.md 的完整内容
  执行：模型按照 Skill 定义的步骤执行

任务执行阶段（另一个任务）：
  用户输入："修复 auth/login.ts 的 token 过期 bug"
  系统匹配：文件 src/auth/login.ts → 检查所有 Skill 的 file_pattern
  结果：没有匹配的 Skill（这是一个简单的 bug 修复，不需要特定流程）
  执行：模型按照 AGENTS.md 中的通用规则执行
```

这个机制意味着你不需要担心"创建太多 Skill 会拖慢系统"。未触发的 Skill 几乎零成本。唯一需要注意的是触发条件的精确性——如果条件写得太宽泛，可能导致不相关的 Skill 被频繁加载，浪费上下文空间。

### 触发条件的设计原则

`when.file_pattern` 应该精确匹配 Skill 真正相关的文件路径，避免过度匹配。

```yaml
# 过于宽泛 -- 任何 TypeScript 文件变更都会触发
when:
  file_pattern:
    - "**/*.ts"

# 合理精确 -- 只在 API 目录下的 TypeScript 文件变更时触发
when:
  file_pattern:
    - "src/api/**/*.ts"
    - "tests/api/**/*.test.ts"

# 过于精确 -- 只匹配一个特定文件，失去了模式匹配的意义
when:
  file_pattern:
    - "src/api/auth/handler.ts"
```

合理的触发条件设计需要考虑两个维度：

第一，覆盖面。条件应该覆盖 Skill 适用的所有场景，但不超过。如果 API 审查 Skill 只对 `src/api/` 下的文件生效，不要把条件写成匹配整个 `src/` 目录。

第二，冲突处理。如果两个 Skill 的触发条件有重叠，应该在 SKILL.md 中明确说明优先级或互斥关系。比如，一个通用的 TypeScript 审查 Skill 和一个专门的 API 审查 Skill，当两者同时匹配时，应该明确是叠加执行还是只执行更具体的那个。

## 常见错误

### 错误一：为一次性任务创建 Skill

这是最常见的新手错误。有人刚学会创建 Skill，就把所有稍微复杂一点的任务都封装成 Skill。结果 Skill 目录里堆了几十个只被用过一次的 Skill，每个都需要维护，但几乎从不被再次触发。

判断标准：如果一个 Skill 创建后一个月内没有被触发过第二次，它大概率不应该被创建。要么删除它，要么回退为 prompt 模板保存在文档中。

### 错误二：Skill 包含应该属于 AGENTS.md 的内容

典型表现是每个 Skill 都重复声明相同的构建命令、相同的编码约定、相同的安全边界。这些信息应该集中在 AGENTS.md 中，Skill 只引用不重复。

```text
错误做法（每个 Skill 重复声明）：
Skill A: "用 pnpm test 跑测试，用 Zod 做验证，不要碰 .env"
Skill B: "用 pnpm test 跑测试，用 Zod 做验证，不要碰 .env"
Skill C: "用 pnpm test 跑测试，用 Zod 做验证，不要碰 .env"

正确做法（AGENTS.md 声明通用规则，Skill 只声明特定流程）：
AGENTS.md: "用 pnpm test 跑测试，用 Zod 做验证，不要碰 .env"
Skill A: "执行 PR 审查流程（构建命令和安全边界见 AGENTS.md）"
Skill B: "执行部署验证流程（构建命令和安全边界见 AGENTS.md）"
```

### 错误三：Skill 的触发条件过于宽泛

当 `when.file_pattern` 写成 `**/*` 时，这个 Skill 会在几乎每次任务中被加载。这违背了 Skill 按需加载的设计初衷，变成了一个更昂贵的 AGENTS.md。

触发条件应该精确到 Skill 真正相关的文件范围。如果你发现一个 Skill 的触发频率远高于预期，检查 `when.file_pattern` 是否需要收窄。

### 错误四：不维护已创建的 Skill

Skill 不是写完就不管的。项目的构建命令可能变了，审查标准可能调整了，依赖的工具可能升级了。如果 Skill 中的流程描述与实际情况脱节，模型按照过时的 Skill 执行，产出质量会下降，而且调试起来更困难——因为你可能先怀疑模型能力，而不是怀疑 Skill 内容过时。

建议把 Skill 的审计纳入月度维护计划。每月花 15 分钟检查所有 Skill：构建命令是否仍然有效、流程步骤是否与当前规范一致、触发条件是否仍然合理。

## 总结

创建 Skill 的决策不是技术问题，而是投资回报问题。每一次封装都意味着维护责任的增加。以下是核心判断框架：

```text
决策路径：

1. 这个工作流只执行一次吗？
   → 是：写一条好的 prompt
   → 否：继续

2. 这个工作流只是一条简单规则吗（没有多步骤和脚本）？
   → 是：加到 AGENTS.md
   → 否：继续

3. 这个工作流手动执行过至少三次了吗？
   → 否：继续在 prompt 中迭代
   → 是：继续

4. 这个工作流有配套的脚本或参考文件吗？
   → 否：可能只需要一段 prompt 模板
   → 是：继续

5. 这个工作流只在特定条件下才需要吗？
   → 否：可能属于 AGENTS.md
   → 是：创建 Skill

创建 Skill 后：
→ 手动验证至少三次
→ 确认输出质量稳定
→ 触发条件精确设置
→ 纳入月度维护计划
```

Skill 是 Codex 工程化体系中能力封装层的核心机制。它和 AGENTS.md、prompt 构成三层递进的知识沉淀体系：prompt 解决一次性问题，AGENTS.md 沉淀常驻规则，Skill 封装特定场景的可复用工作流。三者各有明确的职责边界，不应该混淆或替代。

记住"第三次"规则：第一次探索，第二次确认，第三次封装。在此之前保持耐心，在此之后果断行动。过早创建 Skill 和从不创建 Skill 一样有害。目标是在复用价值和维护成本之间找到平衡点。
