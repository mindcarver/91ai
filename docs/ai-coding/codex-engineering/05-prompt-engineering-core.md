# 指令工程核心原则：行动优先、批量读取和正确性导向

**TL;DR：** Codex 的实际表现不只由模型能力决定，更取决于你给它的指令质量。一条好的指令遵循四个原则：用动词开头告诉模型"做什么"而不是"怎么看"、让模型先并行读取相关文件再动手、宁可多花一轮验证也不省略测试、用四要素（Goal / Context / Constraints / Done-when）描述任务边界。掌握这些原则，同样的模型在同一个仓库里的任务完成率可以从不到 50% 提升到 80% 以上。

## 行动优先原则：告诉模型"做什么"

Codex 的训练目标决定了它的行为偏好。与聊天模型不同，Codex 被训练为一个执行代理（acting agent），核心能力是"做"而不是"说"。这意味着你的 prompt 应该直接描述期望的执行结果，而不是请求模型的意见或分析。

### 差的 prompt 和好的 prompt

差的 prompt 通常以问句或模糊请求开头：

```text
差："你能看看 auth/login.ts 这个文件吗？"
```

模型收到这条指令后，会阅读文件并解释其内容。它可能还会分析潜在问题、给出改进建议。但这些都是"说"，不是"做"。任务结束后，文件没有任何变化，问题没有被修复，你还需要再发一条指令让模型动手。

好的 prompt 直接描述具体的执行目标：

```text
好："在 auth/login.ts 的 handleTokenExpiry 函数中，修复 token 过期后没有清除缓存的 bug。
    具体问题：第 47 行调用了 invalidateToken() 但没有调用 clearCache()。
    修复后在 handleTokenExpiry 的最后添加 clearCache() 调用，然后运行
    pnpm test -- auth/login.test.ts 确认测试通过。"
```

区别在于：好的 prompt 包含了动作（修复）、目标位置（handleTokenExpiry 函数）、具体问题（缺少 clearCache 调用）和验证方式（运行测试）。模型收到后直接执行代码修改，然后运行测试确认。一轮对话完成任务。

### 行动优先的 prompt 结构

一个行动优先的 prompt 通常包含四个部分，按照以下顺序组织：

```text
[动词] + [目标文件/模块] + [具体要求] + [验证方式]
```

每个部分的含义和作用：

| 部分 | 作用 | 示例 |
|------|------|------|
| 动词 | 明确动作类型 | 修复、重构、添加、删除、更新、迁移 |
| 目标文件/模块 | 精确定位操作范围 | auth/login.ts 的 handleTokenExpiry 函数 |
| 具体要求 | 描述期望的变更细节 | 在 invalidateToken() 后添加 clearCache() 调用 |
| 验证方式 | 定义完成标准 | 运行 pnpm test -- auth/login.test.ts |

动词选择直接影响模型的行为模式。对比几组常见动词的效果：

| 动词 | 模型行为 | 适用场景 |
|------|---------|---------|
| 修复 | 定位问题 → 修改代码 → 验证 | Bug 修复、已知问题 |
| 重构 | 理解现有结构 → 调整组织 → 保持行为 | 代码整理、模式迁移 |
| 添加 | 理解现有模式 → 在正确位置插入新代码 | 新功能、新测试 |
| 分析 | 阅读代码 → 输出结构化分析结果 | 代码审查、架构理解 |
| 迁移 | 理解源结构 → 生成目标结构 → 验证等价性 | 框架升级、语言迁移 |

注意"分析"是唯一一个"说"的动词。它的适用场景是理解代码阶段，不是执行变更阶段。混淆这两类动词是 prompt 效率低下的常见原因。

### 什么时候该"问"而不是"做"

行动优先不意味着永远不让模型分析。以下场景适合用"问"的句式：

- **不确定问题在哪**：你只知道现象（"登录偶尔失败"），不知道根因。这时候让模型先调查再动手。
- **方案需要探索**：存在多种可行方案，需要先评估利弊。例如"这个模块的性能瓶颈在哪，有几种优化方案"。
- **理解现有代码**：接手一个不熟悉的模块，需要先建立理解。例如"分析 payment/ 目录的模块划分和依赖关系"。

关键原则是：**先问后做是合理的，但不要混在一起**。一条指令要么让模型分析（Ask 模式），要么让模型执行（执行模式）。"你觉得这段代码有没有问题，有的话帮忙修一下"这种混合指令容易导致模型花大量 token 在分析上，最后给的修改草率且未验证。

正确做法是拆成两步：

```text
第一步（Ask 模式）：
"分析 auth/login.ts 的 handleTokenExpiry 函数，列出所有潜在问题，
每个问题给出严重程度和修复建议。"

第二步（根据分析结果）：
"修复刚才发现的第一个问题（token 过期后未清除缓存）。
修改后运行 pnpm test -- auth/login.test.ts 确认测试通过。"
```

## 批量并行读取策略：先搜索后新增

Codex 的每次工具调用都有 Token 成本，每次交互轮次都有时间成本。让模型一个文件一个文件地顺序读取，是最常见的效率浪费。

### 先搜索后新增的核心逻辑

写新代码之前，模型必须理解现有代码。这不是可选步骤——不理解现有模式的代码一定会与代码库风格不一致，甚至破坏现有功能。但"理解现有代码"不等于"读取所有文件"。正确的策略是：**先让模型并行读取最相关的几个文件，形成足够的上下文理解，再动手写**。

这个策略的执行方式：

```text
1. 识别任务涉及的关键文件（3-5 个）
2. 在一条 prompt 中让模型并行读取所有文件
3. 模型在内部形成完整的上下文理解
4. 基于理解执行修改
5. 运行验证
```

### 并行工具调用的机制

Codex 支持 `parallel_tool_calls`，允许模型在一次推理步骤中同时调用多个工具。当你一次性给出多个读取目标时，模型会并行发起多个读取请求，而不是排队等待。

实际效果对比：

```text
串行读取（反模式）：
  轮次 1: "读取 auth/login.ts"               → 模型读文件 A
  轮次 2: "现在读取 auth/token.ts"            → 模型读文件 B
  轮次 3: "还有 auth/cache.ts 也读一下"       → 模型读文件 C
  轮次 4: "好了，开始修复"                     → 模型终于动手
  总计：4 轮对话，3 轮浪费在读取上

并行读取（推荐）：
  轮次 1: "修复 auth/login.ts 中 handleTokenExpiry 的缓存清除问题。
          先并行读取以下文件了解上下文：
          - auth/login.ts（目标文件）
          - auth/token.ts（token 管理模块）
          - auth/cache.ts（缓存模块）
          - tests/auth/login.test.ts（现有测试）
          读取完成后，修复问题并运行测试。"
  总计：1 轮对话，模型内部并行完成所有读取
```

并行读取不只在时间上更高效。更重要的是，模型在同一个上下文中同时看到所有相关文件，能更准确地理解模块间的依赖关系和调用模式。串行读取时，模型先看到文件 A 的内容，到读文件 C 时可能已经丢失了文件 A 的关键细节（尤其在上下文窗口压力大时）。

### 反模式：逐文件读取

逐文件读取不仅浪费轮次，还会导致更严重的问题——**上下文碎片化**。当模型每次只看一个文件时，它对系统的理解是片面的。这会导致生成的代码与已有模式不一致：

```text
典型反模式对话：

用户: "给 payment/ 模块加一个退款功能"
Codex: "好的，让我先看看 payment/ 目录的结构"
       [读取目录结构]
用户: "继续"
Codex: "让我看看 payment/service.ts"
       [读取 service.ts]
用户: "继续"
Codex: "看看 payment/types.ts"
       [读取 types.ts]
用户: "好了，开始写吧"
Codex: [基于碎片化理解写出不一致的代码]
```

这个对话的问题不是模型能力不足，而是输入方式不对。模型每次只看到一页，就像让一个工程师每次只能看一个文件来理解整个模块——不可能写出与现有代码一致的实现。

正确做法是在第一条 prompt 中就给出完整上下文：

```text
"在 payment/ 模块中添加退款功能。

先并行读取以下文件了解现有模式：
- payment/service.ts（主服务，了解现有的支付和查询方法模式）
- payment/types.ts（类型定义，了解接口约定）
- payment/handlers.ts（HTTP 处理器，了解路由和请求处理模式）
- tests/payment/service.test.ts（测试，了解测试模式）

退款功能的业务规则：
- 只有已完成的支付订单可以退款
- 退款金额不能超过原支付金额
- 退款需要在原支付记录上关联

遵循现有代码风格和错误处理模式。
完成后运行 pnpm test -- payment/ 确认所有测试通过。"
```

### 控制并行读取的文件数量

并行读取不是越多越好。一次读取太多文件会挤占上下文窗口，留给推理和生成的预算就少了。实际建议：

| 任务复杂度 | 并行读取文件数 | 总 Token 预算占比 |
|-----------|-------------|-----------------|
| 简单 Bug 修复 | 2-3 个 | 10-20% |
| 中等功能开发 | 3-5 个 | 20-35% |
| 大型重构任务 | 5-8 个 | 30-50% |

如果任务需要的上下文超过 8 个文件，应该考虑使用 Plan 模式先规划，再分阶段执行。不要试图在一次会话中加载整个代码库。

## 正确性优先于速度

这是一个关于优先级的原则。当"快速完成"和"保证正确"冲突时，应该选择后者。

### 为什么正确性优先

Codex 的一次代码修改如果引入了 bug，后续的定位、修复和验证成本通常是原始修改的 3-5 倍。这不是理论推算——以下是典型的返工链路：

```text
省略验证的代价链：

  模型修改代码（省了 1 轮验证）           → 节省 ~30 秒
  → 修改引入隐蔽 bug
  → 你在后续开发中发现异常                 → 浪费 ~15 分钟排查
  → 让模型重新定位和修复                    → 浪费 ~5 分钟
  → 再次验证                               → ~30 秒
  总成本：~20 分钟（vs 正确流程的 ~30 秒验证）
```

这不是极端情况。模型生成的代码在语法层面通常是正确的，但在语义层面（业务逻辑、边界条件、并发安全等）容易出错。这些错误在简单测试中不会暴露，但会在生产环境中引发问题。

### 验证步骤不能省略

每一条涉及代码修改的指令，都应该包含验证要求。最低限度的验证链路：

```text
代码修改 → Lint 检查 → 类型检查 → 运行相关测试 → 确认通过
```

在 prompt 中，这对应一个明确的验证指令：

```text
"修改完成后，按顺序运行以下命令确认：
1. pnpm lint --fix（Lint 检查）
2. pnpm typecheck（类型检查）
3. pnpm test -- auth/（运行相关测试）
如果任何一步失败，分析原因并修复，然后重新验证。"
```

对于更严格的场景（涉及数据库、API、安全相关代码），验证链路应该更长：

```text
严格验证链路：

  代码修改
  → Lint + 类型检查
  → 单元测试
  → 集成测试（如适用）
  → 手动验证清单（在 prompt 中列出）
  → 检查是否有副作用（搜索受影响的调用方）
```

### 宁可多一轮对话

"多花一轮验证"是正确性优先原则的具体体现。实际操作中，这可以表现为：

**在执行指令中明确要求验证**：

```text
"修复 auth/login.ts 的 token 过期处理。修复后必须运行测试确认。
如果测试失败，分析错误原因、修复代码、重新运行测试，直到全部通过。
不要在测试还没跑完的时候就告诉我'修复完成'。"
```

**使用 Plan 模式处理复杂任务**：

```text
Plan 模式的价值：

用户: codex --plan "重构 payment 模块的错误处理，统一使用 Result 类型"

Codex 输出计划：
  1. 读取现有错误处理模式（并行读取 5 个文件）
  2. 设计 Result 类型定义
  3. 逐模块迁移（先 payment/service.ts → payment/handlers.ts）
  4. 每个模块迁移后运行测试确认
  5. 更新所有调用方
  6. 最终全量测试

用户确认计划后，Codex 按步骤执行，每步都有验证。
```

Plan 模式用一轮对话换来了结构化的执行计划。这一轮"浪费"的 Token 会在后续执行中成倍收回——因为模型不需要在执行过程中反复回退和修正。

## 四要素任务描述法

好的指令不是越长越好，而是每个部分都有明确作用。Codex 工程化实践提炼出四个要素：Goal（目标）、Context（上下文）、Constraints（约束）、Done-when（完成条件）。这四个要素构成一个完整的任务描述，模型拿到后能独立执行到完成。

### Goal：明确目标

Goal 是一句话描述任务要达成什么。好的 Goal 满足三个条件：具体、可执行、可验证。

```text
差的 Goal：
  "改善登录模块的代码质量"

好的 Goal：
  "修复 auth/login.ts 中 handleTokenExpiry 函数的 token 过期后
   缓存未清除问题，确保过期 token 的所有关联缓存都被清理"
```

"改善代码质量"不可执行——模型不知道什么算"改善"。后面那个 Goal 可执行：模型知道要做什么（修复缓存清除问题），也知道什么算完成（过期 token 的所有关联缓存都被清理）。

### Context：相关背景

Context 提供模型完成任务所需的关键信息。内容选择遵循"必要且充分"原则——太多会挤占上下文窗口，太少会导致模型理解不足。

应该包含的 Context：

| 信息类型 | 说明 | 示例 |
|---------|------|------|
| 相关文件路径 | 任务涉及的关键文件 | auth/login.ts, auth/cache.ts |
| 依赖关系 | 文件间的调用和依赖 | login.ts 调用 cache.ts 的 clearUserCache |
| 已知问题 | 已经确认的问题描述 | invalidateToken() 调用后 clearCache() 未被触发 |
| 特殊约定 | 项目特有的编码约定 | 错误处理使用 Result 类型，不抛异常 |

不应该包含的 Context：

| 信息类型 | 为什么不该包含 | 正确做法 |
|---------|-------------|---------|
| 大段代码粘贴 | 挤占上下文窗口 | 给文件路径，让模型自己读 |
| 全量 API 文档 | 模型已有通用知识，只需要项目特有约定 | 只写与项目不同的部分 |
| 冗长的背景故事 | 与任务执行无关 | 只保留影响技术决策的背景 |
| 其他模块的实现细节 | 增加噪音 | 只给与当前任务直接相关的文件 |

### Constraints：约束条件

Constraints 明确告诉模型"不要做什么"和"必须遵循什么"。这是防止模型过度发挥的关键机制。

```text
常见的约束声明：

- 不修改 auth/token.ts（只改 auth/login.ts）
- 不引入新依赖
- 保持现有的错误处理模式（使用 Result 类型，不抛异常）
- 不改动公共 API 的签名
- 性能要求：查询响应时间不超过 50ms
- 兼容性要求：必须支持 Node.js 18+
```

没有 Constraints 的任务，模型倾向于用自己认为"最好"的方式实现。但"最好"不一定适合你的项目——它可能引入你不想要的依赖、破坏现有的编码模式、或者修改了你不想动的文件。Constraints 把模型的创造力限制在安全范围内。

### Done-when：完成条件

Done-when 定义什么样的结果算"完成"。这是防止模型过早宣布完成的关键机制。

```text
差的 Done-when：
  "代码修改完成"

好的 Done-when：
  - [ ] pnpm lint 无错误
  - [ ] pnpm typecheck 通过
  - [ ] pnpm test -- auth/ 全部通过
  - [ ] 手动确认：token 过期后调用 clearCache，缓存确实被清除
  - [ ] 不影响正常登录流程（已有测试覆盖）
```

Done-when 的核心价值是**可客观验证**。"代码修改完成"是主观判断——模型改了代码就可以声称完成。但"测试通过"是客观事实——要么通过要么不通过，不存在灰色地带。

### 任务描述模板

将四个要素组合成一个标准模板。在团队中统一使用这个模板，可以显著降低"指令质量参差不齐"导致的效率波动。

```markdown
## Goal
[一句话描述要达成什么]

## Context
- 相关文件：[路径列表]
- 依赖关系：[文件间的关系]
- 已知问题：[已确认的问题或 bug 描述]

## Constraints
- 不修改 [文件/模块]
- 使用 [特定方法/库/模式]
- 保持 [特定行为或兼容性]

## Done-when
- [ ] [Lint / 类型检查 / 测试] 通过
- [ ] [功能验证方式]
- [ ] [副作用检查]
```

实际使用示例：

```markdown
## Goal
在 payment/refund.ts 中实现退款金额校验逻辑，确保退款金额不超过原支付金额。

## Context
- 相关文件：
  - payment/refund.ts（退款服务，需要添加校验逻辑）
  - payment/types.ts（RefundRequest 类型定义）
  - payment/repository.ts（数据访问，获取原支付金额）
  - tests/payment/refund.test.ts（已有测试框架）
- 依赖关系：refund.ts 调用 repository.ts 的 getPaymentById 获取原支付信息
- 已知问题：当前退款流程缺少金额上限校验，可能出现超额退款

## Constraints
- 不修改 payment/types.ts 中的类型定义
- 不引入新的 npm 依赖
- 错误处理使用现有的 AppError 类（定义在 utils/errors.ts）
- 遵循现有 refund.ts 中的 async/await 模式

## Done-when
- [ ] pnpm lint 无错误
- [ ] pnpm typecheck 通过
- [ ] pnpm test -- payment/refund.test.ts 全部通过
- [ ] 新增测试覆盖：退款金额等于原金额（通过）、退款金额超过原金额（拒绝）、原支付不存在（拒绝）
```

## OpenAI 推荐的七个应用场景

OpenAI 官方指南中列出了 Codex 的七个高价值应用场景。每个场景对应不同的指令策略和验证方式。理解这些场景，能帮助你在遇到实际任务时快速选择正确的 prompt 模式。

### 场景一：理解代码

让模型阅读并解释代码，是 Codex 最基础也最被低估的用法。很多开发者跳过这一步直接让模型修改代码，结果模型在不完全理解的情况下做出错误的修改。

理解代码的 prompt 关键是**结构化输出要求**。不要让模型自由发挥，而是指定输出格式：

```text
"分析 src/core/engine.ts 模块的结构。输出格式：

1. 模块职责：一句话描述这个模块做什么
2. 公共接口：列出所有 export 的函数/类，每个用一句话说明
3. 依赖关系：这个模块依赖哪些内部模块，被哪些模块依赖
4. 关键数据流：描述主要的数据处理链路
5. 潜在风险：你认为需要特别注意的代码段和原因

不要修改任何文件。"
```

这个 prompt 的优点：明确指定输出格式（5 个部分），限制了模型的输出范围（减少无效信息），最后强调"不要修改任何文件"（防止模型自作主张动手）。

### 场景二：重构和迁移

重构任务的核心挑战是**保持行为不变**。好的 prompt 需要明确"改什么"和"不改什么"，并强调测试验证。

```text
"将 src/utils/http.ts 中的 fetch 调用从 callback 风格迁移到 async/await。

约束：
- 不改变任何公共函数的签名和行为
- 不改变错误处理的语义（什么错误对应什么返回值必须保持一致）
- 保持现有的重试逻辑和超时配置

步骤：
1. 先并行读取 src/utils/http.ts 和 tests/utils/http.test.ts
2. 理解所有 callback 的调用模式
3. 逐函数迁移，每个函数迁移后确认测试仍然通过
4. 全部迁移后运行完整测试套件

Done-when：
- [ ] 所有函数使用 async/await
- [ ] pnpm test 全部通过
- [ ] 没有改变公共 API"
```

### 场景三：性能优化

性能优化任务需要**基准测试**。没有基准测试的优化是不可验证的，模型可能做了毫无意义的改动却声称"性能提升"。

```text
"优化 src/search/index.ts 中的全文搜索性能。

当前问题：当文档数量超过 10 万条时，搜索响应时间从 <100ms 增长到 >2s。

步骤：
1. 读取 src/search/index.ts，分析当前搜索算法的时间复杂度
2. 在 tests/search/benchmark.test.ts 中创建基准测试，测量当前性能
3. 实现优化（考虑倒排索引、缓存热点查询、减少不必要的字符串操作）
4. 运行基准测试确认性能提升
5. 运行功能测试确认搜索结果不变

Done-when：
- [ ] 10 万条文档的搜索响应时间 < 200ms
- [ ] 搜索结果与优化前完全一致
- [ ] 内存增长不超过 50%"
```

### 场景四：测试覆盖

补全测试是 Codex 的强项，因为它天然遵循"先读后写"模式——模型先读源代码理解行为，再写测试验证行为。

```text
"给 src/validators/email.ts 补充测试。

先读取以下文件：
- src/validators/email.ts（了解所有导出函数和验证规则）
- tests/validators/（了解现有测试的模式和工具函数）
- tests/helpers/（了解测试辅助工具）

测试要求：
- 覆盖所有导出函数
- 每个函数至少覆盖：正常输入、边界值、异常输入
- 使用项目现有的测试工具（不要引入新的测试库）
- 测试命名：describe('函数名') + it('should [行为] when [条件]')

Done-when：
- [ ] 测试覆盖率 > 90%（运行 pnpm test:cov 验证）
- [ ] 所有测试通过
- [ ] 没有跳过或禁用的测试"
```

### 场景五：加速开发

小到中型的功能开发是 Codex 使用频率最高的场景。关键是在 prompt 中给足上下文，让模型遵循项目现有的模式和约定。

```text
"在 src/features/user-preferences/ 目录下实现用户偏好设置功能。

先并行读取以下文件了解现有模式：
- src/features/user-profile/service.ts（了解 feature 模块的组织模式）
- src/features/user-profile/types.ts（了解类型定义约定）
- src/features/user-profile/handlers.ts（了解 HTTP 处理器模式）
- src/db/schema.ts（了解数据库 schema 约定）

功能要求：
- 用户可以设置主题（light/dark/system）、语言（zh/en/ja）、通知偏好
- 偏好设置存储在 user_preferences 表
- 提供 RESTful API：GET /preferences（读取）、PUT /preferences（更新）
- 更新操作需要验证字段类型和值范围

遵循 user-profile 模块的组织模式（service → types → handlers）。

Done-when：
- [ ] 目录结构和代码风格与 user-profile 模块一致
- [ ] API 端点可以正常请求和响应
- [ ] pnpm test -- user-preferences/ 通过
- [ ] pnpm lint 无错误"
```

### 场景六：保持心流

这个场景解决的是**上下文切换**问题。当你在专注开发某个功能时突然需要处理一个小任务（修个小 bug、查个配置、回答同事的技术问题），Codex 可以帮你快速处理，避免打断心流。

```text
"我在开发支付功能，但 CI 报了一个 lint 错误需要马上修。

错误信息：
  src/middleware/auth.ts:42:5
  Error: 'session' is defined but never used. (@typescript-eslint/no-unused-vars)

快速修复这个 lint 错误。不要改动其他任何东西。修完后运行 pnpm lint 确认。"
```

这种场景的 prompt 特点是**极度聚焦**：只给错误信息、只要求修这一个错误、明确说"不要改动其他任何东西"。不需要给 Context 和复杂的 Done-when，因为任务足够小，过度结构化反而浪费时间。

### 场景七：探索和构思

探索场景是唯一不需要模型"做"的场景。目标是让模型帮你评估方案的可行性、发现潜在问题、或者生成多种备选方案。

```text
"评估在当前项目中引入 WebSocket 实现实时通知的可行性。

先读取以下文件了解当前架构：
- src/index.ts（应用入口，了解 HTTP 服务器配置）
- src/middleware/auth.ts（认证中间件，了解鉴权机制）
- package.json（当前依赖和 Node.js 版本）
- docker-compose.yml（基础设施配置）

输出格式：
1. 架构适配：当前 HTTP 服务器能否直接升级支持 WebSocket？需要哪些改动？
2. 认证方案：WebSocket 连接如何复用现有的 JWT 认证？
3. 部署影响：对现有的 Docker 和反向代理配置有什么影响？
4. 替代方案：除了 WebSocket，还有什么方案可以实现实时通知？各自的利弊？
5. 工作量估算：实现完整实时通知功能需要修改哪些模块？

不要修改任何文件。这是一个纯调研任务。"
```

## 常见反模式

理解正确做法只是解决问题的一半。另一半是识别和避免常见的错误模式。以下是 Codex 使用中最常见的五个反模式。

### 反模式一：过度提示

给模型太多无关背景信息，挤占上下文窗口，导致模型在噪音中丢失关键信息。

```text
过度提示的典型表现：

"我们是一个中型 SaaS 公司，团队有 12 个开发者，使用敏捷开发方法论，
每两周一个 sprint。我们的技术栈是 Next.js + TypeScript + Prisma + PostgreSQL，
部署在 AWS 上，使用 Terraform 管理基础设施。最近我们在做支付模块的重构，
因为之前的实现太耦合了。我的同事张三负责前端，李四负责数据库迁移。
现在我要你帮我修一个 bug：auth/login.ts 第 42 行的 token 过期处理有问题..."

问题：前 6 句话与修复 bug 完全无关，但每句都在消耗上下文窗口。
```

判断标准：如果你的 prompt 中有一段内容删掉后不影响模型的执行结果，那段内容就不应该存在。

### 反模式二：跳过验证

改完代码不跑测试，或者只在 prompt 里写了"测试通过"但没有实际要求模型运行测试命令。

```text
跳过验证的两种表现：

1. Prompt 完全没提验证：
   "修复 login.ts 的 token 过期 bug"
   → 模型改了代码就宣布完成，不跑测试

2. Prompt 提了验证但模型没执行：
   "修复 bug，确保测试通过"
   → 模型改了代码后回复"修改完成，测试应该能通过"
   → 注意"应该能通过"不等于"确认通过"
```

正确的做法是在 prompt 中指定具体的验证命令，并要求模型实际执行：

```text
"修复完成后运行 pnpm test -- auth/ 并报告结果。
如果测试失败，分析原因并修复，直到所有测试通过。"
```

### 反模式三：单线程思维

不让模型并行处理独立任务，导致效率低下。

```text
单线程思维的典型对话：

用户: "在 payment/ 和 notification/ 两个模块中添加日志中间件"
Codex: [处理 payment/]
用户: "好的，notification/ 呢？"
Codex: [处理 notification/]

正确做法：两个模块的修改是独立的，应该在一次 prompt 中一起描述：

"在 payment/ 和 notification/ 两个模块中添加日志中间件。

先并行读取：
- payment/service.ts, payment/handlers.ts
- notification/service.ts, notification/handlers.ts
- src/middleware/logger.ts（日志中间件）

理解现有模式后，同时在两个模块中添加日志中间件。
最后运行 pnpm test 确认两个模块的测试都通过。"
```

### 反模式四：上下文污染

在 prompt 中加载太多文件内容或依赖信息，导致有效信号被淹没。

```text
上下文污染的典型表现：

"这是我们的整个 API 文档（粘贴 2000 行），
这是数据库 schema（粘贴 500 行），
这是团队编码规范（粘贴 300 行）。
请帮我给 login 接口写个测试。"

问题：2800 行背景信息中，与"写 login 测试"相关的可能只有 50 行。
```

正确做法：给文件路径而不是文件内容，让模型自己按需读取。指定需要关注的范围：

```text
"给 src/api/auth/login.ts 的 POST /login 接口写测试。

读取 src/api/auth/login.ts 了解接口行为。
读取 tests/api/ 下的现有测试了解测试模式。

只需要覆盖登录接口的 5 种场景：
1. 正常登录（200）
2. 密码错误（401）
3. 账号锁定（423）
4. 参数缺失（400）
5. 请求频率限制（429）"
```

### 反模式五：过早自动化

工作流还没稳定就想自动化，把不成熟的流程封装成 Skill 或 Automation，导致自动化产出质量不稳定，还要花额外时间调试自动化本身。

```text
过早自动化的信号：

- 同一类任务只做过 2-3 次，就创建 Skill
- Skill 的执行成功率低于 70%
- Automation 触发后的产出经常需要人工修正
- 团队成员对"怎么做"还没有共识，就已经固定了流程

正确节奏：

1. 手动执行 5-10 次同类任务，观察模式
2. 确认 prompt 模板稳定（成功率 > 85%）
3. 写入 AGENTS.md 作为团队共享的指令模板
4. 模板被多人验证后，再考虑封装成 Skill
5. Skill 执行稳定后，再考虑 Automation"
```

## 原则的优先级和冲突处理

四个核心原则在大多数场景下是互补的，但在特定场景中可能冲突。以下是冲突处理指南。

### 行动优先 vs 正确性优先

冲突场景：你希望模型快速完成任务，但任务涉及关键路径代码，错误成本很高。

处理方式：**正确性优先**。行动优先解决的是"让模型动手而不是光说"的问题，正确性优先解决的是"确保动手的结果是对的"的问题。后者更重要。在关键路径代码上，可以先让模型 Plan（牺牲行动优先），规划完成确认后再执行（保证正确性）。

### 批量读取 vs 上下文污染

冲突场景：需要读取很多文件才能理解上下文，但又担心挤占上下文窗口。

处理方式：**按需加载**。第一轮只读取最核心的 3-5 个文件，执行修改。如果发现需要更多上下文，第二轮再补充读取。不要一次性加载所有可能相关的文件"以防万一"。

### 四要素的完整性 vs 小任务效率

冲突场景：每个任务都用四要素模板太繁琐，尤其是 5 分钟能完成的小任务。

处理方式：**按任务复杂度调节模板的完整度**。

| 任务复杂度 | 模板使用方式 |
|-----------|------------|
| 简单（1-2 步，单一文件） | 只写 Goal + 验证命令 |
| 中等（3-5 步，多个文件） | Goal + Context + Constraints + 验证命令 |
| 复杂（5 步以上，多模块） | 完整四要素 + Plan 模式 |

不需要每次都填满模板的每一个格子。关键是确认你没有遗漏影响执行结果的关键信息。

## 从原则到实践

掌握原则只是起点。在实际项目中持续应用这些原则，需要建立两个习惯。

第一个习惯是**每次任务执行后回溯 prompt 质量**。任务失败时，不要只看"模型哪里做错了"，还要问"我的指令哪里写错了"。70% 的 Codex 任务失败可以追溯到 prompt 质量问题：目标不明确、上下文不足、约束缺失、或者完成条件模糊。

第二个习惯是**沉淀有效的 prompt 模板**。当某个类型的任务反复出现（比如"给新模块补测试"、"修复 lint 错误"），把验证过的 prompt 模板写入 AGENTS.md 或团队文档。模板不是死板的脚本，而是一个起点——使用时根据具体任务调整参数（文件路径、约束条件、验证命令）。

这两个习惯的叠加效应是：随着时间推移，团队的 prompt 质量会系统性提升，任务完成率的方差会缩小，新成员也能快速达到老成员的使用水平。这就是 prompt 工程从个人技能升级为工程系统的过程。
