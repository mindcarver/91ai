# Codex vs Claude Code 选型：决策框架和双 Agent 工作流

**TL;DR：** Codex 和 Claude Code 不是"哪个更好"的替代关系，而是两种不同哲学的工程代理。Codex 走"委派-审查-批准"路线，适合验收标准清晰、可以放手让 agent 异步完成的任务；Claude Code 走"结对编程"路线，适合模糊需求、边调试边改方案、需要深度理解代码上下文的场景。选型的核心不是比较模型跑分，而是匹配任务特征到工具哲学。更进一步，两者可以组合成双 Agent 工作流——一个实现、另一个审查，或者一个规划、另一个执行——在各自擅长的环节发挥优势。本文提供完整的选型决策框架、基准对比、双 Agent 工作流模式和面向中国开发者的合规路径分析。

---

## 两个代理，两种哲学

Codex 和 Claude Code 都是 2025-2026 年 AI 编程代理赛道的核心产品，但它们的设计哲学从根上就不一样。理解这个差异是选型的起点，不是终点。

### Codex："委派-审查-批准"

Codex 的核心隐喻是"你是一个技术经理，Codex 是你的下属工程师"。你写清楚任务描述和验收标准，把任务交给 Codex，它在沙箱环境里自主执行，完成后你来审查结果、决定批准还是打回。

这个哲学体现在 Codex 的每个设计决策里：

- **内核级沙箱**。macOS 上用 Seatbelt（`sandbox-exec`），Linux 上用 Landlock + seccomp。这些是操作系统级别的进程隔离机制，Codex 进程即使被恶意 prompt 诱导，也无法突破沙箱边界。代价是可编程性低——你不能在沙箱里安装新的系统包、不能访问未授权的网络端点。
- **异步执行**。Codex Cloud 把任务提交到 OpenAI 基础设施上异步运行，你提交后可以做别的事。Codex App 支持同时管理多个并行线程。这种设计的前提假设是：任务描述已经足够清晰，不需要人在执行过程中不断干预。
- **AGENTS.md 跨工具可移植**。AGENTS.md 是 OpenAI 推出的开放标准，不仅 Codex 用，其他支持该标准的工具也能读取。它的设计目标是"写一次，到处用"——虽然目前生态还在早期。
- **Starlark Rules**。Codex 的命令策略引擎使用 Starlark 语言（Python 的一个安全子集），可以精确控制哪些 shell 命令允许自动执行、哪些需要人工审批、哪些直接拒绝。这是一个可编程的治理层。

### Claude Code："结对编程"

Claude Code 的核心隐喻是"你和一位资深工程师并排坐在同一台电脑前"。你们一起看代码、一起调试、一起做决策。Claude Code 时刻在你身边，你随时可以问它问题、让它解释代码、让它尝试修改、让它回退。

这个哲学同样体现在每个设计决策里：

- **26 个 Hook 事件**。Claude Code 的安全模型不是"关在笼子里"，而是"在每个关键节点都有检查点"。从 `PreToolUse`（工具调用前拦截）到 `PostToolUse`（调用后审计），从 `Notification`（消息通知）到 `Stop`（会话停止），你可以在这些 Hook 上挂自定义逻辑。逃逸抵抗不如内核沙箱——Hook 是应用层的，理论上可以被绕过——但可编程性高得多，你可以根据项目需要做精细化的安全策略。
- **交互式执行**。Claude Code 的默认模式是交互式的。它在终端里运行，你看着它一步步操作，随时可以打断、纠正、调整方向。这种模式不适合"提交后去睡觉"的异步任务，但非常适合"边做边想"的探索性工作。
- **CLAUDE.md 深度集成**。Claude Code 的指令系统与自身执行逻辑深度耦合。CLAUDE.md 支持 `@` 引用其他文件、支持条件加载、支持与 settings.json 的联动。可移植性不如 AGENTS.md，但在 Claude Code 生态内的集成度更高。
- **Hooks + permissions 治理**。Claude Code 通过 `.claude/settings.json` 管理权限白名单，通过 Hook 脚本做运行时策略。没有 Starlark 那样的专用治理语言，但胜在灵活性——任何能跑的脚本都能当 Hook 用。

### 哲学差异的工程含义

两种哲学没有绝对优劣，但适合不同的工作模式：

```text
Codex 模式：
  你写任务 → Codex 在沙箱里执行 → 你审查 diff → 批准或打回
  适合：验收标准明确、执行路径可预测、不需要中途调整方向

Claude Code 模式：
  你描述问题 → Claude 分析 → 你讨论 → Claude 尝试修复 → 你验证 → 继续调整
  适合：问题边界模糊、需要边探索边调整、执行路径不可预测
```

如果强行用 Codex 处理模糊 bug，你会发现它在错误的方向上跑得很远，审查 diff 时需要大量返工。如果强行用 Claude Code 处理结构化任务，你会发现在交互式模式下浪费了本可以异步利用的时间。

## 核心判断框架

选型的第一步不是比较功能列表，而是判断你手上的任务属于哪一类。以下框架来自 yingtu.ai 社区在 2025-2026 年间的实践共识。

### Codex 适合的任务特征

**验收标准可以写清楚的任务。** 你能把"完成条件"列成一条条检查项：lint 通过、测试全绿、特定文件被修改、特定函数签名不变。Codex 的 Plan 模式会先列出执行计划，你可以确认后再执行，降低偏航概率。

```markdown
## Codex 适合的任务示例

### 功能实现（验收标准明确）
- "给 User 模型添加 email_verified_at 字段，创建数据库迁移，更新相关 API 的请求/响应 schema"
- 验收标准：迁移文件生成、API schema 更新、`pnpm test` 通过、`pnpm lint` 通过

### 重构
- "把 src/auth/ 目录下的 callback 风格异步调用全部改为 async/await，保持公共 API 不变"
- 验收标准：无 callback 残留、公共接口签名不变、所有测试通过

### 测试补全
- "给 src/utils/validator.ts 的每个导出函数补充边界值测试，目标覆盖率 90%+"
- 验收标准：覆盖率报告显示 >=90%、所有新测试通过

### PR Review
- "审查 PR #42 的 diff，按安全/性能/可维护性分类输出 findings，每条附严重程度和建议修复"
- 验收标准：结构化输出、无遗漏文件、findings 有明确分类

### 云端异步任务
- "批量修复 issues 列表中标记为 good-first-issue 的前 10 个，每个生成独立 PR"
- 验收标准：10 个独立 PR、每个 PR 只修改相关文件、CI 通过
```

**重构和迁移任务。** 这类任务的特征是"改法明确、涉及文件多"。你知道要改什么（比如把所有 `var` 换成 `let`/`const`，或者把所有 `.then()` 换成 `async/await`），但手动改几百个文件太耗时。Codex 的并行工具调用和批量编辑能力在这里很高效。

**团队 PR review。** Codex Cloud 可以配置为在 PR 创建时自动触发审查。它在云端沙箱里读取 PR diff，输出结构化的 review 意见。这是一种"委派"模式——你不需要坐在那等审查完成。

### Claude Code 适合的任务特征

**模糊 bug 调试。** 你只知道"用户报告登录偶尔失败"，但不清楚是前端问题还是后端问题、是偶发还是必现、是最近的改动引入的还是一直存在的。这种任务需要边探索边假设边验证，Claude Code 的交互式模式正好匹配。

```text
Claude Code 调试模糊 bug 的典型流程：

1. 你："用户报告登录偶尔失败，错误信息是'网络超时'"
2. Claude：分析 auth 相关文件 → 发现 3 个可能的超时点
3. 你："先看 middleware 那个"
4. Claude：深入 middleware → 发现 token 刷新逻辑有竞态条件
5. 你："写个修复试试"
6. Claude：生成修复 → 运行测试 → 测试没覆盖这个场景
7. 你："补个测试"
8. Claude：补充测试 → 测试通过 → 但你发现另一个问题
9. ...
```

**陌生代码库探索。** 你刚加入一个新项目或者接手别人的模块，需要快速理解代码结构、数据流、依赖关系。Claude Code 可以在你本地终端里实时搜索、阅读和分析，你可以随时追问、要求深入某个分支、让它画调用链。

**边调试边改方案。** 有些任务在开始时就知道方案可能需要调整——你可能先试一种改法，跑测试发现问题比想象中复杂，需要换一种策略。Codex 在这种场景下很尴尬：它会在 Plan 阶段承诺一个方案，然后按计划执行，中途很难调整。Claude Code 天然支持"试一下、不行就换"的工作模式。

**架构探索和设计讨论。** 你需要评估"要不要从 REST 迁移到 GraphQL"、"要不要把单体拆成微服务"、"缓存层应该放在哪"。这类问题没有标准答案，需要结合项目具体情况讨论。Claude Code 的对话式交互比 Codex 的任务式交互更适合。

### 判断框架速查表

| 任务特征 | 推荐 | 原因 |
|---------|------|------|
| 验收标准可明确列出 | Codex | 委派模式 + Plan 先规划再执行 |
| 涉及文件多但改法统一 | Codex | 并行工具调用 + 批量编辑 |
| PR 自动审查 | Codex Cloud | 异步执行 + 结构化输出 |
| 测试补全 | Codex | 任务边界清晰 + 可量化验收 |
| Bug 现象模糊、根因不明 | Claude Code | 交互式探索 + 边调试边假设 |
| 陌生代码库理解 | Claude Code | 实时搜索分析 + 追问能力 |
| 方案可能需要中途调整 | Claude Code | 交互式纠正 + 即时方向切换 |
| 架构设计和技术选型讨论 | Claude Code | 对话式推理 + 多方案对比 |
| 大规模代码库重构 | Codex | 并行 subagents + 异步执行 |
| 安全审计和合规检查 | Codex | 内核沙箱 + 只读模式 + 审计日志 |
| 日常开发中的小修改 | 两者皆可 | 根据个人偏好选择 |

## 架构哲学对比

除了任务特征匹配，还需要从工程架构层面理解两者的差异。这些差异决定了你能做哪些定制、安全边界在哪里、长期维护成本如何。

### 全维度对比

| 维度 | Codex | Claude Code |
|------|-------|-------------|
| **哲学** | "委派-审查-批准"（hands-off） | "结对编程"（hands-on） |
| **安全模型** | 内核级沙箱（Seatbelt/Landlock+seccomp） | 26 个 Hook 事件（应用层拦截） |
| **逃逸抵抗** | 高（操作系统级隔离） | 中等（应用层检查点，理论上可绕过） |
| **可编程性** | 低（沙箱内操作受限） | 高（Hook 可挂任意脚本） |
| **指令系统** | AGENTS.md（跨工具可移植的开放标准） | CLAUDE.md（更深集成但绑定生态） |
| **治理语言** | Starlark Rules（Python 安全子集） | Hooks + permissions（任意脚本 + JSON 配置） |
| **并行能力** | subagents + Cloud 多线程 | 原生不支持并行 agent |
| **异步执行** | Codex Cloud 原生支持 | 不支持，必须交互式运行 |
| **IDE 集成** | VS Code / JetBrains / Cursor 等 | VS Code / JetBrains 终端内运行 |
| **MCP 支持** | 原生支持 | 原生支持 |
| **API 模型** | Responses API（SSE 流式） | Anthropic API（工具调用） |
| **上下文管理** | Compaction（自动压缩） | 自动上下文管理 |
| **核心优势** | 委派/审查工作流、并行执行、异步任务 | 深度重构、代码审查、可编程治理、交互式调试 |

### 安全模型的工程含义

安全模型的差异在团队使用场景中尤其重要。

Codex 的内核沙箱提供的是"硬隔离"。即使 prompt 注入攻击成功诱导模型执行恶意命令，沙箱也会在操作系统层面阻止未授权操作。这使得 Codex Cloud 的异步执行在安全上是可接受的——你不需要坐在那里监控每一步操作。

Claude Code 的 Hook 事件提供的是"软拦截"。你在应用层注册的 Hook 脚本可以在工具调用前检查参数、在调用后审计结果。理论上，如果一个精心构造的 prompt 绕过了 Hook 逻辑，或者 Hook 脚本本身有 bug，安全边界就可能被突破。但 Hook 的可编程性意味着你可以做更精细的策略——比如根据文件路径、命令参数、甚至当前 Git 分支来决定是否允许操作。

```json
// Claude Code Hook 示例：阻止在 main 分支上直接提交
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "if git rev-parse --abbrev-ref HEAD | grep -q main; then echo 'BLOCK: 不允许在 main 分支上直接操作' >&2; exit 1; fi"
      }
    ]
  }
}
```

```python
# Codex Starlark Rule 示例：同样目的
def rules():
    return [
        {
            "matcher": "git commit*",
            "action": "require_approval",
            "message": "提交操作需要人工审批",
        }
    ]
```

两种方式都能达到目的，但哲学不同。Codex 的 Rule 是声明式的——你告诉系统"这个命令需要审批"，系统负责执行。Claude Code 的 Hook 是命令式的——你写一段脚本检查条件，脚本自己决定是否放行。声明式更安全但不够灵活，命令式更灵活但需要自己保证脚本正确性。

### 指令系统的工程含义

AGENTS.md 和 CLAUDE.md 解决同一个问题：把项目知识持久化，让 agent 每次会话都能获取上下文。但实现路径不同。

AGENTS.md 追求可移植性。OpenAI 把它定义为开放标准，任何支持 AGENTS.md 的工具都能读取同一份文件。如果你团队同时使用 Codex 和其他工具（比如 Cursor、Windsurf），一份 AGENTS.md 可以减少维护成本。但 AGENTS.md 的功能相对简单——它就是一份 Markdown 文件，没有条件加载、没有 `@` 引用、没有与配置系统的联动。

CLAUDE.md 追求集成度。它支持 `@` 引用其他文件（`@RULES.md`、`@PRINCIPLES.md`），支持分层加载（全局、项目、子目录），支持与 `.claude/settings.json` 联动。如果你完全在 Claude Code 生态内工作，CLAUDE.md 能做更多事。但如果你某天想换工具，CLAUDE.md 的深度集成就成了迁移成本。

实际选择取决于团队工具栈。如果你只用 Codex 或只用 Claude Code，各自的原生指令系统就够了。如果你两者都用，AGENTS.md 作为通用层加上各自的专用配置层是更务实的做法。

## 基准对比

基准测试不是选型的唯一依据，但它提供了客观数据锚点。以下是截至 2026 年 5 月公开可查的关键基准数据。注意：基准测试衡量的是模型在标准化测试集上的表现，与真实工程场景的表现有差距。不要把基准分数当成实际任务完成率的预测器。

### SWE-bench Verified

SWE-bench Verified 测量的是"给定一个真实的 GitHub issue，agent 能否正确修复对应的 bug"。它是最接近真实工程场景的基准之一。

| 模型 | SWE-bench Verified |
|------|-------------------|
| Opus 4.7（Claude Code） | 87.6% |
| GPT-5-Codex（Codex Cloud） | 74.9% |
| GPT-5.4（Codex Cloud 最新） | 数据未公开 |

Opus 4.7 在这个基准上领先约 13 个百分点。这个差距在工程上的含义是：对于真实 bug 修复任务，Claude Code 背后的模型有更高的单次成功率。但要注意，SWE-bench 的任务通常有明确的 issue 描述，这恰好是 Claude Code 擅长的场景——理解模糊描述、定位问题、修复验证。

### Terminal-Bench 2.0

Terminal-Bench 测量的是"在终端环境中完成多步骤工程任务"的能力，更接近 Codex 的核心使用场景。

| 模型 | Terminal-Bench 2.0 |
|------|-------------------|
| GPT-5.4 | 75.1% |
| Opus 4.7 | 69.4% |

GPT-5.4 在终端任务基准上领先约 6 个百分点。这反映了 Codex 在结构化、多步骤、可规划任务上的优势——Plan 模式先列计划再执行，与终端环境的操作模式天然匹配。

### 如何解读基准差距

两个基准的结果看起来矛盾——Opus 在 bug 修复上领先，GPT 在终端任务上领先。但这恰恰说明选型不能只看一个维度：

- SWE-bench 偏向"理解问题 → 定位代码 → 精确修复"的能力，这需要深度推理和对代码上下文的细腻理解。
- Terminal-Bench 偏向"规划步骤 → 执行命令 → 处理输出 → 调整策略"的能力，这需要结构化规划和工具调用效率。

两者结合起来看，Opus 4.7 在需要深度理解的任务上更强，GPT-5.x 在需要结构化执行的任务上更强。这与前面的任务特征判断框架一致。

### 基准之外的因素

基准不测量的因素在实际选型中往往更重要：

- **指令系统质量**：AGENTS.md 或 CLAUDE.md 写得好不好，对实际表现的影响可能比模型差距更大。一个好指令可以让弱模型表现稳定，一个差指令可以让强模型频繁偏航。
- **工具生态**：Codex 的 MCP + Skills + Automations 三层封装 vs Claude Code 的 Hook + Skills 体系，哪个更匹配你的工作流？
- **安全需求**：你的代码是否涉及敏感数据？是否需要通过合规审计？内核沙箱 vs Hook 拦截的选择可能比模型跑分更关键。
- **团队规模**：单人使用和 50 人团队使用的选型标准完全不同。Codex 的标准化治理能力在团队规模放大时更有价值。
- **网络环境**：对于中国大陆开发者，API 访问路径是硬约束，可能直接排除某些选项。

## 双 Agent 工作流

选型不是"选一个、放弃另一个"。在实际工程中，Codex 和 Claude Code 可以组合成互补的双 Agent 工作流。核心原则是：让每个 agent 做它最擅长的事，用另一个 agent 做交叉验证。

### 模式 1：Codex 实现 → Claude Code 审查

这是最简单的双 Agent 模式。Codex 负责实现，Claude Code 负责审查。利用的是 Codex 的批量执行能力和 Claude Code 的深度代码理解能力。

```text
工作流步骤：

1. 人工写任务描述（四要素：目标、上下文、约束、完成条件）
2. 提交给 Codex 执行（使用 Plan 模式先确认计划）
3. Codex 生成代码修改和 diff
4. 人工快速浏览 diff（确认方向正确）
5. 将 diff 交给 Claude Code 做深度审查
6. Claude Code 输出结构化 findings：安全风险、性能问题、可维护性问题
7. 人工根据 findings 决定：接受、修改或打回
```

这个模式的优势在于"不同视角的交叉验证"。Codex 作为实现者关注的是"如何完成任务"，Claude Code 作为审查者关注的是"这段代码有什么问题"。两个 agent 的训练数据、推理模式和偏见不同，一个容易忽略的问题另一个可能发现。

工程上的关键规则：**Claude Code 在审查模式下只做只读操作**。它不能直接修改 Codex 的代码，只能输出 findings 和建议。修改决策权在人类手里。这避免了"两个 agent 互相覆盖"的混乱局面。

### 模式 2：Claude Code 规划 → Codex 执行

当任务复杂但执行路径明确时，先让 Claude Code 做分析和规划，再让 Codex 按计划执行。

```text
工作流步骤：

1. 用 Claude Code 分析代码库（交互式探索、理解架构、定位关键文件）
2. Claude Code 输出执行计划：哪些文件需要修改、修改顺序、预期影响范围
3. 人工审查计划，调整不合理之处
4. 将计划转化为 Codex 的任务描述（加入四要素约束）
5. Codex 按计划执行（Plan 模式 + 自动执行）
6. 人工审查最终 diff
```

这个模式利用了 Claude Code 的深度理解能力来做"侦察"，利用 Codex 的结构化执行能力来做"施工"。类比工程实践：先让资深工程师做方案设计，再让执行团队按图纸施工。

适用场景：大规模重构、跨模块修改、依赖升级。这些任务的共同特征是需要先深入理解现状，再按明确计划批量执行。

### 模式 3：五阶段工作流

对于高风险、高复杂度的任务，可以使用更严格的五阶段工作流。每个阶段由不同 agent 负责，人工在关键节点做决策。

```text
五阶段工作流：

阶段 1 - Claude 规划
  Claude Code 分析需求 → 生成技术方案 → 列出修改计划
  人工确认方案可行性

阶段 2 - Codex 审查计划
  Codex 以只读模式审查 Claude 的方案
  检查：遗漏的边界情况、与现有代码的冲突、测试覆盖缺口
  输出计划修正建议

阶段 3 - Claude 实现
  Claude Code 按修正后的计划实现代码修改
  人工审查关键修改点

阶段 4 - Codex 测试
  Codex 运行完整测试套件、检查 lint、验证构建
  补充 Claude 可能遗漏的边界测试
  输出测试报告

阶段 5 - Claude 精化
  Claude Code 根据测试结果精化代码
  处理 Codex 发现的边界情况
  最终输出经过双重验证的代码
```

这个工作流的开销很大——每个任务至少需要两轮 agent 执行和三轮人工审查。它只适合高风险任务：核心模块修改、安全相关代码、生产环境变更。对于日常的 bug 修复和功能开发，模式 1 或模式 2 就够了。

### 双 Agent 安全规则

无论使用哪种模式，都需要遵守以下安全规则：

**规则一：一个 Agent 实现，另一个只做审查或小范围修正。** 不要让两个 agent 在同一次任务中都可以做大规模修改。否则你会遇到"A 改了 B 又改回去"的死循环，或者两个 agent 各自向不同方向修改的混乱局面。

**规则二：人工审 diff 是必须的，不能省略。** 双 Agent 工作流不等于"全自动"。每次 agent 产出的 diff 都需要人工审查。审查的重点不是代码风格（那应该由 linter 处理），而是逻辑正确性和安全影响。

**规则三：审查模式的 agent 必须是只读的。** 无论用 Codex 审查还是 Claude Code 审查，审查 agent 不能有写权限。Codex 用 `--ask` 模式或 read-only 沙箱，Claude Code 在 Hook 中限制工具调用为只读操作。

**规则四：任务描述必须明确标注"实现"还是"审查"。** 不要给 agent 一个模糊的任务让它自己判断是应该写代码还是只分析。明确的角色定义减少歧义。

**规则五：版本控制是最终的回退机制。** 每次 agent 执行前，确保当前代码已经 commit。如果 agent 的修改有问题，`git checkout .` 或 `git revert` 是最快的回退方式。不要依赖 agent 自己的"回退"功能。

```bash
# 双 Agent 工作流的安全实践
# 步骤 1：确保干净的工作区
git stash  # 暂存未提交的修改
git checkout -b feature/agent-task-xxx  # 在新分支上工作

# 步骤 2：Codex 实现
codex "实现 XXX 功能，约束条件见 AGENTS.md"

# 步骤 3：commit Codex 的修改
git add -A && git commit -m "feat: Codex 实现 XXX 功能"

# 步骤 4：Claude Code 审查（只读模式）
claude --allowedTools Read,Bash:"git diff HEAD~1" "审查上一个 commit 的修改"

# 步骤 5：根据审查结果决定
# 如果没问题：合并到主分支
# 如果有问题：人工修改或让 Claude 做小范围修正（限制修改范围）
```

## 中国开发者的特殊考量

对于中国大陆的开发者，选型还需要考虑一个硬约束：API 访问路径。OpenAI 和 Anthropic 的第一方 API 都不官方支持中国大陆的直接访问。

### API 可用性现状

截至 2026 年 5 月：

| 服务商 | 直接访问 | 中国大陆合规路径 |

| OpenAI API | 不支持 | Azure OpenAI（世纪互联运营或国际版） |
| Anthropic API | 不支持 | AWS Bedrock（ap-southeast-1 新加坡 / ap-northeast-1 东京） |
| Google Vertex AI | 不支持 | Vertex AI（asia-east2 香港） |

Codex Cloud（SaaS 版本）的情况更复杂。它的任务提交、执行和结果返回都在 OpenAI 基础设施上完成。如果你的代码不能离开本地环境，Codex Cloud 可能不可用。替代方案是使用 Codex CLI 的 `exec` 模式在本地运行。

### 国内模型的现实评估

DeepSeek、千问（Qwen）、Kimi 等国内模型在代码生成能力上已经取得显著进步。但"代码生成"和"agent loop"是两个不同层次的能力：

- **代码生成**：给定一个明确的 prompt，生成一段代码。国内前沿模型在这个层次上已经接近或达到国际一流水平。
- **Agent loop**：自主规划任务步骤、调用工具、处理工具返回结果、根据结果调整策略、在多步骤中保持上下文一致性。这个层次需要模型在工具调用格式、多轮推理、错误恢复等方面有高可靠性。

截至 2026 年 5 月，完整的 agent loop 可靠性仍然以前沿模型（Opus 4.7、GPT-5.x）为优。国内模型在 agent loop 场景下的主要挑战包括：

1. **工具调用格式稳定性**：长会话中偶尔输出不合法的工具调用格式，导致 agent loop 中断。
2. **上下文一致性**：多步骤任务中，模型可能"忘记"前面的决策，导致后续步骤与前面矛盾。
3. **错误恢复**：当工具调用失败时，模型需要理解错误信息并调整策略。这个能力在复杂错误场景下不够稳定。
4. **指令遵循精度**：在长上下文窗口中，模型对 AGENTS.md 或 CLAUDE.md 中规则的遵循精度会随会话长度下降。

如果你选择使用国内模型，建议的适配策略：

```text
国内模型适配策略：

1. 使用适配器网关（如 LiteLLM、One API）统一接口
2. 将 AGENTS.md/CLAUDE.md 控制在 50 行以内（减少上下文压力）
3. 将大任务拆成更小的子任务（减少单次会话的步骤数）
4. 增加人工审查频率（每 2-3 个步骤审查一次）
5. 建立评测基准（用 20 个真实任务量化实际表现）
6. 关注模型更新日志（国内模型迭代速度快，可靠性在快速提升）
```

## 实际使用建议

以上是理论和框架层面的分析。以下是来自一线工程实践的具体建议。

### 第一条：不要用"哪个更好"来选

"Codex 好还是 Claude Code 好"是一个错误的问题。正确的问题是"这个任务适合哪个工具"。两个工具在各自擅长的场景下都很好，在各自不擅长的场景下都很差。选错工具比没有工具更糟糕——你会得到一个看起来合理但实际上偏航的结果。

### 第二条：从你已有的开始

如果你已经在使用 Codex，不需要因为读了这篇文章就切换到 Claude Code。反之亦然。先用好手上的工具，遇到它的边界时再引入另一个工具作为补充。工具切换有学习成本和流程调整成本，不值得为了"理论上更好"而频繁切换。

### 第三条：AGENTS.md 和 CLAUDE.md 可以共存

如果你同时使用两个工具，可以在仓库根目录同时维护 AGENTS.md 和 CLAUDE.md。两者内容可以重叠——都写构建命令和编码规范——但各自保留自己特有的配置（AGENTS.md 写 Codex 的 Rules 引用，CLAUDE.md 写 Claude Code 的 Hook 配置）。维护两份文件的代价不高，但换来的是两个工具都能获取完整上下文。

```text
项目根目录文件结构：

/my-project
  /AGENTS.md          ← Codex 读取（也作为通用项目说明）
  /CLAUDE.md          ← Claude Code 读取
  /.codex/
    config.toml       ← Codex 配置
    rules/            ← Codex Starlark Rules
  /.claude/
    settings.json     ← Claude Code 权限配置
    hooks/            ← Claude Code Hook 脚本
    skills/           ← Claude Code Skills
```

### 第四条：评测基准是最值得的投资

无论选哪个工具，花半天时间建立 20 个真实任务的评测基准。这些任务应该来自你的实际工作——你上周做过的任务、你知道下周要做的任务。用这 20 个任务分别测试两个工具，记录完成率、返工次数和耗时。

评测基准的价值不在比较两个工具谁更好，而在于：

- 量化你的指令质量（同模型、同任务，不同指令的表现差距有多大）
- 发现工具的边界（哪类任务成功率低于 70%，需要调整策略）
- 追踪改进效果（修改 AGENTS.md 或 CLAUDE.md 后，成功率是否提升）

### 第五条：工具是手段，流程是核心

Codex 和 Claude Code 都是工具。工具的能力边界在短期内会快速变化——模型升级、功能更新、新的竞品出现。但工程流程是相对稳定的：

- 任务描述的四要素（目标、上下文、约束、完成条件）不会因为换了工具就失效
- 验证闭环（lint → typecheck → test → review）不会因为换了工具就不需要
- 安全规则（人工审 diff、审查 agent 只读、版本控制兜底）不会因为换了工具就可以省略

投资流程建设比投资工具选择更有长期价值。先把流程跑通，再根据流程需要选工具，而不是反过来。

### 第六条：关注成本结构差异

两个工具的计费模型不同，对成本敏感的团队需要关注：

- **Codex**：按 Token 计费（通过 API Key），或者按 ChatGPT Pro/Team/Enterprise 订阅。Cloud 任务使用 Credits。subagents 并行执行会成倍消耗 Token。
- **Claude Code**：按 Anthropic API Token 计费。Opus 4.7 的 Token 单价高于 GPT-5.x，但单次任务成功率也更高。需要根据实际任务类型计算总成本——如果 Claude 的单次成功率比 Codex 高 20%，但 Token 单价贵 50%，在需要多次重试的复杂任务上，Claude 的总成本可能更低。

建议每个团队根据自己的任务分布做一次成本测算。用 20 个评测任务分别跑两个工具，记录每个任务的 Token 消耗和成功/失败次数，然后计算总成本。

### 第七条：团队采纳从低风险任务开始

不管选哪个工具，团队采纳都应该从低风险任务开始。不要一上来就让 agent 改核心模块。推荐的采纳阶梯：

```text
团队采纳阶梯（适用于两个工具）：

阶段 1：代码理解（只读）
  - 让 agent 解释代码逻辑、回答架构问题
  - 风险：零（不修改任何文件）
  - 目标：团队建立信任、熟悉交互模式

阶段 2：测试补全（低风险修改）
  - 让 agent 给已有代码补充测试
  - 风险：低（测试文件独立，不影响主代码）
  - 目标：验证 agent 的代码生成质量

阶段 3：Bug 修复（中风险修改）
  - 让 agent 修复明确的、有复现步骤的 bug
  - 风险：中（需要审查修改逻辑）
  - 目标：验证 agent 在真实 bug 修复场景的表现

阶段 4：功能实现（高风险修改）
  - 让 agent 实现新功能
  - 风险：高（需要设计审查 + 代码审查）
  - 前提：前三个阶段的成功率 >80%

阶段 5：自动化流程（生产级）
  - PR 自动审查、CI 集成、批量任务
  - 风险：高（自动化意味着问题会规模化）
  - 前提：前四个阶段稳定运行 2-4 周
```

## 结论

Codex 和 Claude Code 代表了 AI 编程代理的两种哲学：委派式和协作式。选型的核心不是比较功能列表或基准分数，而是匹配任务特征到工具哲学。

对于验收标准清晰、执行路径可预测的任务——功能实现、重构、测试补全、PR review——Codex 的"委派-审查-批准"模式更高效。对于问题边界模糊、需要边探索边调整的任务——模糊 bug、陌生代码库、架构设计——Claude Code 的"结对编程"模式更合适。

更进一步，两者可以组合成双 Agent 工作流。核心原则是让每个 agent 做它最擅长的事：Codex 做批量执行，Claude Code 做深度审查；或者 Claude Code 做分析规划，Codex 做结构化执行。安全规则是底线：一个 agent 实现、人工审 diff、另一个 agent 只做审查或小范围修正。

对于中国开发者，API 访问路径是硬约束。Azure OpenAI、AWS Bedrock、Google Vertex AI 是三条合规路径。国内模型在代码生成上进步显著，但完整 agent loop 的可靠性仍以前沿模型为优。建议通过适配器网关接入，并采取更保守的任务拆分和审查策略。

最终，工具是手段，流程是核心。任务描述的四要素、验证闭环、安全规则、评测基准——这些工程实践不会因为换了工具就失效。先把流程跑通，再根据流程需要选择和组合工具。
