# Skill 评测：欠触发、误触发和执行失败怎么修

**TL;DR：** Skill 的质量直接决定 Codex 工作流的可靠性。一个写得差的 Skill 比没有 Skill 还糟糕——它浪费 token、引入错误的执行路径、消耗调试时间。Skill 的故障模式分为三类：欠触发（该激活时不激活）、误触发（不该激活时乱激活）和执行失败（激活了但产出的结果不对）。每种故障的根因不同，修复方法也不同。系统化的 Skill 评测需要构造 10 个测试场景（5 个应该触发、5 个不该触发），计算精确率和召回率，目标精确率 > 90%、召回率 > 80%。调试工作流遵循"识别故障模式 -> 隔离根因 -> 定向修复 -> 回归测试"的循环。长期来看，Skill 需要指标追踪和定期审计；当修复成本持续高于收益时，应该果断退役而不是继续修补。

---

## 为什么 Skill 质量至关重要

### 一个坏 Skill 比没有 Skill 更费 token

Skill 不是零成本的抽象。每次模型在推理过程中加载一个 Skill，SKILL.md 的内容（触发条件、执行步骤、错误处理策略、示例）都会被注入到上下文窗口中。这意味着两件事：

第一，Skill 被加载时消耗 token，无论它最终是否被执行。一个 200 行的 SKILL.md 文件，按每行平均 15 个 token 估算，每次加载消耗约 3000 token。如果这个 Skill 被误触发（不该激活却激活了），这 3000 token 就纯粹浪费了。如果它被频繁误触发，浪费的 token 会迅速累积。

第二，Skill 的存在会影响模型的决策。当上下文中有一个 Skill 的完整定义时，模型会倾向于尝试使用它，即使当前场景并不完全匹配。这就像一个手握锤子的人，看什么都像钉子。误触发的 Skill 不仅浪费加载 token，还可能引导模型走上错误的执行路径，产生更多无效的 tool call，最终消耗的 token 远超 Skill 本身的体积。

用一个具体数字说明：假设一个 PR Review 的 Skill 每次加载消耗 3000 token，被误触发后模型尝试执行，调用 GitHub MCP 工具读取 diff、分析代码、生成报告，平均额外消耗 8000-15000 token。一次误触发的总成本约 11000-18000 token。如果这个 Skill 每天被误触发 3 次，一周的浪费约为 23000-54000 token，换算成实际费用取决于模型定价，但更关键的是这些 token 占用了上下文窗口，挤占了其他有效信息的空间。

### 没有评测就没有质量保证

大部分团队写 Skill 的流程是这样的：根据需求写 SKILL.md，跑一两个场景确认"能触发、能执行"，然后上线。问题在于，一两个场景只验证了 Skill 在理想条件下的行为，没有验证边界情况。欠触发和误触发问题通常出现在边界场景中——触发条件描述稍有偏差、文件路径匹配不够精确、上下文措辞恰好命中了不该命中的关键词。

系统化评测的核心思想是：把 Skill 的触发行为当作一个分类器来评估。每个输入场景对应一个期望结果（应该触发或不应该触发），Skill 的实际行为是预测结果。通过比较期望和实际，计算出精确率和召回率，量化 Skill 的触发质量。只有通过这种定量评估，才能发现那些隐藏在"大部分时候能用"背后的触发问题。

## 三种故障模式

### 欠触发（Under-triggering）：该来的时候不来

**定义**：场景完全符合 Skill 的预期使用范围，但 Skill 没有被激活。模型收到了一个本该由 Skill 处理的请求，但因为触发条件没有匹配上，模型用自己的通用能力去处理，产出的结果不如 Skill 定义的流程可靠。

**典型表现**：

```text
用户输入："帮我审查这个 PR，重点看看安全问题"
期望行为：触发 pr-review Skill，按 SOP 执行结构化审查
实际行为：模型直接用自己的方式审查，输出格式随意，检查项不完整
```

**根因分析**：

欠触发几乎总是触发条件配置不当导致的。具体原因分三类：

**原因一：file_pattern 太窄。** SKILL.md 的 frontmatter 中通过文件路径模式匹配来决定是否触发。如果模式定义得过窄，很多合法的文件路径无法匹配。

```yaml
# 欠触发示例：file_pattern 过窄
---
name: pr-review
trigger:
  file_pattern: "src/**/*.ts"
---

# 问题：只匹配 src/ 目录下的 .ts 文件
# 以下场景不会被触发：
# - 修改了 lib/auth/login.ts（lib/ 不在匹配范围内）
# - 修改了 src/config.ts（根目录下的文件不在 ** 匹配范围）
# - 修改了 src/utils/helpers.js（.js 文件不在匹配范围）
# - 修改了 src/types/auth.d.ts（.d.ts 文件不在匹配范围）
```

修复方式是拓宽匹配模式，确保覆盖所有合理的文件路径：

```yaml
# 修复后：覆盖更多路径和文件类型
---
name: pr-review
trigger:
  file_pattern: "**/*.{ts,tsx,js,jsx}"
---
```

**原因二：description 太模糊。** Skill 的 description 字段是模型判断"这个 Skill 是否适用于当前场景"的主要依据。如果描述太抽象，模型可能无法确认 Skill 是否匹配，从而选择不触发。

```yaml
# 欠触发示例：description 太模糊
---
name: code-review
description: "审查代码变更"
---

# 问题：描述太短、太泛，模型无法判断：
# - 审查什么类型的代码？前端？后端？全栈？
# - 审查哪些方面？安全？性能？风格？架构？
# - 什么时候该用这个 Skill？PR review？commit review？ad-hoc？
```

修复方式是提供具体、可匹配的描述，包含使用场景、触发关键词和能力范围：

```yaml
# 修复后：具体的描述
---
name: code-review
description: |
  对代码变更进行结构化审查。适用于以下场景：
  - 用户要求 review PR、审查代码、检查变更
  - PR 中的代码需要审查
  - 代码变更涉及安全、性能、架构方面的检查
  审查流程包含架构变更分析、逻辑变更检查、测试覆盖评估三个阶段。
  输出结构化的审查报告，包含问题列表和审查建议。
---
```

**原因三：when 条件过于严格。** 有些 Skill 在 frontmatter 中定义了多重触发条件，要求所有条件同时满足才触发。条件越多，联合概率越低，欠触发的风险越大。

```yaml
# 欠触发示例：多重条件联合
---
name: deploy-verification
trigger:
  file_pattern: "k8s/**/*.yaml"
  description_match: "部署验证|deploy.*verify"
  context_keywords: ["production", "kubernetes", "deployment"]
---

# 问题：三个条件必须同时满足
# 如果用户说"帮我验证一下刚才的部署"，但没有提 Kubernetes，
# context_keywords 条件不满足，Skill 不会触发
```

修复原则是减少条件的"与"逻辑，改用"或"逻辑或放宽条件：

```yaml
# 修复后：放宽触发条件
---
name: deploy-verification
trigger:
  file_pattern: "k8s/**/*.yaml"
  description_match: "部署验证|deploy.*verify|验证部署|检查部署状态"
---
# 去掉 context_keywords 条件，description_match 增加更多同义表述
```

### 误触发（Over-triggering）：不该来的时候乱来

**定义**：场景不在 Skill 的预期使用范围内，但 Skill 被激活了。模型在不需要这个 Skill 的上下文中加载了它，浪费了 token，还可能引导模型走入错误的执行路径。

**典型表现**：

```text
用户输入："帮我写一个用户注册表单"
期望行为：直接生成前端代码，不触发任何审查类 Skill
实际行为：pr-review Skill 被加载，模型尝试审查不存在的 PR
```

**根因分析**：

误触发是欠触发的镜像问题，根因恰好相反——触发条件太宽。

**原因一：file_pattern 太宽。** 匹配模式覆盖了不该覆盖的文件路径。

```yaml
# 误触发示例：file_pattern 太宽
---
name: pr-review
trigger:
  file_pattern: "**/*"
---

# 问题：匹配所有文件
# 即使是修改 README.md、更新 package.json，
# 也会触发 PR 审查 Skill
```

修复方式是收窄匹配范围，只覆盖 Skill 真正需要处理的文件：

```yaml
# 修复后：限定为源代码文件
---
name: pr-review
trigger:
  file_pattern: "src/**/*.{ts,tsx,js,jsx}"
  exclude_pattern: "src/**/*.test.{ts,tsx,js,jsx}"
---
```

**原因二：description 匹配了太多上下文。** Skill 的描述包含了一些常见词汇，导致在很多不相关的上下文中也被匹配。

```yaml
# 误触发示例：description 包含常见词
---
name: code-review
description: "审查代码，检查问题，提供建议"
---

# 问题：描述中的"检查"、"建议"等词太常见
# 用户说"检查一下这个配置文件有没有问题"（不是代码审查）
# 用户说"给我一些建议怎么优化这个查询"（不是代码审查）
# 都可能触发这个 Skill
```

修复方式是在描述中增加限定条件，明确不适用场景：

```yaml
# 修复后：增加限定条件
---
name: code-review
description: |
  对 Pull Request 中的代码变更进行结构化审查。
  仅在涉及 PR review、代码审查请求时触发。
  不适用于：一般性代码问题咨询、代码风格建议、
  单文件修改的快速检查、配置文件调整。
  关键触发词：review PR、审查代码、code review、PR review。
---
```

**原因三：缺少负向条件。** 只定义了"什么时候触发"，没有定义"什么时候不触发"。对于容易混淆的场景，显式排除比隐式期望更可靠。

```yaml
# 误触发修复：增加负向条件
---
name: deploy-verification
trigger:
  file_pattern: "k8s/**/*.yaml"
  description_match: "部署验证|deploy.*verify"
  exclude_context:
    - "本地开发"    # 本地开发不触发生产部署验证
    - "测试环境"    # 测试环境有独立的验证流程
    - "文档"        # 文档修改不触发部署验证
---
```

### 执行失败（Execution Failure）：来了但做不对

**定义**：Skill 正确触发了，模型按照 Skill 的步骤执行，但产出的结果不符合预期。可能是输出格式错误、步骤执行不完整、判断标准应用错误，或者是 Skill 引用的外部资源不可用。

执行失败和触发失败的本质区别：触发失败是"该不该来"的问题，执行失败是"来了之后怎么做"的问题。前者是 frontmatter 配置问题，后者是 Skill 正文内容的问题。

**典型表现**：

```text
Skill: pr-review
触发：正确（PR review 场景，Skill 被激活）
执行：模型按步骤执行，但输出结果不完整
  - 只做了逻辑变更审查，跳过了架构变更和测试覆盖审查
  - 或者报告格式混乱，不是 Skill 定义的结构化格式
  - 或者引用了不存在的 MCP 工具
```

**根因分析**：

**原因一：指令模糊。** SKILL.md 中的执行步骤描述不够精确，模型在执行时有多种理解方式，选择了错误的那一种。

```markdown
<!-- 模糊的指令 -->
## 执行步骤

1. 检查代码变更
2. 分析问题
3. 生成报告
```

这段描述每个步骤都有大量歧义。"检查代码变更"——检查什么方面？安全？性能？风格？全部？"分析问题"——按什么标准判断什么是问题？严重程度怎么分级？"生成报告"——什么格式？包含哪些章节？模型只能靠猜测，而猜测的结果不可控。

修复方式是把每个步骤写得精确到模型不需要做任何推断：

```markdown
<!-- 精确的指令 -->
## 执行步骤

### Step 1: 获取变更范围
Tool: github.get_pull_request_diff
Input: { owner: "{repo_owner}", repo: "{repo_name}", pull_number: {pr_number} }
Output: diff 字符串
Assert: diff 不为空
On Error: 报告获取失败，终止执行

### Step 2: 文件分类
Logic: 将 diff 中的文件分为以下类别
- 架构文件: 路径包含 /types/、/interfaces/、/config/，或文件名为 *.d.ts、*.schema.json
- 逻辑文件: 路径包含 /src/、/lib/，文件扩展名为 .ts、.tsx、.js、.jsx
- 测试文件: 路径包含 /test/、/tests/、/__tests__/，或文件名包含 .test.、.spec.
- 文档文件: 文件扩展名为 .md、.txt，或路径包含 /docs/
Output: 每个类别的文件列表

### Step 3: 架构变更审查
Condition: 仅当 Step 2 的架构文件列表非空时执行
Check Items:
- 接口定义变更是否与其他模块的类型引用兼容
- 配置变更是否包含破坏性参数调整（删除必填项、修改默认值）
- 新增依赖的许可证是否合规（GPL、AGPL 需标记）
Output: 问题列表，每个问题包含 [严重程度] [文件:行号] 描述

### Step 4: 逻辑变更审查
Condition: 仅当 Step 2 的逻辑文件列表非空时执行
Check Items:
- 空值/undefined 访问风险
- 未处理的异常和错误
- 资源泄漏（未关闭的连接、未释放的锁）
- 并发安全问题（竞态条件、死锁风险）
Output: 同 Step 3 格式

### Step 5: 测试覆盖审查
Condition: 仅当 Step 2 的逻辑文件列表非空时执行
Check Items:
- 变更的函数/方法是否有对应测试
- 测试是否覆盖了核心边界条件
- 是否存在只测实现不测行为的测试
Output: 覆盖评估（充分/部分/缺失）

### Step 6: 生成报告
Format: Markdown，包含以下章节（缺一不可）
- ## 摘要：PR 的一句话概述 + 变更影响范围
- ## 架构变更：问题列表（如无则写"无架构级变更"）
- ## 逻辑问题：问题列表（如无则写"未发现逻辑问题"）
- ## 测试评估：覆盖评估结果
- ## 建议：批准 / 需要修改 / 需要讨论，附理由
```

对比修复前后的差异：修复后的指令为每个步骤定义了输入、输出、判断标准和条件。模型不需要猜测"检查什么"和"怎么检查"，只需要按照定义执行。

**原因二：脚本或引用失效。** Skill 引用的外部脚本、MCP 工具或配置文件路径已经过时。

```markdown
<!-- 引用失效示例 -->
### Step 1: 运行 lint 检查
Run: `./scripts/lint.sh`   # 脚本已被迁移到 scripts/check/lint.sh
```

```markdown
<!-- MCP 工具失效示例 -->
### Step 2: 获取 PR diff
Tool: github.get_pr_diff    # 工具名已改为 github.get_pull_request_diff
```

这种问题在 Skill 刚创建时不会出现，但随时间推移、项目结构演变、MCP Server 升级后会逐渐冒出来。修复方式是定期审计 Skill 引用（后面会详细讲 Skill 审计流程）。

**原因三：指令过长导致注意力稀释。** SKILL.md 的正文部分过长，模型在执行时丢失了对关键指令的注意力。这和 AGENTS.md 的约 100 行原则一致——当指令文件过长时，模型无法对所有部分保持同等关注。

```text
Skill 指令长度 vs 执行准确率（经验观察）：
- < 50 行：执行准确率高，但可能覆盖不够
- 50-100 行：执行准确率和覆盖度的最佳平衡
- 100-200 行：准确率开始下降，模型开始遗漏后半部分的指令
- > 200 行：准确率显著下降，模型只关注开头和结尾的指令
```

修复方式是精简指令，遵循以下原则：

- 每个步骤只用一句话描述核心动作，不要用段落
- 判断标准用列表，不要用散文
- 删掉"注意事项"和"温馨提示"之类的非操作性文字
- 把复杂的多分支逻辑拆成独立的子 Skill

## Skill 评测方法论

### 构造测试场景集

评测 Skill 的触发质量需要一个结构化的测试场景集。建议为每个 Skill 构造 10 个测试场景：5 个应该触发 Skill 的正例（positive），5 个不该触发 Skill 的负例（negative）。

正例的构造原则：

```text
1. 标准场景：完全匹配 Skill 的核心使用场景
   例（pr-review）："帮我 review 这个 PR"

2. 变体表述：换一种说法表达同样的意图
   例："这个 PR 有什么问题？"
   例："审查一下 #247 的代码变更"

3. 部分匹配：请求只涉及 Skill 的部分功能
   例："检查一下这个 PR 的测试覆盖够不够"

4. 隐式需求：没有直接说要做审查，但上下文暗示需要
   例："我刚提了一个 PR，你能看看吗？"

5. 带限定条件的请求：有额外约束，但核心需求匹配
   例："帮我 review 这个 PR，重点关注安全问题"
```

负例的构造原则：

```text
1. 语义相近但类型不同
   例（不是 PR review）："帮我审查一下这个函数的时间复杂度"
   → 这是代码分析，不是 PR review

2. 包含触发词但意图不同
   例："帮我看看 review 流程怎么配置"
   → 用户在问如何配置流程，不是要执行 review

3. 不同操作的类似表述
   例："帮我检查一下这个 PR 能不能合并"
   → 这是合并冲突检查，不是代码审查

4. 无关但包含部分关键词
   例："写一个代码审查的自动化脚本"
   → 用户在写工具，不是在做 review

5. 完全无关场景
   例："帮我部署一下这个服务"
   → 部署操作，和代码审查无关
```

### 执行评测并记录结果

评测的执行方式是对每个测试场景，在隔离的会话中输入请求，观察 Skill 是否被触发。记录格式：

```text
| # | 场景描述 | 期望结果 | 实际结果 | 判定 |
|---|----------|----------|----------|------|
| 1 | "帮我 review 这个 PR" | 触发 | 触发 | TP  |
| 2 | "这个 PR 有什么问题？" | 触发 | 触发 | TP  |
| 3 | "检查测试覆盖" | 触发 | 未触发 | FN  |
| 4 | "看看 #247 的变更" | 触发 | 触发 | TP  |
| 5 | "review PR 重点关注安全" | 触发 | 触发 | TP  |
| 6 | "审查函数的时间复杂度" | 不触发 | 触发 | FP  |
| 7 | "review 流程怎么配置" | 不触发 | 未触发 | TN  |
| 8 | "检查 PR 能不能合并" | 不触发 | 未触发 | TN  |
| 9 | "写一个代码审查脚本" | 不触发 | 触发 | FP  |
| 10 | "帮我部署服务" | 不触发 | 未触发 | TN  |
```

其中：
- TP（True Positive）：应该触发且确实触发
- TN（True Negative）：不该触发且确实没触发
- FP（False Positive）：不该触发但误触发了（误触发）
- FN（False Negative）：应该触发但没触发（欠触发）

### 计算精确率和召回率

```text
精确率（Precision）= TP / (TP + FP)
含义：在所有触发的次数中，正确触发的比例
衡量：误触发的严重程度

召回率（Recall）= TP / (TP + FN)
含义：在所有应该触发的次数中，实际触发的比例
衡量：欠触发的严重程度

上表的数据：
TP = 4, FP = 2, TN = 2, FN = 1
精确率 = 4 / (4 + 2) = 66.7%
召回率 = 4 / (4 + 1) = 80.0%
```

### 评测质量目标

| 指标 | 及格线 | 目标值 | 说明 |
|------|--------|--------|------|
| 精确率 | > 80% | > 90% | 精确率过低意味着大量 token 浪费在误触发上 |
| 召回率 | > 70% | > 80% | 召回率过低意味着用户经常得不到 Skill 的帮助 |
| F1 分数 | > 75% | > 85% | 精确率和召回率的调和平均值 |

精确率和召回率之间通常存在此消彼长的关系。放宽触发条件可以提高召回率但降低精确率，收紧触发条件可以提高精确率但降低召回率。取舍原则：

- **精确率优先**：当 Skill 的执行成本高（加载大量 token、调用多个 MCP 工具）或执行后果严重（修改生产代码）时，宁可欠触发也不要误触发
- **召回率优先**：当 Skill 的价值高且用户不会手动指定 Skill 名称触发时，宁可多触发一些再在执行阶段判断是否适用

## 迭代调试工作流

### 调试循环

Skill 调试遵循一个标准的迭代循环：

```text
  识别故障模式
       |
       v
  隔离根因位置 ----> 无法定位？
       |                  |
       v                  v
  定向修复          增加 logging 或
       |            缩小测试范围
       v                  |
  回归测试               |
       |                 |
       v                 |
  通过？---- 否 -------->+
       |
      是
       |
       v
  更新评测记录
```

### 识别故障模式

第一步是确定 Skill 的故障属于哪种模式。判定方法：

```text
问题：Skill 在某个场景下行为不符合预期

问：Skill 被触发了吗？
├─ 没有触发
│   └─ 这是欠触发（Under-triggering）
│       → 检查 frontmatter 的触发条件
│
├─ 触发了，但这个场景不该触发
│   └─ 这是误触发（Over-triggering）
│       → 检查 frontmatter 的触发条件是否太宽
│
└─ 触发了，场景也对了，但执行结果不对
    └─ 这是执行失败（Execution Failure）
        → 检查 SKILL.md 正文中的指令内容
```

这个判定很重要，因为不同故障模式的修复位置不同。欠触发和误触发修 frontmatter，执行失败修正文。如果判断错了故障模式，会在错误的位置找原因，浪费调试时间。

### 隔离根因

确定了故障模式后，需要进一步隔离具体原因。

**欠触发根因隔离**：

```text
Step 1: 确认 file_pattern 是否匹配
方法：用 glob 工具测试 file_pattern 对测试场景中的文件路径是否能匹配
  - 能匹配 → file_pattern 不是原因，检查 description
  - 不能匹配 → 调整 file_pattern

Step 2: 确认 description 是否匹配
方法：把用户输入和 Skill 的 description 并排阅读，判断语义是否匹配
  - 匹配 → 可能是其他原因（如 Skill 加载优先级问题）
  - 不匹配 → 调整 description 的措辞

Step 3: 确认是否有多重条件的"与"逻辑问题
方法：逐个去掉非核心条件，看是否能触发
  - 去掉某条件后能触发 → 该条件过严，需要放宽
  - 去掉所有非核心条件仍不能触发 → 回到 Step 1 重新检查
```

**误触发根因隔离**：

```text
Step 1: 确认 file_pattern 是否过宽
方法：列出 file_pattern 匹配的所有文件路径，检查是否包含不该匹配的路径
  - 包含不该匹配的路径 → 收窄 file_pattern
  - 匹配范围合理 → 检查 description

Step 2: 确认 description 是否过于宽泛
方法：把误触发的场景描述和 Skill 的 description 对比，看是否有模糊匹配
  - 有模糊匹配 → 增加 description 的限定条件
  - 没有模糊匹配 → 检查是否需要增加 exclude_pattern

Step 3: 确认是否缺少负向条件
方法：分析误触发场景的共同特征，看是否能用一条规则排除
  - 能 → 增加 exclude_pattern 或 exclude_context
  - 不能 → 可能需要拆分 Skill
```

**执行失败根因隔离**：

```text
Step 1: 确认指令是否清晰
方法：找一个人（不是 Skill 作者）阅读 SKILL.md 的执行步骤，看能否无歧义地执行
  - 有歧义 → 精简和明确指令
  - 无歧义 → 检查外部依赖

Step 2: 确认外部依赖是否可用
方法：手动执行 Skill 引用的脚本、MCP 工具调用、文件路径，确认都可用
  - 有不可用的 → 更新引用
  - 全部可用 → 检查指令长度

Step 3: 确认指令是否过长
方法：检查 SKILL.md 正文的行数
  - > 100 行 → 精简指令，拆分子 Skill
  - < 100 行 → 可能是模型能力限制，考虑简化任务逻辑
```

### 常见修复模式

每种故障模式都有高频出现的修复手段。把这些修复手段标准化，可以加速调试。

**欠触发高频修复**：

| 修复手段 | 适用场景 | 示例 |
|----------|----------|------|
| 扩展 file_pattern | 文件路径遗漏 | `src/**/*.ts` 改为 `**/*.{ts,tsx}` |
| 增加同义词 | description 只有一种表述 | "PR review" 增加同义词 "审查代码、review PR、代码审查" |
| 减少与条件 | 多重条件过严 | 去掉 `context_keywords` 条件 |
| 增加隐式触发词 | 用户表述不直接 | 增加 "帮我看看" 等间接表述 |
| 降低匹配阈值 | 精确匹配太严 | 从精确匹配改为包含匹配 |

**误触发高频修复**：

| 修复手段 | 适用场景 | 示例 |
|----------|----------|------|
| 收窄 file_pattern | 匹配了无关文件 | `**/*` 改为 `src/**/*.{ts,tsx}` |
| 增加排除列表 | 特定路径不该匹配 | 增加 `exclude_pattern: "**/*.test.*"` |
| 明确不适用场景 | description 太宽泛 | 增加 "不适用于：一般代码咨询、配置调整" |
| 增加必要触发词 | 缺少强信号 | 要求用户请求中包含 "PR"、"pull request" |
| 拆分 Skill | 一个 Skill 混合了多种职责 | 拆为 pr-review 和 code-analysis |

**执行失败高频修复**：

| 修复手段 | 适用场景 | 示例 |
|----------|----------|------|
| 步骤结构化 | 指令模糊 | 为每步增加 Tool/Input/Assert/On Error |
| 更新引用 | 脚本路径或工具名变更 | 更新 SKILL.md 中的路径和工具名 |
| 精简指令 | SKILL.md 过长 | 删除非操作性描述，控制在 100 行内 |
| 增加示例 | 模型对输出格式不确定 | 增加 good-example 和 bad-example |
| 拆分子 Skill | 单个 Skill 逻辑过复杂 | 提取公共步骤为独立 Skill |

### 回归测试

每次修复后，必须对完整的 10 个测试场景重新执行评测，而不仅仅是修复前失败的那个场景。原因有两个：

第一，修复一个故障可能引入新故障。放宽 file_pattern 可能解决欠触发但引入误触发；增加排除条件可能解决误触发但引入新的欠触发。只有全量回归测试才能发现这种连带影响。

第二，评测数据需要更新。每次修复后重新计算精确率和召回率，记录变化趋势。这是一个增量积累的过程：

```text
Skill 评测记录：pr-review

v1.0（初始版本）：
  精确率 66.7% | 召回率 80.0%
  问题：误触发到代码分析和工具编写场景

v1.1（收窄 description + 增加 exclude）：
  精确率 83.3% | 召回率 80.0%
  问题：场景 9 仍误触发，场景 3 仍欠触发

v1.2（扩展 file_pattern + 增加 "测试覆盖" 关键词）：
  精确率 83.3% | 召回率 100%
  问题：场景 9 仍误触发

v1.3（增加 "不适用于：写工具" 限定）：
  精确率 100% | 召回率 100%
  通过评测目标
```

## Skill 指标追踪

### 建立指标看板

Skill 的质量不是一次评测就能保证的。随着项目结构变化、团队工作流调整、新 Skill 的加入，已通过评测的 Skill 也可能出现退化。长期维护 Skill 需要建立指标追踪机制。

核心追踪指标：

| 指标 | 采集方式 | 频率 | 阈值 |
|------|----------|------|------|
| 触发精确率 | 采样评测 | 每月 | > 90% |
| 触发召回率 | 采样评测 | 每月 | > 80% |
| 执行成功率 | 日志统计 | 每周 | > 85% |
| 平均 token 消耗 | 日志统计 | 每周 | 无固定阈值，追踪趋势 |
| 误触发成本占比 | 日志计算 | 每月 | < 10% 总 Skill token 开销 |

### 追踪实现方式

不需要复杂的工具。一个简单有效的做法是在项目的 Skill 目录中维护一个评测记录文件：

```text
skills/
  pr-review/
    SKILL.md
    EVAL.md              # 评测记录
    test-scenarios.md    # 测试场景集
  deploy-verification/
    SKILL.md
    EVAL.md
    test-scenarios.md
```

EVAL.md 的格式：

```markdown
# Skill 评测记录：pr-review

## 版本历史

### v1.3 (2026-05-10)
- 精确率：100%（5/5 正例全部正确触发，5/5 负例全部正确不触发）
- 召回率：100%（5/5 应触发的场景全部触发）
- 变更：增加 "不适用于：写工具" 限定条件
- 测试人：张三

### v1.2 (2026-05-03)
- 精确率：83.3%（1 个负例误触发）
- 召回率：100%
- 变更：扩展 file_pattern + 增加 "测试覆盖" 关键词
- 测试人：张三

### v1.1 (2026-04-26)
- 精确率：83.3%（1 个负例误触发）
- 召回率：80.0%（1 个正例欠触发）
- 变更：收窄 description + 增加 exclude
- 测试人：李四

## 下次评测时间
2026-06-10（距上次评测 1 个月）
```

### 触发指标退化的信号

以下情况出现时，应该立即重新评测 Skill，而不是等到定期评测周期：

```text
退化信号：
1. Skill 引用的 MCP 工具接口变更（版本升级、参数调整）
2. 项目目录结构重构（src/ 拆分为 packages/）
3. 新 Skill 上线后，旧 Skill 的触发行为异常（Skill 之间竞争触发）
4. 团队成员反馈 "Skill 好像不好使了"
5. Skill 的 token 消耗突然上升（可能是误触发频率增加）
6. Skill 执行日志中出现新的错误类型
```

## 何时退役一个 Skill

### 退役的判断标准

不是所有 Skill 都值得持续维护。以下条件满足两条或以上时，应该认真考虑退役：

```text
退役条件清单：
[ ] 连续 3 次定期评测无法同时达到精确率 > 80% 且召回率 > 70%
[ ] 过去一个月内被修复了 5 次以上，每次修复后不超过一周又出问题
[ ] Skill 的实际使用频率极低（每月触发不到 2 次）
[ ] Skill 的功能已被其他更通用的 Skill 或模型自身能力覆盖
[ ] 维护 Skill 的成本（调试时间 + token 浪费）超过它节省的时间
```

### 退役不是删除

退役一个 Skill 不意味着删除它。推荐的做法是把退役的 Skill 移到归档目录，保留其评测记录和历史版本：

```text
skills/
  _archived/
    legacy-code-review/    # 已退役，被 pr-review 替代
      SKILL.md
      EVAL.md
      RETIREMENT.md        # 退役原因和替代方案
  pr-review/              # 替代者
    SKILL.md
    EVAL.md
```

RETIREMENT.md 记录退役决策的理由：

```markdown
# Skill 退役记录：legacy-code-review

退役日期：2026-04-01
替代方案：pr-review (v2.0+)

退役原因：
1. 触发精确率连续 3 个月低于 80%，最佳调整后仅 77%
2. 与 pr-review 的功能重叠度 > 70%，维护两套成本高
3. 月触发次数 < 2，团队成员已习惯使用 pr-review
4. Skill 正文引用的 MCP 工具已有 2 个 deprecated

替代迁移方案：
- 将 legacy-code-review 中独有的 "安全扫描" 步骤合并到 pr-review
- 更新 AGENTS.md 中的 Skill 引用
- 通知团队成员统一使用 pr-review
```

### 避免过早退役

有一种情况需要注意：Skill 的故障不是 Skill 本身的问题，而是外部环境变化导致的临时性问题。比如 MCP Server 升级后工具名变更，导致 Skill 执行失败。这种情况不应该退役 Skill，而是更新 Skill 中的工具引用。

判断原则：如果修复成本是一次性的（更新引用、调整路径），就不应该退役。如果修复成本是持续性的（频繁需要调整触发条件、反复出现执行失败），才考虑退役。

## 总结

Skill 的质量工程是 Codex 工程化体系中容易被忽视但实际成本很高的环节。一个有触发问题的 Skill 不是一个"偶尔不灵"的工具，而是一个持续浪费 token 和消耗调试时间的隐患。

核心要点：

1. **三种故障模式对应不同的修复位置**：欠触发和误触发修 frontmatter 的触发条件，执行失败修 SKILL.md 正文的指令内容。判断故障模式是调试的第一步。

2. **10 场景评测法是性价比最高的验证手段**：5 个正例 + 5 个负例，计算精确率和召回率，目标精确率 > 90%、召回率 > 80%。不需要更复杂的工具，只需要有意识地构造覆盖面够广的测试场景。

3. **调试是迭代循环**：识别故障模式 -> 隔离根因 -> 定向修复 -> 回归测试。每次修复后全量回归，避免修复一个问题时引入新问题。

4. **长期维护需要指标追踪**：定期评测、记录指标趋势、监控退化信号。当维护成本持续高于收益时，果断退役而不是继续修补。

5. **Skill 指令遵循约 100 行原则**：过长的 SKILL.md 会导致模型注意力稀释，执行准确率下降。精简、结构化、可操作是指令撰写的核心原则。

Skill 评测和调试不是一次性工作，而是 Skill 生命周期中的持续工程活动。把它当作代码的测试来对待——不测的代码不可信，不评测的 Skill 同样不可信。
