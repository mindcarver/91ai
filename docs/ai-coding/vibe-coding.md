## 什么是 Vibe Coding

2025 年 Andrej Karpathy 提出了这个词。他的意思不是"随便聊几句就能做出好软件"，而是：**程序员的关注点从写代码转移到引导 AI 写代码**。

用一句话区分三种模式：

| 模式 | 程序员做什么 | AI 做什么 |
| --- | --- | --- |
| 传统编程 | 写代码 | 不参与 |
| AI 辅助编程 | 写代码，AI 补全和建议 | 补全当前行、解释选中代码 |
| Vibe Coding | 描述意图、审查输出、迭代方向 | 写代码、跑测试、处理细节 |

Vibe Coding 不是"不写代码"，而是写代码的比例从 80% 降到 20%。剩下 80% 是：想清楚要什么、跟 AI 沟通清楚、检查 AI 做得对不对、决定下一步往哪走。

### 什么不是 Vibe Coding

- ❌ 在 ChatGPT 里问"帮我写一个 Todo App"，复制粘贴到文件里——这叫**一次性生成**，不是迭代式开发
- ❌ 在 IDE 里按 Tab 接受所有补全建议——这叫**被动接受**，没有方向感
- ❌ 没有任何验证（测试、预览、部署）就认为"做好了"——缺少反馈循环

关键区别是**有没有迭代循环**：描述 → AI 生成 → 验证 → 调整 → 再描述。没有这个循环，AI 生成的代码大概率不可用。

## 实战框架

不管用什么工具、做什么项目，Vibe Coding 的核心循环是一样的：

```
1. 定义边界          → 这个项目做到什么程度算"完成"？
2. 选工具            → 根据项目类型选最合适的 AI Coding 工具
3. 写初始 prompt     → 给 AI 足够的上下文和明确的交付物
4. 迭代              → 验证 → 描述问题 → AI 修复 → 再验证
5. 收束              → 修 bug、加细节、确认边界内的功能都完成
```

下面逐个展开。

### 1. 定义边界

最常见的失败不是 AI 写不出代码，而是你不知道什么时候该停。

**好的边界定义：**

- "做一个单页落地页，包含标题、3 个功能介绍、CTA 按钮，部署到 Vercel"
- "做一个命令行工具，接收 CSV 文件路径，输出按日期排序的 JSON，支持 `--output` 参数"
- "做一个 Chrome 扩展，在 GitHub PR 页面自动标记超过 500 行的文件"

**坏的边界定义：**

- "做一个电商网站"（太宽泛，永远不会"完成"）
- "做一个比 XX 更好的产品"（无法验证）
- "做一个 app"（什么 app？什么平台？什么功能？）

**判断标准：** 如果你能用一句话说清楚"做完了长什么样"，边界就够了。如果不能，先缩小范围。

### 2. 选工具

不同项目类型适合不同工具。没有万能工具。

| 项目类型 | 推荐工具 | 原因 |
| --- | --- | --- |
| 静态网站 / 落地页 | [Cursor](./cursor.md)、v0.dev、Bolt.new | 前端生成是当前 AI 最擅长的方向，所见即所得迭代快 |
| 命令行工具 / 脚本 | [Claude Code](./claude-code.md)、[Codex](./openai-codex.md) | 终端工具天然适合 CLI Agent，文件操作和命令执行一体化 |
| 全栈 Web 应用 | Cursor、[Cline](./cline.md) | 需要 IDE 上下文理解，前后端同时编辑 |
| Chrome 扩展 / 浏览器插件 | Claude Code、Cursor | 需要理解 manifest 和文件结构，Agent 模式处理多文件更可靠 |
| 数据处理 / 自动化脚本 | Claude Code、ChatGPT | 单文件为主，逻辑清晰，生成质量高 |
| API 对接 / 集成 | Claude Code、Codex | 需要读文档、写配置、测试接口，Agent 工具调用有优势 |

**选型原则：**

- 有 IDE 需求 → Cursor / Cline（在编辑器里工作）
- 纯终端 / 脚本 → Claude Code / Codex（CLI 原生）
- 零代码 / 快速原型 → Bolt.new / v0.dev / Lovable（模板化生成）
- 不确定 → 先用你最顺手的那个，比选"最优"更重要

更详细的工具对比见 [AI Coding 工具总览](./README.md)。

### 3. 写初始 prompt

初始 prompt 决定了 AI 的方向。花 5 分钟写好 prompt，比花 30 分钟修方向偏了的结果更划算。

**一个好的初始 prompt 包含：**

```
项目目标：做什么东西、给谁用
技术栈：用什么语言、框架、库
项目结构：目录怎么组织（如果在意的话）
核心功能：必须有的 3-5 个功能
边界说明：明确不做的东西
风格要求：UI 风格、代码风格（如果在意的话）
```

**示例——做一个个人博客：**

```
帮我从零创建一个个人技术博客。

技术栈：Next.js + Tailwind CSS + MDX，部署到 Vercel。
项目结构：
  - app/ 下是页面路由
  - content/posts/ 下放 MDX 文章
  - components/ 放可复用组件

核心功能：
1. 首页展示最新 10 篇文章列表
2. 每篇文章有独立的 slug 路由
3. 支持标签分类，点击标签过滤文章
4. 暗色模式切换
5. RSS 订阅

不做：
- 评论系统
- 用户登录
- 后台管理
- SEO 优化（后续迭代再加）

UI 风格：简洁、大量留白、字体用 Inter，参考 danluu.com 的极简风格。
```

**反面示例——同样的项目，糟糕的 prompt：**

```
帮我做一个博客网站。
```

这种 prompt 的问题：AI 不知道用什么技术栈、做什么功能、做到什么程度。它可能给你一个 WordPress 方案，也可能给你一个 Hugo 方案，大概率不是你想要的。

### 4. 迭代

Vibe Coding 的核心不是第一次就写对，而是快速迭代。

**迭代节奏：**

1. **看结果** —— 跑起来、预览、测试，不要只看代码
2. **定位问题** —— 是方向错了（功能不对），还是实现有 bug（行为不对）
3. **描述调整** —— 告诉 AI 具体改什么、改成什么样
4. **验证修复** —— 确认改对了再进入下一轮

**常见迭代模式：**

| 迭代类型 | 你怎么说 | AI 做什么 |
| --- | --- | --- |
| 修 bug | "点击提交按钮后页面白屏了，控制台报 TypeError" | 定位问题并修复 |
| 调样式 | "导航栏在手机上溢出了，改成汉堡菜单" | 修改响应式布局 |
| 加功能 | "新增一个搜索框，实时过滤文章列表" | 在现有代码上添加功能 |
| 改方向 | "不用 MDX 了，改用数据库存文章" | 可能需要重构部分代码 |

**迭代中的技巧：**

- **提供错误信息** —— 把终端报错、浏览器控制台错误直接粘贴给 AI，比自己描述问题效率高 10 倍
- **截图比文字强** —— UI 问题直接截图，说"像左边这样"比描述 3 段话管用
- **一次改一个方向** —— 不要一次提 5 个不相关的修改，AI 处理并行修改容易遗漏
- **频繁提交** —— 每个成功的迭代都 `git commit`，方便回滚

### 5. 收束

项目接近完成时，需要从"加功能"切换到"收束"模式：

- **修已知 bug** —— 把积攒的小问题一次性处理
- **处理边界情况** —— 空状态、加载状态、错误状态
- **检查部署** —— 本地能跑不代表部署能跑，提前试
- **写 README** —— 至少写清楚怎么安装、怎么运行

## 什么时候不该 Vibe Coding

Vibe Coding 不是万能的。以下场景建议换方式：

| 场景 | 为什么不适合 | 该怎么做 |
| --- | --- | --- |
| 已有大型代码库的新功能 | AI 对现有架构理解不深，容易引入不一致的代码 | 用 AI 辅助写局部模块，手动集成 |
| 性能敏感的核心逻辑 | AI 生成的代码通常不优化到极致 | 自己写核心算法，用 AI 处理周边 |
| 安全关键代码（认证、加密） | AI 可能引入安全漏洞且难以发现 | 自己写 + 人工审计 |
| 需要精确控制每行代码的场景 | Vibe Coding 的价值在于减少对代码细节的关注 | 直接手写 |

简单的判断标准：**如果项目从零开始、核心复杂度在功能而非性能/安全、你有能力验证 AI 的输出**，适合 Vibe Coding。否则慎用。
