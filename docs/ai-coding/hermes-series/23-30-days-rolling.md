# 我用 Hermes 30 天（滚动栏目）

> 系列第 23 篇 · 读者预设：想知道真实长期使用体验、不只想看功能介绍的人 · 最后核实：2026-07

**TL;DR：** 这是滚动栏目，不是一次性文章。本篇是栏目开篇，讲清楚我打算怎么测、用什么场景、记录什么指标、什么算成功、什么算失败。后续按周和月更新真实数据：哪些 skill 真有用、哪些是噱头、token 成本实际多少、什么时候它帮倒忙。第 1 期还在积累，不是结论。长期实测需要时间，我不假装用了 30 天。如果你读这篇时发现第 1 周的内容还是空的，那是栏目刚开始的诚实状态，不是漏写。回来再看。

## 为什么单独开个滚动栏目

前面 22 篇做的都是机制拆解和工程指南，依据是官方文档、源码分析和社区讨论。这种写法在讲清楚它是什么、怎么工作这件事上有用，但回答不了它长期到底好不好用。Hermes 这种自托管的长期陪伴型工具，跟一次性跑个 prompt 的客户端不一样，它的核心卖点是会越用越懂你：skill 库会自动扩充，curated memory 会沉淀你的偏好和上下文，cron 任务会按你的节奏跑后台自动化。这些承诺只有在几个月的日常使用里才能验证。

具体要回答的问题包括：skill 库到底会不会越用越准，还是越攒越乱、最后变成一堆互相冲突的 markdown？curated memory 记进来的东西有多少是真有用的偏好，有多少是无意义噪音、甚至被错误事实污染？后台 self-improvement 的 token 成本真实是多少，是省了时间还是把预算烧在 Curator 和 GEPA 上了？哪些自动化任务真的省时间，哪些是花架子维护起来比手动做还累？这些只能用时间回答。

社区已经出现的反面案例值得警惕。Reddit 上 r/hermesagent 有用户写我的令人沮丧的 Hermes 体验，明确指出 memory 可能被无意义内容冲刷，导致后续对话漂移。另一个用户说维护了一个月 skills 和 memories，全是上下文膨胀、污染、幻觉，最后放弃。还有人说我的 Hermes agent 感觉非常蠢，反复训练但总是回归到不会用 skill 的状态。也有反例：一位用户分享使用 Hermes 的体验与疑问，长期使用同一个 profile 也没遇到 memory 问题，最大挑战反而是 skill 的配置要花心思。这些反馈加在一起勾勒出一幅复杂的图景：同样的工具在不同人手里结果差很多，单点体验推不出通用结论。这正是开滚动栏目的理由：必须自己跑，必须长期跑，必须记录原始数据。这些抱怨不是孤立现象，是长期使用后才暴露的问题，短期的 14 天测评根本看不见。所以我想开个滚动栏目，定期更新真实数据，而不是写一篇我用了 30 天感觉很棒的伪体验文。

更深一层的问题是：Hermes 这种自我改进型工具，它的失败模式是慢性病，不是急性病。一次 prompt 调用错了你立刻就知道；一个 skill 慢慢被改偏你可能在第二十次使用才隐约感觉不对劲，到第五十次才发现已经累积成问题。memory 的污染更隐蔽：错误事实一旦写进 curated memory，会在你完全不知情的情况下影响接下来所有对话。这种慢性问题没法靠用户主动发现，必须靠定期 review 和 git diff 这种被动观测。栏目的一个隐性目标是建立这套被动观测的习惯，看它能不能在不依赖用户主动警觉的前提下，把慢性问题挡在积累成灾难之前。

伪体验文是这种工具评测的重灾区。作者安装好之后跑两三个 demo 场景，看见 skill 自动创建成功、记忆写入成功、cron 任务按时触发，就写 30 天深度体验，效率提升 300%。这种文章的问题不在于数据夸张，而在于它根本没经历 30 天。skill 还没漂移、记忆还没污染、token 账单还没爆、自动化还没出诡异事故，所有长期问题都被时间挡在门外。读者看完以为找到了银弹，自己装上跑了两个月才发现要每天手动清理 memory。我不想写这种东西。

## 第 1 期状态（栏目开篇的诚实交代）

栏目开篇时，我自己还没积累满 30 天的使用数据。强行编我用了 30 天的发现等于欺骗。所以这一篇不讲结论，讲方法：怎么测、记什么、什么算成功。第一周跑通核心场景、建立 baseline；第二到第三周观察 self-improvement 开始生效，记录第一批自动创建的 skill 质量；第四周做第一次系统 review，看命中率、成本、事故；之后每月一篇更新，对比 baseline 看趋势。

如果你现在读这篇发现第 1 周 baseline 那一节内容是空的，说明栏目刚开始，我还没攒到那个时间点。这是滚动栏目的诚实状态，不是漏写。我也会在每篇更新顶部标明当时对应的天数和实际样本量，避免读者误以为这是大规模结论。

为什么不在第一周就发？因为第一周的数据没有信息量。skill 刚装好，命中率本来就是零或低；记忆刚初始化，没东西可污染；后台成本刚启动，还没爬到稳定状态。这种数据发出来只会误导。第一篇有信息量的内容至少要等到第三周：self-improvement 跑了几轮，第一批自动改 skill 的 diff 出来了，第一批 curated memory 写进去了，才能开始判断方向对不对。

栏目开篇这种只讲方法不讲结果的写法本身有风险。读者可能觉得那你写这篇干什么，等三周再来。这个质疑合理，但开篇仍有必要。原因有几个。第一，方法本身需要被审视：如果六维度的设计有漏洞、baseline 没建对、幸存者偏差对抗没做到位，越往后跑偏越多。开篇把方法摆出来，让读者提前挑刺，比三个月后回头看才发现方法错了更省时间。第二，开篇是承诺。把我会持续更新、会诚实记录失败、不会美化这些写出来，等于给自己上了一道软约束。后面某周想偷懒不更新，会想起开篇的承诺。第三，开篇是档案。后面读到任何一期更新的读者，都能回头查这篇搞清楚栏目一开始的设定是什么，避免读到第三期发现数据格式变了不知道是版本升级还是我自己改了方法。

## 测什么：六个维度的具体方法

栏目会持续记录六个维度。每个维度都要有可操作的测量方法，不能停在感觉变好了这种主观判断。

**第一个维度是 skill 命中率。** 测量方法是每周统计 skill 被自动调用和手动调用的比例。Hermes 有 `/skills` 命令可以列出当前 skill 库，配合 TG 客户端和 desktop 的实际对话日志，统计某个 skill 在多少次对话里被自动触发、多少次是我手动 `/use skill-name` 强制调用。成功标准不是命中率越高越好。命中率虚高有可能是 skill 写得太宽泛，把不该匹配的场景也匹配上了，反而是问题。我盯的是命中率曲线的趋势：稳定上升说明 skill 库在变好，剧烈波动或者持续走低说明 skill 改坏了。这个维度对应第 9 篇 skill 自我改进的观察方法。具体记录格式是每周一张表：skill 名称、本周被触发次数、自动触发占比、命中后是否真的解决了问题。最后一列特别重要，因为有些 skill 被触发了但输出没用，这种假命中比没命中更危险，它会让你误以为 skill 库在变好。

**第二个维度是 skill 质量。** 命中率高不等于改得对。每周抽样 review 一批 skill 的 git diff，看自动修补是改对了还是改偏了。具体做法是 `git log --since="1 week ago" -- skills/` 看本周的提交，然后挑出 5 到 10 个改动比较大的 skill，对比改动前后的语义。判断标准分三档：改对了（更通用、更鲁棒、修复了边界）、改偏了（变得更窄、更专用、引入了错误假设）、中性（格式调整、注释更新）。成功标准是改对了 / 总改动 > 70% 且没有错误固化。Reddit 上我的 Hermes agent 感觉非常蠢那条抱怨，很可能就是 skill 长期被改偏、回归到不会用的状态。这是我要重点盯的反面案例。review 的时候还要特别警惕一种隐性漂移：skill 的核心逻辑没变，但前置 prompt 被悄悄改了，导致执行风格偏移。比如 PR 审查 skill 的检查项没变，但 system prompt 里严格被改成了宽松，长期下来审查质量就滑坡了。这种漂移 git diff 看得见，但要细读才能识别，光看变更行数不行。

**第三个维度是记忆准确性。** Hermes 有两层记忆：curated memory（被 Curator agent 主动整理的）和 insights（自动从对话里抽取的）。每周用 `/insights` 命令扫一遍，配合 `git diff` 看本周新增的记忆条目。重点不是看条目数量，是看准确率：记下来的偏好是不是我真实表达的，记下来的事实是不是对的。成功标准是没有错误事实被长期沉淀。Reddit 上 memory 可以被无意义内容冲刷那条是直接警告：一旦错误事实进了 curated memory，它会持续影响后续所有对话，这就是所谓的内存中毒。第 11 篇详细讲了 curated memory 的干预机制，本栏目长期跟踪的是干预够不够、信噪比在什么水平。具体到测量上，每周分三类统计：准确（真实表达或真实事实）、噪音（无关紧要的细节，比如用户提到了 git 这种）、错误（记反了或编造的）。错误类的恢复成本要单独记，因为某些错误可能要回滚好几天甚至清空某条 memory 才能消除影响。curated memory 和 insights 要分开统计，因为这两层的污染途径和恢复难度不一样。curated 是被 Curator 主动整理过的，理论上信噪比应该高，但也意味着错误更隐蔽；insights 是自动抽取的，噪音多但用户警觉性也高。

**第四个维度是 token 成本。** 这是所有 self-improving agent 的隐藏账单。前台对话的 token 是显性的，用户能直接感受到；后台的 self-improvement、Curator 整理、nudge 提醒这些是隐性的，不知不觉就烧掉了。每周用 `/usage` 监控前后台消耗，把后台成本拆成几个来源：skill 创建和修补、memory 整理、cron 任务本身。成功标准是后台成本占比稳定且可控，不超过某个我自己设定的阈值（具体阈值第一周 baseline 后再定）。第 18 篇专门讲了成本监控，本栏目跟踪的是这个成本在 30 天、60 天、90 天的曲线，是平的、上升的、还是突然失控。这里有一个容易忽略的细节：成本不仅要算钱，还要算延迟。后台 self-improvement 跑的时候，前台对话的响应可能会变慢，这种体验损失不会体现在账单上，但直接影响工具的可用性。栏目会顺便记录这种隐性延迟：是不是后台跑 Curator 的时候前台就明显卡顿，是偶尔发生还是规律性发生。

**第五个维度是自动化收益。** 每个 cron 任务记录两件事：省了多少时间和维护成本是多少。省时间容易算：任务代替了我原本要手动做的多少分钟工作。维护成本难算但要记：包括修 cron 触发条件、改 skill 让它别再产生错误输出、清理它生成的脏数据。成功标准是省时大于维护。如果某个 cron 任务每周省 30 分钟但维护花 1 小时，那就是负收益，应该停掉。第 16 篇有一个 cron 任务的实际案例，本栏目会跟踪这类任务在长期使用里是持续正收益，还是慢慢变成维护负担。还有一个判断维度是如果我停掉这个 cron 任务一周，会不会真的想念它。如果停了之后没觉得生活变差，说明这个任务本来就可有可无，self-improvement 帮你创建的不一定都是你真需要的。这是用反向验证来对抗自动化惯性：人一旦习惯了某个自动化，就倾向于保留它，哪怕实际收益是负的。

**第六个维度是意外事故。** 静默失败、内存中毒、误操作的次数。这个维度的成功标准最简单：趋于零。但更重要的是记录事故的严重程度和恢复成本：是几分钟手动修一下，还是要回滚一整周的 memory 状态。事故分级我用三档：一级是轻微影响（单次对话答错，不影响后续）、二级是中等影响（污染了 memory 或 skill，需要手动清理但能恢复）、三级是严重事故（污染范围大，需要回滚多天状态或重置整个 profile）。Reddit 上那个维护一个月放弃的用户，本质上就是事故频率和恢复成本超过了收益。这个维度我特别想盯，因为短期的 14 天测评根本看不见事故，事故是积累出来的。还有一种隐性事故容易被忽略：skill 写得过于自信，把错的答案包装得像是对的。这种事故最难发现，因为表面上看 Hermes 给了答案、你接受了、对话继续了，但事实上你被误导了。栏目会专门盯这种高置信度错误，它在学习助手和 PR 审查场景里最危险。

## 用什么场景测：真实工作流不是构造的

测试场景必须是我真实会做的事，不能是为测而造的人造场景。人造场景的问题是它不反映真实分布：构造一个完美匹配某个 skill 的查询，命中率当然是 100%，但这不代表 skill 在真实对话里有用。

**第一个场景是日常查询。** 我日常会从 TG 随时问技术问题、查文档、问代码细节。这种场景的特点是高频、碎片化、上下文不完整。它考验 skill 库的覆盖度（有没有现成的 skill 能直接回答）和 memory 的连贯性（昨天聊过的东西今天还能不能接上）。这是 baseline 场景，也是判断 Hermes 整体有没有用的最直接窗口。日常查询还有一个隐藏价值：它是 memory 污染的主要来源。碎片化的提问很容易产生噪音，因为单条查询信息量小、容易被 Curator 误读成偏好。所以这个场景除了看命中率，还要特别看它产生的 memory 条目信噪比。如果日常查询把 memory 污染得厉害，那其他场景再准也救不回来。

**第二个场景是 PR 和代码审查。** 让 Hermes 按团队 checklist 审 PR，检查风格、安全、测试覆盖、潜在 bug。这个场景考验 skill 的执行严格度（每次审都按同一套标准、还是漂移）和长上下文处理（一个 PR 几百行 diff 能不能稳定看完）。它也是观察 skill 漂移的好场景：如果审 PR 的 skill 一周比一周宽松，那就是在漂移。

**第三个场景是每周简报。** cron 任务汇总上周的关键信息：issue 状态、PR 合并、commit 活动、社区讨论。这是第 16 篇讲过的案例。它考验 cron 的稳定性和 memory 的过滤能力（哪些信息重要、哪些是噪音）。简报类的任务最容易暴露自动化是省时间还是制造新工作：如果每周简报都需要我事后手动改，那就是负收益。判断标准我设了一个简单的：简报发到我面前，我读完之后有没有哦原来还有这事的瞬间。如果整篇简报我读起来都是这我早知道了，说明 cron 任务没在做信息过滤，只是在做信息堆砌，self-improvement 没帮上忙。这个判断标准主观，但比准确率这种指标更接近真实价值。

**第四个场景是学习助手。** 让 Hermes 帮整理学习笔记、对比概念、生成练习题。这个场景的特点是低风险但高频，适合做 skill 库的覆盖率测试。学习场景的好处是正确性容易验证：我自己懂这些概念，能直接判断 Hermes 输出对不对，不像代码审查有时要查文档才能确认。学习场景也是观察高置信度错误的好窗口：概念对比时 Hermes 容易把两个相似但不同的概念混着说，听起来头头是道但实际错了。这种错误如果进了 memory，长期下来会污染我对自己的学习状态的认知。所以学习场景特别要看 Hermes 写进 memory 的概念总结是不是干净的。

**第五个场景是写作辅助。** 让 Hermes 批评和优化草稿，包括这篇栏目本身。写作场景考验 long-form 上下文处理和 memory 的偏好学习：我写作的风格偏好记进去了没、改完是不是真的更合我意。这个场景也方便做自我观测：我把这篇栏目喂给 Hermes 让它批评，看它能不能识别出 AI 高频词和营销腔，这本身就是栏目的一部分。

这五个场景覆盖了 Hermes 的主要能力：记忆、skill、自动化、多平台。更重要的是，它们覆盖了不同的失败模式：日常查询考验覆盖度，PR 审查考验严格度，每周简报考验自动化稳定性，学习助手考验正确性，写作辅助考验偏好学习。一个工具在不同场景下的表现可以差很多，单场景的好不能推广。

## 更新节奏：四周加月度的具体安排

栏目不是每天写的，按阶段更新。每个阶段有明确的目标和判定点。

**第 1 周：建立 baseline。** 这一周不追求发现洞察，追求把基础数据记下来。具体动作：记录 skill 库初始状态（多少个 skill、什么类型、命中率初始值）、记忆初始状态（有没有预填的内容、初始条目数）、token 初始消耗（前台多少、后台 self-improvement 启动后多少）。这些数字本身没什么意义，但它们是后续所有对比的基准。第 18 篇讲成本监控时强调过，没有 baseline 就没有变好了还是变差了的判断依据。

**第 2 到 3 周：观察 self-improvement。** 这阶段开始有数据。第一批自动创建的 skill 出来了，要看它们的来源（什么场景触发的）、质量（有没有改对）、是否被实际命中。第一批 curated memory 写进去了，要看它们的准确性（记对了没）、信噪比（有用的多还是噪音多）。token 成本开始爬坡，要看后台 self-improvement 的曲线斜率。这个阶段的判定点是方向对不对：不是要求立刻有效果，但要确认没有立刻坏掉的迹象。

**第 4 周：第一次系统 review。** 一个月节点做完整复盘：命中率曲线、skill 质量统计、memory 信噪比、token 总账单、自动化收益盘点、事故清单。这次 review 是栏目第一个有信息量的内容，前面的更新都是铺垫。重点是对比 baseline 看变化的方向，而不是绝对数字。

**后续月度：每月一篇。** 30 天之后转入月度更新。每月一篇对比上个月的 baseline，看趋势：skill 命中率是平稳、上升还是下降；memory 信噪比是稳定还是恶化；token 成本是平的还是失控；事故频率是降低还是升高。月度更新的核心是趋势，不是单点数字。

这种节奏的取舍是：开篇和前几周内容看起来空，因为没有时间积累；但长期看，栏目积累的洞察比一次性水文值钱得多。我接受这个取舍。

## 已知要重点观察的几个怀疑点

基于前面 22 篇的分析和社区出现的反面案例，长期实测要重点盯这几个怀疑。这些不是凭空担心，是有具体证据的怀疑。

**怀疑一：skill 漂移。** 第 9 篇详细讲过 skill 自我改进的机制和它的风险。30 天后某个 skill 被自动修补了多少次？是改对了还是改偏了？Reddit 上我的 Hermes agent 感觉非常蠢那条抱怨，明确说反复训练后 agent 总是回归到不会用 skill，这很可能就是 skill 被改偏后丢失了原本的行为。重点盯的是 PR 审查、写作辅助这类需要严格度的 skill，看它们的执行标准是不是随时间漂移。还有一种特殊的漂移要警惕：skill 之间互相干扰。当 skill 库扩充到几十个之后，某些 skill 的语义开始重叠，触发条件冲突，结果就是同一个查询被不同 skill 抢着处理，输出风格分裂。这种漂移不容易从单个 skill 的 diff 看出来，要从对话日志层面才能识别：某类问题这周答得很一致，下周开始答得乱七八糟，往往是 skill 之间在抢主导权。

**怀疑二：curated memory 信噪比。** 第 11 篇讲了 curated memory 的 nudge 和干预机制。问题在于：记进来的东西有用的多还是噪音多？Reddit 上令人沮丧的体验那条直接说 memory 可以被无意义内容冲刷，导致漂移。维护一个月放弃那条说 memories 全是上下文膨胀、污染、幻觉。这是社区反复出现的痛点，必须重点跟踪。具体盯的是每周新增 memory 条目里，准确和有用的占比多少。这里还有一个判断难点：什么是有用本身是模糊的。比如用户偏好简洁回答这种条目，看起来有用，但如果你问的就是需要详细展开的问题，这条偏好反而帮倒忙。所以栏目要跟踪的不仅是信噪比，还有偏好类条目的副作用：记进去的偏好有没有让 Hermes 在不该简洁的场景变得过于简洁。

**怀疑三：后台 token 成本。** 第 18 篇讲成本监控。self-improvement、Curator、nudge 这些后台进程烧的 token 占总成本多大比例？有没有失控？这个怀疑的本质是 self-improving 的代价是不是超过了它带来的收益。30 天、60 天、90 天的曲线会说话。

**怀疑四：Issue #25833 的实际表现。** 第 1 篇讲过这个结构性问题：author、executor、reviewer 是同一个 agent，缺乏独立 reviewer 制衡。长期使用是不是真的导致 skill 越改越偏、没人拦得住？这个怀疑最难验证，因为它不是某个具体的 bug，是结构性的偏差。要靠对比第 1 周和第 12 周的 skill 质量这种长跨度数据才能看出来。

**怀疑五：内存中毒。** 第 20 篇讲过内存中毒的预防和恢复。30 天内有没有发现错的事实被记下来？记下来之后影响了多少后续对话？恢复成本多大？这个维度的事故一旦发生，影响范围是全局的：所有后续对话都会基于错误事实回答，比单条 skill 改偏严重得多。

这五个怀疑点不是要否定 Hermes，是要带着具体问题去观察，而不是泛泛地感觉变好了。每个怀疑都有对应的测量维度，每周更新里会专门有一节报这几个数。

## 真实坑：栏目开篇就预见到的几个

长期实测有几个方法论上的坑，开篇就讲清楚，避免后面掉进去。

**第一个坑是没有 baseline 没法判断。** 不记录初始状态，30 天后没法说变好了还是变差了。比如某天 skill 命中率从 60% 跌到 40%，如果第 1 周没记初始值，根本不知道 60% 是高还是低。所以第 1 周建 baseline 是关键。所有六个维度都要有初始数字，哪怕这些数字看起来没意义。

**第二个坑是幸存者偏差。** 只记录成功案例会美化结论。一个 skill 被命中了、一份简报生成了、一次查询答对了，这些都是成功。但如果只记这些，栏目会变成软文。失败、帮倒忙、误操作、生成错误内容这些必须如实记下来。具体做法是：每周强制至少记录一个这次 Hermes 帮倒忙的案例，没有也得找。这个约束是为了对抗只记好不记坏的自然倾向。

**第三个坑是样本量。** 30 天加单人使用，样本量小到不能推广。同一套 skill 在不同人手里表现可能完全不同：有人每天用 PR 审查、有人每天用学习助手，命中率曲线没可比性。所以栏目强调我的体验，不是通用结论。读者可以参考方法和维度，但具体数字不能照搬到自己的场景。

**第四个坑是测试者污染测试。** 我自己既是用户又是评测者，会有意识无意识地调整行为去配合 Hermes。比如知道某个 skill 应该被命中，提问时会下意识用更标准的措辞。这种污染没法完全消除，只能靠提问时不刻意匹配 skill 关键词这种自我约束缓解。还有一种更隐蔽的污染：我作为评测者，对 Hermes 的失败有更宽容的解读倾向。普通用户一次失败就放弃了，我会归因到这是长期实测要记录的案例，于是继续用。这意味着栏目的持续使用本身可能不代表普通用户的真实留存。要诚实标注这一点：栏目测的是在愿意持续投入维护的用户手里 Hermes 表现如何，不是在普通用户手里表现如何。

**第五个坑是版本漂移。** Hermes 是活跃开发的项目，30 天里可能升好几次版本。某个版本升级后命中率突然变了，可能是 skill 改进，也可能是底层模型变了。所以每次更新都要记录当前的 Hermes 版本和底层模型版本，避免把版本变化误判成 self-improvement 的效果。处理方法是每次大版本升级后单独建一个版本后 baseline，前后两段数据不直接对比，必须标注 4.2 到 4.3 之间数据不可比。这种细节枯燥但必要，否则栏目可能给出完全相反的结论：明明是版本回退导致的退步，被误判成 self-improvement 失败。

这些坑不是说不要做长期实测，是说做的时候要诚实标注边界。栏目的可信度建立在把这些坑讲清楚，而不是假装它们不存在。一个诚实的长期实测栏目，它的价值恰恰在于让读者看见测的是什么、测的不是什么、可信区间在哪、哪里要打折看。这种透明度比任何一句推荐都更有信息量，也是本栏目想立的标准。

## 权衡：开滚动栏目的取舍

滚动栏目的核心交易是真实需要时间和读者想立刻看结论之间的拉锯。我选真实，不写我用了 30 天感觉很棒的伪体验，而是建框架、慢慢填数据。代价是开篇时内容看起来空，第 1 周和第 2 周的更新信息量不大。读者想立刻知道 Hermes 好不好用，栏目给不出，因为我自己也还没用够时间。

这个取舍的合理性在于：Hermes 的核心承诺就是长期陪伴、越用越懂你，验证这种承诺的唯一方式就是时间。任何 14 天或 30 天的封闭测评，都回答不了 60 天之后 skill 库会不会乱掉这种问题。社区已经有人维护一个月放弃，这说明一个月都不够。所以栏目要持续至少 90 天，最好 180 天，才能给出有信息量的判断。读者如果只想要一个装或不装的二元结论，社区里短期测评已经够多，本栏目不重复造这类内容；本栏目服务的是那些已经决定要长期用、想看清楚长期用到底会怎样的读者。

另一个取舍是栏目的工作量。每周记录六个维度、抽样 review、整理事故清单，本身是一项不小的工作。如果 Hermes 真的能帮忙，栏目本身的维护就是它的一个测试场景：让 Hermes 帮我整理栏目数据。如果它帮不上，那也是一个数据点：连评测自己的栏目都帮不上，说明能力有限。这里有一个有趣的递归：栏目既评测 Hermes，又依赖 Hermes 帮忙维护评测。如果某周栏目数据是 Hermes 整理的，那这一周的自动化收益维度就得把这件事记进去；如果 Hermes 整理错了，那本身就是栏目要记录的事故。评测和被评测纠缠在一起，这是长期实测的常态，栏目不假装它们是分离的。

最后是诚实和读者预期的取舍。开篇就讲这不是结论、是框架，会筛掉一部分想立刻看结论的读者。但留下来的读者是真正关心长期判断的，他们才是栏目的目标受众。这也是为什么栏目放在系列第 23 篇：前面 22 篇已经把机制讲透，到这一篇才有底气说现在开始测。读者读完前 22 篇已经知道 Hermes 的架构、风险点、调优方法，到这一篇自然理解为什么不能立刻给结论。因为前 22 篇里识别出的所有风险（skill 漂移、memory 污染、结构性 reviewer 缺失）都是慢性问题，慢性问题必须用时间观测。

还有一个常被忽略的取舍：长期实测栏目本身会被版本演进影响。Hermes 是活跃开发的项目，栏目跑 90 天期间可能经历好几次大版本更新，某些早期发现的问题可能在某个版本后被修复，某些新的问题可能在某个版本后引入。这意味着栏目的 baseline 不是一条直线，而是会被版本升级打断的折线。每次大版本更新都要重建部分 baseline，否则会把版本变化误判成 self-improvement 的效果。这种复杂性本身是长期实测的真实成本，栏目会如实记录哪一周因为版本升级数据失真。

## 结论：栏目要回答的核心问题

栏目开篇交代清楚框架：测什么、怎么测、什么算成功。后续按周和月更新真实数据。重点是诚实记录，包括失败、漂移、事故，不是美化。如果你读到这发现数据还空，栏目刚开始，回来再看。

栏目最终要回答的核心问题不是 Hermes 好用还是不好用这种二元判断，而是更具体的几个：在什么场景下、什么样的使用强度、什么样的维护投入下，Hermes 的 self-improving 是净收益？在什么情况下它会变成负债？什么样的用户适合用、什么样的用户不该碰？这些问题只能用真实的时间和数据回答，栏目就是用来积累这个回答的。

具体到要交付的产物，栏目结束时希望给出三件东西。第一是一张长期数据表，记录 90 天里每个维度每周的数值，让读者自己看趋势而不是听我下结论。第二是一份场景适配图，标注哪类场景 Hermes 表现稳定、哪类容易出问题、哪类要看具体用法，把看人这件事从模糊变具体。第三是一份维护手册，记录栏目期间实际用过的清理、回滚、调优操作，以及这些操作的成本和效果。这三件东西比一句推荐或不推荐对读者有用得多。

栏目的另一个隐性产出是方法论。怎么客观地长期评测一个 self-improving 工具，本身没有标准答案。社区里的短期测评套路化了，长期实测几乎是空白。栏目摸索出的 baseline 建立、被动观测、幸存者偏差对抗、版本漂移处理这些方法，能不能复用到其他 self-improving 工具（比如别的 agent 框架、别的记忆系统），是栏目结束后值得单独再写一篇讨论的话题。

我也会在栏目更新里诚实地标注：什么时候我自己也快放弃了、什么时候出现了让我重新看好它的事、什么时候我不得不改 baseline 因为版本升级了。这些都是判断长期工具真实状态的必要信息，不是噪音。
