# Cron 定时任务 + 子 Agent 委派（spawn / RPC）

> 系列第 15 篇 · 读者预设：想让 Hermes 24/7 自动干活的人 · 最后核实：2026-07

**TL;DR：** 这两块能力是 Hermes 从「聊天工具」变成「个人助手」的关键分水岭。**Cron 定时**让它按自然语言描述的时间表自动起活，结果投递到任意 IM 平台；**子 Agent 委派（spawn）**让主 agent 派出隔离的子 agent 干并行的子任务，互不污染上下文；**RPC 工具调用（`execute_code`）**让模型自己写 Python，通过沙箱化的 RPC 桥直接调用 Hermes 全部工具，把多步流水线压成「零上下文成本的一轮」。三者拼起来：cron 到点，spawn 子 agent，RPC 脚本确定性跑完，结果投递到你的 Telegram / Discord / 飞书。这就是「二十四小时不睡觉的个人助手」的真实形态：不是它主动找你聊，是它在后台按时干活、把活干完、把结果送到你眼前。

## 这两块为何是「助手」的关键

要把这件事说清楚，得先回到一个根本问题：什么样的系统算「助手」，什么样的系统只能算「聊天工具」。

聊天工具的本质是「你问我答」。你不说话，它就什么都不干。它的价值密度集中在「当下这一轮对话」里，一旦你关掉窗口，它就彻底停摆。绝大多数人最早接触的 LLM 形态，网页对话框，都是这一类。它们很聪明，但它们是被动的。

助手则完全不同。助手的本质是「你交办一件事，它在合适的时间、用合适的方式把这件事做完，并且把结果交给你」。这里的关键词不是「聪明」，而是「时间」和「闭环」。助手会自己等到该干活的时候；助手会把结果送到你指定的位置；助手会处理「你没在场」的情况。这些都不是单轮对话能解决的，需要的是一套从「触发」到「执行」到「投递」的完整闭环。

Hermes 的第 13、14 篇我们讲的是「能触达到多少平台」：它把 IM、邮件、浏览器、终端都接到自己的工具集里。这解决了「最后一公里」：助手干完活，能找到你。但只解决最后一公里还不够，前面还有两个洞。

第一个洞：谁来按按钮？传统方案是系统 crontab，但 crontab 写起来门槛不低，而且它只能跑脚本，没法跑「需要一点判断力」的任务。Hermes 用内置的 cron 调度器堵了这个洞，而且用的是自然语言配置。你说一句「每天早上九点把昨天的 PR 总结发到我 TG」，它就把这句话解析成调度计划存下来。这把「配置定时任务」这件事从「写 shell 的人的特权」变成了「会说人话的人都能干」。

第二个洞：活儿太大怎么办？一个 cron 触发的任务可能很重，比如「审计 staging 日志，把异常归因后发 Discord」。如果让一个 agent 串行把所有事做完，上下文会膨胀得非常厉害，token 烧得飞快，而且任何一步出错都会污染后续判断。Hermes 的 spawn 机制堵了这个洞：主 agent 把任务拆成几个独立子任务，每个子任务派一个隔离的子 agent 去干，每个子 agent 有自己的对话历史、自己的终端会话、自己受限的工具集。它们之间互不可见，干完只把最终结果汇报回主 agent。

第三个洞，也是最隐蔽的一个：有些流程根本不需要模型每轮决策。比如「读昨天的 PR 列表，逐个分析，推到飞书表格」这种流水线，步骤是固定的，让模型一轮一轮跑工具调用纯粹是浪费 token。Hermes 给的答案是 `execute_code`：让模型自己写一段 Python，在沙箱里通过 RPC 桥一次性调用所有需要的工具。逻辑写在代码里，跑的是确定性逻辑，不靠模型每轮即兴发挥。这就是「零上下文成本的一轮」：把多步流水线压成一个执行单元。

把三个洞堵上，Hermes 才算完整地具备了「助手」的三要素：能等到该动的时候（cron）、能并行处理大活儿（spawn）、能用确定性方式跑固定流程（RPC）。这篇就拆这三件事，讲它们的机制是什么、适合什么、边界在哪、怎么组合、坑在哪。

## Cron 定时：自然语言起闹钟

先看 cron。Hermes 内置了一个调度器，它对外暴露的不是一堆零散的工具，而是一个统一的 `cronjob` 工具，用「动作式」的参数来管理：加任务、列任务、改任务、暂停、删除都走这一个入口。这种设计哲学贯穿整个 Hermes：把高频操作收敛到少数几个语义清晰的工具上，避免「二十个调度相关的小函数」那种碎片化。

它和系统 crontab 最大的区别，是配置方式。系统 crontab 要你写 `0 9 * * *` 这种五行表达式，写错一个星号任务就跑错时间，而且你心里得有「分时日月周」这张表。Hermes 不要求你记这张表，它接受自然语言：

```
# 概念性示例，具体语法以版本文档为准
cronjob action=add schedule="每天早上 9 点" task="把昨天的 PR 总结发到 Telegram"
cronjob action=add schedule="每周一 10 点" task="审计 staging 日志，异常归因后发 Discord"
cronjob action=add schedule="工作日 18:30" task="把今天 Kanban 上完成的卡片汇总成日报，推到飞书"
```

底层发生的事情是：调度器拿到「每天早上 9 点」这句话，用 LLM 解析成一份合法的调度计划（cron 表达式 + 时区 + 投递目标），存到 gateway 内部的调度表里。这一点很关键：解析的不是死规则，是带语义的自然语言，所以你写「工作日」它能懂，写「周一到周五」它也懂，写「每个工作日的傍晚」它照样能解析。这就把配置门槛降到了「会说中文」的程度。

到点之后，机制是这样的：

第一步，调度器触发，启动一轮完整的 agent loop，把任务描述塞进上下文。第二步，agent 像处理任何用户请求一样干活：可能调工具、查数据、跑脚本、生成内容。第三步，结果通过 gateway 投递到你指定的平台。投递是 cron 价值闭环里不可省的一环，任务跑完没人看等于没跑。所以 cron 的设计从一开始就把「结果送到哪」当成一等公民，不是事后补丁。

适合 cron 的任务有几个共同特征：重复、时间触发、结果要送达。具体来说：

第一类是**每日简报**。早上起来想看昨天的代码提交、工单进展、监控异常、新闻摘要，这些都是「到了点就要、内容是动态生成的」的典型。比起你自己点开十个网页扫一眼，让 agent 在后台把信息聚合好、推到你 TG 上，效率高得多。

第二类是**定时备份与同步**。比如每天凌晨把当天的笔记推到备份仓库、每周把 Kanban 状态导出成 Markdown 归档。这类任务的痛点不是「难做」，是「容易忘」。cron 的价值就是消除「忘了」这个失败模式。

第三类是**周期性审计**。每周一扫一遍依赖有没有新版本、有没有安全公告；每天晚上检查一下 staging 环境的日志有没有异常模式；每小时拉一下某个 API 的延迟是不是在阈值内。这些事没有时效压力，但需要持续盯着。交给 agent 比交给一个值班的人便宜得多，也比一个死脚本灵活得多（agent 能做归因，不是只发原始日志）。

第四类是**定期汇总**。每周把团队这周完成的 PR、合并的分支、关闭的工单汇总成一份「周报草稿」，自动推到飞书文档。每个月把当月的开销、收入、订阅状态汇总成「月度财务简报」。

但 cron 也有清晰的边界，不能当万能锤。

**边界一：依赖模型在线。** cron 触发的本质是「起一轮 agent loop」，这一轮要调用 LLM。如果到点的时候上游 provider 挂了、API 限流了、网络断了，这一轮就失败了。Hermes 不是「本地脚本」，它的智能来自模型，模型不可用就是不可用。对延迟极敏感、对可用性要求极高的任务，传统 crontab + 纯本地脚本仍然更可靠。Hermes 也提供了「script-only cron」这个降级形态：任务完全跳过 LLM，直接跑一段固定脚本，把脚本输出投递到 IM。这等于回到经典 watchdog 模式，换来了「provider 挂了我也能跑」的可用性。两者各有所长，看你这个任务是要「智能」还是要「稳」。

这里有一个工程上容易被低估的细节：智能和稳定之间不是非此即彼，而是可以分层的。一个常见的成熟做法是「智能层 + 兜底层」的双轨结构。主任务走完整 agent loop 拿到高质量的智能结果，同时配一个 script-only 的兜底 cron，时间错开半小时。如果智能层在 deadline 之前成功跑完，就抑制兜底层；如果智能层失败或超时，兜底层自动顶上，至少保证「有结果送达」这件事不破。这套思路借鉴的是传统运维里的「主备切换」，用在 agent 自动化上完全成立。代价是要维护两套配置、要确保抑制机制可靠，但换来的是「单点失败不致命」的可用性，对真正重要的日报、监控类任务很值得。

**边界二：成本。** 每次触发都是一轮完整的 agent loop，每轮都要消耗 token。一个任务本身不贵，但你一旦配了十几个 cron、每个每小时跑一次，月底账单会很难看。社区里已经多次出现「我以为这些 cron 不花多少钱，结果一个月烧了几百刀」的吐槽。这一点下面「真实坑」里还会专门讲。

**边界三：时区。** 「每天早上九点」是哪个时区的九点？配的时候必须想清楚。Hermes 的调度器通常以你配置时所在的时区为准，但如果你在跨时区团队里、或者部署在云上某个地区的实例上，时区不一致是经典翻车点。养成习惯：配置时显式带上时区，比如「北京时间每天早上九点」「UTC 周一零点」。

## 子 Agent 委派：spawn 隔离并行

接下来看 spawn。这是 Hermes 处理「活儿太大」的核心机制。

主 agent 在跑复杂任务的时候，可以选择调用 `delegate_task` 这个工具，把自己的一部分工作「外包」出去。这个调用会启动一个新的子 agent 实例，不是同一个对话里的另一段消息，而是一个全新的、独立的 agent 实例。这个子 agent 拿到的是：自己的对话历史（干净的）、自己的终端会话（独立的）、自己的工具集（可以受限的）。它干完之后，只把最终结果汇报回主 agent。中间过程不会泄漏到主 agent 的上下文里。

这套隔离机制的意义，可以从三个维度来看。

**第一是上下文隔离。** 主 agent 的上下文窗口是宝贵的，要装着用户的需求、之前的对话、关键决策。如果它每处理一个子任务都把中间细节塞进上下文，很快就会爆；而且爆的不只是「放不下」，更重要的是「噪声太多导致后续判断质量下降」。spawn 把子任务的中间细节关在子 agent 自己的上下文里，主 agent 只看到「我外包了这件事，结果是这个」。这等于把上下文当成了「主管的工作记忆」：主管不需要看每个执行者的草稿纸，只需要看汇报。

**第二是工具集隔离。** `delegate_task` 接受一个工具集参数，你可以限制子 agent 只能用哪些工具。比如让它做调研，就只给 web 搜索和浏览工具；让它做代码审查，就只给文件读取和 git 工具；不给孩子危险的玩具：不让它动删除、不让它发消息、不让它跑高权限命令。这种「最小权限」原则在自动化场景里非常重要。你不会希望一个负责「调研三个数据库方案」的子 agent 顺手把生产库的某个文件给改了。

**第三是并行。** 多个无依赖的子任务可以同时 spawn 出去并行跑。如果你要「调研 A、B、C 三个工具的优劣」，串行做要跑三轮 agent loop，并行做只要一轮。时间省到原来的三分之一，token 成本也省，因为主 agent 不需要在上下文里同时维持三个调研的状态。这是 spawn 最直观的收益。社区里 MarkTechPost 在 2026 年 6 月专门报道过 Hermes 上线「异步子 agent」这件事：之前的 spawn 是阻塞的（主 agent 等子 agent 干完才能继续），异步化之后主 agent 可以继续处理别的，子 agent 干完再回来汇报，这对交互体验是质的提升。

适合 spawn 的场景，可以归纳成几类：

**第一类：可拆分的并行调研。** 「帮我比较 A、B、C 三个方案」「并行查这三个数据源」「同时审这三段代码」，只要子任务之间互相独立，spawn 并行都比串行处理快几个数量级。

**第二类：上下文很大的子任务。** 比如「读这五十页文档，提取关键点」。让主 agent 自己读，它的上下文会被这五十页撑爆，后续做任何判断都受影响。spawn 一个子 agent 专门读，读完只把关键点汇报回来，主 agent 的上下文保持清爽。

**第三类：需要不同工具集的任务。** 一个任务里既要做 web 调研、又要改本地文件、又要发消息。把这些拆给不同的子 agent，每个只拿到自己需要的工具，权限隔离更干净，也避免了「一个 agent 同时拥有所有权限」带来的风险。

但 spawn 也有它**不适合**的场景，必须说清楚。

**不适合一：子任务强依赖前一步结果。** spawn 的价值在并行，如果 B 必须等 A 出结果才能开始，那 spawn 出去只是徒增协调成本，不如让主 agent 串行做。判断标准很简单：能不能把 A、B、C 同时交给三个不认识彼此的人各干各的。能就是适合 spawn。不能就别 spawn。

**不适合二：子任务需要主 agent 的完整上下文。** spawn 的子 agent 是空手起活的，它只拿到你 delegate 时给它的任务描述，拿不到主 agent 这一刻拥有的全部上下文。如果一个子任务必须基于「之前对话积累的全部信息」才能做对，spawn 出去会丢上下文。这时候要么让主 agent 自己做，要么在 delegate 时把必要上下文显式打包传过去（这又会增加 token 成本，要权衡）。

**不适合三：需要持久化的子任务。** 这是个容易被忽略的点。子 agent 不是 durable 的，它们的生命周期跟着主 agent 的会话走。会话结束，子 agent 的所有状态就丢了。如果你需要「这个子任务跑五分钟、跑完结果存到磁盘、过两天再来取」这种跨会话的持久化，spawn 不是合适的工具，你应该用 cron + RPC 脚本，把结果写到磁盘或数据库。

理解 spawn 的非持久性，要把它和「记忆」区分开。Hermes 是有持久记忆的（第 14 篇专门讲），但记忆是「事实和偏好」层面的，不是「执行状态」层面的。子 agent 干完之后，它的结论可以沉淀到主 agent 的记忆里：「调研过 A 工具，结论是 X」这条事实是可持久化的。但子 agent 干到一半的中间状态（比如「正读到第三页，已经提取了五个要点，还差两个」）这种执行快照不会自动持久化，会话一结束就蒸发。所以判断 spawn 适不适合，要问自己一个问题：「如果这个子任务在干到一半的时候会话突然断了，重启动之后我能不能接受它从零开始重做」。能接受就是适合 spawn。不能接受就要把执行状态外化到磁盘或数据库，而那已经不是 spawn 模式擅长的事了。

## RPC 工具调用：写 Python 直接调工具

第三块是最进阶的，也是最难讲清楚的一块：RPC 工具调用。在 Hermes 里对应的工具叫 `execute_code`。

理解 `execute_code` 之前，先理解一个痛点。假设你想让 agent 做这件事：「读昨天的 PR 列表，对每个 PR 跑一次摘要，把摘要按表格格式推到飞书」。这件事在传统 agent loop 里要怎么跑？大概是：

1. 调用 GitHub 工具，列出昨天的 PR。
2. 对每个 PR，调用读取工具拿 diff。
3. 对每个 diff，调用模型生成摘要。
4. 调用飞书工具，建表格。
5. 对每个摘要，调用飞书工具写入一行。

每一「调用」都是一轮 LLM 决策：模型要先想「下一步该调哪个工具」，再生成调用参数，再等工具返回，再看返回结果决定再调什么。这个循环里，模型每一步都要重新加载上下文、重新做决策，token 烧得飞快，而且每一步都有「模型这次会决策成什么样」的不确定性。流程是固定的，执行却是不确定：这是浪费。

`execute_code` 给的答案是：让模型写一段 Python，把上面的整个流程写在代码里，然后一次性执行。代码里通过 RPC 桥调用 Hermes 的工具（读 GitHub、调模型摘要、写飞书），全在一段 Python 里串起来。执行结果是确定性的：代码怎么写就怎么跑，不会这次跑成 A 下次跑成 B。

伪代码长这样：

```python
# 概念性伪代码，具体 RPC 接口以版本文档为准
from hermes.rpc import tools

# 第一步：读昨天的 PR 列表（一次性批量）
prs = tools.github.list_prs(since="1d", repo="myorg/myrepo")

# 第二步：对每个 PR 生成摘要（确定性循环，不靠模型每轮决策）
summaries = []
for pr in prs:
    diff = tools.github.get_diff(pr["number"])
    summary = tools.ai.summarize(diff, max_tokens=200)
    summaries.append({
        "number": pr["number"],
        "title": pr["title"],
        "author": pr["author"],
        "summary": summary,
    })

# 第三步：批量推到飞书表格
tools.feishu.create_table(
    app="weekly_report",
    columns=["PR 编号", "标题", "作者", "摘要"],
    rows=summaries,
)
```

这段代码被沙箱化的 RPC 执行环境跑一次，就把整条流水线跑完了。对模型来说，它只需要一轮：写代码、看结果。中间的每一步都不需要模型决策，决策已经固化在代码里。这就是「零上下文成本的一轮」的字面意义：你不用把每一步的输入输出都塞回模型上下文，因为决定下一步的不是模型，是代码。

这种模式带来几个根本性的好处：

**好处一：确定性优先。** 代码跑的是固定逻辑，今天跑是这个结果，明天跑还是这个结果（输入一样的前提下）。这对生产场景极其重要：你不能接受「同一个日报任务，昨天跑出来的格式和今天不一样」。

**好处二：可测试、可版本管理。** Python 脚本是文本，可以提交到 git，可以写单元测试，可以做 code review。比起「藏在某次对话里的隐式流程」，脚本是工程师友好的产物。团队协作时，脚本是「大家都能看懂、都能改」的载体，而 prompt 是「只有写 prompt 的人真正懂」的载体。

**好处三：复用。** 同一段流水线脚本可以在多个 cron 任务里复用，也可以被不同的 agent 调用。一旦写好、测好，就是一份资产。Prompt 的复用就难得多：同一个 prompt 在不同模型版本、不同上下文下表现都不一样。

**好处四：成本可控。** 多步 pipeline 压成一轮，意味着 LLM 调用次数从 N 次降到 1 次（外加 N 次工具 RPC 调用，RPC 不走 LLM）。token 成本能差一个数量级。社区里 Alpha Signal 那篇「Search as Code」的文章专门测过：把搜索流程从「轮式调用」改成「代码式调用」，token 能省到原来的 15%。

但 RPC 模式不是万能的，它的**适用边界**也很清楚。

**适合 RPC：** 流程固定但步骤多的任务（数据 ETL、批量处理、固定格式的报告生成）；对确定性要求高、不希望模型「创造发挥」的任务；需要集成到现有 Python 工作流的任务（比如你的项目已经有数据 pipeline，想让 Hermes 的某些能力嵌入进去）；需要可测试、可版本管理的核心业务流程。

**不适合 RPC：** 流程高度动态、每一步都依赖前一步的语义判断的任务（这种任务硬写成代码会变得一坨 if-else，不如让 agent loop 自己跑）；需要模型即兴发挥、跨领域推理的任务（比如「读这篇论文，找出它的核心创新点和潜在缺陷」，这种任务的「下一步做什么」高度依赖当前内容，写不成固定流程）。

判断标准是：**这个流程如果让人做，他会需要每一步都重新思考吗？** 如果是，用 agent loop。如果不是，用 RPC 脚本。

这里还有一层工程上的微妙判断：**RPC 脚本的「确定性」是相对的，不是绝对的**。脚本里如果调用了模型（比如 `tools.ai.summarize`），那这一步本质上仍然是不确定的：同一个输入，今天摘要出来的措辞和明天可能不一样。但脚本结构本身是确定的：先读 PR、再生成摘要、再推飞书，这个顺序不会变；摘要这一步的不确定性被限制在了「单步输出」层面，不会扩散到「整个流程」。这是 RPC 模式相对纯 agent loop 的核心优势：它把不确定性装进了「可控的小盒子」，每个盒子有明确的输入输出契约，盒子之间的连接是确定性的代码。理解这一点，就不会把 RPC 当成「绝对确定」的银弹，而是当成「不确定性可以分层管理」的工程手段。

## 三者组合：二十四小时助手的实际形态

把三件事单讲清楚之后，真正的力量在组合。下面是一个典型的「二十四小时助手」实际形态，用 ASCII 图把数据流画出来：

```
┌──────────────────────────────────────────────────────────────────┐
│  ①  cron 调度器（gateway 内置）                                  │
│      schedule: "每天 9:00 (Asia/Shanghai)"                       │
│      task:     "把昨天 GitHub 的 PR 汇总成日报，推到飞书"        │
└────────────────┬─────────────────────────────────────────────────┘
                 │ 到点触发
                 ▼
┌──────────────────────────────────────────────────────────────────┐
│  ②  主 agent loop 启动                                           │
│      读取任务描述，决定拆分策略                                   │
└────────────────┬─────────────────────────────────────────────────┘
                 │ delegate_task
        ┌────────┴────────┬────────────────┐
        ▼                 ▼                 ▼
┌────────────────┐ ┌────────────────┐ ┌────────────────┐
│ ③ 子 agent A   │ │ ③ 子 agent B   │ │ ③ 子 agent C   │
│ 工具集: github │ │ 工具集: github │ │ 工具集: github │
│ 范围: backend  │ │ 范围: frontend │ │ 范围: infra    │
│ 独立终端会话   │ │ 独立终端会话   │ │ 独立终端会话   │
└───────┬────────┘ └───────┬────────┘ └───────┬────────┘
        │                  │                  │
        │ 各自跑 execute_code（RPC 脚本）     │
        │ 列 PR → 拉 diff → 调模型摘要        │
        │ 全部在一段 Python 里串好            │
        └──────────┬───────┴──────────────────┘
                   │ 只回传最终摘要列表（中间过程不污染主上下文）
                   ▼
┌──────────────────────────────────────────────────────────────────┐
│  ④  主 agent 汇总三个子 agent 的结果                             │
│      按「重要更新 / 待 review / 已合并」分类                      │
│      调 execute_code 写一段渲染脚本，生成 Markdown 表格           │
└────────────────┬─────────────────────────────────────────────────┘
                 │ gateway 投递
                 ▼
┌──────────────────────────────────────────────────────────────────┐
│  ⑤  投递到目标平台                                               │
│      → 飞书文档（完整日报）                                       │
│      → Telegram（简短摘要 + 飞书链接）                            │
│      失败时 → Telegram 告警频道（带错误堆栈）                     │
└──────────────────────────────────────────────────────────────────┘
```

这张图里出现了五个角色，每个角色都有自己的边界：

cron 是「闹钟」：只负责到点触发，不关心后面怎么干。主 agent 是「项目经理」：负责拆任务、协调子 agent、汇总结果。子 agent 是「执行者」：拿到独立子任务、用受限工具集干完、汇报结果。RPC 脚本是「流水线」：把多步固定流程压成一段确定性代码。gateway 投递是「快递员」：把最终结果送到指定平台，包括失败时送告警。

这套组合的真实价值在哪？不在「它能做的事比单 agent 多」，而在「它能做的事是单 agent 做不了的」。一个不带 cron 的 agent，你必须每天九点手动跟它说「该做日报了」，这就回到了「聊天工具」的被动模式。一个不带 spawn 的 agent，处理大任务时上下文会爆、token 会烧、并行做不了。一个不带 RPC 的 agent，每个固定流程都要走多轮 LLM 决策，既贵又不确定。三者拼齐，才有了「二十四小时无人值守的助手」这件事。

这里有一个常被忽略的工程判断：哪些步骤该 cron、哪些该 spawn、哪些该 RPC，不是显而易见的。一般原则是：**时间触发的部分用 cron，并行可拆的部分用 spawn，流程确定的部分用 RPC**。但实际任务里这三者经常交织：一个 cron 触发的任务，可能 spawn 出三个子 agent，每个子 agent 内部又跑 RPC 脚本。设计这种复合流程的能力，是用好 Hermes 的核心技能之一，也是第 16 篇要专门展开的「端到端实战」。

## 真实坑：四个一定会踩的雷

讲完理想形态，必须讲坑。下面四个坑都是社区反复踩过、GitHub issue 反复出现过的。

**坑一：静默失败。** cron 任务最大的风险是「凌晨跑挂了你不知道」。一个本该每天早上九点推日报的任务，可能因为 GitHub API 限流、飞书 token 过期、模型 provider 抖动而失败。如果失败时没有任何通知，你会连续几天收不到日报，等发现的时候已经攒了一周没处理的事。这种「沉默失败」对自动化系统是致命的：比「任务跑挂了」更糟的是「任务跑挂了你以为它还在跑」。

防御措施是强制的失败通知：每个 cron 任务都必须配置一个失败回调，失败时把错误堆栈投递到一个独立的告警频道（比如专门的 TG 告警群、Discord 的 ops 频道）。Hermes 的 cron 系统支持这种失败投递，但不是默认开启的，你必须显式配。养成习惯：**配 cron 的同时配失败通知，没有失败通知的 cron 不算配完**。更进一步，可以再加一个「心跳」cron：每天某个时间发一条「所有 cron 任务都正常」的消息，如果某天没收到这条心跳，说明某个任务卡死了。这是「监控的监控」。

**坑二：并行冲突。** spawn 出去的子 agent 是真并行，它们各自有独立的终端会话和工具调用。如果两个子 agent 同时操作同一个文件、同一个 git 分支、同一个外部资源（数据库行、API 配额），就会撞车。常见的悲剧是：两个子 agent 同时往同一个 README 追加内容，后写的覆盖先写的，数据丢了。

防御措施是在 spawn 之前明确划分「势力范围」。每个子 agent 只能改自己负责的目录、自己的分支、自己的资源。如果任务结构上做不到完全隔离，那就别并行，串行做更安全。判断标准：**如果两个子任务都试图写同一个资源，不要并行**。读可以并行（多个读者不冲突），写不能并行。这条规则和数据库的并发控制是一回事，没有因为换了 agent 就变。

**坑三：RPC 脚本权限。** `execute_code` 跑的 Python 脚本，调用 Hermes 的工具时拥有和 agent 同等的权限：脚本能删文件就真删，能发消息就真发，能调 GitHub 就真调。**没有「这是脚本所以再确认一下」的环节**。这是 RPC 模式的设计取舍：确定性优先，意味着去掉了人工确认这一步。

这带来的风险是：一段写错的脚本、或者一段来自外部的脚本，可能造成真实破坏。一段写错了路径的清理脚本可能把不该删的目录删光；一段从某个论坛复制来的「自动化神器」脚本可能偷偷把你的 token 上传到某个地址。防御措施是：**所有要跑的 RPC 脚本必须 code review**，特别是涉及写操作、外部调用、敏感数据访问的部分。陌生脚本绝不直接跑：先在一个隔离的测试环境里跑一遍，看它实际调用了哪些工具、访问了哪些资源。把 RPC 脚本当生产代码对待：版本管理、code review、测试覆盖、变更审批。这不是过度工程，这是基本卫生。

**坑四：成本失控。** 这是 GitHub issue #23419 专门反映的问题：cron 任务和 agent 派发会「无声地」烧 provider credits，没有预算上限、没有成本预估。一个高频 cron（比如每五分钟跑一次）+ 每次跑都 spawn 三四个子 agent + 每个子 agent 都跑一段中等长度的 RPC 流水线，这套组合一周烧掉几十刀不夸张。社区里那位吐槽「token 成本预投影离谱」的用户，问题就出在背景 cron 配置不当。

Hermes 本身目前缺少「触发前预估成本并拦截」的内建机制（这正是 #23419 提的改进点）。在它落地之前，务实的做法有三层：第一，给每个 cron 任务设一个粗略的「月度预算上限」，用第三方工具（比如 Revenium 或 hermes-telemetry 插件）做实际预算拦截；第二，定期检查实际开销，把那些「以为不花钱但其实很贵」的高频任务降频或砍掉；第三，能用 script-only cron 解决的就不要走完整 agent loop，纯脚本任务的成本是固定的，不会因为模型「多想了几个回合」而飙升。

成本失控的根因不是 Hermes 设计差，而是「智能」本身是付费的：每一轮 agent loop 都要付 token 钱，cron 让「智能」可以自动触发，自然就自动花钱。把这一点内化成默认认知：**任何 cron 都是开着的水龙头**，不是设完就忘，要定期查水表。

还有一个隐性成本维度容易被忽略：**重试成本**。一个 cron 任务失败了，Hermes 默认可能不会自动重试，但如果你配了重试策略（或者通过外部 watchdog 触发重跑），每次重试都是一轮完整的 agent loop，都要付费。如果一个任务因为上游故障反复失败、反复重试，一天的 token 成本可能比平时高十几倍。防御措施是给重试也设上限：最多重试 N 次，N 次失败后停止并告警，不要无脑重试。同时监控「重试开销 / 正常开销」这个比值，比值异常高通常说明上游有持续故障，应该让任务暂停而不是继续烧钱。

## 权衡：自动化的核心交易

把三件事和四个坑放在一起，能提炼出一条核心权衡：**你用「省下盯活的精力」交换了「无人值守时的不确定性」**。

完全自动化的代价是：你必须接受静默失败的风险、并行冲突的可能、脚本权限滥用的隐患、成本失控的潜在账单。这些风险不是「装上 Hermes 就消失」的，是要靠工程纪律持续管理的：配失败通知、划分势力范围、code review 脚本、监控预算，每一项都是要持续投入的运维成本。

完全手动的代价是：你必须每天亲自按按钮、亲自等结果、亲自处理每一件事。这看起来「安全」，但它的成本是你的注意力，而注意力是最稀缺的资源。

务实的选择不是「全自动」或「全手动」两端，而是分层：

**高频、低风险、流程固定的任务，全自动。** 每日简报、定时备份、日志扫描，这类任务值得花时间配好 cron + 失败通知 + 预算上限，让它 24/7 自己跑。失败有告警，跑飞了能立刻知道。

**低频、高风险、需要判断的任务，半自动。** 比如周度代码审查、月度架构决策、季度依赖升级评估。这类任务可以 cron 触发「准备好素材」，但最后一步判断留给人。agent 把数据准备好、把候选方案列出来、把权衡写清楚，人来拍板。

**一次性、探索性、跨领域的任务，纯手动。** 「调研这个新框架值不值得换」「这个产品方向对不对」，这种任务的本质是探索，没有固定流程可写，强行自动化反而限制发现。让 agent 当场配合你思考就好。

这套分层的关键判断是：**越确定的事越值得自动化，越需要判断的事越要留给人**。听起来像废话，但实践中很常见的一个反模式是：把所有事都往「全自动」上推，结果高风险任务被自动跑飞了，又退回来把所有事都改回手动。正确的做法是从「高频低风险」开始自动化，稳定了再逐步往「中频中风险」扩，每扩一步都先把失败通知和预算上限配齐。

还有一条不那么显眼但同样重要的权衡：**自动化的边界要写在代码里，不能只写在人的脑子里**。意思是：「这个任务不能自动发消息，必须人来审一下」这种约束，要变成配置里的一道闸门（比如 cron 任务标 `requires_human_review: true`，到这一步暂停等人工确认），而不是依赖「我记得这个任务要人工审」。人会忘、会偷懒、会赶时间，配置不会。把约束固化到流程里，是自动化系统能长期稳定运行的真正秘密。所有「我以为没事」的事故，根因几乎都是某条本该硬化的约束还停在「靠人记得」的状态。

## 结论：三件事都用上才算用到家

回到开头的判断：cron 定时 + 子 agent 并行 + RPC 确定性脚本，这三件事拼起来，Hermes 才从「聊天工具」升级成「自动化助手」。任何一件单独用，都只是发挥了一部分能力。三者协同，才有了真正的二十四小时无人值守闭环：cron 负责到点触发，spawn 负责把大活儿拆开并行干，RPC 负责把固定流程确定性跑完，最后由 gateway 把结果送到你眼前。

但越自动化，越要把三件事管住：**失败通知要响、并行边界要清、脚本权限要审、成本预算要控**。这四件事不是「配完一次就忘」的，是日常运维要持续盯的。自动化不是「装上就省心」，是「装上之后省下了按按钮的时间，但要花一部分时间管自动化的边界」。净收益仍然是正的，但前提是你愿意持续管理它。

最后一句话给准备上这套组合的人：**先从一个低风险的 cron 开始**，比如「每天早上把昨天的 git 提交汇总推到 TG」。把这个跑稳（失败通知响了你能收到、成本在预算内、结果格式稳定）再考虑加 spawn 和 RPC。复杂度要逐步加，每加一层都要确保前一层是稳的。直接上「cron + spawn + RPC」三件套全开的复杂自动化，几乎一定会出事；从最简单的 cron 开始逐步往上搭，才能把这套助手的真实价值榨出来。
