# 踩过的坑和不会再用它的场景

写这篇之前我犹豫了一下。一个全是好评的工具生态是不健康的，要么说明没人真用，要么说明大家在说谎。但反过来，专门写一篇讲缺点，又怕显得在唱衰。

最后我还是写了。因为一个工具只有边界被清楚认识，才可能被正确使用。盲目信任会翻车，盲目排斥会错失价值。这篇讲我踩过的坑、失败模式、明确不会再用它的场景，帮你建立"什么时候用、什么时候不用"的判断。

声明：以下踩坑基于 2026-07 这个时间点的实际使用和社区反馈。工具在迭代，有些坑可能在新版修了。但趋势性的问题，不是某个 bug 而是设计层面或成熟度的局限，值得说清楚。

## Craft 模式多文件改写的坑

先讲 Craft 模式（IDE 多文件改写）的坑。这是 CodeBuddy 的招牌能力，也是最容易翻车的地方。

改动范围失控。Craft 模式描述需求后，它自主决定改哪些文件。多数时候它判断得对，但偶尔会改了不该改的文件。我有一次让它加一个 API，它除了改 API 文件，还顺手"优化"了它认为相关的其他文件，而这些顺手的改动我不想要。审 diff 的时候差点没注意到。

对策是审 diff 时不要只看主文件，要看完整的改动文件列表。看到不相关的改动，直接拒绝那个文件的 diff。我在项目上下文里加了一条"不要修改任务范围外的文件"的约束，有所改善但没根除。

关联文件感知不全。Craft 号称能感知关联文件并同步更新。我的体验是，它对直接依赖的关联做得不错，改了接口会更新调用方。但对间接依赖经常漏，改了数据结构，影响序列化、影响测试、影响文档，这些它经常不提。

对策是改完后用 Ask 模式问一遍"这次改动可能影响哪些地方，有没有遗漏"，做一轮检查。

复杂改动的 diff 质量波动。简单改动的 diff 质量稳定，加个字段改个方法什么的。但当一次改动涉及五六个文件、上百行变更时，diff 里偶尔出现逻辑错误、不存在的 API 调用、或前后不一致的改动。复杂度上来，一次成功率明显下降。

我现在对复杂改动的做法是不要一次性让 Craft 全做完。先用 Plan 模式拆成小步骤，每个步骤一个小改动，分批 Craft 加审 diff。把一次大改动拆成几次小改动，每次的成功率和可控性都更高。

## CLI 的工程稳定性问题

长会话上下文丢失。在一个会话里交互很久，几十轮对话大量代码进出，偶尔会出现 AI 忘了前面说过什么。表现是它开始重复问已经确认过的事，或者给出跟前面约定矛盾的方案。这是上下文窗口管理的成熟度问题。

我的对策是长会话定期用 /clear 重置，把关键结论写在项目上下文文件里而不是依赖会话记忆。一个复杂任务拆成多个短会话，比一个超长会话更可靠。

复杂任务中途失败。CLI 处理高复杂度任务时，偶尔会在中途卡住或产出不完整结果。不像 Claude Code 在复杂任务上的表现那么稳。我的对策是高复杂度任务不硬上 CLI，要么切回 IDE 的 Craft，要么切回 Claude Code。

## 模型相关的问题

混元在英文或国际化项目上没有优势。混元的中文优势是真实的，但反过来，在纯英文项目、国际化代码库、英文技术栈里，它的优势消失，甚至偶尔不如直接用海外模型。不要因为"默认是混元"就不切，英文重度项目我会在意模型选择。

DeepSeek R1 慢且贵，用在简单任务上很亏。R1 推理深，但每次响应要等它"想"，token 消耗也高。具体的踩坑经历（比如用 R1 做简单补全等了快十秒）和应对，见 [多模型策略](./07-multi-model-strategy.md) 的"我踩过的坑"。一句话结论：简单任务别用 R1。

模型切换的上下文断裂。不同模型的对话风格和能力有差异，中途切换偶尔衔接不顺畅。我的做法是一个会话尽量用一个模型，要切就复述约定——一次完整的翻车经历见 [多模型策略](./07-multi-model-strategy.md)。

## 我不会再用 CodeBuddy 的场景

讲完具体的坑，直接列几个我明确"不会再用 CodeBuddy"的场景。这不是说 CodeBuddy 差，是这些场景有更合适的工具。

高复杂度的架构重构。涉及全项目架构调整、跨十几个模块的重构，Claude Code 在我目前的使用经验里一次成功率更高、稳定性更好。CodeBuddy CLI 在这种任务上的稳定性差距让它不是我的首选。

深度 CI/CD 自动化。要在 GitHub Actions 里跑 AI 自动 review PR、自动修 bug，Claude Code 的 Headless 模式和 CI 集成更成熟。如果核心需求是"把 AI 装进 CI 流水线"，现阶段 Claude Code 更可靠。

已有深度 Claude Code 生态投入。如果团队已经在 Claude Code 的 Subagents、Skills、Hooks、Plugins 上有大量积累，迁移到 CodeBuddy 的成本会吃掉它的优势。生态投入是沉没成本，也是迁移阻力。这种情况下不迁移不是因为 CodeBuddy 不行，是因为迁移不划算。

纯英文、海外技术栈、对生态丰富度要求高的项目。这种场景 CodeBuddy 的两个核心优势，中文和国产合规，都用不上，而海外工具的生态优势更突出。选 Claude Code 或 Cursor 更合适。

## 反过来我一定会用 CodeBuddy 的场景

为了平衡，也列一下我一定会优先选 CodeBuddy 的场景。

国内受监管行业。合规是硬门槛，CodeBuddy 可能是唯一选项。

中文为主的项目。混元的中文理解有真实加成，体感更准。

国内网络环境下想用稳定的 AI 编程工具。不用折腾代理，体验稳定。

成本敏感的个人或小团队。CodeBuddy 的免费额度对个人开发者和小团队友好，具体以官网为准。

中等复杂度的日常开发。这个区间 CodeBuddy 完全够用，没必要为用 Claude Code 承担网络和成本负担。

## 怎么持续判断边界

工具在迭代，今天的坑明天可能修了。怎么持续判断 CodeBuddy 的边界适不适用你的场景？

关注版本更新。每次大版本更新，重新测试你之前踩过坑的场景，看修没修。CodeBuddy 迭代不慢，趋势性的问题会逐步改善。

维护一个自己的评测任务集。我攒了大概十五个真实典型任务，每次工具大更新后跑一遍，对比效果。这比看任何评测文章都准，因为它是针对我的真实场景的。

看社区但要带批判性。掘金、知乎、GitHub issue 里的反馈有参考价值，但要注意区分个体体验和普遍问题，区分 bug 和设计局限。一篇吐槽不等于普遍问题，一个好评也不等于真的成熟。

## 最后的判断

回到最开始那个问题：CodeBuddy 到底能不能打。

我的结论比一开始更具体了。CodeBuddy 不是万能的，也不是不能用的。它是一个在特定场景有明显优势（中文、国产合规、国内网络）、在特定场景有明显短板（复杂任务稳定性、生态成熟度、CI 深度集成）的工具。

正确的用法是在它的优势场景里认真用，拿到真实的提效。在它的短板场景里诚实承认局限，换更合适的工具。不要非此即彼，双轨甚至多轨是务实的策略。持续评估，因为工具在迭代，边界在变化。

国内 AI 编程工具正在快速追赶。CodeBuddy 现在的短板，半年一年后可能补上不少。作为使用者，保持关注、持续测试、按场景选择，比站队吹或黑都更有价值。

这是我自己用了一段时间之后最真实的判断。
