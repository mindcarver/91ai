# 让 CodeBuddy 真正理解你的项目：上下文与记忆怎么做

有一次我让 CodeBuddy 给一个项目加个工具函数。它加完之后，我一看，项目里已经有一个功能完全一样的函数，就在它隔壁的文件里。AI 不知道，因为没人告诉它。

还有一次，它改完代码，风格跟项目里其他代码完全不一致。项目用 async/await，它给了一堆 .then() 链。项目错误处理有统一封装，它直接 throw。

这些问题的根子都不在模型，在上下文。AI 每次会话从零开始，它不知道你的项目用什么规范、有哪些已有工具、哪些地方不能动。它只能基于自己被训练过的通用代码习惯来改，而这些习惯不一定跟你项目一致。

让 AI 真正读懂项目，本质是把人类开发者长期协作积累的隐性知识，变成 AI 能消费的显性上下文。这件事做得好不好，直接决定 CodeBuddy 在你项目里表现好不好。

## CodeBuddy 能自动感知什么，感知不到什么

先搞清楚 CodeBuddy 自己能拿到什么，缺什么。

它能自动感知的：当前打开的文件和光标位置的代码，项目目录结构（它能看到文件树），你在对话里贴的代码片段，你用 @提到的特定文件。这些是地基。

它感知不到的：项目的命令（怎么装依赖、怎么跑测试），代码规范和风格约定，架构说明（哪个目录放什么、模块之间怎么依赖），安全边界（哪些文件不能动），已有的工具函数和封装。这些得你主动写下来。

大多数"瞎改一气"的问题，都出在它感知不到的那一层有缺口。

在 CodeBuddy 里，这份文件有标准落点：**`CODEBUDDY.md`**。把它放在项目根目录（`./CODEBUDDY.md` 或 `./.codebuddy/CODEBUDDY.md`），CodeBuddy 启动时会自动加载进上下文，不需要每次手动 @。团队把它提交进 git，所有人读同一份，保证 AI 行为一致。个人跨项目的偏好放 `~/.codebuddy/CODEBUDDY.md`；按主题拆分的模块化规则放 `.codebuddy/rules/*.md`；`/init` 命令能扫描项目自动生成初版。它也能直接复用 CLAUDE.md（兼容读取）。下面这份样例，就是 CODEBUDDY.md 该写的内容，控制在 100 到 150 行：

## 项目上下文不是写给人看的 README

很多人犯的第一个错，是把项目上下文写成给人看的 README。给人看的文档解释"为什么这么做"，给 AI 看的上下文强调"必须怎么做、不能怎么做"。两回事。

给 AI 写的上下文，是一份操作手册。命令要可执行，写具体的命令不写"用包管理器安装依赖"这种模糊描述。架构说明要带约束，不只说"src/models 放数据模型"，还要说"改动前需确认是否影响迁移"。

我自己的 CODEBUDDY.md 大概长这样：

```markdown
# 项目上下文

## Commands
- Install: pnpm install
- Test: pnpm test
- Typecheck: pnpm typecheck
- Build: pnpm build

## Architecture
- src/api/        后端路由，每个资源一个文件
- src/models/     数据模型，改动前确认是否影响迁移
- src/utils/      工具函数，新增前先检查这里有没有现成的
- src/middleware/ 中间件，鉴权逻辑在 auth.ts
- migrations/     数据库迁移，不要修改已有的，只新增

## Conventions
- 用 async/await，不要用 .then() 链
- 错误处理用 src/utils/errors.ts 里的统一封装
- 新增 API 参考 src/api/ 下同类文件的风格

## Safety
- 不要修改 migrations/ 下已有的文件
- 不要改 .env* 文件
- 改 auth.ts 前先问我
```

这里面最有价值的是 Conventions 那一块。它写的是项目特有的规范。AI 默认用通用规范，你得告诉它这个项目的例外：用什么风格、复用什么工具、怎么命名。这一块写得好，能直接消除"风格不一致"和"重复造轮子"两个最常见的问题。

Safety 那一块不是可选的。一个能改文件、跑命令的 AI，没有安全边界就是隐患。

## 喂上下文的几个实战技巧

光有一份上下文文件不够，实际用的时候有几个技巧我摸索出来了。

@文件精确投喂。当任务只涉及某几个文件时，用 @把相关文件直接喂给 AI，比让它自己找更准。比如改一个 API，把这个 API 文件、它依赖的 model、相关的测试一起 @ 进来。AI 拿到精确上下文，改出来的东西更贴合现有代码。

范围要限定。大项目里，不要让 AI 在整个代码库里找答案。明确告诉它"只看 src/api/ 这个目录"或"先看 auth.ts 理解鉴权逻辑"。范围不限定，AI 容易被无关代码干扰，还会浪费上下文窗口。

先让它复述。喂完上下文后，先让 AI 用自己的话复述一遍项目规范。它复述不清楚的地方，就是你上下文写得不够明白的地方。这个检查能暴露很多你没意识到的信息缺口。

分批喂，别一次塞爆。上下文窗口是有限的。把所有项目文档一次性塞进去，会挤占有效上下文空间，AI 反而表现更差。核心规范放上下文文件里常驻，任务相关的具体代码用 @按需投喂。

## 上下文文件是要迭代的

一个常见误区是以为上下文写一次就完事。实际上它需要迭代。

每次遇到 AI 行为不符合预期，我会问自己：这个问题是因为上下文里缺了什么吗？如果是，就把那条规则补进去。比如 AI 改完代码没更新测试，就在 Conventions 里加一条"改了公共行为要同步改测试"。

经过几轮迭代，上下文文件会越来越准确，AI 的表现也会越来越好。这是一个用得越久越值钱的资产。我现在回头看自己项目最早的上下文文件，跟现在差了三倍内容，每一条都是踩坑之后补上的。

## CodeBuddy 项目理解能力的边界

讲了怎么做，也要说清楚边界在哪，不然你会对它有错误期待。

它能做到的：读懂你明确提供的上下文文件和 @进来的代码，感知当前文件和项目结构，基于提供的上下文生成贴合项目风格的代码。

它做不到的：自动发现整个代码库里的隐式约定，你不写它就不知道；跨大型项目的全局推理，上下文窗口有限，超出的部分它看不到；准确判断哪些旧代码有特殊历史原因不能动，它倾向于"优化"所有看起来能优化的代码。

这些边界意味着，上下文工程在 CodeBuddy 里依然是你的责任。工具会越来越强，但"把项目知识显式化"这件事，短期内还得人来做。

## 一个检验标准

怎么判断你的项目上下文写得好不好？我有一个简单的检验标准：如果 AI 改出来的代码风格跟项目不一致，或者重复造了已有的工具，说明你的上下文有缺口。补上，迭代。这个文件会越来越值钱。

反过来，如果 AI 改出来的代码每次都贴合项目规范、能复用已有工具，说明上下文写得差不多了。这时候你的注意力可以从"写上下文"转到"用 AI 干活"上。
