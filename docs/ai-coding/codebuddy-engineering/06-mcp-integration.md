# 给 CodeBuddy 装上真本事：MCP 接入实战与取舍

我第一次听说 MCP 的时候，以为它是"让 AI 什么都能干"的万能开关。装了一堆 MCP Server 之后发现，AI 反而变笨了，因为工具描述把上下文窗口吃光了。

这是我对 MCP 最大的误解。MCP 不是万能开关，它是一个工具连接协议。搞清楚它解决什么、不解决什么，才不会对它有错误期待。

## MCP 到底解决什么

CodeBuddy 默认只能操作本地文件和跑本地命令。但你开发时还要跟很多外部系统打交道：GitHub 上的 Issue 和 PR、数据库里的数据、浏览器里的页面、监控系统的指标。这些靠本地文件和命令够不到。

MCP 就是解决这个的。它是一个开放标准协议，让 CodeBuddy 通过统一接口连接各种外部工具和数据源。连上之后，AI 能直接读 GitHub Issue、查数据库、操作浏览器，不用你手动复制粘贴。

它解决的问题是让 AI 安全地访问本地之外的系统。它不解决的问题：它不替代你的业务权限系统，不替代 CI/CD，也不是万能开关。理解了这个定位，才不会对它抱有不切实际的期待。

## 两种配置方式

CodeBuddy 的 MCP 配置有两种方式。

IDE 图形界面。在 CodeBuddy 面板里点 Settings，切到 MCP 标签页，可以手动添加 MCP Server，也可以在 MCP Market 里一键安装。适合不熟悉配置文件、想快速上手的人。

配置文件。在项目根目录的 `.mcp.json` 里手写（用户级在 `~/.codebuddy/.mcp.json`）。一个 Server 配置大致长这样：

```json
{
  "mcpServers": {
    "github": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}
```

配置文件方式的好处是可以版本管理、团队共享。团队所有人用同一套 MCP 配置，靠 git 同步就行。

我翻官方文档发现几个值得注意的能力。配置有三种作用域：user（全局用户，所有项目）、project（项目级，跟随仓库，适合团队共享）、local（本地/会话级，临时工具），生效优先级 local > project > user。传输类型支持 stdio（本地进程）、SSE 和 HTTP（远程服务）三种。可以控制每个 Server 暴露哪些工具给 AI，做最小权限收敛。

## 哪些 MCP 真值得装

这是我觉得市面上教程最缺的部分。MCP Market 里东西很多，但不是每个都值得装。我按真实价值分三档。

GitHub MCP 是最值得装的之一。连上之后 AI 能直接读 Issue、看 PR diff、查代码上下文。做"根据 Issue 改 bug"或"review 这个 PR"这类任务时，不用你手动把 Issue 内容复制给 AI，它自己能读。我省了大量复制粘贴的时间。

只读数据库 MCP 也值得装。连数据库让 AI 查表结构、跑只读查询，对理解数据模型、写涉及数据库的代码非常有帮助。但我的原则是初期只接只读权限，验证稳定后再考虑写权限。一个能写数据库的 AI，权限控制做不好就是事故。

文件系统 MCP 我装过但后来卸了。CodeBuddy 本身能读文件，文件系统 MCP 提供的能力跟原生重叠，装了只是增加上下文负担，没有增量价值。

浏览器 MCP（比如 Playwright）做前端开发、需要 AI 操作浏览器验证页面时有用。纯后端开发装了也用不上，我做过后端项目装了它，从来没被触发过。

来源不明、没有审计的第三方 MCP 我绝对不装。MCP Server 是能执行代码、访问数据的程序。装一个没审计过的第三方 MCP，等于给它开了访问你系统的权限。这不是 MCP 协议的问题，是供应链安全的问题。只装来源可信、有维护的 MCP。

## 一次装一大堆是最大的坑

我踩过最大的坑就是贪多。刚发现 MCP 的时候，我一口气装了七八个，觉得"能力越多越好"。结果 AI 反而变笨了。

原因很简单：每个 MCP Server 的工具描述都进上下文。装多了，有效上下文窗口被吃掉，AI 反而表现下降。后来我精简到三个最常用的，AI 的表现立刻回来了。

官方文档里有个 `defer_loading` 机制，可以延迟加载不常用的 MCP。善用它，不常用的别常驻。我的原则是按需装，不用的及时关掉。

## 自己写一个 MCP Server

市面上的 MCP 不满足需求时，可以自己写。CodeBuddy 支持标准 MCP 协议，官方 SDK 支持 Python、Node.js、Go。

写一个自定义 MCP Server 的最小步骤：选实现语言用官方 SDK，定义工具接口（工具名、参数 schema、返回格式），处理工具调用请求返回结果，在 CodeBuddy 里配置这个 Server。

我写过一个内部部署系统的 MCP。我们公司有个部署系统，没有现成的 MCP。我写了一个自定义 MCP，暴露"触发部署""查部署状态"几个工具，配进 CodeBuddy。之后让 AI 帮我查部署状态，它直接调我的 MCP，不用我切到浏览器手动操作。

自定义 MCP 的价值在于把你的团队专属工作流接进 AI。这是现成 MCP 给不了的。

## 几个风险和坑

Token 爆炸前面讲过了，每个 MCP 的工具描述都进上下文，装多了吃光窗口。用 `defer_loading` 延迟加载。

权限失控。一个能读代码、改文件、跑命令、连数据库的 AI，再加上一堆 MCP，权限面很大。我的原则是初期只接只读类工具，写权限和危险操作要严格审批。

工具描述注入。恶意的 MCP Server 可以在工具描述里塞提示注入内容，诱导 AI 做不该做的事。只装可信来源的 MCP，这是基本卫生。

超大响应。有些 MCP 工具（比如查日志、查数据库）可能返回超大响应，直接塞进上下文会把窗口撑爆。CodeBuddy 有个超大响应落盘机制，把大结果写到文件而不是直接塞上下文。了解这个机制，避免被大响应卡住。

## 一个判断

MCP 是 CodeBuddy 从"能改本地文件"升级到"能连外部世界"的关键能力。但它不是越多越好，核心是按真实需求取舍。

优先装 GitHub MCP 和只读数据库 MCP，这两个对绝大多数开发者有实打实的提效。按场景装浏览器、搜索类，跟你工作内容匹配才装。谨慎装来源不明的第三方 MCP，供应链安全是真实风险。有专属工作流就自己写，自定义 MCP 是把团队 SOP 接进 AI 的最佳方式。

MCP 的价值不在数量，在每一个都真正减少了你的手动操作。装一个用一个，比装十个用两个强。这是我自己装多了踩坑之后的血泪教训。
