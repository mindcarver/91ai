# 写文档流程

> 更新日期：2025/06

**TL;DR：** 用 Claude Code 写文档分五类：README、ADR、API 文档、迁移说明、变更摘要。每类有固定的提示词模板和产出标准。关键是先让 Claude Code 读代码，再让它写文档——没读过代码的文档一定是废话连篇。

## 为什么写文档需要流程

让 Claude Code 写文档，最容易出现的三个问题：

1. **写了一堆正确的废话。** README 里充斥着"本项目是一个高性能、可扩展的现代化应用"这种话。看的人什么也学不到。
2. **和代码对不上。** 文档里说接口返回 `user_id`，代码里实际返回的是 `userId`。文档比代码过期得更快。
3. **不知道该写什么。** 空文档页面摆在那，你不确定 README 该写到什么程度、ADR 要不要记、迁移说明给谁看。

这三个问题都有同一个解法：给 Claude Code 一个结构化的流程，让它先读代码再动手，并且用模板约束它的输出格式。

前置技能：本篇依赖第 51 篇「新项目快速理解」。如果你还不熟悉怎么让 Claude Code 理解一个项目，先看那篇。

## Claude Code 能写什么文档

不是所有文档都适合交给 Claude Code。先分清楚：

| 文档类型 | Claude Code 适合度 | 原因 |
|---------|-------------------|------|
| README | 高 | 可以从代码和配置中提取准确信息 |
| ADR（架构决策记录） | 中高 | 能记录"做了什么决定"，但"为什么这么决定"需要你补充 |
| API 文档 | 高 | 可以从路由定义、类型声明、注释中生成 |
| 迁移说明 | 高 | 可以对比新旧代码，列出具体变更点 |
| 变更摘要 / Changelog | 高 | 可以从 git diff 中提取结构化摘要 |
| 教程 / Getting Started | 中 | 流程描述可以生成，但用户体验需要你验证 |
| 设计文档 | 低 | 需要产品判断和权衡分析，不适合全自动生成 |

下面按类型逐个讲。

## README

README 是项目的门面。一个好的 README 回答三个问题：这是干什么的、怎么装、怎么跑。

### 提示词模板

```text
阅读项目的 package.json（或 pyproject.toml / Cargo.toml / go.mod）、
顶层目录结构和入口文件，然后生成 README.md。

要求：
1. 一句话描述这个项目是干什么的（不要用"高性能""现代化"之类的修饰词）
2. 列出环境要求（Node 版本、Python 版本等）
3. 安装步骤（从 clone 到装完依赖）
4. 启动命令（开发模式）
5. 其他常用命令（测试、构建、lint）
6. 项目结构（只列顶层目录，每个目录一句话）
7. 如果有配置文件需要填写（.env 等），列出必填项

不要包含：贡献指南、License、Changelog——这些是单独的文件。
不要编造任何信息。如果某个信息在代码里找不到，标注 [待补充]。
```

### 关键约束

- **"不要编造"这条必须写。** Claude Code 在找不到信息时，倾向于编造一个看起来合理的命令或配置。`[待补充]` 标记比假信息强一万倍。
- **不要让它写"贡献指南"。** 这是 README 的常见膨胀点。贡献指南应该单独放在 `CONTRIBUTING.md` 里，而且需要你根据团队实际情况写。
- **控制长度。** README 超过 200 行就没人看了。在提示词里加"控制在 150 行以内"。

### 更新已有的 README

不要让 Claude Code 从头重写已有的 README。重写会丢失你手动维护的内容。

```text
对比当前 README.md 和实际代码，找出过时或不准确的部分。
列出需要更新的条目，每条包含：
- 当前 README 里的描述
- 实际代码中的情况
- 建议的修改

不要直接修改文件，先给我看差异列表。
```

这样你可以在 Claude Code 的建议基础上做编辑，保留 README 中仍然有效的部分。

## ADR（架构决策记录）

ADR 记录"我们做了一个技术决定，原因是这些"。它的价值不在于记录决定本身（代码已经体现了），而在于记录"当时为什么这么做"——这是代码里看不出来的。

### ADR 模板

让 Claude Code 按 Michael Nygard 的 ADR 格式生成：

```text
为以下架构决策生成一份 ADR 文档：
[描述决策内容，比如"使用 PostgreSQL 替代 MongoDB 作为主数据库"]

使用以下格式：
1. 标题：简短的决策名称
2. 状态：Proposed / Accepted / Deprecated / Superseded
3. 背景：为什么需要做这个决定？触发条件是什么？
4. 决策：我们决定怎么做？
5. 考虑过的方案：至少列出 2 个备选方案及其优缺点
6. 后果：这个决策带来的正面和负面影响

文件命名：docs/adr/NNNN-简短标题.md
```

### Claude Code 能做的部分

- **背景**：从代码变更历史中提取触发条件
- **决策**：从代码实现中反推做了什么决定
- **考虑过的方案**：如果代码里有注释或废弃的实现，可以识别出来

### Claude Code 做不好的部分

- **"为什么选 A 而不选 B"的理由**：这个需要你补充。Claude Code 不知道团队的政治因素、预算限制、时间压力。
- **后果评估**：它倾向于只写正面后果。你需要追问"有什么潜在风险或负面后果"。

### 批量生成 ADR

如果项目从来没有记录过 ADR，可以从代码中的关键转折点反推：

```text
扫描 git log，找出以下类型的提交：
1. 涉及框架或数据库更换的
2. 涉及目录结构重大调整的
3. 涉及外部依赖增删的
4. commit message 中包含 "migrate"、"replace"、"switch"、"refactor" 的

为每个关键提交生成一份简短的 ADR 摘要（不需要完整格式），
包含：提交哈希、日期、做了什么变更、可能的原因。

只列清单，不生成完整文档。
```

拿到清单后，你决定哪些值得展开成完整的 ADR。

## API 文档

API 文档最怕的是和实际接口不一致。用 Claude Code 从代码生成可以最大程度保证准确性。

### 从路由定义生成

```text
阅读 src/routes/ 目录下所有路由文件，生成 API 文档。

每个接口包含：
- HTTP 方法和路径
- 简要描述（一句话，不要超过 20 字）
- 请求参数（Query、Body、Path 参数分开列出，标注类型和是否必填）
- 响应格式（成功的状态码和返回体结构）
- 需要的认证方式（如果有）

按资源分组，不要按文件分组。
输出格式：Markdown 表格。
```

### 从类型声明生成

如果项目用了 TypeScript 或类似的类型系统：

```text
阅读 src/types/ 目录下的类型定义文件，
为每个对外暴露的 API 响应类型生成文档。

格式：
- 类型名称
- 每个字段的含义、类型、是否可能为空
- 字段之间的关系（如果有嵌套）
- 枚举值（如果有）

如果类型定义里有 JSDoc 注释，优先使用注释中的描述。
```

### 保持 API 文档同步

每次改接口后，不要手动更新文档。让 Claude Code 对比：

```text
对比 docs/api.md 和 src/routes/ 下的实际路由定义，
列出所有不一致的地方：
- 文档里有但代码里没有的接口
- 代码里有但文档里没有的接口
- 参数或响应格式不一致的接口

只列差异，不自动修改文档。
```

把这种检查加到 CI 或 pre-push hook 里更好。但至少每次发版前跑一次。

## 迁移说明

迁移说明写给用你项目的人看。它回答的问题是："我从旧版本升到新版本，要改什么？"

### 从 git diff 生成

```text
对比 v2.0.0 和 v3.0.0 之间的 git diff，生成迁移说明。

重点关注：
1. 破坏性变更（Breaking Changes）——API 签名变了、配置格式变了、行为变了
2. 废弃的功能——还在但将来会删的东西
3. 新增的功能——不需要迁移但需要知道的
4. 配置变更——环境变量、配置文件格式变化
5. 依赖变更——新增或升级了什么

格式：
- 按影响程度排序：破坏性变更 > 废弃 > 配置变更 > 新功能
- 每条包含：旧写法 → 新写法 的对比
- 不需要改动的部分不要写（迁移说明不是 Changelog）
```

### 单次变更的迁移说明

不是每次都有大版本升级。一个小改动如果影响下游使用者，也需要说明：

```text
这次 PR 把 UserService.login() 的返回值从
{ token: string } 改成了 { accessToken: string, refreshToken: string }。

生成一份面向下游使用者的迁移说明：
1. 什么变了
2. 旧代码怎么写的
3. 新代码应该怎么写
4. 如果不改会怎样
```

### 迁移说明和 Changelog 的区别

它们不是同一个东西：

- **Changelog**：按时间列出所有变更，面向维护者。
- **迁移说明**：只列出影响使用者的变更，给出"旧→新"的对照，面向使用者。

如果变更很小，Changelog 够用。如果有破坏性变更，迁移说明不能省。

## 变更摘要

变更摘要是从 git diff 或 PR diff 中提炼出"到底改了什么"的短文档。最常见的场景是 PR 描述和发版说明。

### PR 描述

```text
根据当前分支和 main 分支的 diff，生成 PR 描述。

格式：
## Summary
1-3 句话说明这个 PR 做了什么。

## Changes
- 按功能模块分组列出变更
- 每条以动词开头（添加、修复、修改、删除）
- 不列出纯格式化或重命名的变更

## Test Plan
- 列出可以验证这些变更的测试方法
- 包括自动化测试和手动验证步骤

不要包含：代码片段、diff 片段、实现细节。
PR 描述是给 review 的人看的，不是给自己看的。
```

### 发版说明（Release Notes）

```text
对比 v1.4.0..v1.5.0 的所有提交，生成 Release Notes。

分类：
- Features：新功能
- Fixes：Bug 修复
- Breaking：破坏性变更
- Dependencies：依赖更新

每条以动词开头，简洁描述。
如果一个功能有多个相关提交，合并成一条。

不要包含：内部重构、CI 配置变更、不影响使用者的改动。
```

### 自动生成变更摘要的技巧

如果你经常需要生成变更摘要，可以在 CLAUDE.md 里加一条规则：

```markdown
## Documentation Rules
- 每次 PR 都自动生成 PR 描述（使用上面的模板）
- 发版时生成 Release Notes
- API 变更时更新 docs/api.md
```

这样 Claude Code 会在你提交代码或发版时自动提醒你做文档同步。

## 常见问题

### Claude Code 生成的文档太长怎么办

在提示词里加长度限制。"控制在 150 行以内"或"每个接口描述不超过 3 行"。如果不限制，Claude Code 倾向于写得很详细——详细到没人愿意看。

### 文档和代码不一致怎么办

不要手动维护文档。每次改代码后，让 Claude Code 对比文档和代码的差异。差异检查比全文重写更安全——你不会丢失手动维护的内容。

### ADR 写了没人看怎么办

ADR 的价值不在于被反复阅读，而在于被搜索。当有人问"为什么用 PostgreSQL 不用 MySQL"时，能搜到一份记录就够了。把 ADR 放在 `docs/adr/` 目录下，文件名用 `NNNN-标题.md` 格式，方便排序和引用。

### 迁移说明该在什么时候写

有破坏性变更时必须写。"破坏性变更"的定义：使用你项目的人需要改自己的代码才能升级。内部重构不叫破坏性变更——它不影响使用者。

### 变更摘要和 commit message 什么关系

commit message 记录单次提交。变更摘要汇总多次提交。好的 commit message 让生成变更摘要更容易——如果每个 commit 都说清了"改了什么"，Claude Code 只需要归类合并；如果 commit 都是"fix bug"这种，它只能硬猜。

## 关键要点

- **先读代码再写文档。** 没读过代码的文档一定是废话。所有提示词模板的第一步都是"阅读相关代码"。
- **用模板约束格式和长度。** 不加约束的文档生成会产出又长又空的内容。
- **不要从头重写已有文档。** 对比差异、增量更新，比重写安全。
- **API 文档从代码生成，不从文档复制。** 路由定义、类型声明是 source of truth，文档是它的投影。
- **迁移说明和 Changelog 不是同一件事。** 迁移说明面向使用者，回答"我要改什么"。Changelog 面向维护者，回答"改了什么"。
- **ADR 记录的是"为什么"，不是"是什么"。** 代码已经体现了决定本身，ADR 补充的是上下文和理由。
