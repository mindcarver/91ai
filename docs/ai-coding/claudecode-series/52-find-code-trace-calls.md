# 找代码与追调用链

> 更新日期：2025/06

**TL;DR：** 用 Claude Code 在陌生代码库里找代码，有三种入口：功能名、错误栈、接口入口。找到文件后，追踪数据流用固定模板——从调用点往前追源头、往后追影响。关键是每次只追一条链，不要一口气让 Claude Code "分析整个模块"。

## 为什么这很重要

上一篇（第 51 篇）讲了用 Claude Code 快速理解新项目的五步流程。那篇解决的是"项目长什么样"的问题。这一篇解决的是更具体的场景：你已经在项目里了，需要找到某个功能的代码在哪，或者搞清楚一段数据从哪来、到哪去。

实际工作里，这种需求出现频率极高：

- 产品说"支付失败的那个弹窗文案改一下"——你得先找到弹窗在哪个文件里
- 线上报了一个 `TypeError` ——你得从错误栈定位到出问题的函数，再追溯到数据来源
- 需求是"在用户注册后发一封欢迎邮件"——你得先找到注册逻辑在哪、邮件发送在哪，才能把两段代码串起来

这些场景的共同点：你不知道代码在哪，但你知道你要找什么。区别在于你手里拿的线索不同——有时候是功能名，有时候是错误栈，有时候是一个 URL。

下面按三种入口分别讲操作方法。

## 三种入口

在进入具体方法之前，先理清三种找代码的场景和它们对应的线索类型：

| 入口 | 你手里的线索 | 适用场景 |
|------|-------------|---------|
| 功能名 | "购物车"、"用户登录"、"导出报表" | 需求驱动，知道业务含义 |
| 错误栈 | `TypeError: Cannot read... at cart.js:47` | Bug 驱动，有报错信息 |
| 接口入口 | `POST /api/users`、`/dashboard` 路由 | 架构驱动，知道 URL 或路由 |

三种入口的搜索策略不一样。功能名需要模糊匹配到精确，错误栈可以直接定位到行号，接口入口需要理解框架的路由注册机制。

## 从功能名定位

最常见也最容易翻车的场景。你说"找购物车的代码"，Claude Code 可能找到十几个文件，里面混杂着测试文件、类型定义、文档、废弃代码。要精准定位，分两步走。

### 第一步：模糊搜索，缩小范围

```text
搜索这个项目里和"购物车"相关的代码。
列出所有包含 cart、shopping、checkout 关键词的文件，
排除测试文件和类型定义文件。
```

Claude Code 会用 Grep 工具搜索关键词，然后用 Glob 按文件名搜索。排除测试和类型定义是为了减少噪音——这两个类别的文件通常数量多但不是你要改的地方。

### 第二步：精确定位，找到入口文件

拿到文件列表后，挑出最关键的 2-3 个文件来读：

```text
从上面列出的文件里，找出购物车的核心业务逻辑入口。
重点看：
1. 哪个文件定义了购物车的主要数据结构或状态
2. 哪个文件包含了"加入购物车"和"结算"的核心逻辑
3. 列出这些核心函数的函数名和所在文件

不要列出所有函数，只列核心的 3-5 个。
```

限制输出数量（"只列 3-5 个"）很重要。不加这个约束，Claude Code 会把文件里每个函数都列出来，淹没真正关键的信息。

### 一个具体例子

假设你在一个电商项目里找"优惠券"相关代码：

```text
找这个项目里处理优惠券的核心代码。

步骤：
1. 搜索 coupon、discount、promo、voucher 关键词
2. 排除 test 文件和 mock 数据
3. 找到优惠券的核心业务文件（不是 UI 组件）
4. 列出 3 个最核心的函数：创建优惠券、应用优惠券、验证优惠券

每个函数给出一行说明：文件路径、函数名、它做什么。
```

Claude Code 返回的结果可能像这样：

```
1. src/services/coupon.service.ts → createCoupon() → 创建优惠券，写入数据库
2. src/services/coupon.service.ts → applyCoupon() → 校验优惠券有效性，计算折扣
3. src/utils/coupon-validator.ts → validateCoupon() → 检查过期时间、使用次数、适用商品
```

拿到这个结果后，你可以直接打开这三个文件开始工作。

## 从错误栈定位

有报错信息时，定位比功能名搜索精确得多。错误栈直接告诉了你文件名和行号。但报错位置和真正的 bug 位置经常不在同一个文件——这就是为什么需要追调用链。

### 第一步：从报错位置读取上下文

```text
项目报了一个错误：

TypeError: Cannot read properties of undefined (reading 'price')
  at calculateTotal (src/cart/calculator.js:47)
  at processCheckout (src/cart/checkout.js:23)
  at handlePost (src/routes/cart.js:15)

请读取 src/cart/calculator.js 的第 40-55 行，
解释第 47 行出了什么问题。
只分析报错位置，不追踪调用链。
```

先看报错位置本身。`price` 是 `undefined`，说明传入的对象里没有 `price` 字段。到这一步你只知道"这里炸了"，还不知道"为什么这里的数据不对"。

### 第二步：往前追溯数据来源

```text
从 calculateTotal 函数开始，往前追踪数据来源：
1. 谁调用了 calculateTotal？参数是怎么传进来的？
2. 传进来的数据（商品列表）在哪构造的？
3. 在什么条件下商品对象会缺少 price 字段？

列出每一跳的文件、函数名和行号。
```

这一步是关键。Claude Code 会沿着调用栈往回追，逐层读取代码。它的工作方式是：读取调用方函数 → 看参数怎么来的 → 再追上一层的调用方 → 直到找到数据的构造点。

### 第三步：确认根因

```text
根据上面的追踪结果，price 变成 undefined 的可能原因是什么？
给出最可能的那个，并说明判断依据。
```

让 Claude Code 做判断而不是列一堆可能性。"给出最可能的那个"迫使它给出一个明确的答案，你再去验证这个答案是否正确。

### 错误栈定位的注意事项

**不要只看栈顶。** 栈顶是报错位置，但根因可能在栈底的某个调用方里。比如上面的例子，栈顶是 `calculator.js:47`，但根因可能是 `checkout.js:23` 在组装商品列表时漏掉了 `price` 字段。

**不要忽略中间层。** 有时候中间层的某个函数做了数据转换，把 `price` 字段改名了或者过滤掉了。Claude Code 有时候会跳过中间层直接从栈顶跳到栈底，你需要提醒它逐层检查。

**异步代码的栈可能不完整。** `Promise`、`async/await`、`setTimeout` 里的错误，栈信息经常被截断。这时候需要结合日志或者手动搜索调用方来补充。

## 从接口入口定位

第三种入口：你知道一个 URL 或者路由名称，需要找到处理这个请求的代码。这在后端项目里特别常见。

### 从 URL 到路由处理函数

```text
找到处理 POST /api/users 请求的代码。
从这个接口入口出发，列出：
1. 路由定义在哪个文件
2. 路由处理函数（handler/controller）在哪个文件
3. 处理函数调用的 service 层函数
4. service 层调用的数据访问层函数

每一跳给出文件路径和函数名。
```

Claud Code 的工作流程是：先在路由配置文件里找到 `/api/users` 的注册位置，然后读取对应的 handler 函数，再追到 service 层和 repository 层。不同框架的路由注册方式不同：

| 框架 | 路由注册位置 | 搜索方式 |
|------|------------|---------|
| Express | `app.post('/api/users', ...)` | 搜索 `'/api/users'` 或 `"/api/users"` |
| NestJS | `@Controller('users')` + `@Post()` | 搜索 `users` controller |
| Next.js | `app/api/users/route.ts` | 按文件路径直接找 |
| Spring Boot | `@RequestMapping("/api/users")` | 搜索 `"/api/users"` |

如果你知道项目用的是什么框架，在提示词里说出来会让搜索更精准：

```text
这是一个 Express 项目。找到 POST /api/users 的完整处理链。
```

### 从页面路由到组件

前端项目里类似的场景：知道页面 URL，找对应的组件代码。

```text
找到 /dashboard/settings 页面对应的 React 组件。
从这个页面出发，列出：
1. 路由定义在哪个文件
2. 页面组件在哪个文件
3. 页面用了哪些子组件
4. 页面数据从哪些 API 获取
```

React Router 的路由定义方式很多样（集中式路由配置 vs 文件系统路由），Vue Router 也一样。让 Claude Code 先找路由配置文件，再从配置里追踪到具体组件。

## 追数据流

找到代码位置只是第一步。大多数情况下你还需要理解数据怎么在多个函数和文件之间流动。这一节讲怎么用 Claude Code 追踪一条完整的数据流。

### 追踪模板

数据流追踪有一个固定的提示词模板，适用于大多数场景：

```text
追踪 [具体功能] 的完整数据流。

从 [起点] 到 [终点]，列出数据经过的每一跳：
1. [起点]：数据怎么产生的
2. [中间处理]：数据经过了哪些函数、做了什么转换
3. [终点]：数据最终怎么被消费（存储、显示、发送）

每一跳给出：
- 文件路径和行号
- 函数名
- 数据在这一步发生了什么变化

格式示例：
src/api/order.ts:15 → fetchOrders() → 从后端拿到订单原始数据
src/services/order.ts:23 → normalizeOrders() → 把日期字符串转成 Date 对象
src/stores/order.ts:30 → setOrders() → 存入 Zustand store
src/pages/OrderList.tsx:45 → OrderList 组件 → 渲染订单列表
```

这个模板的核心是"每一跳"——强制 Claude Code 逐层列出数据传递过程，而不是给一段概括性的描述。

### 追源头：从消费点反向追踪

有时候你只有一个终点——比如一个显示用户头像的组件——需要追到头像数据的源头。

```text
在 src/components/UserAvatar.tsx 里，头像图片的 URL 来自哪个 prop？
追踪这个 prop 的数据源头：
1. UserAvatar 的父组件是谁？
2. 父组件传了什么值给 avatarUrl prop？
3. 这个值从哪个 API 或 store 来？
4. API 返回的原始数据格式是什么？

一直追到数据最初产生的地方。
```

"一直追到数据最初产生的地方"——这句很重要。没有这个约束，Claude Code 可能只追一层就停了。

### 追影响：从变更点正向追踪

另一种常见场景：你要改一个函数，需要知道改了之后会影响谁。

```text
我要修改 src/utils/formatDate() 函数的返回格式。
请列出所有调用 formatDate 的地方：
1. 直接调用方（在同一文件或不同文件里调用这个函数）
2. 间接影响方（调用方被其他代码引用，形成调用链）

对每个调用方，说明它怎么使用 formatDate 的返回值，
以及返回格式改变会不会导致它出问题。
```

这类"影响分析"在重构和修 bug 时特别有用。改一行代码可能影响十几个地方，不提前查清楚就会引入回归。

## 实战技巧

### 技巧 1：用 Glob 找文件比 Grep 快

如果你要找的是"购物车相关的文件"，先按文件名搜比按内容搜快得多：

```text
列出文件名包含 cart、checkout、basket 的所有文件。
排除 node_modules、dist、.git 目录。
```

Claude Code 的 Glob 工具扫描文件名比 Grep 扫描文件内容快一个数量级。先靠文件名缩小范围，再在少数文件里做内容搜索。

### 技巧 2：搜索时排除噪音

大项目里搜索结果经常被测试文件、mock 文件、自动生成的文件污染。在提示词里加排除条件：

```text
搜索 "processPayment" 函数的定义和调用，
排除：
- *.test.* 和 *.spec.* 文件
- *.mock.* 文件
- src/generated/ 目录
- node_modules/ 目录
```

排除条件越具体，搜索结果越干净，Claude Code 需要读取的文件越少，响应速度越快。

### 技巧 3：用子代理做大规模搜索

当项目文件数超过几百个时，搜索和追踪会消耗大量上下文。用子代理（详见第 65 篇）来做搜索，主会话保持干净：

```text
用子代理完成以下搜索任务：

搜索 src/ 目录下所有调用 PaymentService.process() 的地方。
对每个调用方，说明：
1. 在哪个文件的哪一行
2. 传了什么参数
3. 怎么处理返回值和异常

返回一份结构化的调用方清单。
```

子代理在独立上下文里完成搜索，把结果汇总成一份简洁的报告返回给主会话。主会话拿到结果后继续做分析或修改，不用被搜索过程中的大量文件内容占满上下文窗口。

### 技巧 4：搜索结果要求结构化输出

Claude Code 默认的搜索结果是叙述式的——"这个函数在某某文件里被调用，那个文件也有调用"。改成表格或列表格式，后续引用更方便：

```text
列出 calculateTotal 的所有调用方。
用以下格式输出：

| 调用方文件 | 调用方函数 | 行号 | 传参方式 | 是否处理异常 |
|-----------|-----------|------|---------|-------------|
```

结构化输出的好处：一眼看清全貌，而且可以在后续提示词里直接引用表格中的条目——"修改第 3 行那个调用方的传参逻辑"。

### 技巧 5：分步追踪，不要一口气追完

一条完整的数据流可能跨越 5-8 个文件。让 Claude Code 一次追完，它会因为中间读太多文件导致上下文膨胀，后面的分析质量下降。

更稳的做法是分段追：

```text
第一轮：
追踪 POST /api/orders 请求从路由到 service 层的路径。
列出经过的文件和函数。

第二轮（基于第一轮结果）：
从 orderService.create() 开始，追踪到数据库写入。
列出经过的文件和函数。

第三轮（基于前两轮结果）：
追踪 create() 返回后，响应怎么回到客户端。
列出经过的中间件和响应格式化逻辑。
```

每轮只追 2-3 跳，保持上下文精简。三轮的结果拼起来就是完整的数据流。

## 关键要点

- **三种找代码入口**：功能名（模糊到精确）、错误栈（从栈顶往回追）、接口入口（从路由到 handler）。选哪种取决于你手里的线索。
- **搜索先排噪音**：排除测试文件、mock 文件、生成目录。搜索范围越小，结果越精准，速度越快。
- **追数据流用固定模板**：每一跳列出文件、函数、数据变化。不要接受概括性描述。
- **分步追踪比一口气追完更靠谱**：每轮 2-3 跳，保持上下文精简。多轮结果拼出完整链路。
- **追影响和追源头是两个方向**：从变更点正向追影响（谁会受影响），从消费点反向追源头（数据从哪来）。方向搞反了会浪费很多时间。
- **搜索结果要求结构化输出**：表格或列表格式比叙述式更容易验证和引用。
