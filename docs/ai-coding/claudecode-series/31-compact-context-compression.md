# `/compact` 与上下文压缩

> 更新日期：2025/06

**TL;DR：** 上下文窗口是 Claude Code 最稀缺的资源。对话越长、读的文件越多，可用空间越少。`/compact` 会把对话历史压缩成一段摘要，释放空间继续工作。代价是丢失细节——具体代码片段、中间讨论、之前读过的文件内容都会被精简。学会判断什么时候压缩、压缩后怎么补回丢失的信息，是长对话的基本功。

## 上下文怎么消耗

### 窗口大小

Claude Code 的上下文窗口大约 200K token。听起来很大，但系统提示词、工具定义、CLAUDE.md 这些固定内容已经占了 25-35K。剩下大约 100-120K 给对话内容。

### 什么吃 token 最多

按消耗量排序：

1. **文件内容**：Claude 读取的源代码文件。一个大文件可能占 5-10K token
2. **工具输出**：`grep`、`ls`、测试结果等。跑一次 `find` 输出 50 个文件名，几千 token 就没了
3. **对话历史**：你说的话 + Claude 的回复。每一轮都在累积
4. **Claude 的思考过程**：扩展思考模式（extended thinking）会让回复更占空间

### 消耗速度

一个典型的工作会话：

- 前 5 轮：轻量对话 + 读几个小文件 → 占 20-30%
- 10-15 轮：开始改代码、跑命令 → 占 50-60%
- 20 轮以上：上下文接近饱和 → Claude 开始"忘记"之前的讨论

## 自动 vs 手动压缩

### 自动压缩

Claude Code 在上下文占用接近上限时会自动触发压缩。你会看到一条提示："Compressing conversation history..."

自动压缩的特点：

- 不需要你做任何操作
- 压缩时机由系统判断，通常在占用达到 85-90% 时触发
- 压缩后会生成一段摘要替代之前的对话

问题在于：自动压缩通常发生在你最不想丢上下文的时候——比如正在做复杂的重构，对话里积累了大量上下文。

### 手动压缩：`/compact`

在任意时刻输入 `/compact`，主动压缩对话历史。

```text
> /compact
```

可以附加指令，告诉 Claude 压缩时保留什么：

```text
> /compact 保留所有 API 接口设计的讨论，可以丢弃代码格式化的细节
```

手动压缩的优势：

- 你可以选择时机——比如一个大任务刚完成、准备开始下一个时
- 可以指定保留什么、丢弃什么
- 避免 Claude 在关键时刻"断片"

### 什么时候该手动压缩

| 场景 | 建议 |
|------|------|
| 一个任务完成，准备开始下一个 | 压缩，保留任务结论 |
| 对话超过 15 轮 | 看一眼 Ctrl+O，超过 60% 就压缩 |
| Claude 开始重复问同样的问题 | 上下文已经乱了，压缩或 `/clear` |
| 准备讨论新的、不相关的话题 | `/clear` 比压缩更合适 |

## 压缩后丢失什么

### 会保留的

- 对话中的关键决策和结论
- 当前正在处理的文件路径和任务状态
- 压缩指令中明确要求保留的内容

### 会丢失的

- **具体的代码片段**：Claude 之前读过的文件内容会被精简成摘要
- **中间讨论过程**：试探性的方案、被否决的思路会被删除
- **工具调用的详细输出**：之前的 grep 结果、测试输出等不会保留
- **精确的行号和位置信息**：压缩后变成"某文件的某处"

### 实际影响

举个例子。压缩前：

```
你：src/api/auth.ts 第 42 行的 token 校验逻辑有问题
Claude：我看了，JWT 的过期判断少了一个条件。应该改成...
[详细代码讨论 5 轮]
```

压缩后，这段对话可能变成：

```
摘要：讨论了 src/api/auth.ts 中 JWT token 校验逻辑的修复。
```

"第 42 行"这个精确信息没了。Claude 重新工作时需要再读一遍文件。

## 恢复策略

压缩丢信息是必然的。关键是减少影响。

### 策略 1：压缩前让 Claude 做总结

```text
> 总结一下我们目前讨论的所有改动，列出每个文件的修改要点
```

等 Claude 回复完，再执行 `/compact`。这样压缩时会以这段总结为基础，比 Claude 自己摘要更准确。

### 策略 2：用 CLAUDE.md 记录持久信息

如果你在对话中确定了重要的架构决策，把它写进 CLAUDE.md 或 rules 文件。这些文件不受对话压缩影响。

### 策略 3：关键文件重新读取

压缩后，Claude 对之前读过的文件内容只剩摘要。需要继续修改某个文件时，先让 Claude 重新读一遍：

```text
重新读一下 src/api/auth.ts，然后继续刚才的修改
```

### 策略 4：该用 `/clear` 就用 `/clear`

如果压缩后 Claude 还是不在状态，不如直接 `/clear` 重新开始。新的对话上下文干净，Claude 会重新读 CLAUDE.md 和项目文件，比带着压缩后的残缺信息工作效果好。

## 关键要点

- 上下文窗口大约 200K token，实际可用约 100-120K
- 文件内容和工具输出是最大的 token 消耗者
- 超过 15 轮对话或占用超过 60% 时考虑手动 `/compact`
- 压缩会丢失具体代码片段、行号、中间讨论
- 压缩前让 Claude 做总结，关键决策写入 CLAUDE.md
- 压缩效果不好就 `/clear` 重来，别硬撑
