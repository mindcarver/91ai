# 修 Bug 标准流程

> 更新日期：2025/06

**TL;DR：** 修 bug 分五步：复现问题 → 定位根因 → 比较方案 → 最小修复 → 验证修复。每一步都有对应的 Claude Code 提示词模板。不要直接说"帮我修这个 bug"——那是把方向盘交给一个还没看过代码的人。

## 为什么修 bug 需要流程

没有流程时，Claude Code 修 bug 常见三种翻车：

1. **改了一堆不该改的文件。** 你说"登录有问题"，它把 auth 模块重写了一遍，顺带改了中间件和路由。
2. **治了症状，没治根因。** 报错是 `NullPointerException`，它加了个 `if (x != null)` 打发了事。真正的问题是上游没传数据。
3. **改完了不验证。** 代码看起来对了，但没跑测试。上线之后发现引入了新 bug。

这三个问题的根源一样：你让 Claude Code 自由发挥，没有给它结构和约束。修 bug 的工作流就是用固定的步骤把这个约束建立起来。

## 第一步：复现问题

修 bug 的前提是能复现。如果你自己都不确定怎么触发这个 bug，Claude Code 更不可能知道。

### 你需要提供什么

- **报错信息**：完整的错误栈，不是截断的第一行
- **复现步骤**：做了什么操作、在什么条件下出现的
- **预期行为**：正常应该是什么样子

### 提示词模板

```text
Bug 描述：用户在购物车页面点击"结算"后，页面白屏。

复现步骤：
1. 登录后往购物车加 3 件商品
2. 删除第 2 件
3. 点"结算"

报错信息：
TypeError: Cannot read properties of undefined (reading 'price')
  at calculateTotal (cart.js:47)

预期行为：点结算后跳转到订单确认页。

请先阅读相关代码，确认你能理解这个报错出现的原因，
再提出修复方案。
```

关键是最后一句——"先阅读、再提方案"。这阻止了 Claude Code 看到 bug 描述就立刻动手改代码。

### 如果 bug 不能稳定复现

有些 bug 是间歇性的，或者只在生产环境出现。这时候把你有的信息都给出去：

```text
这个 bug 在生产环境间歇性出现，本地无法稳定复现。
已知条件：
- 出现在高并发场景
- 日志里看到 timeout 错误
- 涉及的接口是 /api/orders

请分析 src/api/orders/ 目录下的代码，
找出可能导致并发 timeout 的代码路径。
只分析，不改代码。
```

"只分析，不改代码"这个约束很重要。没有复现条件时，直接让 Claude Code 改代码风险太高——它可能改到错误的地方。

## 第二步：定位根因

复现之后，需要从报错位置追溯到真正出问题的代码。报错位置和根因经常不在同一个文件。

### 提示词模板

```text
根据上一步的分析，现在需要定位根因。

报错位置是 cart.js:47 的 calculateTotal 函数，
它访问了一个 undefined 的 price 属性。

请从 calculateTotal 开始，往前追踪数据来源：
1. 这个函数的参数从哪里传入？
2. 商品列表是怎么构造的？
3. 删除商品时，列表的更新逻辑是什么？

只追踪调用链，不做任何代码修改。
```

这一步对应了系列第 52 篇的调用链追踪方法。如果你已经读过那篇，这里的提示词就是把它直接用在 bug 场景里。

### 用子代理做独立分析

如果代码量大、调用链复杂，用子代理来做根因分析，避免撑爆主会话的上下文：

```text
用子代理分析这个 bug 的根因：

错误：TypeError: Cannot read properties of undefined (reading 'price')
位置：cart.js:47 calculateTotal

子代理需要：
1. 阅读 cart.js 和相关的数据源文件
2. 追踪从"删除商品"到"计算总价"的完整数据流
3. 找出 price 变成 undefined 的具体原因
4. 返回：根因文件名、行号、原因解释
```

子代理跑完后，你会拿到一份简洁的根因报告，而不是把一堆文件内容塞进主会话。

## 第三步：评估修复方案

找到根因后，不要急着改代码。先让 Claude Code 列出可能的方案，你来做选择。

### 提示词模板

```text
根因已经定位：删除商品时，cart.items 数组用 splice 移除了元素，
但后续的 calculateTotal 没有对空位做检查。

请给出至少两种修复方案，每种方案包含：
- 具体改哪个文件的哪些行
- 改动描述
- 优点
- 缺点或风险

不要修改任何代码。
```

### 怎么选方案

几个判断标准：

| 标准 | 说明 |
|------|------|
| 改动文件数 | 越少越好，理想情况只改 1 个文件 |
| 改动行数 | 越少越好 |
| 是否引入新逻辑 | 不引入最好。如果引入，复杂度要可控 |
| 是否影响其他调用方 | 改公共函数要格外小心 |
| 有没有现成的测试能验证 | 有测试的方案比没有的更安全 |

Claude Code 给的方案不一定是最优的。它倾向于给出"改动最少"的方案，但有时候改动最少不等于最正确。比如加一个 null check 改动最少，但从上游保证数据完整才是更好的做法。

## 第四步：最小修复

选定方案后，开始改代码。核心原则：**只改该改的，不夹带私货。**

### 提示词模板

```text
采用方案 B：在 removeItem 函数里，删除商品后
重新映射 items 数组，确保索引连续。

要求：
- 只修改这一个函数
- 不优化其他代码
- 不重构文件结构
- 不添加"顺便改进"的逻辑

修改完成后，展示改动前后的 diff。
```

"不优化、不重构、不顺便改进"——这三条约束需要每次都写。Claude Code 有一个倾向：它觉得既然在改这个文件了，不如顺便把旁边那个不太好的写法也改了。这种"顺手优化"是引入回归的头号原因。

### 改完之后立刻检查

不要等 Claude Code 说"修改完成"就算了。让它自己展示 diff：

```text
展示刚才修改的 diff，逐行说明每一处改动的理由。
标注哪些改动是必须的，哪些是可选的。
如果发现任何"顺手改动"，立刻撤回。
```

如果发现了不该改的地方：

```text
撤回对 utils.js 的修改。那个改动和这个 bug 无关。
只保留对 cart.js 的修改。
```

如果 Claude Code 改了太多东西，直接用 `Esc + Esc`（或 `/rewind`）回退到修改前的状态，重新来。

## 第五步：验证修复

修复完成不等于 bug 修好了。需要验证三件事：

### 1. 跑测试

```text
运行和购物车相关的测试。如果测试失败，分析失败原因，
判断是测试本身需要更新还是修复引入了新问题。
不要自动修改测试来让测试通过。
```

"不要自动修改测试"这一条很关键。Claude Code 有时候会把测试改成匹配自己写的代码，而不是检查代码是否正确。

### 2. 手动验证复现场景

```text
根据最初的复现步骤：
1. 登录后往购物车加 3 件商品
2. 删除第 2 件
3. 点"结算"

请确认现在的代码逻辑能正确处理这个流程。
列出每一步涉及的函数调用和返回值。
```

### 3. 检查回归

```text
检查这个修改是否影响了其他调用 calculateTotal 的地方。
列出所有调用方，说明为什么不会受到影响，
或者如果会受到影响，影响是什么。
```

### 完整的验证提示词

把三步合成一个提示词：

```text
修复已完成。现在执行验证：

1. 运行相关的单元测试（npm test -- --grep cart）
2. 走一遍原始复现步骤，确认不再白屏
3. 检查所有调用 calculateTotal 的地方，确认没有回归

最后给出验证总结：
- 测试结果
- 复现验证结果
- 回归风险评估
- 剩余需要注意的点
```

## 一个完整的 bug 修复例子

**Bug：** 用户修改个人资料后，头像消失了。

**第一步——复现：**

```text
Bug：修改个人资料（昵称、签名）后，头像变成默认头像。

复现步骤：
1. 用户已上传头像
2. 进入"编辑资料"页面
3. 修改昵称，不碰头像
4. 点保存
5. 头像消失了

报错信息：无报错，但数据库里 avatar_url 变成了 null。

预期行为：修改昵称不应该影响头像字段。
```

**第二步——定位根因：**

```text
追踪数据流：编辑资料页提交 → updateProfile API → updateUser Service → 数据库。

关键问题： updateUser 接收的前端数据里有没有 avatar_url 字段？

请阅读以下文件并追踪：
1. 前端提交了哪些字段（src/pages/Profile.tsx）
2. API 接收了哪些字段（src/api/user.ts）
3. updateUser 怎么构建 UPDATE 语句（src/services/user.ts）

只追踪，不改代码。
```

Claude Code 分析后发现：前端编辑页只传了 `{ nickname, bio }`，但 `updateUser` 服务用 `Object.assign` 把传入数据覆盖到用户对象上——包括 `avatar_url: undefined`，数据库就把这个字段写成了 `null`。

**第三步——方案比较：**

```text
根因确认：updateUser 用 Object.assign 覆盖整个用户对象，
前端没传 avatar_url，导致它变成 undefined，写入数据库成了 null。

给出修复方案，至少两种，只列出方案不改代码。
```

Claude Code 给出两种方案：

- 方案 A：前端在提交时把当前头像 URL 一起传回去
- 方案 B：updateUser 只更新传入的字段，跳过 undefined 的字段

方案 B 更好——它从服务端保证部分更新不会意外清空其他字段，以后不管哪个调用方漏传字段都不会出问题。

**第四步——最小修复：**

```text
采用方案 B：修改 updateUser，只更新传入的非 undefined 字段。

只改 src/services/user.ts 的 updateUser 函数。
不改前端，不改其他函数。
改完后展示 diff。
```

**第五步——验证：**

```text
执行验证：
1. 运行用户服务相关的测试
2. 确认编辑资料不会清空头像
3. 检查所有调用 updateUser 的地方，确认没有回归

给出验证总结。
```

## 常见坑

### 让 Claude Code 直接修，不复现不定位

"帮我修这个 bug" 是最差的提示词。Claude Code 会猜测根因然后直接改代码。猜对了是运气，猜错了你得到一个没修好的 bug 加上一堆无关改动。

### 修完不跑测试

Claude Code 改完代码会说"修复完成"。没有测试验证的"完成"不算完成。每次修 bug 都要让它跑测试，哪怕只有一条相关的测试。

### 信任 Claude Code 的方案选择

Claude Code 倾向于给"改动最少"的方案。加一个 null check 改动最少，但可能不是正确的修复方向。看方案的时候想一想：这个修复是让代码更健壮了，还是只是把报错藏起来了？

### 一个会话修多个 bug

修完一个 bug 之后开新会话（`/clear`）。上一个 bug 的上下文会干扰下一个 bug 的定位和修复。如果修到一半发现 Claude Code 走偏了两次以上，也建议 `/clear` 重新来，带上你前面试错积累的认知写一个更精确的初始提示词。

### 上下文窗口用完还在改

修 bug 读了大量文件后，上下文可能快满了。这时候 Claude Code 开始"遗忘"前面的约束，出现范围蔓延。留意上下文使用量，快满时用 `/compact` 压缩，或者开新会话继续。

## 关键要点

- 修 bug 五步走：复现 → 定位 → 比较方案 → 最小修复 → 验证。跳过任何一步都会增加风险
- 每一步都加"不改代码"的约束，直到修复阶段才放开
- 最小修复 = 只改 1-2 个文件，不夹带优化
- 验证必须包含：跑测试 + 复现场景 + 回归检查
- 一个会话只修一个 bug，修完 `/clear` 再开始下一个
- 方案选择是你的责任，不是 Claude Code 的。它给方案，你做判断
