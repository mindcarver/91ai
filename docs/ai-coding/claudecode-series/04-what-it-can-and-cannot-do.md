# Claude Code 能做什么，不能做什么

> 更新日期：2025/06

**TL;DR：** Claude Code 擅长需要多步骤、跨文件、需要验证的工程任务——调试、重构、理解代码库、批量修改、写脚本。它不擅长需要视觉反馈的工作、纯创意设计、从零到一的架构决策、以及需要长期维护上下文的超大项目。最重要的是：不管它做得好不好，验证结果的始终是你。

## 为什么需要知道边界

[第 1 篇](01-what-is-claude-code.md)里说 Claude Code 是一个能自主读代码、改文件、跑命令的编程代理。[第 3 篇](03-agent-loop-explained.md)拆解了它的工作引擎——agent loop，一轮一轮地理解、计划、行动、观察、判断。

知道这些机制后，一个实际的问题摆在你面前：**什么时候该用 Claude Code，什么时候不该用？**

这不是一个理论问题。根据 [Morph 的 15 款 AI 编程代理横评](https://www.morphllm.com/ai-coding-agent)，Claude Code 被评为"解决难题能力最强的代理"，但同时也是"最贵的"。在简单任务上用 Claude Code，相当于开卡车去买菜——能到，但浪费。在复杂任务上用错了方法，比如指望它自己设计整个系统架构，你会得到一个看起来完整但经不起推敲的结果。

更根本的问题是：**Claude Code 是一个工具，不是一个工程师。** 它没有判断力，没有责任感，不会在凌晨三点因为线上事故而焦虑。它能在你的指导下完成很多工程任务，但决定做什么、判断做得对不对、为最终结果负责的，始终是你。

这篇文章把 Claude Code 的能力边界摊开来讲——哪些任务它做得好，哪些任务它容易搞砸，什么时候你该自己来。

## Claude Code 擅长的 5 类任务

### 1. 定位和修复 bug

这是 Claude Code 最被低估的能力。不是因为它能直接给你答案，而是因为它会自己跑完整个"搜索 → 读取 → 分析 → 修改 → 验证"的循环。

一个真实的例子：后台任务间歇性超时六周没人查。把错误信息给 Claude Code，它自己搜索日志相关的代码、追踪调用链、检查超时配置、找到根因。根据 [dev.to 上的 30 天对比评测](https://dev.to/dextralabs/claude-code-vs-cursor-vs-github-copilot-honest-comparison-after-30-days-1030)，这种任务在 20 分钟内定位到了原因。

为什么它擅长这个？因为调试本身就是 agent loop 的理想场景——每一轮循环都是一次"提出假设 → 验证 → 调整方向"。Claude Code 能在同一会话里反复执行这个过程，而你只需要在它卡住的时候给提示。

**有效的方式**：给它具体的错误信息、复现步骤、或者失败的测试用例。越具体，它越快找到问题。

**无效的方式**："这个系统有 bug，帮我找。"没有线索的搜索和盲猜一样低效。

### 2. 理解不熟悉的代码库

接手一个五年的遗留项目，文档残缺，调用链像意大利面。Claude Code 能直接在代码库里搜索、追踪引用、理解模块间关系，帮你快速建立心智地图。

根据 [ArXiv 上的配置机制研究](https://arxiv.org/html/2602.14690v2)，Claude Code 用户使用的配置机制最广泛（包括 CLAUDE.md、Skills、Hooks 等），这意味着它特别适合需要深度理解代码库上下文的任务。

实际操作中，你可以让它：
- "这个函数的所有调用方在哪？"
- "数据库连接是在哪里初始化的？"
- "这个项目的认证流程是怎样的？从入口到 token 校验的完整链路。"

它不是简单地在文件里搜关键词——它会追踪调用链、理解数据流、识别设计模式。当然，它可能也会误解某些隐含的约定或者历史包袱，所以你仍然需要用自己的判断去验证它的理解。

**有效的方式**：问具体的问题，而不是"帮我理解整个项目"。一个问题一个答案，逐步建立理解。

**无效的方式**：一次给它整个项目让它"解释一下"。上下文窗口有限，它会只能看到局部。

### 3. 跨多文件的重构

把一个 600 行的"上帝类"拆分成独立模块。给 20 个接口统一加错误处理。批量重命名文件并更新所有引用。

这类任务的特点是：**模式固定，但涉及文件多，手动操作容易漏。** Claude Code 能跨文件搜索、理解依赖关系、逐个修改、跑测试验证。它的 Edit 工具做的是精确替换——只改需要改的部分，不是重写整个文件。

根据 [LinkedIn 上的开发者调查数据](https://www.linkedin.com/posts/witanowski_some-interesting-insights-last-night-reverse-activity-7425282357072498688-n77x)，Claude Code 用户的提交频率比非代理式工具用户高 437%。这个数字背后的原因之一就是批量重构任务的效率提升——代理可以自主处理大量重复性修改。

**有效的方式**：先让它分析要改的范围，确认它的理解是对的，再让它动手。

**无效的方式**：不给验证机会就直接让它改完——一次改 30 个文件，如果方向错了，回滚很麻烦。

### 4. 写脚本和自动化工具

"帮我写一个脚本，从 CSV 导入数据到数据库，加上去重和错误处理。"Claude Code 能直接在项目里创建文件、写代码、运行验证——不是给你一段代码片段让你自己跑，而是完整地写完、测完、确认能工作。

这类任务之所以适合 Claude Code，是因为：
- 需求明确，验证标准清晰（脚本跑通了就对）
- 通常涉及多个步骤（读文件、处理数据、写数据库、处理异常）
- 可以即时验证（跑一遍就知道行不行）

**有效的方式**：说清楚输入输出格式、错误处理策略、边界条件。

**无效的方式**："帮我写个脚本"——写什么脚本？做什么？输入什么？输出什么？

### 5. 代码搜索和代码审查辅助

"这个项目里所有使用了 `eval()` 的地方在哪？""这个接口的每个调用方都正确处理了错误吗？"

Claude Code 的搜索能力（Grep、Glob）加上它的理解能力，让它能做比 `grep -r` 更智能的代码搜索。它不只是找关键词匹配——它能理解上下文，判断一个 `eval()` 是真的安全隐患还是一个测试用例里的正常使用。

代码审查辅助也是一个实用场景。你可以让它审查一个 PR："看看这个改动有没有遗漏的错误处理、有没有性能问题、有没有破坏现有接口的兼容性。"它不会替代人工 code review，但能帮你先过一遍，筛掉明显的问题。

**有效的方式**：给出审查的具体维度，而不是笼统地"帮我看看这段代码"。

**无效的方式**：让它审查几千行的改动——上下文窗口放不下。

## Claude Code 容易失败的 5 类任务

### 1. 需要视觉反馈的前端工作

调整 CSS 像素对齐、调试动画效果、打磨 UI 细节。Claude Code 运行在终端里，它看不到你看到的画面。你可以用文字描述"这个按钮偏左了 10 像素"，但这比在编辑器里直接拖一下慢得多。

这不是能力问题，而是**反馈环路的问题**。前端视觉迭代的循环是"改一点 → 看一眼 → 再改一点"，每次调整的幅度很小，依赖即时视觉反馈。用文字描述视觉差异的往返成本远高于直接操作。

如果你用 VS Code 集成或者浏览器截图工具，可以部分缓解这个问题，但本质上这仍然不是 Claude Code 的主场。Cursor、Windsurf 这类编辑器原生工具在这种场景下效率更高。

### 2. 从零开始的架构设计

"帮我设计一个微服务架构"——Claude Code 会给你一个看起来完整的方案，包含服务拆分、通信协议、数据库选型。但架构设计的核心不是画图和写方案，而是在特定业务约束下做取舍。

Claude Code 不知道你的团队有几个人、哪些服务是核心路径、你们能接受多大的运维复杂度、性能瓶颈在哪里、预算是多少。这些信息它无法自动获取，而你很难在一次对话里把所有约束条件说清楚。

根据社区实践，合理的方式是：**你做架构决策，让它帮你实现具体部分。** 比如你决定了用事件驱动架构，让它帮你实现某个消息消费者的具体逻辑。把"做决策"和"写代码"分开，让 Claude Code 做它擅长的那部分。

Reddit 上有人直言："Claude Code 做不了大多数人声称它能做的事——构建真正能用的软件仍然需要时间，不管用什么工具。" ([r/ClaudeCode 讨论](https://www.reddit.com/r/ClaudeCode/comments/1rx1l7d/so_i_tried_using_claude_code_to_build_actual/))这话虽然偏激，但点出了一个事实：架构决策需要的是判断力，不是代码生成能力。

### 3. 超大项目的全局修改

一个 200 个文件的项目，你要做一次涉及所有文件的 API 迁移。Claude Code 能改，但会遇到几个硬限制：

- **上下文窗口**：标准约 200K token。一个大项目的结构信息加对话历史很容易填满。虽然有自动压缩机制（`/compact`），但压缩会丢失细节。
- **单线程主循环**：同一时间只做一件事。200 个文件的修改必须一个一个来，中间不能并行。
- **成本累积**：每一轮循环都是一次 API 调用。200 个文件的任务可能跑几百轮，API 费用会很高。

根据 [Product Compass 的成本分析](https://www.productcompass.pm/p/stop-hitting-claude-code-limits)，很多用户遇到的使用限制问题根源不是额度不够，而是工作流配置不当。合理的做法是把大任务拆成小块，每块独立处理。

### 4. 需要领域专家知识的任务

医疗数据处理、金融风控逻辑、法律合规代码——这些任务需要的不是通用编程能力，而是对特定领域的深度理解。Claude Code 能写代码，但它不理解 HIPAA 合规的具体要求、不清楚 Basel III 对风险计算的规定、不知道某个法律条文在不同司法管辖区的解释差异。

你可以说"把这些规则写在 CLAUDE.md 里"。确实，提供领域知识可以改善结果。但问题是：如果你已经能把领域知识写得足够精确和完整到让 AI 正确实现，你可能已经可以自己写代码了。

**合理的使用方式**：你提供领域知识和业务规则，让 Claude Code 负责把规则翻译成代码。你来审查逻辑是否正确。

### 5. 长时间无人值守的自主开发

"我下班了，Claude Code 你帮我把这个功能做完，明天早上我看。"这个想法很诱人，但实际效果不好。

原因：
- **没有反馈修正机制**：人不在，没法在它走偏的时候纠正方向。一个小误解可能在几十轮循环后放大成完全错误的方向。
- **上下文窗口限制**：长时间运行会触发压缩，丢失早期的重要决策。
- **使用额度**：跑几小时的任务可能很快耗尽你的额度。根据 [SitePoint 的调查](https://www.sitepoint.com/claude-code-rate-limits-explained/)，一些用户在额度的 6% 时就已经触发了限制。

根据 [Hacker News 上的讨论](https://news.ycombinator.com/item?id=44596472)，Claude Code 的正确使用方式是"人在旁边看着，适时打断"，而不是"放手让它自己跑"。它的价值在于加速你的工作，不是替代你的参与。

## 什么时候该人工接管

不是所有任务都适合从头到尾交给 Claude Code。下面这些信号出现时，你应该考虑接手。

### 信号一：它在同一问题上循环超过 3 次

Claude Code 尝试修一个 bug，改了代码，跑测试，失败了，又改，又失败，又改。连续三次走不通，说明它对问题的理解可能有误。

这时候你应该：停下来，自己看看测试输出和错误信息，判断它的修复方向对不对。如果方向错了，给它更具体的指引或者自己接手修。

### 信号二：修改范围超出预期

你说"修一下这个接口的错误处理"，它开始改 15 个文件，涉及数据库层、中间件、前端调用。范围蔓延是一个危险的信号——它可能在改不该改的东西。

这时候你应该：中断它，看看它已经改了什么。确认修改范围是否合理。如果超出了，用 `git checkout` 撤回不必要的改动，重新给它更明确的边界。

### 信号三：它开始"猜测"而不是"搜索"

好的 Claude Code 行为是先搜索代码、读取文件、理解上下文，再做修改。坏的行为是凭记忆编造函数名、猜测文件路径、假设某个库的 API。

出现猜测行为时，说明它的上下文可能已经丢失（通常发生在长会话中）。这时候你应该：手动 `/compact` 压缩对话，或者用 `/clear` 重新开始一个会话，给它干净的上下文。

### 信号四：任务需要主观判断

"这个函数名好不好？""这个 API 设计够不够 RESTful？""这段代码可读性怎么样？"这些问题没有客观标准答案。Claude Code 会给你一个回答，但这个回答反映的是它训练数据中的统计模式，不是真正的设计品味。

设计决策应该由你来做。你可以让 Claude Code 提供选项和分析（"列出三种可能的命名方案和各自的优缺点"），但最终选择权在你。

### 信号五：涉及生产环境的敏感操作

直接在生产数据库上执行 SQL、修改线上配置、发布关键服务。不管 Claude Code 多能干，这些操作都需要人类亲自确认和执行。

权限系统可以帮你把控这一点——对生产相关的命令保持手动确认模式，不要为了图快切换到 auto 模式。具体权限配置在[第 43 篇](43-permission-modes-explained.md)会详细讲。

## 验证是你的责任

这是整篇文章最重要的一个观点：**Claude Code 生成的每一行代码，你都需要理解并验证。**

为什么？三个原因：

**第一，它会犯错。** 不是偶尔，是经常。函数名拼错、参数类型搞反、逻辑条件写反、忽略边界情况——这些错误人类也会犯，但人类会在 code review 时发现。如果你不审查 Claude Code 的输出，这些错误就直接进入了代码库。

**第二，它的"正确"可能只是"测试通过"。** 测试通过只说明代码满足了测试覆盖的场景，不代表它在所有情况下都正确。Claude Code 可能写了一段通过测试但存在竞态条件的代码，或者通过了单元测试但在集成环境中表现不同的代码。

**第三，你要为结果负责。** 不管代码是谁写的——是你还是 Claude Code——当代码出了问题，追责到你头上。用户不会因为"这是 AI 写的"就原谅你的系统故障。

### 验证的实践方法

**用 `git diff` 审查每一处改动。** Claude Code 改完之后，不要直接说"好的"。先看看它改了什么。这是最基本的验证步骤。

**跑完整的测试套件。** 不要只看 Claude Code 跑的那几个测试。它可能只跑了和改动直接相关的测试，而忽略了回归风险。

**理解你合并的每一行代码。** 如果你看不懂 Claude Code 写的某段代码，这是一个危险信号——要么这段代码有问题，要么你需要先学习相关知识再决定是否接受。

**不要因为"测试通过了"就跳过审查。** 测试通过是必要条件，不是充分条件。

## 关键要点

- Claude Code 擅长 5 类任务：定位修复 bug、理解不熟悉代码库、跨多文件重构、写脚本和自动化工具、代码搜索和审查辅助。这些任务的共同特点是：需求可明确描述、验证标准客观、多步骤但有模式可循。
- Claude Code 容易失败的 5 类任务：需要视觉反馈的前端工作、从零开始的架构设计、超大项目的全局修改、需要领域专家知识的任务、长时间无人值守的自主开发。这些任务的共同特点是：需要主观判断、反馈环路依赖人类、或者超出上下文窗口的容量。
- 5 个该人工接管的信号：同一问题循环超过 3 次、修改范围超出预期、开始猜测而不是搜索、需要主观判断、涉及生产环境敏感操作。
- 验证是你的责任。Claude Code 是工具，不是工程师。它生成的每一行代码都需要你理解、审查、验证。用 `git diff` 看改动，跑完整测试套件，理解你合并的每一行代码。
- 正确的心态不是"让 AI 做完我来验收"，而是"我和 AI 协作完成"。你在关键节点做决策，它负责执行和加速。角色分工不同，但最终负责的始终是你。
