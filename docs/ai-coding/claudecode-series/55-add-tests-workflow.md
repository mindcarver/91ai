# 补测试流程

> 更新日期：2025/06

**TL;DR：** 补测试分四步：发现未覆盖 → 编写用例 → 运行修复 → 提交。核心原则是"先搞清楚缺什么，再写，最后跑通"。不要一上来就说"帮我写测试"——没有覆盖分析的测试，大概率测了已经能跑的路径，漏了真正会炸的地方。

## 为什么补测试需要流程

让 Claude Code 补测试，最常见的三种浪费：

1. **测了不该测的。** 你让它给 `utils.js` 加测试，它给每个导出函数写了 10 个用例。但 `utils.js` 里 80% 的函数是纯函数、逻辑简单，真正容易出 bug 的是 `formatMoney` 这个函数——它没测到，因为没分析过哪里容易出错。
2. **测了已经能跑的路径。** Claude Code 生成的测试倾向于测正常流程：传入合法参数、拿到合法返回。但你的项目真正需要保护的是边界情况——空数组、undefined、并发请求。
3. **测了但跑不通。** 生成了 30 个测试，20 个报错。Claude Code 开始逐个修，改了测试又改了源码，改了源码又引入新 bug。一轮下来比不写测试还糟糕。

这三个问题的根源一样：没有先分析覆盖缺口就动手写。补测试的工作流就是把"先分析"这件事固定下来。

## 第一步：发现未覆盖

写测试之前，先搞清楚缺什么。盲写等于浪费。

### 用覆盖率工具找缺口

如果你的项目已经配了覆盖率工具（Jest 的 `--coverage`、pytest 的 `--cov`、Go 的 `go test -cover`），直接跑：

```text
运行测试并生成覆盖率报告。
列出覆盖率低于 80% 的文件，按覆盖率从低到高排列。
对每个文件，列出未被测试覆盖的函数和分支。
只报告，不改代码。
```

拿到报告后，你不是所有低覆盖率的文件都要补。有些文件是配置文件、类型定义、常量——这些东西不需要测试。需要补的是业务逻辑。

### 没有覆盖率工具时的分析

项目可能还没配覆盖率工具。这时候让 Claude Code 做手动分析：

```text
阅读 src/services/ 目录下的所有文件。
对每个文件，列出：
1. 导出了哪些函数
2. 每个函数的正常路径是否被现有测试覆盖
3. 每个函数的异常路径（错误处理）是否被覆盖
4. 边界情况是否被覆盖（空输入、null、超大值等）

输出格式：表格，列为 文件名 | 函数名 | 正常路径 | 异常路径 | 边界情况
用 ✅ 和 ❌ 标记。
```

这个分析本身就需要 Claude Code 读代码，所以它比盲目写测试费时间，但产出的结果让你知道该补哪里。

### 确定优先级

不是所有未覆盖的代码都需要立刻补测试。按这个优先级排：

| 优先级 | 场景 | 原因 |
|--------|------|------|
| P0 | 修完 bug 的代码 | 刚修的 bug 最容易回归，立刻补测试保护 |
| P1 | 核心业务逻辑 | 支付、权限、数据计算——错了影响大 |
| P2 | 公共工具函数 | 被多处调用，改一处影响全局 |
| P3 | 复杂条件分支 | 分支多、容易漏改 |
| P4 | 简单 CRUD 和静态方法 | 错了容易发现，影响范围小 |

一个实际的做法：让 Claude Code 给未覆盖代码打优先级标签，你确认后再开始写测试。

```text
根据上一步的覆盖分析结果，按以下优先级排序：
P0：最近修过 bug 的函数（检查 git log --since="2 weeks ago"）
P1：涉及支付、权限、数据计算的函数
P2：被 3 个以上文件调用的公共函数
P3：包含 3 个以上 if/else 分支的函数
P4：其他

输出排序后的清单，只包含 ❌ 标记的条目。
```

## 第二步：编写用例

确定了要补什么之后，开始写。这一步的关键是约束 Claude Code 的测试风格，让它和项目已有的测试保持一致。

### 匹配项目现有风格

Claude Code 会读你项目里已有的测试文件，但有时候它读得不够仔细，写出来的风格和现有的对不上。明确告诉它：

```text
阅读 tests/ 目录下的已有测试文件，确认以下风格：
1. 测试框架（Jest / Vitest / pytest / Go testing）
2. describe/it 的组织方式
3. 断言风格（expect vs assert vs should）
4. mock 和 stub 的使用模式
5. 测试数据的构造方式（factory / fixture / inline）

然后用同样的风格，为 src/services/payment.ts 编写测试。
```

如果你在 CLAUDE.md 里提前记录了测试规范，这一步可以简化：

```markdown
## Testing Conventions
- Framework: Vitest
- 测试文件放在 tests/ 目录，镜像 src/ 的目录结构
- 用 describe 按函数分组，it 描述具体场景
- Mock 外部依赖，不 mock 被测函数内部调用
- 测试数据用 factory 函数构造，不用硬编码
```

### 边界用例清单

正常路径的测试 Claude Code 自己就能想到。你需要提醒它的是边界情况：

```text
为 calculateDiscount 函数编写测试，覆盖以下场景：

正常路径：
- 标准折扣计算（输入合法的价格和折扣率）
- 多级折扣叠加

边界情况：
- 折扣率为 0（不打折）
- 折扣率为 1（全免）
- 价格为 0
- 价格为负数
- 折扣率为负数
- 非常大的价格（浮点精度）
- 非数字输入（字符串、null、undefined）

异常路径：
- 数据库查询失败时的错误处理
- 用户没有折扣权限时的拒绝

不要用 any 类型断言绕过类型检查。
```

这个提示词的核心是"边界情况"那一栏。Claude Code 不会主动想到"价格为负数"这种场景，除非你列出来。

### 一次性补一个模块的测试

有时候你需要给整个模块补测试，不是单个函数：

```text
为 src/utils/ 目录下所有未被测试覆盖的函数编写单元测试。

要求：
1. 每个函数一个 describe 块
2. 每个函数至少覆盖：正常输入、边界值、异常输入
3. 纯函数优先测，有副作用的函数用 mock
4. 测试文件放在 tests/utils/ 目录下

写完后统计：新增了多少测试用例，覆盖了多少函数。
```

这种批量操作要注意一个坑：Claude Code 可能会在一次会话里生成大量测试代码，撑爆上下文窗口。如果模块超过 10 个函数，建议分批：

```text
分两批为 src/utils/ 补测试：

第一批（先做）：
- formatMoney
- parseDate
- validateEmail

完成后运行测试确认通过，再做第二批。

第二批（做完第一批后再说）：
- generateId
- slugify
- truncateText
```

## 第三步：运行修复

写完测试，下一步是跑。这里最容易翻车：测试失败后，Claude Code 开始同时改测试和改源码，改着改着两边都对不上了。

### 正确的失败处理流程

测试跑不通时分两种情况，处理方式完全不同：

**情况一：测试写错了。** 测试的断言和代码的实际行为不一致，但代码本身是对的。

```text
运行刚编写的测试。如果有失败：
1. 分析每个失败用例的原因
2. 区分是测试写错了还是代码有 bug
3. 如果是测试写错了，修正测试
4. 如果是代码有 bug，只记录 bug 信息，不修改源码

判断标准：
- 如果代码的其他测试都通过，大概率是新测试写错了
- 如果同一个函数的其他测试也失败，大概率是代码有 bug
```

"不修改源码"这条约束很关键。补测试的过程中改源码，你就分不清是在修 bug 还是在让代码适配测试。

**情况二：测试暴露了代码 bug。** 新测试发现了代码的问题。

这时候应该停下来，把 bug 修好之后再继续补测试流程。修 bug 的流程参考系列第 53 篇。修完后重新跑测试。

### 批量补测试时的执行策略

一次性补了大量测试（比如 20 个以上）时，不要一口气全跑。先跑一批，确认能过，再跑下一批：

```text
运行 tests/utils/format.test.ts 的测试。
如果全部通过，继续运行 tests/utils/parse.test.ts。
如果有失败，按上面的失败处理流程修复，全部通过后再继续。
不要同时运行所有测试文件。
```

分批跑的好处是：失败范围小，定位快。20 个文件一起跑，10 个失败了，你得花时间搞清楚哪个文件失败的原因是什么。一个文件一个文件跑，失败了立刻知道是哪里的问题。

### 跑通后的检查

所有测试都通过后，做一次最终检查：

```text
确认以下事项：
1. 所有新增测试都能独立通过（不依赖执行顺序）
2. 没有跳过的测试（it.skip / test.skip）
3. 没有空的测试（只有 placeholder 断言的）
4. 没有硬编码的等待时间（setTimeout、sleep）
5. 测试文件中没有 console.log 或调试代码
```

Claude Code 在赶进度时可能用 `it.skip` 跳过跑不通的测试，或者留一个 `expect(true).toBe(true)` 占位。最终检查能抓到这些偷懒行为。

## 一个完整的补测试例子

**场景：** `src/services/order.ts` 有 6 个导出函数，测试覆盖率只有 32%。刚修了一个"订单金额计算错误"的 bug。

**第一步——发现未覆盖：**

```text
运行测试覆盖率报告，重点关注 src/services/order.ts。
列出每个函数的覆盖情况。
用 ✅/❌ 标记。

然后检查 git log，最近两周有没有修过这个文件的 bug。
如果有，标注哪些函数是刚修过 bug 的（P0 优先级）。
```

结果：

| 函数 | 正常路径 | 异常路径 | 边界 | 优先级 |
|------|---------|---------|------|--------|
| createOrder | ✅ | ❌ | ❌ | P2 |
| calculateTotal | ✅ | ❌ | ❌ | P0 |
| applyDiscount | ❌ | ❌ | ❌ | P1 |
| validateOrder | ✅ | ✅ | ❌ | P3 |
| cancelOrder | ❌ | ❌ | ❌ | P2 |
| getOrderStatus | ✅ | ✅ | ✅ | P4 |

`calculateTotal` 刚修过 bug，优先补。`applyDiscount` 涉及金额计算，P1。

**第二步——编写用例：**

```text
先补 calculateTotal 的测试。这个函数刚修过一个 bug：
输入空商品列表时，原代码返回 NaN，修复后应该返回 0。

覆盖以下场景：
1. 正常：多商品计算总价
2. 正常：含折扣商品的总价
3. Bug 回归：空商品列表返回 0（不是 NaN）
4. 边界：单个商品、总价为 0 的商品
5. 异常：商品缺少 price 字段时的错误处理

匹配项目现有测试风格（tests/services/order.test.ts 已有部分测试）。
```

Claude Code 生成了 8 个测试用例。

**第三步——运行修复：**

```text
运行 calculateTotal 相关的测试。
如果有失败，分析原因：是测试写错了还是代码有问题。
只修测试，不改源码。
全部通过后展示最终结果。
```

全部通过。

**第四步——继续下一个函数：**

```text
继续补 applyDiscount 的测试。

覆盖以下场景：
1. 正常：百分比折扣
2. 正常：固定金额折扣
3. 边界：折扣金额大于订单金额
4. 边界：折扣率为 0
5. 边界：折扣率为 1（100%）
6. 异常：无效的折扣码
7. 异常：过期的折扣码

运行测试，确认通过。
```

逐个函数推进，每次一个函数，跑通再继续。

**第五步——提交：**

```text
提交新增的测试代码。
只包含 tests/ 目录下的文件。
commit message: "test(order): add coverage for calculateTotal and applyDiscount"
不包含任何源代码修改。
```

## 和修 bug、重构的关系

补测试不是孤立的流程，它和修 bug、重构紧密关联：

- **修 bug 后补测试**（系列第 53 篇）：修完 bug 立刻补回归测试，防止同一个 bug 再出现。P0 优先级的来源。
- **重构前补测试**（系列第 54 篇）：重构前写特征测试锁定当前行为。特征测试和普通单元测试的区别是，特征测试捕获代码的实际行为（包括 bug），普通测试捕获期望行为。
- **TDD 模式**：先写测试再写代码。这不是本篇的范围，但如果你已经用 TDD 模式，Claude Code 也能配合——你写测试，它写实现，跑不通它修实现，直到测试通过。

## 常见问题

**Q：Claude Code 生成的测试跑不通怎么办？**

不要让它同时改测试和源码。分两步走：先确认测试逻辑对不对，再确认源码对不对。如果两边都在改，你很快就会失控。

**Q：测试太多，一次写不完怎么办？**

分批写。按优先级从高到低，每次补 2-3 个函数的测试。每批写完跑通、提交，再做下一批。宁可慢一点，不要一口气生成 50 个测试然后花两小时修。

**Q：补测试时要不要顺便改代码？**

不要。补测试的目的只有一个：给现有代码加保护。如果测试过程中发现了 bug，记录下来，单独修。补测试和修 bug 混在一起做，到最后你分不清"这个测试是为了保护修的 bug 还是为了覆盖新路径"。

**Q：项目没有测试框架怎么办？**

先配框架，再写测试。让 Claude Code 帮你配：

```text
项目目前没有测试框架。请帮我配置：
1. 推荐适合这个项目技术栈的测试框架
2. 安装依赖
3. 创建配置文件
4. 写一个示例测试文件验证配置正确
5. 在 package.json 的 scripts 里加 test 命令

不要修改任何现有源代码。
```

框架配好了再开始补测试。

**Q：测试覆盖率目标是多少？**

没有万能的数字。80% 对大多数项目是个合理的起点。但覆盖率的数字本身不等于质量——100% 覆盖率的代码如果只测了正常路径，照样会出 bug。关注"关键路径是否被覆盖"比关注数字更有意义。

## 关键要点

1. **补测试之前先做覆盖分析**：搞清楚缺什么再动手。盲写是浪费时间
2. **按优先级排**：刚修过 bug 的 > 核心业务逻辑 > 公共函数 > 复杂分支 > 简单函数
3. **测试风格匹配项目现有风格**：不要引入新的写法，保持一致性
4. **边界用例需要你显式列出**：Claude Code 不会主动想到所有边界情况
5. **测试失败时分清"测试写错了"还是"代码有 bug"**：不要同时改两边
6. **补测试和修 bug 分开做**：不要混在一起
7. **分批写、分批跑、分批提交**：不要一口气全写完
8. **最终检查**：没有 skip、没有空测试、没有调试代码
